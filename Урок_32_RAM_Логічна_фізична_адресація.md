## 1. Оперативна пам'ять (RAM) — нагадування та поглиблення

**Оперативна пам'ять** (RAM, Random Access Memory) — це енергозалежна пам'ять комп'ютера, яка зберігає дані та код програм під час їх виконання. При вимкненні живлення вміст RAM втрачається.

RAM є "робочим столом" процесора: усе, з чим процесор працює прямо зараз (операційна система, відкриті програми, активні дані), розташоване саме тут.

### 1.1. Типи RAM: DRAM та SRAM

Існує два основних типи оперативної пам'яті:

#### DRAM (Dynamic RAM) — динамічна пам'ять

- Зберігає біт даних у вигляді **електричного заряду в конденсаторі**
- Потребує постійного **оновлення (refresh)** сотні разів на секунду, бо заряд поступово "витікає"
- **Повільніша**, але **дешевша** та має **більшу ємність**
- Використовується як **основна оперативна пам'ять** комп'ютера (ті самі "планки" RAM)

#### SRAM (Static RAM) — статична пам'ять

- Зберігає біт даних за допомогою **тригера на транзисторах** (зберігає стан напруги)
- **НЕ потребує оновлення** — дані зберігаються, поки є живлення
- **Значно швидша**, але **дорожча** і займає більше місця на кристалі
- Використовується як **кеш-пам'ять процесора** (L1, L2, L3)

```
┌───────────────────────────────────────────────────────────┐
│              Ієрархія пам'яті комп'ютера                  │
├───────────────────────────────────────────────────────────┤
│                                                           │
│   Регістри CPU    ◄── найшвидші, найменший обсяг (SRAM)  │
│        ▼                                                  │
│   Кеш L1 / L2 / L3  ◄── дуже швидка SRAM                │
│        ▼                                                  │
│   Оперативна пам'ять (RAM)  ◄── швидка DRAM              │
│        ▼                                                  │
│   SSD / HDD         ◄── повільні, великий обсяг          │
│                                                           │
│   Швидкість ▲                        Обсяг ▼             │
│   Ціна за ГБ ▲                       Ціна за ГБ ▼        │
└───────────────────────────────────────────────────────────┘
```

**Порівняльна таблиця SRAM vs DRAM:**

| Характеристика | SRAM | DRAM |
|---|---|---|
| Елемент зберігання | Транзистори (тригер) | Конденсатор + транзистор |
| Потребує оновлення? | Ні | Так (сотні разів/с) |
| Швидкість | Висока (~1-10 нс) | Нижча (~50-70 нс) |
| Вартість | Висока | Низька |
| Щільність | Низька | Висока |
| Застосування | Кеш CPU | Основна пам'ять |

### 1.2. Покоління DDR: DDR4 та DDR5

DDR (Double Data Rate) — технологія, що передає дані двічі за один такт (на фронті та спаді сигналу). Основна оперативна пам'ять сучасних комп'ютерів використовує саме DDR SDRAM.

**Порівняння DDR4 та DDR5:**

| Параметр | DDR4 | DDR5 |
|---|---|---|
| Базова частота | 2133-3200 МГц | 4800-6400+ МГц |
| Максимальна (розгін) | ~4000+ МГц | ~8400+ МГц |
| Напруга живлення | 1.2 В | 1.1 В |
| Макс. обсяг модуля | 32 ГБ | 128 ГБ |
| Кількість каналів на модулі | 1 x 64-біт | 2 x 32-біт |
| Довжина пакету (burst length) | 8 | 16 |
| ECC на кристалі | Ні | Так (on-die ECC) |
| Кількість контактів | 288 | 288 (інший ключ!) |
| Управління живленням | На материнській платі | PMIC на самому модулі |

> **Важливо:** DDR4 і DDR5 модулі **НЕ взаємозамінні** — вони мають різне розташування "ключа" (вирізу на планці), тому фізично не встановлюються в слот іншого покоління.

**Що краще обрати?**
- **DDR4** — чудовий вибір для бюджетних збірок або оновлення існуючих систем (платформи Intel 10-11 покоління, AMD AM4)
- **DDR5** — стандарт для нових платформ (Intel 12+ покоління, AMD AM5), кращий для ресурсомістких завдань (відеомонтаж, 3D, ШІ)

### 1.3. Характеристики RAM: обсяг, частота, латентність, канали

**Обсяг (Capacity)** — кількість даних, яку може зберігати RAM. Вимірюється в гігабайтах (ГБ). Типові значення: 8, 16, 32, 64 ГБ. Чим більше обсяг — тим більше програм та даних можуть одночасно перебувати в пам'яті.

**Частота (Frequency/Speed)** — швидкість передачі даних, вимірюється в МГц (мегагерцах) або MT/s (мегатрансферів на секунду). Наприклад, DDR4-3200 виконує 3200 мільйонів передач даних за секунду.

> **Нюанс DDR:** DDR означає "подвійна швидкість передачі даних". Тому реальна частота тактового сигналу вдвічі менша за заявлену. Наприклад, DDR4-3200 працює на тактовій частоті 1600 МГц, але передає дані двічі за такт. У Диспетчері завдань Windows або CPU-Z ви можете бачити частоту 1600 МГц — це нормально, просто множте на 2.

**Латентність (CAS Latency, CL)** — затримка (в тактах) між запитом процесора і видачею даних. Наприклад, CL16 означає 16 тактів очікування. Чим менше значення CL при тій самій частоті — тим краще.

- DDR4 зазвичай має CL14-CL19
- DDR5 зазвичай має CL30-CL40 (але завдяки вищій частоті реальна затримка в наносекундах може бути подібною)

**Канали пам'яті (Memory Channels):**

| Режим | Опис | Пропускна здатність |
|---|---|---|
| Single Channel | 1 модуль, 1 канал | 1x (базова) |
| Dual Channel | 2 модулі в парних слотах | 2x (подвоєна) |
| Quad Channel | 4 модулі (серверні/HEDT платформи) | 4x (вчетверо) |

> **Практична порада:** Завжди краще ставити RAM **парами** (2 модулі по 8 ГБ замість 1 модуля на 16 ГБ) — так активується двоканальний режим, що дає помітний приріст швидкості.

### 1.4. Як дізнатися параметри RAM у Windows

#### Спосіб 1: Диспетчер завдань (Task Manager)

1. Натисніть **Ctrl + Shift + Esc**
2. Перейдіть на вкладку **"Швидкодія" (Performance)**
3. Оберіть **"Пам'ять" (Memory)**

Тут відображається: загальний обсяг RAM, скільки використовується, частота, кількість зайнятих слотів, форм-фактор.

#### Спосіб 2: Командний рядок (wmic)

Відкрийте командний рядок (cmd) і введіть:

```
wmic memorychip get BankLabel, Capacity, Speed, Manufacturer, MemoryType
```

Корисні команди:
- `wmic memorychip get speed` — частота кожного модуля
- `wmic memorychip get capacity` — обсяг кожного модуля (у байтах)
- `wmic memorychip list full` — повна інформація
- `wmic memorychip get formfactor` — форм-фактор (8 = DIMM, 12 = SODIMM)

#### Спосіб 3: Програма CPU-Z

Безкоштовна утиліта CPU-Z (cpuid.com) показує:
- Тип пам'яті (DDR4/DDR5)
- Реальну тактову частоту (DRAM Frequency)
- Таймінги (CL-tRCD-tRP-tRAS)
- Кількість модулів та вільних слотів
- Виробника та серійний номер модуля

```
┌─────────────────────────────────────────────┐
│  Способи перевірки RAM у Windows            │
├──────────────┬──────────────┬───────────────┤
│  Диспетчер   │   wmic       │   CPU-Z       │
│  завдань     │  memorychip  │  (cpuid.com)  │
├──────────────┼──────────────┼───────────────┤
│  Обсяг    +  │  Обсяг    +  │  Обсяг     +  │
│  Частота  +  │  Частота  +  │  Частота   +  │
│  Слоти    +  │  Слоти    +  │  Слоти     +  │
│  Тип      +  │  Тип     (+) │  Тип       +  │
│  Виробник -  │  Виробник +  │  Виробник  +  │
│  Таймінги -  │  Таймінги -  │  Таймінги  +  │
└──────────────┴──────────────┴───────────────┘
```

## 2. Адресний простір

Щоб процесор міг прочитати або записати дані в RAM, кожна комірка пам'яті повинна мати свою унікальну **адресу** — числовий ідентифікатор, за яким процесор "знає", до якої саме комірки звертатися.

### 2.1. Фізична адреса

**Фізична адреса** — це реальна адреса конкретної комірки в мікросхемі оперативної пам'яті. Це той номер, який передається по шині адреси безпосередньо на модулі RAM.

- Фізичні адреси починаються з 0 і йдуть до максимального значення, що визначається обсягом встановленої пам'яті
- Наприклад, якщо в комп'ютері 4 ГБ RAM, то фізичні адреси — від 0x00000000 до 0xFFFFFFFF

### 2.2. Логічна (віртуальна) адреса

**Логічна (віртуальна) адреса** — це адреса, яку генерує процесор під час виконання програми. Це та адреса, яку "бачить" програма.

- Програма працює зі своїм **власним адресним простором**, починаючи від адреси 0
- Програма "не знає", де саме у фізичній пам'яті розміщені її дані
- Логічна адреса **не відповідає** безпосередньо фізичній адресі

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Програма A          Програма B          Програма C     │
│  ┌──────────┐        ┌──────────┐        ┌──────────┐   │
│  │ Логічна  │        │ Логічна  │        │ Логічна  │   │
│  │ адреса 0 │        │ адреса 0 │        │ адреса 0 │   │
│  │ адреса 1 │        │ адреса 1 │        │ адреса 1 │   │
│  │   ...    │        │   ...    │        │   ...    │   │
│  │ адреса N │        │ адреса M │        │ адреса K │   │
│  └────┬─────┘        └────┬─────┘        └────┬─────┘   │
│       │                   │                   │         │
│       └───────────┬───────┴───────────┬───────┘         │
│                   ▼                   ▼                  │
│         ┌─────────────────────────────────────┐         │
│         │       MMU (перетворення адрес)       │         │
│         └──────────────────┬──────────────────┘         │
│                            ▼                            │
│         ┌─────────────────────────────────────┐         │
│         │      Фізична пам'ять (RAM)          │         │
│         │  ┌───┬───┬───┬───┬───┬───┬───┬───┐  │         │
│         │  │ A │ B │ C │ A │ C │ B │ A │ B │  │         │
│         │  └───┴───┴───┴───┴───┴───┴───┴───┘  │         │
│         │  0   1   2   3   4   5   6   7       │         │
│         └─────────────────────────────────────┘         │
└─────────────────────────────────────────────────────────┘
```

> Кожна програма "думає", що має весь адресний простір для себе, починаючи з нуля. Насправді її дані розкидані по різних ділянках фізичної пам'яті.

### 2.3. Навіщо розділяти логічну та фізичну адресацію

| Причина | Пояснення |
|---|---|
| **Ізоляція процесів** | Кожна програма працює у своєму "віртуальному світі" і не може випадково зіпсувати дані іншої програми |
| **Безпека** | Один процес не може прочитати пам'ять іншого (наприклад, пароль з браузера) |
| **Гнучкість** | ОС може розміщувати програму в будь-якому вільному місці RAM, не змінюючи саму програму |
| **Спрощення програмування** | Програміст не думає про те, які саме фізичні адреси вільні |
| **Віртуальна пам'ять** | Можна використовувати більше пам'яті, ніж фізично встановлено, переносячи частину даних на диск |

## 3. MMU — Memory Management Unit

**MMU (Memory Management Unit)** — це спеціалізований **апаратний блок** (частина процесора або окрема мікросхема), який перетворює логічні (віртуальні) адреси у фізичні адреси при кожному зверненні до пам'яті.

### Як працює MMU

```
     Процесор генерує                MMU перетворює
     логічну адресу                  на фізичну адресу
           │                                │
           ▼                                ▼
  ┌─────────────────┐           ┌─────────────────┐
  │  Логічна адреса │           │ Фізична адреса  │
  │    0x14303      │           │    0x2303        │
  │                 │           │                  │
  │ Номер    Зміщ.  │           │ Номер    Зміщ.   │
  │ сторінки (offset)│          │ фрейму  (offset) │
  │  0x14  +  0x303 │  ──MMU──▶│  0x2   +  0x303  │
  └─────────────────┘           └─────────────────┘
```

**Покроковий алгоритм:**

1. Процесор генерує **логічну адресу** (наприклад, `0x14303`)
2. MMU розбиває адресу на дві частини:
   - **Номер сторінки** (page number) — `0x14`
   - **Зміщення** (offset) — `0x303`
3. MMU шукає номер сторінки у **таблиці сторінок** (page table)
4. Знаходить відповідний **номер фрейму** (physical frame) — `0x2`
5. Формує **фізичну адресу**: фрейм `0x2` + зміщення `0x303` = `0x2303`
6. Дані зчитуються з фізичної адреси `0x2303` у RAM

### TLB — Translation Lookaside Buffer

Оскільки таблиця сторінок зберігається в самій оперативній пам'яті, кожне звернення до пам'яті вимагало б **двох** звернень до RAM (спочатку до таблиці, потім до даних). Щоб уникнути цього, використовується **TLB** — невеликий надшвидкий кеш всередині MMU.

```
  Логічна адреса
       │
       ▼
  ┌──────────┐     Знайдено (TLB Hit)
  │   TLB    │ ─────────────────────────▶  Фізична адреса
  │ (кеш MMU)│                              (дуже швидко!)
  └────┬─────┘
       │ Не знайдено (TLB Miss)
       ▼
  ┌──────────────┐
  │ Таблиця      │
  │ сторінок     │ ──────────────────────▶  Фізична адреса
  │ (в RAM)      │                          (повільніше)
  └──────────────┘
```

- **TLB Hit** (влучення) — потрібний запис є в TLB, переклад адреси відбувається миттєво
- **TLB Miss** (промах) — запису немає в TLB, MMU звертається до таблиці сторінок у RAM

## 4. Зв'язування адрес (Address Binding)

**Зв'язування адрес** (address binding) — це процес визначення, за якою фізичною адресою в RAM будуть розміщені інструкції та дані програми. Може відбуватися на трьох етапах:

### 4.1. На етапі компіляції (Compile-Time Binding)

- Компілятор **заздалегідь знає**, за якою фізичною адресою програма буде розміщена в пам'яті
- Генерується **абсолютний код** з "вшитими" фізичними адресами
- Якщо потрібно змінити адресу — програму треба **перекомпілювати**
- Використовувалось у старих ОС (наприклад, MS-DOS) та вбудованих системах

```
  ┌──────────┐     Компілятор знає:        ┌──────────────┐
  │ Вихідний │     "Програма завжди за     │ RAM          │
  │ код      │ ──▶  адресою 0x1000"   ──▶  │ [0x1000] ◄── │
  │ (.c)     │                              │  програма    │
  └──────────┘                              └──────────────┘
```

### 4.2. На етапі завантаження (Load-Time Binding)

- Компілятор генерує **переміщуваний код** (relocatable code)
- Конкретна фізична адреса визначається **в момент завантаження** програми в пам'ять
- Завантажувач (loader) замінює відносні адреси на абсолютні
- Більш гнучко: програму можна завантажити в різні ділянки пам'яті

```
  ┌──────────┐     ┌───────────┐     ┌──────────────────┐
  │ Код з    │     │ Завантаж. │     │ RAM              │
  │ відносн. │ ──▶ │ (loader)  │ ──▶ │ [будь-яка вільна │
  │ адресами │     │ визначає  │     │  адреса]         │
  └──────────┘     │ де саме   │     └──────────────────┘
                   └───────────┘
```

### 4.3. На етапі виконання (Execution-Time / Dynamic Binding)

- Зв'язування відкладається до **моменту виконання** програми
- Програма працює з **логічними адресами**, а MMU перетворює їх на фізичні "на льоту"
- Програму можна **переміщувати** в пам'яті навіть під час роботи
- Потребує **апаратної підтримки** (MMU, базовий/граничний регістри)
- Використовується у **всіх сучасних ОС** (Windows, Linux, macOS)

```
  ┌──────────┐     ┌───────────┐     ┌──────────────────┐
  │ Програма │     │   MMU     │     │ RAM              │
  │ працює з │ ──▶ │ перетвор. │ ──▶ │ [динамічна адр.] │
  │ логічними│     │ кожну     │     │ може змінюватись │
  │ адресами │     │ адресу    │     │ під час роботи!  │
  └──────────┘     └───────────┘     └──────────────────┘
```

**Порівняльна таблиця:**

| Характеристика | Компіляція | Завантаження | Виконання |
|---|---|---|---|
| Коли визначається адреса | При компіляції | При завантаженні | При виконанні |
| Гнучкість | Низька | Середня | Висока |
| Можливість переміщення | Ні | Тільки до запуску | Так, навіть під час роботи |
| Потребує MMU? | Ні | Ні | Так |
| Приклад ОС | MS-DOS, вбудовані | Ранні ОС | Windows, Linux, macOS |

## 5. Розподіл пам'яті

Операційна система повинна розподілити оперативну пам'ять між кількома процесами, що виконуються одночасно. Існує кілька підходів.

### 5.1. Одноперегородочний розподіл

Найпростіший варіант: уся оперативна пам'ять поділяється на **дві частини**:

```
  ┌────────────────────────────┐
  │     Операційна система     │  ◄── фіксована область
  ├────────────────────────────┤
  │                            │
  │                            │
  │     Один процес            │  ◄── вся решта пам'яті
  │     користувача            │      для ОДНОГО процесу
  │                            │
  │                            │
  └────────────────────────────┘
```

- ОС займає фіксовану область (наприклад, нижні адреси)
- Увесь інший простір — для **одного процесу**
- **Переваги:** простота реалізації
- **Недоліки:** неможлива багатозадачність, неефективне використання пам'яті

### 5.2. Багатоперегородочний розподіл

#### Фіксовані (статичні) розділи

Пам'ять поділяється на **фіксовану кількість розділів** при завантаженні ОС. Розміри розділів не змінюються.

```
  ┌────────────────────────────┐
  │     Операційна система     │  ◄── фіксована
  ├────────────────────────────┤
  │     Розділ 1 (2 МБ)        │  ◄── Процес A (1.5 МБ)
  │     [0.5 МБ не використ.]  │      внутрішня фрагментація!
  ├────────────────────────────┤
  │     Розділ 2 (4 МБ)        │  ◄── Процес B (4 МБ)
  ├────────────────────────────┤
  │     Розділ 3 (3 МБ)        │  ◄── порожній
  ├────────────────────────────┤
  │     Розділ 4 (5 МБ)        │  ◄── Процес C (3 МБ)
  │     [2 МБ не використ.]    │      внутрішня фрагментація!
  └────────────────────────────┘
```

- **Переваги:** проста реалізація, низькі накладні витрати
- **Недоліки:**
  - **Внутрішня фрагментація** — процес менший за розділ, залишок пропадає
  - Обмежена кількість процесів (стільки, скільки розділів)
  - Процес не може бути більшим за найбільший розділ

#### Динамічні (змінні) розділи

Розділи створюються **під час завантаження процесу** і мають **саме той розмір**, який потрібен процесу.

```
  Початковий стан:              Після завершення B і D:

  ┌────────────────────┐        ┌────────────────────┐
  │       ОС           │        │       ОС           │
  ├────────────────────┤        ├────────────────────┤
  │   Процес A (2 МБ)  │        │   Процес A (2 МБ)  │
  ├────────────────────┤        ├────────────────────┤
  │   Процес B (3 МБ)  │        │   Вільно (3 МБ)    │  ◄── "діра"
  ├────────────────────┤        ├────────────────────┤
  │   Процес C (1 МБ)  │        │   Процес C (1 МБ)  │
  ├────────────────────┤        ├────────────────────┤
  │   Процес D (4 МБ)  │        │   Вільно (4 МБ)    │  ◄── "діра"
  ├────────────────────┤        ├────────────────────┤
  │   Вільно (6 МБ)    │        │   Вільно (6 МБ)    │
  └────────────────────┘        └────────────────────┘
```

- **Переваги:** немає внутрішньої фрагментації, більша гнучкість
- **Недоліки:** **зовнішня фрагментація** — вільна пам'ять розбита на маленькі "діри"

### 5.3. Проблема фрагментації пам'яті

| Тип фрагментації | Опис | Де виникає |
|---|---|---|
| **Внутрішня** | Невикористаний простір **всередині** виділеного розділу | Фіксовані розділи |
| **Зовнішня** | Вільна пам'ять є, але розбита на малі несуміжні блоки | Динамічні розділи |

**Приклад зовнішньої фрагментації:**

```
  ┌───┬───────┬───┬───────┬───┬──────────┐
  │ A │ вільно│ C │ вільно│ E │  вільно   │
  │2МБ│  3 МБ │1МБ│  2 МБ │1МБ│   4 МБ   │
  └───┴───────┴───┴───────┴───┴──────────┘

  Загалом вільно: 3 + 2 + 4 = 9 МБ
  Але процес розміром 8 МБ НЕ поміститься!
  (бо 9 МБ розкидано по різних місцях)
```

**Рішення — ущільнення (compaction):**

ОС переміщує всі процеси до одного краю пам'яті, об'єднуючи вільні "діри" в один суцільний блок.

```
  До ущільнення:                Після ущільнення:
  ┌───┬───────┬───┬───────┐     ┌───┬───┬──────────────┐
  │ A │ вільно│ C │ вільно│     │ A │ C │   вільно     │
  │2МБ│  3 МБ │1МБ│  5 МБ│     │2МБ│1МБ│    8 МБ      │
  └───┴───────┴───┴───────┘     └───┴───┴──────────────┘
```

> **Сучасне рішення:** Замість ущільнення, сучасні ОС використовують **сторінкову організацію пам'яті** (paging), яка повністю усуває зовнішню фрагментацію. Це тема наступних уроків.

## 6. Захист пам'яті

Коли кілька програм одночасно працюють в оперативній пам'яті, необхідно **захистити** кожну програму від несанкціонованого доступу інших програм.

### Базовий та граничний регістри

Найпростіший механізм захисту використовує два апаратних регістри:

- **Базовий регістр (Base Register)** — зберігає **початкову фізичну адресу** області пам'яті, виділеної процесу
- **Граничний регістр (Limit Register)** — зберігає **розмір** виділеної області (кількість допустимих адрес)

```
  Базовий регістр = 300                Граничний регістр = 200
  (початок області)                    (розмір області)

  Дозволений діапазон: адреси від 300 до 499

  ┌────────────────────────────────────────────────────┐
  │  0    100   200   300   400   500   600   700      │
  │  ├─────┼─────┼─────┼─────┼─────┼─────┼─────┤      │
  │                    │◄────────────▶│                 │
  │                    300           499                │
  │                    Область процесу                  │
  └────────────────────────────────────────────────────┘
```

**Як працює перевірка:**

```
  Процес генерує логічну адресу
           │
           ▼
  ┌─────────────────────┐
  │ Адреса < Limit?     │
  │ (адреса < 200?)     │
  └──────┬──────────────┘
     Так │           Ні
         ▼            ▼
  ┌──────────┐   ┌──────────────┐
  │ Додати   │   │   ПОМИЛКА!   │
  │ Base     │   │ Порушення    │
  │ (+ 300)  │   │ захисту      │
  └────┬─────┘   │ пам'яті.    │
       │         │ Процес       │
       ▼         │ зупинено.    │
  Фізична       └──────────────┘
  адреса
  (дозволено)
```

**Важливо:**
- Тільки **операційна система** (ядро, привілейований режим) може змінювати значення базового та граничного регістрів
- При **перемиканні контексту** (context switch) диспетчер ОС завантажує правильні значення регістрів для кожного процесу
- Якщо процес намагається звернутися за межі своєї області — генерується **апаратне переривання** (trap), і ОС зупиняє порушника

## 7. 32-бітні vs 64-бітні ОС: вплив на адресний простір

Розрядність процесора та ОС безпосередньо визначає розмір адресного простору — тобто скільки оперативної пам'яті може адресувати (бачити) система.

### 32-бітні системи

- Регістр адреси має ширину **32 біти**
- Максимальна кількість адрес: 2^32 = **4 294 967 296** (приблизно 4.3 мільярди)
- Максимальний адресний простір: **4 ГБ**
- На практиці доступно ще менше: **~3.2-3.5 ГБ**
  - Частина адресного простору зарезервована для відеокарти, BIOS, контролерів
  - Ядро ОС резервує ~1 ГБ для своїх потреб

### 64-бітні системи

- Регістр адреси має ширину **64 біти**
- Теоретичний максимум: 2^64 = **16 ексабайт** (16 ЕБ = 18 446 744 ТБ)
- Практичний ліміт в сучасних ОС: **128 ГБ - 2 ТБ** (залежить від версії ОС)
- Цього більш ніж достатньо для будь-яких сучасних задач

```
  ┌───────────────────────────────────────────────────────────┐
  │          Порівняння адресного простору                     │
  ├───────────────────────────────────────────────────────────┤
  │                                                           │
  │  32-бітна ОС:                                             │
  │  ┌──────────────────────────────┐                         │
  │  │  4 ГБ (2^32 адрес)          │                         │
  │  │  ████████████████████████████│  ◄── це ВСЕ            │
  │  └──────────────────────────────┘                         │
  │                                                           │
  │  64-бітна ОС:                                             │
  │  ┌──────────────────────────────────────────────────────┐ │
  │  │  16 ЕБ (2^64 адрес)                                 │ │
  │  │  ████...                                             │ │
  │  │  (в мільярди разів більше, ніж 32-бітна)             │ │
  │  └──────────────────────────────────────────────────────┘ │
  │                                                           │
  │  Якби 32-бітний простір = 1 сторінка книги,               │
  │  то 64-бітний = бібліотека з 4 мільярдів книг!            │
  └───────────────────────────────────────────────────────────┘
```

### Порівняльна таблиця

| Характеристика | 32-бітна ОС | 64-бітна ОС |
|---|---|---|
| Ширина адреси | 32 біти | 64 біти |
| Макс. адресний простір | 4 ГБ | 16 ЕБ (теор.) |
| Практичний ліміт RAM | ~3.2-3.5 ГБ | 128 ГБ - 2 ТБ |
| Запуск 32-бітних програм | Так | Так (сумісність) |
| Запуск 64-бітних програм | Ні | Так |
| Безпека | Базова | Покращена (DEP, ASLR) |
| Поточний статус | Застаріла | Стандарт |

### PAE — часткове рішення для 32-бітних систем

**PAE (Physical Address Extension)** — технологія, що розширює фізичну адресну шину з 32 до 36 біт, дозволяючи 32-бітній ОС адресувати до **64 ГБ** фізичної пам'яті. Проте кожна окрема 32-бітна програма все одно обмежена 4 ГБ віртуального простору.

> **Висновок:** Сьогодні 64-бітні системи є стандартом. 32-бітні ОС вважаються застарілими — Intel та AMD припинили їх підтримку в нових процесорах.

## 9. Глосарій

| Термін | Визначення |
|---|---|
| **RAM (Random Access Memory)** | Оперативна пам'ять — енергозалежна пам'ять для зберігання даних та коду програм під час їх виконання |
| **DRAM (Dynamic RAM)** | Динамічна пам'ять, що зберігає біти в конденсаторах і потребує постійного оновлення; використовується як основна оперативна пам'ять |
| **SRAM (Static RAM)** | Статична пам'ять на транзисторних тригерах; швидша та дорожча за DRAM; використовується як кеш процесора |
| **DDR (Double Data Rate)** | Технологія передачі даних двічі за такт; стандарт для модулів оперативної пам'яті (DDR4, DDR5) |
| **CAS Latency (CL)** | Затримка між запитом процесора та видачею даних з пам'яті, вимірюється в тактах |
| **Dual Channel** | Двоканальний режим роботи пам'яті, що подвоює пропускну здатність за рахунок одночасної роботи двох модулів |
| **Фізична адреса** | Реальна адреса комірки в мікросхемі оперативної пам'яті |
| **Логічна (віртуальна) адреса** | Адреса, яку генерує процесор під час виконання програми; перетворюється MMU на фізичну |
| **Адресний простір** | Множина всіх адрес, доступних процесу (логічний) або системі (фізичний) |
| **MMU (Memory Management Unit)** | Апаратний блок процесора для перетворення логічних адрес у фізичні |
| **TLB (Translation Lookaside Buffer)** | Кеш MMU для швидкого перетворення адрес без звернення до таблиці сторінок у RAM |
| **Таблиця сторінок (Page Table)** | Структура даних, що зберігає відповідність між номерами логічних сторінок та фізичних фреймів |
| **Address Binding (зв'язування адрес)** | Процес визначення фізичної адреси для інструкцій та даних програми |
| **Compile-Time Binding** | Зв'язування адрес на етапі компіляції; адреси "вшиті" у програму |
| **Load-Time Binding** | Зв'язування адрес під час завантаження програми в пам'ять |
| **Execution-Time Binding** | Динамічне зв'язування адрес під час виконання програми через MMU |
| **Фіксовані розділи** | Метод розподілу пам'яті, при якому RAM ділиться на розділи фіксованого розміру при завантаженні ОС |
| **Динамічні розділи** | Метод розподілу пам'яті, при якому розділи створюються динамічно під розмір кожного процесу |
| **Внутрішня фрагментація** | Втрата пам'яті всередині виділеного розділу, коли процес менший за розділ |
| **Зовнішня фрагментація** | Втрата пам'яті через розбиття вільного простору на малі несуміжні блоки |
| **Ущільнення (Compaction)** | Переміщення процесів у пам'яті для об'єднання вільних блоків у один суцільний |
| **Базовий регістр (Base Register)** | Апаратний регістр, що зберігає початкову фізичну адресу області пам'яті процесу |
| **Граничний регістр (Limit Register)** | Апаратний регістр, що зберігає розмір дозволеної області пам'яті процесу |
| **PAE (Physical Address Extension)** | Технологія розширення фізичної адресної шини 32-бітних процесорів з 32 до 36 біт |
| **Ексабайт (ЕБ)** | Одиниця вимірювання обсягу даних; 1 ЕБ = 2^60 байт (приблизно 1 мільярд ГБ) |
| **PMIC** | Мікросхема управління живленням, вбудована в модулі DDR5 для ефективного регулювання напруги |
| **On-die ECC** | Вбудована корекція помилок на кристалі пам'яті DDR5 для підвищення надійності |
