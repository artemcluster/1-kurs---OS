## 1. Вступ

Коли ми працюємо за комп'ютером — відкриваємо браузер, слухаємо музику, друкуємо документ — ми одночасно використовуємо десятки різних ресурсів: процесор обчислює, оперативна пам'ять зберігає дані програм, жорсткий диск читає файли, мережева карта завантажує сторінки. Хто координує все це? **Операційна система**.

ОС виступає в ролі **менеджера ресурсів** — вона вирішує, яка програма отримає доступ до процесора, скільки пам'яті виділити кожному додатку і як організувати черговість доступу до принтера, якщо друкувати хочуть одразу три програми.

Крім того, ОС створює **середовище** для виконання програм. Програма не працює "в порожнечі" — їй потрібен набір сервісів, бібліотек та інтерфейсів, які надає операційна система. Це називається **операційне середовище** (runtime environment).

У цьому уроці ми детально розглянемо:
- які ресурси існують у комп'ютері та як ОС ними керує;
- що таке операційне та програмне середовище;
- як програми взаємодіють з ОС через системні виклики та API;
- що таке сумісність, емуляція та контейнеризація.

## 2. Поняття ресурсу в контексті операційної системи

**Ресурс** — це будь-який апаратний або програмний компонент комп'ютерної системи, який має обмежену доступність і необхідний для виконання програм.

Ключова властивість ресурсу — він **обмежений**. Процесор один (або декілька ядер), оперативної пам'яті конкретний обсяг, принтер може друкувати лише одне завдання одночасно. Саме через обмеженість ресурсів виникає необхідність в їх управлінні.

Ресурси поділяються на три основні категорії: **апаратні**, **програмні** та **інформаційні**.

---

### 2.1. Апаратні ресурси

Апаратні ресурси — це фізичні компоненти комп'ютера, які використовуються для виконання програм.

**Процесор (CPU)** — головний обчислювальний ресурс. Виконує інструкції програм. Сучасні процесори мають декілька ядер (наприклад, 4, 8, 16), але кількість програм, які хочуть працювати одночасно, зазвичай значно більша за кількість ядер. ОС вирішує, яка програма в який момент отримає доступ до процесора.

**Оперативна пам'ять (RAM)** — тимчасова швидка пам'ять, у якій зберігаються дані програм під час їх виконання. Коли ви відкриваєте браузер із 10 вкладками, текстовий редактор та месенджер — кожна програма займає свою частину RAM. ОС розподіляє пам'ять між програмами та слідкує, щоб одна програма не "залізла" в пам'ять іншої.

**Дискова пам'ять (HDD/SSD)** — довготривале сховище для файлів, програм та самої ОС. ОС керує доступом до диска через файлову систему, організовуючи дані у вигляді файлів та папок.

**Пристрої введення-виведення (I/O)** — клавіатура, миша, монітор, принтер, мережева карта, USB-пристрої тощо. ОС забезпечує доступ програм до цих пристроїв через драйвери.

```
┌─────────────────────────────────────────────────────┐
│              АПАРАТНІ РЕСУРСИ КОМП'ЮТЕРА            │
├──────────────┬──────────────┬───────────────────────┤
│  Процесор    │  Оперативна  │   Дискова пам'ять     │
│  (CPU)       │  пам'ять     │   (HDD / SSD)         │
│              │  (RAM)       │                       │
├──────────────┴──────────────┴───────────────────────┤
│              Пристрої введення-виведення             │
│  Клавіатура | Миша | Монітор | Принтер | Мережа    │
└─────────────────────────────────────────────────────┘
```

---

### 2.2. Програмні ресурси

Програмні ресурси — це нематеріальні компоненти, необхідні для роботи програм.

**Файли** — програми зберігаються у файлах, а також створюють та використовують файли для зберігання даних. Файли є обмеженим ресурсом: наприклад, два процеси не можуть одночасно записувати в один файл без ризику пошкодження даних.

**Бібліотеки** — готові набори коду, які програми використовують для виконання типових завдань. Наприклад, бібліотека для роботи з графікою або для шифрування даних. В ОС Windows це файли з розширенням `.dll` (Dynamic Link Library), в Linux — `.so` (Shared Object).

**Змінні середовища** — іменовані значення, які зберігають системну конфігурацію і доступні всім програмам. Наприклад, змінна `PATH` повідомляє системі, де шукати виконувані файли.

**Мережеві з'єднання** — порти, сокети та канали зв'язку, через які програми обмінюються даними по мережі. Кількість одночасних з'єднань обмежена.

---

### 2.3. Інформаційні ресурси

Інформаційні ресурси — це дані, якими оперує комп'ютерна система.

**Бази даних** — структуровані колекції даних. Наприклад, база даних інтернет-магазину зберігає інформацію про товари, замовлення та клієнтів. ОС забезпечує доступ до баз даних та захищає їх від несанкціонованого доступу.

**Файли даних** — документи, зображення, відео, конфігураційні файли тощо. ОС керує доступом до цих файлів, визначаючи, хто може їх читати, змінювати або видаляти.

**Системний реєстр** (у Windows) — централізована база конфігурації ОС та встановлених програм.

---

### 2.4. Порівняльна таблиця типів ресурсів

| Тип ресурсу | Приклади | Що обмежено | Хто керує |
|---|---|---|---|
| **Апаратні** | Процесор, RAM, диск, принтер | Кількість, швидкість, обсяг | Ядро ОС, драйвери |
| **Програмні** | Файли, бібліотеки, мережеві порти | Кількість одночасних доступів | Файлова система, мережевий стек |
| **Інформаційні** | Бази даних, документи, реєстр | Обсяг, права доступу | ОС, СУБД, система безпеки |

## 3. Управління ресурсами — ОС як менеджер ресурсів

### 3.1. Роль ОС у розподілі ресурсів

Операційну систему часто називають **менеджером ресурсів** (resource manager), оскільки вона виконує п'ять ключових функцій:

**1. Розподіл ресурсів (Resource Allocation).** ОС вирішує, яка програма (процес) отримає доступ до якого ресурсу та на який час. Наприклад, процесорний час розподіляється між десятками процесів щомілісекунди.

**2. Моніторинг ресурсів (Resource Monitoring).** ОС відстежує, які ресурси зайняті, скільки пам'яті використовує кожна програма, наскільки завантажений процесор. Ви можете побачити це у Диспетчері завдань Windows або команді `top` у Linux.

**3. Захист ресурсів (Resource Protection).** ОС захищає ресурси від несанкціонованого доступу. Одна програма не може прочитати пам'ять іншої програми, звичайний користувач не може видалити системні файли.

**4. Спільне використання ресурсів (Resource Sharing).** Деякі ресурси можуть використовуватися кількома процесами одночасно. Наприклад, багато програм можуть одночасно читати один і той самий файл бібліотеки.

**5. Облік ресурсів (Resource Accounting).** ОС веде статистику використання ресурсів, що необхідно для оптимізації продуктивності та діагностики проблем.

```
┌──────────────────────────────────────────────┐
│          ОПЕРАЦІЙНА СИСТЕМА                   │
│         (Менеджер ресурсів)                   │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ Розподіл │  │Моніторинг│  │  Захист  │   │
│  └──────────┘  └──────────┘  └──────────┘   │
│  ┌──────────┐  ┌──────────┐                  │
│  │ Спільне  │  │  Облік   │                  │
│  │ викорис- │  │          │                  │
│  │ тання    │  │          │                  │
│  └──────────┘  └──────────┘                  │
└──────────────────────────────────────────────┘
         │              │             │
    ┌────┴────┐   ┌─────┴────┐  ┌────┴─────┐
    │Процесор │   │  Пам'ять │  │Пристрої  │
    │  (CPU)  │   │  (RAM)   │  │  (I/O)   │
    └─────────┘   └──────────┘  └──────────┘
```

### 3.2. Проблеми при розподілі ресурсів

Коли кілька програм (процесів) одночасно хочуть використовувати один і той самий ресурс, виникає **конкуренція за ресурси** (resource contention).

**Приклад з життя:** Уявіть, що в офісі є один принтер, а роздрукувати документи хочуть одночасно п'ять співробітників. Хтось має друкувати першим, хтось — останнім. Потрібні правила черговості.

У комп'ютері така ситуація виникає постійно:
- Кілька програм хочуть виконуватися на процесорі одночасно.
- Декілька процесів намагаються записати дані на один диск.
- Два потоки намагаються змінити один і той самий файл.

Якщо конкуренцію не контролювати, можуть виникнути серйозні проблеми: втрата даних, зависання програм або навіть повне зависання системи.

---

### 3.3. Взаємне блокування (Deadlock)

Найсерйозніша проблема при розподілі ресурсів — це **взаємне блокування** (deadlock) — ситуація, коли два або більше процеси заблоковані назавжди, бо кожен чекає ресурс, який утримує інший.

**Аналогія з життя:** Два водії зустрілися на вузькому мосту, рухаючись назустріч. Жоден не може проїхати вперед, бо шлях заблокований іншим. Жоден не хоче здавати назад. Обидва стоять назавжди.

**Приклад у комп'ютері:**
- Процес A захопив принтер і чекає на сканер.
- Процес B захопив сканер і чекає на принтер.
- Обидва заблоковані — жоден не може продовжити роботу.

```
    Процес A                    Процес B
    ┌───────┐                   ┌───────┐
    │Тримає:│                   │Тримає:│
    │Принтер│ ──── чекає ────> │Сканер │
    │       │ <─── чекає ───── │       │
    │Хоче:  │                   │Хоче:  │
    │Сканер │                   │Принтер│
    └───────┘                   └───────┘
           DEADLOCK — тупик!
```

**Чотири умови виникнення deadlock (умови Коффмана):**

Deadlock може виникнути лише тоді, коли одночасно виконуються всі чотири умови:

1. **Взаємне виключення (Mutual Exclusion)** — ресурс може використовуватися лише одним процесом одночасно. Наприклад, принтер не може друкувати два документи одночасно.

2. **Утримування та очікування (Hold and Wait)** — процес утримує хоча б один ресурс і одночасно чекає на отримання додаткових ресурсів, які утримуються іншими процесами.

3. **Неможливість примусового вивільнення (No Preemption)** — ресурс не можна примусово забрати у процесу; він має бути звільнений добровільно.

4. **Циклічне очікування (Circular Wait)** — існує замкнутий ланцюг процесів, де кожен чекає ресурс, який утримує наступний процес у ланцюгу.

**Способи боротьби з deadlock:**
- **Запобігання** — проєктувати систему так, щоб хоча б одна з чотирьох умов не могла виконатися.
- **Уникнення** — перед виділенням ресурсу перевіряти, чи не призведе це до deadlock.
- **Виявлення та відновлення** — дозволити deadlock виникнути, виявити його та примусово завершити один із процесів.
- **Ігнорування** — якщо deadlock виникає дуже рідко, деякі ОС (Windows, Linux) просто ігнорують цю проблему, покладаючись на перезавантаження.

---

### 3.4. Методи розподілу ресурсів

ОС використовує різні методи для справедливого та ефективного розподілу ресурсів:

**Черга (Queue / FIFO — First In, First Out).** Найпростіший метод: хто першим подав запит, той першим отримає ресурс. Працює як черга в магазині — перший прийшов, перший обслужений.

*Приклад:* Завдання друку стають у чергу до принтера. Перший документ, відправлений на друк, друкується першим.

**Пріоритети (Priority Scheduling).** Кожному процесу присвоюється пріоритет — числове значення, яке визначає важливість. Процеси з вищим пріоритетом отримують ресурс першими, навіть якщо вони прийшли пізніше.

*Приклад:* Системний процес, що відповідає за роботу мережі, має вищий пріоритет, ніж гра, яку ви запустили. Якщо обидва хочуть процесорний час — спочатку отримає мережевий процес.

**Квоти (Quotas).** Кожному процесу або користувачу виділяється максимально допустимий обсяг ресурсу. Це запобігає ситуації, коли одна програма забере всю пам'ять або весь процесорний час.

*Приклад:* На сервері кожен користувач має дискову квоту 5 ГБ. Навіть якщо на диску є 100 ГБ вільного місця, один користувач не зможе зайняти більше 5 ГБ.

**Мультиплексування** — це поділ ресурсу у **часі** або в **просторі**:
- **У часі** — процеси по черзі використовують ресурс. Наприклад, процесор виконує код програми A протягом 10 мс, потім перемикається на програму B на 10 мс, потім на C і так далі. Це відбувається настільки швидко, що здається, ніби всі програми працюють одночасно.
- **У просторі** — ресурс ділиться на частини. Наприклад, оперативна пам'ять ділиться між програмами: браузер отримує 500 МБ, текстовий редактор — 100 МБ, месенджер — 200 МБ.

| Метод | Принцип | Приклад | Переваги | Недоліки |
|---|---|---|---|---|
| Черга (FIFO) | Перший прийшов — перший обслужений | Черга друку | Простота, справедливість | Важливий процес може чекати |
| Пріоритети | Важливіший — першим | Системні vs користувацькі процеси | Гнучкість | "Голодування" низькопріоритетних |
| Квоти | Максимальний ліміт для кожного | Дискова квота користувача | Захист від зловживань | Може бути нестача для легітимних потреб |
| Мультиплексування | Поділ у часі або просторі | Процесорний час, RAM | Ефективне використання | Накладні витрати на перемикання |

---

## 4. Операційне середовище (Runtime Environment)

### 4.1. Що таке операційне середовище

**Операційне середовище** (runtime environment) — це сукупність засобів та сервісів, які надає операційна система для виконання програм. Це "умови", в яких програма живе та працює.

Аналогія: уявіть акваріум для рибки. Рибка — це програма. Акваріум з водою, фільтром, підсвіткою та обігрівачем — це операційне середовище. Без акваріуму рибка не зможе жити, так само як програма не зможе працювати без операційного середовища.

Операційне середовище надає програмі:
- доступ до апаратних ресурсів (процесор, пам'ять, диск) через контрольований інтерфейс;
- набір готових функцій (бібліотеки), які програма може використовувати;
- механізми управління пам'яттю (виділення, звільнення);
- механізми введення-виведення (читання з клавіатури, вивід на екран, запис у файл);
- систему безпеки (контроль доступу, ізоляція процесів).

```
┌──────────────────────────────────────────┐
│           ПРИКЛАДНА ПРОГРАМА             │
│        (браузер, текстовий редактор)     │
├──────────────────────────────────────────┤
│       ОПЕРАЦІЙНЕ СЕРЕДОВИЩЕ              │
│  ┌──────────┬──────────┬──────────────┐  │
│  │Бібліотеки│Системні  │Змінні        │  │
│  │часу      │виклики   │середовища    │  │
│  │виконання │(API)     │(Environment  │  │
│  │(Runtime  │          │ Variables)   │  │
│  │Libraries)│          │              │  │
│  └──────────┴──────────┴──────────────┘  │
├──────────────────────────────────────────┤
│           ЯДРО ОПЕРАЦІЙНОЇ СИСТЕМИ       │
├──────────────────────────────────────────┤
│              АПАРАТУРА                   │
└──────────────────────────────────────────┘
```

---

### 4.2. Системні виклики (API) як інтерфейс між програмою та ОС

**Системний виклик** (system call, syscall) — це механізм, через який програма користувача запитує послугу у ядра ОС. Програми не можуть безпосередньо працювати з апаратурою — це було б небезпечно та нестабільно. Замість цього вони звертаються до ядра через системні виклики.

**Аналогія:** Ви в ресторані (операційне середовище). Ви не йдете самі на кухню (апаратура) готувати їжу. Замість цього ви робите замовлення (системний виклик) офіціанту (API), який передає його на кухню (ядро ОС). Кухар (ядро) готує страву та повертає результат через офіціанта.

**Як працює системний виклик:**

```
1. Програма викликає функцію API
   (наприклад, fopen() — відкрити файл)
            │
            ▼
2. Бібліотека перетворює виклик у системний виклик
   (наприклад, open() з номером 2 у Linux)
            │
            ▼
3. Процесор перемикається з режиму користувача
   в режим ядра (через переривання)
            │
            ▼
4. Ядро виконує запит
   (знаходить файл на диску, перевіряє права)
            │
            ▼
5. Результат повертається програмі
   (дескриптор файлу або код помилки)
            │
            ▼
6. Процесор повертається в режим користувача
```

**Приклади типових системних викликів:**

| Категорія | Приклади | Що робить |
|---|---|---|
| Робота з файлами | `open`, `read`, `write`, `close` | Відкрити, прочитати, записати, закрити файл |
| Управління процесами | `fork`, `exec`, `exit`, `wait` | Створити, запустити, завершити процес |
| Робота з пам'яттю | `mmap`, `brk` | Виділити або звільнити пам'ять |
| Мережеві операції | `socket`, `connect`, `send`, `recv` | Створити з'єднання, відправити/отримати дані |
| Управління пристроями | `ioctl`, `read`, `write` | Керування пристроями введення-виведення |

**Що таке API?** API (Application Programming Interface, інтерфейс програмування додатків) — це набір готових функцій, які спрощують роботу з системними викликами. Замість того щоб безпосередньо робити низькорівневий системний виклик, програміст використовує зручну функцію з бібліотеки.

Наприклад, у мові C замість системного виклику `write()` для виводу на екран програміст використовує функцію `printf()` — вона зручніша і працює однаково на різних ОС.

---

### 4.3. Бібліотеки часу виконання (Runtime Libraries)

**Бібліотеки часу виконання** (runtime libraries) — це набори готового коду, які завантажуються в пам'ять разом з програмою під час її виконання та надають їй базові функції.

**Навіщо потрібні бібліотеки?** Кожна програма потребує типових операцій: вивести текст на екран, прочитати файл, обчислити математичну функцію, працювати з мережею. Замість того щоб кожен програміст писав цей код з нуля, ОС та мови програмування надають готові бібліотеки.

**Приклади:**
- **C Runtime Library (CRT)** — базова бібліотека для програм, написаних на мові C. Надає функції `printf()`, `malloc()`, `fopen()` тощо.
- **msvcrt.dll** — реалізація C Runtime Library у Windows.
- **glibc** — реалізація C Runtime Library у Linux (GNU C Library).
- **.NET Base Class Library** — бібліотека для програм, написаних на C#.
- **Java Class Library** — бібліотека для Java-програм.

**Два типи бібліотек:**

| Тип | Опис | Приклад | Переваги |
|---|---|---|---|
| **Статичні** (.lib, .a) | Вбудовуються в програму при компіляції | libm.a (математична бібліотека) | Програма самодостатня |
| **Динамічні** (.dll, .so) | Завантажуються окремо під час виконання | kernel32.dll, libc.so | Економія пам'яті, легке оновлення |

Динамічні бібліотеки особливо важливі: якщо 10 програм використовують одну й ту саму бібліотеку, в пам'яті зберігається лише одна її копія, а не десять.

---

### 4.4. Змінні середовища (Environment Variables)

**Змінні середовища** — це іменовані значення (пари "ім'я = значення"), які зберігаються в операційній системі та доступні всім програмам. Вони визначають конфігурацію середовища виконання.

**Аналогія:** Змінні середовища — це як написи на дверях у будівлі. Будь-яка людина (програма), що зайшла в будівлю, може прочитати ці написи та зрозуміти, де знаходиться їдальня (TEMP), де вихід (PATH), де її кабінет (HOME).

**Найважливіші змінні середовища:**

**`PATH`** — список каталогів, де ОС шукає виконувані файли. Коли ви набираєте команду в терміналі (наприклад, `python` або `notepad`), ОС переглядає всі каталоги, перелічені в PATH, щоб знайти відповідний файл.

```
Windows: PATH = C:\Windows\System32;C:\Program Files\Python311;C:\Users\User\bin
Linux:   PATH = /usr/local/bin:/usr/bin:/bin:/home/user/bin
```

Каталоги розділяються символом `;` у Windows та `:` у Linux/macOS.

**`TEMP` / `TMP`** — каталог для тимчасових файлів. Програми створюють тут тимчасові файли під час роботи (наприклад, автозбереження документа).

```
Windows: TEMP = C:\Users\User\AppData\Local\Temp
Linux:   TMPDIR = /tmp
```

**`HOME` / `USERPROFILE`** — домашній каталог користувача. Тут зберігаються особисті файли, налаштування програм, документи.

```
Windows: USERPROFILE = C:\Users\User
Linux:   HOME = /home/user
macOS:   HOME = /Users/user
```

**Інші важливі змінні:**

| Змінна | ОС | Призначення |
|---|---|---|
| `COMPUTERNAME` / `HOSTNAME` | Windows / Linux | Ім'я комп'ютера |
| `USERNAME` / `USER` | Windows / Linux | Ім'я поточного користувача |
| `OS` | Windows | Тип операційної системи |
| `LANG` | Linux | Мова та кодування системи |
| `JAVA_HOME` | Кросплатформна | Каталог встановленої Java |

**Як переглянути змінні середовища?**
- **Windows:** команда `set` у командному рядку або `Система` → `Додаткові параметри системи` → `Змінні середовища`.
- **Linux/macOS:** команда `env` або `printenv` у терміналі.

**Як програми використовують змінні середовища?** При запуску нового процесу ОС передає йому копію всіх змінних середовища. Програма може читати ці змінні та змінювати свою поведінку відповідно. Наприклад, програма-архіватор зчитує змінну TEMP, щоб знати, куди розпаковувати тимчасові файли.

## 5. Програмне середовище

### 5.1. Сумісність програм з ОС

Програма, написана для однієї операційної системи, зазвичай **не може працювати на іншій** без додаткових зусиль. Це пов'язано з тим, що кожна ОС має:
- свій набір системних викликів;
- свій формат виконуваних файлів (.exe у Windows, ELF у Linux, Mach-O у macOS);
- свої бібліотеки та API;
- свій спосіб роботи з пам'яттю, файловою системою, пристроями.

**Приклад:** Програма, написана для Windows, використовує функції з бібліотеки `kernel32.dll` та формат файлу `.exe`. Linux не знає, що таке `.dll` та `.exe`, і не має відповідних системних функцій. Тому запустити таку програму на Linux напряму неможливо.

**Програмне середовище** — це сукупність засобів, які забезпечують виконання програм, написаних для певної платформи. ОС може підтримувати одне або кілька програмних середовищ.

---

### 5.2. Підсистеми середовища та рівні сумісності

Сучасні ОС мають механізми для запуску програм, призначених для інших систем:

**Windows Subsystem for Linux (WSL)** — підсистема Windows, яка дозволяє запускати Linux-програми безпосередньо у Windows без віртуальної машини.

- **WSL 1** працювала як шар сумісності — перетворювала системні виклики Linux у системні виклики Windows "на льоту". Це не потребувало повного ядра Linux, але деякі програми працювали некоректно.
- **WSL 2** використовує справжнє ядро Linux у легковагій віртуальній машині. Це забезпечує повну сумісність з Linux-програмами та працює значно швидше (до 20 разів швидше при деяких операціях). WSL 2 стала стандартом у Windows 10/11 та Windows Server 2025.

**Wine** ("Wine Is Not an Emulator") — вільний шар сумісності для запуску Windows-програм у Linux та macOS. Wine не емулює Windows, а перетворює виклики Windows API у відповідні виклики API поточної ОС.

**Rosetta 2** (macOS) — транслятор від Apple, який дозволяє запускати програми, написані для процесорів Intel (x86-64), на комп'ютерах Apple з процесорами M1/M2/M3/M4 (ARM). Rosetta автоматично перетворює інструкції однієї архітектури процесора в іншу.

**Режим сумісності у Windows** — можливість запускати програми, написані для старіших версій Windows (наприклад, Windows 7 або Windows XP). Windows емулює поведінку старої версії ОС для конкретної програми.

---

### 5.3. Емуляція та віртуальні машини

Коли шарів сумісності недостатньо, використовують більш потужні інструменти:

**Емуляція** — програмне відтворення роботи однієї системи в середовищі іншої. Емулятор імітує апаратне забезпечення цільової платформи повністю.

*Приклади:*
- Емулятори ігрових консолей (NES, PlayStation) на ПК.
- Емулятор Android (для розробників мобільних додатків).
- QEMU — універсальний емулятор, що може імітувати різні процесори.

**Віртуальна машина (Virtual Machine, VM)** — програмне забезпечення, яке створює ізольований "віртуальний комп'ютер" всередині реального. На віртуальну машину можна встановити будь-яку ОС та запускати програми для цієї ОС.

```
┌──────────────────────────────────────────────┐
│              Реальний комп'ютер               │
│  ┌────────────────────────────────────────┐  │
│  │     Основна ОС (наприклад, Windows)    │  │
│  │  ┌──────────────┐  ┌───────────────┐  │  │
│  │  │  Віртуальна  │  │  Віртуальна   │  │  │
│  │  │  машина 1    │  │  машина 2     │  │  │
│  │  │ ┌──────────┐ │  │ ┌───────────┐ │  │  │
│  │  │ │  Ubuntu   │ │  │ │  Windows  │ │  │  │
│  │  │ │  Linux    │ │  │ │  Server   │ │  │  │
│  │  │ └──────────┘ │  │ └───────────┘ │  │  │
│  │  └──────────────┘  └───────────────┘  │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

**Популярні системи віртуалізації:**
- **VirtualBox** (Oracle) — безкоштовна, підходить для навчання.
- **VMware Workstation** — професійне рішення.
- **Hyper-V** — вбудований у Windows гіпервізор.
- **Parallels Desktop** — для macOS (запуск Windows на Mac).

**Переваги віртуальних машин:**
- Повна ізоляція — можна безпечно тестувати вірусне ПЗ або нестабільні програми.
- Запуск будь-якої ОС на будь-якому комп'ютері.
- Створення знімків (snapshots) стану для швидкого відкату.

**Недоліки:**
- Потребують значних ресурсів (RAM, процесорний час, дисковий простір).
- Працюють повільніше, ніж програми, запущені безпосередньо на "рідній" ОС.

---

## 6. Win32 API, .NET, Java Runtime — приклади програмних середовищ

### 6.1. Win32 API

**Win32 API** — це базовий набір програмних інтерфейсів операційної системи Windows. Це "рідний" API Windows, через який програми безпосередньо взаємодіють з ядром ОС.

**Характеристики:**
- Програми написані на C/C++ та компілюються в машинний код.
- Прямий доступ до функцій ОС: створення вікон, робота з файлами, управління процесами, мережа.
- Працюють лише на Windows.
- Максимальна продуктивність, бо немає проміжних рівнів.
- Складний у використанні — потрібно знати багато деталей роботи ОС.

**Приклади програм на Win32 API:** Блокнот (Notepad), Провідник Windows, Диспетчер завдань.

---

### 6.2. .NET Runtime (CLR)

**.NET** — це платформа від Microsoft, яка надає програмне середовище для розробки та виконання програм. Ключовий компонент — **Common Language Runtime (CLR)**, середовище виконання, яке бере на себе управління пам'яттю, безпеку та виконання коду.

**Як працює:**
1. Програміст пише код на C#, VB.NET або F#.
2. Компілятор перетворює код у проміжну мову — **IL** (Intermediate Language), подібно до байт-коду Java.
3. Під час запуску CLR компілює IL у машинний код для конкретного процесора (JIT — Just-In-Time компіляція).
4. CLR виконує код, автоматично керуючи пам'яттю (збирач сміття — garbage collector).

**Переваги:**
- Автоматичне управління пам'яттю — програміст не мусить вручну звільняти пам'ять.
- Безпека типів — CLR перевіряє правильність роботи з даними.
- Кросплатформність — .NET (починаючи з .NET Core / .NET 5+) працює на Windows, Linux та macOS.
- Підтримка багатьох мов програмування в одному середовищі.

---

### 6.3. Java Runtime Environment (JRE)

**Java Runtime Environment (JRE)** — це середовище виконання для програм, написаних мовою Java. Головний принцип Java: **"Write Once, Run Anywhere"** ("Напиши один раз — запускай будь-де").

**Як працює:**
1. Програміст пише код на Java.
2. Компілятор Java (`javac`) перетворює код у **байт-код** — проміжний формат, який не є машинним кодом.
3. **Java Virtual Machine (JVM)** — віртуальна машина Java — виконує байт-код. JVM існує для всіх основних ОС.
4. JVM під час виконання компілює байт-код у машинний код конкретної платформи (JIT-компіляція).

```
         Код Java (.java)
              │
              ▼
    Компілятор (javac)
              │
              ▼
       Байт-код (.class)
              │
     ┌────────┼────────┐
     ▼        ▼        ▼
  JVM на   JVM на   JVM на
  Windows  Linux    macOS
     │        │        │
     ▼        ▼        ▼
 Машинний  Машинний Машинний
   код       код      код
  x86-64   x86-64    ARM
```

**Ключові компоненти JRE:**
- **JVM** — виконує байт-код, керує пам'яттю, забезпечує безпеку.
- **Бібліотека класів** (Java Class Library) — тисячі готових класів для роботи з файлами, мережею, графікою, базами даних.
- **ClassLoader** — завантажувач класів, що динамічно завантажує потрібні класи в пам'ять.

## 7. Контейнеризація (Docker) — сучасний підхід до ізоляції середовищ

**Контейнеризація** — це сучасний метод ізоляції додатків, який є легшою альтернативою віртуальним машинам.

**Проблема, яку вирішує контейнеризація:** Програма, яка працює на комп'ютері розробника, може не працювати на сервері, бо відрізняються версії бібліотек, налаштування ОС, змінні середовища. Класична фраза розробників: "У мене все працює!" ("It works on my machine!").

**Docker** — найпопулярніша платформа контейнеризації. Docker дозволяє "упакувати" програму разом із усім її середовищем (бібліотеки, конфігурація, залежності) у **контейнер** — ізольований пакет, який гарантовано працює однаково на будь-якому комп'ютері з Docker.

**Контейнер vs Віртуальна машина:**

```
    Віртуальна машина              Контейнер (Docker)
┌────────────────────┐        ┌────────────────────┐
│     Додаток A      │        │     Додаток A      │
├────────────────────┤        ├────────────────────┤
│  Бібліотеки A      │        │  Бібліотеки A      │
├────────────────────┤        ├────────────────────┤
│  Гостьова ОС       │        │                    │
│  (повна копія!)    │        │   (ОС не потрібна!)│
├────────────────────┤        ├────────────────────┤
│  Гіпервізор        │        │  Docker Engine     │
├────────────────────┤        ├────────────────────┤
│  Основна ОС        │        │  Основна ОС        │
├────────────────────┤        ├────────────────────┤
│  Апаратура          │        │  Апаратура          │
└────────────────────┘        └────────────────────┘
     ~декілька ГБ                  ~десятки МБ
     Запуск: хвилини               Запуск: секунди
```

**Ключові відмінності контейнерів від віртуальних машин:**

| Параметр | Віртуальна машина | Контейнер (Docker) |
|---|---|---|
| Розмір | Гігабайти (повна ОС) | Мегабайти (лише додаток) |
| Час запуску | Хвилини | Секунди |
| Ізоляція | Повна (окреме ядро) | Часткова (спільне ядро ОС) |
| Продуктивність | Нижча (накладні витрати) | Близька до "рідної" |
| Застосування | Запуск різних ОС | Ізоляція додатків |

Docker активно використовується разом з **WSL 2** у Windows: Docker Desktop інтегрується з WSL 2 і запускає Linux-контейнери без повноцінної віртуальної машини, використовуючи справжнє ядро Linux, яке надає WSL 2.

**Де використовується Docker:**
- Веб-розробка — запуск веб-серверів, баз даних у контейнерах.
- Хмарні сервіси (AWS, Azure, Google Cloud) — масштабування додатків.
- DevOps — автоматизація розгортання програмного забезпечення.
- Тестування — створення ідентичних середовищ для тестувальників.

## 8. Підсумок

У цьому уроці ми розглянули ключові концепції, пов'язані з ресурсами ОС та середовищами виконання програм:

1. **Ресурси** бувають апаратні (процесор, RAM, диски, пристрої I/O), програмні (файли, бібліотеки, змінні середовища) та інформаційні (бази даних, дані). Усі ресурси обмежені, тому потребують управління.

2. **ОС як менеджер ресурсів** — розподіляє, моніторить, захищає та обліковує ресурси. Проблеми: конкуренція за ресурси та взаємне блокування (deadlock). Методи розподілу: черга, пріоритети, квоти, мультиплексування.

3. **Операційне середовище** — сукупність засобів ОС для виконання програм: системні виклики (API), бібліотеки часу виконання, змінні середовища (PATH, TEMP, HOME).

4. **Програмне середовище** — забезпечує сумісність програм з конкретною платформою. Підсистеми сумісності (WSL, Wine, Rosetta 2), емуляція та віртуальні машини дозволяють запускати програми для інших ОС.

5. **Win32 API, .NET, JRE** — конкретні приклади програмних середовищ з різними підходами до виконання коду (машинний код, проміжна мова + JIT-компіляція).

6. **Контейнеризація (Docker)** — сучасний легковаговий підхід до ізоляції середовищ, альтернатива важким віртуальним машинам.

## 10. Глосарій ключових термінів

| Термін | Визначення |
|---|---|
| **Ресурс** | Будь-який апаратний або програмний компонент обмеженої доступності, необхідний для виконання програм |
| **Апаратні ресурси** | Фізичні компоненти комп'ютера: процесор, RAM, диски, пристрої I/O |
| **Програмні ресурси** | Нематеріальні компоненти: файли, бібліотеки, мережеві з'єднання, змінні середовища |
| **Інформаційні ресурси** | Дані та їх колекції: бази даних, файли даних, реєстр ОС |
| **Менеджер ресурсів** | Функція ОС з розподілу, моніторингу, захисту та обліку ресурсів |
| **Конкуренція за ресурси** | Ситуація, коли кілька процесів одночасно потребують один ресурс |
| **Deadlock (взаємне блокування)** | Ситуація, коли два або більше процеси заблоковані назавжди, кожен чекаючи ресурс, утримуваний іншим |
| **Умови Коффмана** | Чотири умови, одночасне виконання яких призводить до deadlock: взаємне виключення, утримування та очікування, неможливість примусового вивільнення, циклічне очікування |
| **FIFO (First In, First Out)** | Метод розподілу ресурсів за принципом черги: перший запит обслуговується першим |
| **Пріоритет** | Числове значення, що визначає важливість процесу при розподілі ресурсів |
| **Квота** | Максимально допустимий обсяг ресурсу, виділений процесу або користувачу |
| **Мультиплексування** | Поділ ресурсу між кількома процесами у часі (по черзі) або у просторі (на частини) |
| **Операційне середовище (Runtime Environment)** | Сукупність засобів ОС, що забезпечують виконання програм: системні виклики, бібліотеки, змінні середовища |
| **Системний виклик (System Call)** | Механізм запиту послуги у ядра ОС з боку програми користувача, перемикання з режиму користувача в режим ядра |
| **API (Application Programming Interface)** | Набір готових функцій, що спрощують взаємодію програми з ОС або іншими програмами |
| **Бібліотека часу виконання (Runtime Library)** | Набір готового коду, який завантажується разом з програмою та надає базові функції |
| **Динамічна бібліотека (.dll / .so)** | Бібліотека, що завантажується в пам'ять під час виконання програми та може спільно використовуватись кількома програмами |
| **Змінна середовища (Environment Variable)** | Іменоване значення в ОС, доступне всім програмам (наприклад, PATH, TEMP, HOME) |
| **PATH** | Змінна середовища зі списком каталогів, де ОС шукає виконувані файли |
| **Програмне середовище** | Сукупність засобів, що забезпечують виконання програм, написаних для певної платформи |
| **WSL (Windows Subsystem for Linux)** | Підсистема Windows для запуску Linux-програм без віртуальної машини |
| **Емуляція** | Програмне відтворення роботи однієї системи в середовищі іншої |
| **Віртуальна машина (VM)** | Програмний "віртуальний комп'ютер", на який можна встановити окрему ОС |
| **Win32 API** | Базовий набір програмних інтерфейсів ОС Windows для створення нативних програм |
| **.NET CLR (Common Language Runtime)** | Середовище виконання .NET-програм від Microsoft з автоматичним управлінням пам'яттю |
| **JRE (Java Runtime Environment)** | Середовище виконання Java-програм, що включає JVM та стандартні бібліотеки |
| **JVM (Java Virtual Machine)** | Віртуальна машина, що виконує байт-код Java на будь-якій платформі |
| **JIT-компіляція (Just-In-Time)** | Компіляція проміжного коду в машинний код безпосередньо під час виконання програми |
| **Байт-код** | Проміжний формат коду (Java, .NET), який виконується віртуальною машиною, а не процесором напряму |
| **Контейнеризація** | Технологія ізоляції додатків у легковагових контейнерах зі спільним ядром ОС |
| **Docker** | Платформа контейнеризації, що дозволяє упакувати додаток з усіма залежностями в ізольований контейнер |
| **Контейнер** | Ізольований пакет з програмою та всіма її залежностями, що використовує спільне ядро ОС |
| **Гіпервізор** | Програмне забезпечення для створення та управління віртуальними машинами (VirtualBox, Hyper-V, VMware) |
