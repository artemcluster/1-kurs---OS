## 1. Вступ: навіщо потрібне перетворення адрес

Кожна програма під час виконання працює з **віртуальними (логічними) адресами**. Програма «думає», що вона одна в пам'яті і що їй доступний увесь адресний простір. Але насправді в оперативній пам'яті (ОЗП) одночасно знаходяться десятки та сотні процесів, і їхні дані розташовані в різних місцях.

Тому операційна система разом із апаратним модулем **MMU** (Memory Management Unit — блок керування пам'яттю) виконує **перетворення адрес**: кожна віртуальна адреса, яку видає процесор, перетворюється на фізичну адресу в оперативній пам'яті.

Існує два основні підходи до організації цього перетворення:

- **Сторінкова організація (Paging)** — поділ пам'яті на блоки фіксованого розміру
- **Сегментація (Segmentation)** — поділ пам'яті на блоки змінного розміру за логічним призначенням

```
  Програма (віртуальний простір)          Оперативна пам'ять (фізична)
  ┌──────────────────────┐                ┌──────────────────────┐
  │  Код програми        │───┐            │  Дані процесу B      │
  │  (адреси 0x0000...)  │   │   MMU      │                      │
  ├──────────────────────┤   │ перетворює  ├──────────────────────┤
  │  Глобальні змінні    │───┼──адреси──> │  Код процесу A       │ <──
  │  (адреси 0x4000...)  │   │            ├──────────────────────┤
  ├──────────────────────┤   │            │  Стек процесу C      │
  │  Стек                │───┘            ├──────────────────────┤
  │  (адреси 0xFF00...)  │               │  Змінні процесу A    │ <──
  └──────────────────────┘                └──────────────────────┘
  Програма "бачить"                       Реальне розташування
  безперервний простір                    може бути будь-яким
```

## 2. Сторінкова організація пам'яті (Paging)

### 2.1. Сторінки та фрейми

При сторінковій організації пам'яті:

- **Віртуальний адресний простір** процесу поділяється на блоки однакового фіксованого розміру — **сторінки (pages)**
- **Фізична оперативна пам'ять** поділяється на блоки такого самого розміру — **фрейми (frames)**, або кадри

Кожна сторінка процесу може бути розміщена в **будь-якому** вільному фреймі фізичної пам'яті. Сторінки одного процесу **не обов'язково** розташовані поруч у фізичній пам'яті — вони можуть бути розкидані по різних фреймах.

```
  Віртуальний простір              Фізична пам'ять (ОЗП)
  процесу A:                       (розділена на фрейми):

  ┌─────────────────┐              ┌─────────────────┐  Фрейм 0
  │  Сторінка 0     │─────────────>│  Сторінка 0 (A) │
  ├─────────────────┤              ├─────────────────┤  Фрейм 1
  │  Сторінка 1     │──┐           │  Сторінка 3 (B) │
  ├─────────────────┤  │           ├─────────────────┤  Фрейм 2
  │  Сторінка 2     │  │     ┌────>│  Сторінка 2 (A) │
  ├─────────────────┤  │     │     ├─────────────────┤  Фрейм 3
  │  Сторінка 3     │  │     │     │  (вільний)      │
  └─────────────────┘  │     │     ├─────────────────┤  Фрейм 4
                       │     │     │  Сторінка 1 (B) │
                       └─────│────>├─────────────────┤  Фрейм 5
                             │     │  Сторінка 1 (A) │
                             │     ├─────────────────┤  Фрейм 6
                             └─────│  Сторінка 3 (A) │
                                   ├─────────────────┤  Фрейм 7
                                   │  Сторінка 2 (A) │
                                   └─────────────────┘
```

### 2.2. Типовий розмір сторінки

Розмір сторінки визначається архітектурою процесора і є **однаковим** для сторінок та фреймів:

| Архітектура | Типовий розмір сторінки | Примітка |
|-------------|------------------------|----------|
| **x86 / x86-64** | **4 КБ** (4096 байт) | Найпоширеніший. Також підтримуються великі сторінки 2 МБ та 1 ГБ |
| **ARM** | **4 КБ** або **16 КБ** | ARM64 (AArch64) підтримує 4 КБ, 16 КБ, 64 КБ |
| **RISC-V** | **4 КБ** | Також підтримує суперсторінки |

Чому саме 4 КБ? Це компроміс:
- **Менший розмір** (наприклад, 1 КБ) — менше втрат пам'яті через внутрішню фрагментацію, але таблиця сторінок стає дуже великою
- **Більший розмір** (наприклад, 64 КБ) — менша таблиця, але більше внутрішніх втрат у кожній сторінці

### 2.3. Як формується віртуальна адреса

Віртуальна адреса, яку генерує процесор, складається з двох частин:

```
  Віртуальна адреса (наприклад, 32 біти):
  ┌──────────────────────┬────────────────┐
  │  Номер сторінки (p)  │  Зміщення (d)  │
  │     (20 біт)         │   (12 біт)     │
  └──────────────────────┴────────────────┘

  - Номер сторінки (p) — вказує, ЯКА саме сторінка потрібна
  - Зміщення (d) — вказує, ЯКИЙ БАЙТ всередині сторінки потрібен

  При розмірі сторінки 4 КБ = 4096 байт = 2^12:
  - Зміщення: 12 біт (щоб адресувати кожен з 4096 байт)
  - Номер сторінки: 32 - 12 = 20 біт (до 2^20 = 1 048 576 сторінок)
```

## 3. Таблиця сторінок (Page Table)

### 3.1. Призначення таблиці сторінок

**Таблиця сторінок (Page Table)** — це структура даних, яку підтримує операційна система для **кожного** процесу. Вона зберігає відповідність між номерами віртуальних сторінок та номерами фізичних фреймів.

Коли процесор генерує віртуальну адресу, блок MMU бере номер сторінки (p), шукає його в таблиці сторінок і отримує номер фрейму (f). Потім номер фрейму поєднується зі зміщенням (d) — і виходить фізична адреса.

```
  Таблиця сторінок процесу A:

  ┌───────────────┬────────────┬─────────┬──────────┬──────────┐
  │ Номер сторінки│ Номер      │ Біт     │ Біт      │ Біт      │
  │ (індекс)      │ фрейму     │присутн. │модифікац.│ захисту  │
  ├───────────────┼────────────┼─────────┼──────────┼──────────┤
  │      0        │     5      │    1    │    0     │   R/W    │
  │      1        │     2      │    1    │    1     │   R/W    │
  │      2        │     —      │    0    │    —     │    —     │ <-- на диску!
  │      3        │     8      │    1    │    0     │    R     │
  │      4        │     1      │    1    │    0     │   R/W    │
  │     ...       │    ...     │   ...   │   ...    │   ...    │
  └───────────────┴────────────┴─────────┴──────────┴──────────┘
```

### 3.2. Структура запису таблиці сторінок (Page Table Entry — PTE)

Кожен рядок таблиці сторінок називається **PTE** (Page Table Entry) — запис таблиці сторінок. Він містить:

| Поле | Опис |
|------|------|
| **Номер фрейму (Frame Number)** | Номер фізичного фрейму, де зараз знаходиться ця сторінка |
| **Біт присутності (Present / Valid bit)** | 1 — сторінка завантажена в ОЗП; 0 — сторінка зараз на диску (у файлі підкачки). Якщо 0 — виникає **сторінкове переривання (page fault)** |
| **Біт модифікації (Dirty bit)** | 1 — вміст сторінки змінювався з моменту завантаження; 0 — не змінювався. Якщо 0, при вивантаженні з ОЗП сторінку можна не записувати на диск (економія часу) |
| **Біт доступу (Referenced / Accessed bit)** | 1 — до сторінки зверталися (читання або запис). Використовується алгоритмами заміщення сторінок |
| **Біти захисту (Protection bits)** | Визначають дозволені операції: R (читання), W (запис), X (виконання). Наприклад, сторінка з кодом — R+X, сторінка з даними — R+W |

```
  Структура одного запису PTE (спрощено):

  ┌──────────────┬───────┬───────┬──────────┬──────────┬──────────┐
  │ Номер фрейму │Present│ Dirty │Referenced│ Захист   │ Інші     │
  │  (20 біт)    │(1 біт)│(1 біт)│ (1 біт)  │ (3 біти) │ (6 біт)  │
  └──────────────┴───────┴───────┴──────────┴──────────┴──────────┘
  |<───────────────── Загалом 32 біти (4 байти) ────────────────>|
```

### 3.3. Схема перетворення адреси

```
  ПЕРЕТВОРЕННЯ ВІРТУАЛЬНОЇ АДРЕСИ У ФІЗИЧНУ:

  Віртуальна адреса від процесора:
  ┌──────────────────┬───────────────┐
  │ Номер сторінки p │ Зміщення d    │
  └────────┬─────────┴───────────────┘
           │
           ▼
  ┌──────────────────────────┐
  │    ТАБЛИЦЯ СТОРІНОК      │
  │  ┌─────┬──────────────┐  │
  │  │  0  │  Фрейм 5     │  │
  │  │  1  │  Фрейм 2     │  │
  │  │  2  │  (на диску)   │  │
  │  │ [p] │  Фрейм f  ◄──│──── знайдено!
  │  │  4  │  Фрейм 1     │  │
  │  └─────┴──────────────┘  │
  └──────────────────────────┘
           │
           ▼
  Фізична адреса:
  ┌──────────────────┬───────────────┐
  │ Номер фрейму f   │ Зміщення d    │
  └──────────────────┴───────────────┘
  (зміщення залишається тим самим!)
```

## 4. Приклад перетворення віртуальної адреси у фізичну

Розглянемо покроковий числовий приклад.

**Дано:**
- Розмір сторінки: 4 КБ = 4096 байт
- Віртуальна адреса: **8196** (десяткове)
- Таблиця сторінок процесу:

| Сторінка | Фрейм |
|----------|--------|
| 0        | 5      |
| 1        | 3      |
| 2        | 7      |
| 3        | 1      |

**Крок 1.** Знаходимо номер сторінки та зміщення:

```
  Номер сторінки p = 8196 / 4096 = 2 (ціла частина)
  Зміщення d       = 8196 % 4096 = 8196 - 2 * 4096 = 8196 - 8192 = 4
```

Тобто віртуальна адреса 8196 — це **сторінка 2, зміщення 4**.

**Крок 2.** Знаходимо номер фрейму в таблиці сторінок:

```
  Сторінка 2 → Фрейм 7
```

**Крок 3.** Обчислюємо фізичну адресу:

```
  Фізична адреса = Номер фрейму * Розмір сторінки + Зміщення
  Фізична адреса = 7 * 4096 + 4 = 28672 + 4 = 28676
```

**Результат:** віртуальна адреса **8196** перетворюється у фізичну адресу **28676**.

```
  Віртуальна адреса: 8196
  ┌──────────────┬──────────────┐
  │ Сторінка = 2 │ Зміщення = 4 │
  └──────┬───────┴──────────────┘
         │
         ▼  (таблиця сторінок: стор. 2 → фрейм 7)
         │
  ┌──────┴───────┬──────────────┐
  │ Фрейм = 7   │ Зміщення = 4 │
  └──────────────┴──────────────┘
  Фізична адреса: 7 * 4096 + 4 = 28676
```

## 5. Багаторівневі таблиці сторінок

### 5.1. Проблема: однорівнева таблиця занадто велика

У системі з 32-бітною адресацією та сторінкою 4 КБ:
- Кількість сторінок: 2^20 = 1 048 576
- Розмір одного PTE: 4 байти
- Розмір таблиці сторінок: 1 048 576 * 4 = **4 МБ на кожен процес**

Якщо в системі працює 100 процесів, лише на таблиці сторінок піде 100 * 4 МБ = **400 МБ**! При цьому більшість процесів використовують лише невелику частину свого адресного простору — більшість записів таблиці сторінок порожні.

### 5.2. Рішення: дворівнева таблиця сторінок

Замість однієї великої таблиці створюється **ієрархія** з маленьких таблиць:

- **Каталог сторінок (Page Directory)** — таблиця першого рівня, яка завжди в пам'яті. Містить 1024 записи, кожен вказує на таблицю другого рівня.
- **Таблиці сторінок другого рівня** — 1024 маленькі таблиці, кожна з 1024 записів. Створюються **лише за потребою** — тільки для тих ділянок віртуального простору, які процес реально використовує.

```
  Віртуальна адреса (32 біти):
  ┌────────────┬────────────┬──────────────┐
  │  10 біт    │  10 біт    │   12 біт     │
  │ Індекс у   │ Індекс у   │  Зміщення    │
  │ каталозі   │ таблиці    │  на сторінці │
  └─────┬──────┴─────┬──────┴──────────────┘
        │            │
        ▼            │
  ┌──────────────┐   │
  │   Каталог    │   │
  │  сторінок    │   │
  │ (1024 записи)│   │
  │  ┌────────┐  │   │
  │  │ PDE 0  │──│───│──> Таблиця сторінок 0 (може не існувати)
  │  │ PDE 1  │──│───│──> Таблиця сторінок 1
  │  │  ...   │  │   │
  │  │ PDE n ◄│──│───┘
  │  │  ...   │  │        ┌──────────────────┐
  │  │PDE 1023│  │        │ Таблиця сторінок │
  │  └────────┘  │        │  n (1024 записи) │
  └──────────────┘        │  ┌──────────┐    │
                          │  │  PTE 0   │    │
                          │  │  PTE 1   │    │
                          │  │   ...    │    │
                          │  │  PTE m  ◄├────│──> Фрейм f
                          │  │   ...    │    │
                          │  └──────────┘    │
                          └──────────────────┘
```

### 5.3. Економія пам'яті

**Приклад:** процес використовує лише 2 ділянки пам'яті — код (початок адресного простору) та стек (кінець).

- **Однорівнева таблиця:** потрібна повна таблиця на 1 048 576 записів = 4 МБ, навіть якщо 99% записів порожні
- **Дворівнева таблиця:** потрібен каталог (4 КБ) + лише 2 таблиці другого рівня (2 * 4 КБ) = **12 КБ**

Економія: з 4 МБ до 12 КБ — майже в **340 разів**!

### 5.4. Сучасні системи

У 64-бітних системах (x86-64) використовуються **4-рівневі** та навіть **5-рівневі** таблиці сторінок, оскільки адресний простір значно більший:

| Рівень | Назва (x86-64) | Записів |
|--------|----------------|---------|
| 1 (верхній) | PML4 (Page Map Level 4) | 512 |
| 2 | PDPT (Page Directory Pointer Table) | 512 |
| 3 | PD (Page Directory) | 512 |
| 4 (нижній) | PT (Page Table) | 512 |

## 6. TLB — кеш для таблиці сторінок

### 6.1. Проблема швидкості

При кожному зверненні до пам'яті процесор має спочатку знайти потрібний запис у таблиці сторінок. Але таблиця сторінок сама знаходиться в оперативній пам'яті! Тобто для **одного** звернення до даних потрібно **два** (або більше, при багаторівневій таблиці) звернення до ОЗП: спочатку до таблиці, потім до самих даних. Це суттєво уповільнює роботу.

### 6.2. Що таке TLB

**TLB (Translation Lookaside Buffer)** — це невеликий, але надзвичайно швидкий апаратний кеш, вбудований у процесор (у блок MMU). Він зберігає **нещодавно використані** відповідності «номер сторінки → номер фрейму».

TLB зазвичай містить від 64 до 1024 записів. Цього достатньо, оскільки програми зазвичай працюють із невеликою кількістю сторінок у кожен момент часу (принцип **локальності**).

### 6.3. Як працює TLB

```
  Процесор генерує віртуальну адресу
            │
            ▼
  ┌─────────────────────┐
  │  Перевірити TLB      │
  │  (пошук за номером   │
  │   сторінки)          │
  └──────────┬──────────┘
             │
      ┌──────┴──────┐
      │             │
  TLB HIT       TLB MISS
  (знайдено!)   (не знайдено)
      │             │
      ▼             ▼
  Фрейм          Звернення до
  отримано        таблиці сторінок
  миттєво         в ОЗП (повільно)
  (~1 нс)              │
      │                ▼
      │           Фрейм знайдено
      │           в таблиці
      │                │
      │                ▼
      │           Оновити TLB
      │           (записати новий запис)
      │                │
      ▼                ▼
  Сформувати фізичну адресу
  (фрейм + зміщення)
            │
            ▼
      Доступ до даних
      у фізичній пам'яті
```

### 6.4. TLB Hit та TLB Miss

| Подія | Опис | Час |
|-------|------|-----|
| **TLB Hit** (влучення) | Потрібна відповідність знайдена в TLB. Перетворення адреси виконується майже миттєво | ~1 нс |
| **TLB Miss** (промах) | Відповідності немає в TLB. Потрібне звернення до таблиці сторінок в ОЗП (page walk). Після знаходження запис додається в TLB | ~10-100 нс |
| **Page Fault** (сторінкове переривання) | Сторінки взагалі немає в ОЗП — вона на диску. Потрібне завантаження з диска | ~1 000 000 нс та більше |

На практиці **TLB Hit Rate** (частка влучень) досягає **95-99%**, що робить сторінкову організацію ефективною.

### 6.5. Чому TLB такий ефективний

Ефективність TLB пояснюється **принципом локальності**:

- **Часова локальність** — якщо програма звернулася до якоїсь адреси, вона, ймовірно, звернеться до неї знову найближчим часом (цикли, повторні виклики)
- **Просторова локальність** — якщо програма звернулася до адреси N, вона, ймовірно, звернеться і до адрес поруч (N+1, N+2...), тобто до тієї ж самої сторінки

Тому невеликий кеш TLB покриває переважну більшість звернень.

## 7. Сегментація (Segmentation)

### 7.1. Що таке сегмент

**Сегмент** — це логічна частина програми, що відповідає певному призначенню. На відміну від сторінок, сегменти мають **змінний розмір** і відповідають структурі програми.

Типові сегменти програми:

| Сегмент | Призначення | Приклад вмісту |
|---------|-------------|----------------|
| **Сегмент коду (Code / Text)** | Виконуваний код програми | Машинні інструкції |
| **Сегмент даних (Data)** | Глобальні та статичні змінні | Ініціалізовані змінні |
| **Сегмент стеку (Stack)** | Локальні змінні, адреси повернення | Параметри функцій |
| **Сегмент купи (Heap)** | Динамічно виділена пам'ять | Об'єкти, масиви |

```
  Програма з погляду сегментації:

  ┌──────────────────────────┐
  │   Сегмент 0: КОД         │  Розмір: 8 КБ
  │   (інструкції програми)  │
  ├──────────────────────────┤
  │   Сегмент 1: ДАНІ        │  Розмір: 3 КБ
  │   (глобальні змінні)     │
  ├──────────────────────────┤
  │   Сегмент 2: СТЕК        │  Розмір: 12 КБ
  │   (локальні змінні)      │
  ├──────────────────────────┤
  │   Сегмент 3: КУПА        │  Розмір: 20 КБ
  │   (динамічна пам'ять)    │
  └──────────────────────────┘
  Кожен сегмент має свій розмір!
```

### 7.2. Таблиця сегментів (Segment Table)

Операційна система підтримує **таблицю сегментів** для кожного процесу. Кожен запис таблиці містить два поля:

| Поле | Опис |
|------|------|
| **Базова адреса (Base)** | Фізична адреса початку сегмента в оперативній пам'яті |
| **Ліміт (Limit)** | Максимальна довжина сегмента (розмір у байтах) |

```
  Таблиця сегментів процесу:

  ┌─────────┬──────────────┬─────────────┐
  │ Сегмент │ Базова адреса│   Ліміт     │
  ├─────────┼──────────────┼─────────────┤
  │    0    │    2000      │    8192     │  (код, 8 КБ)
  │    1    │   14000      │    3072     │  (дані, 3 КБ)
  │    2    │    6000      │   12288     │  (стек, 12 КБ)
  │    3    │   20000      │   20480     │  (купа, 20 КБ)
  └─────────┴──────────────┴─────────────┘
```

### 7.3. Перетворення адреси при сегментації

Віртуальна адреса при сегментації складається з двох частин:
- **Номер сегмента (s)** — який сегмент потрібен
- **Зміщення (d)** — яке місце всередині сегмента

```
  Перетворення адреси при сегментації:

  Віртуальна адреса:
  ┌─────────────────┬───────────────┐
  │ Номер сегмента s│ Зміщення d    │
  └────────┬────────┴───────────────┘
           │
           ▼
  ┌──────────────────────────┐
  │    ТАБЛИЦЯ СЕГМЕНТІВ     │
  │  ┌────┬───────┬────────┐ │
  │  │ s  │ Base  │ Limit  │ │
  │  └──┬─┴───┬───┴───┬────┘ │
  └─────│─────│───────│──────┘
        │     │       │
        │     │  Перевірка:
        │     │  d < Limit?
        │     │       │
        │     │   Так ─┘ Ні → ПОМИЛКА
        │     │         (вихід за межі сегмента)
        │     ▼
        │  Фізична адреса = Base + d
        │
        ▼
  Приклад: сегмент 1, зміщення 500
  Base = 14000, Limit = 3072
  500 < 3072? Так ✓
  Фізична адреса = 14000 + 500 = 14500
```

### 7.4. Переваги сегментації

1. **Логічне групування** — сегменти відповідають структурі програми, що полегшує розуміння та організацію пам'яті
2. **Захист** — кожен сегмент може мати свої права доступу (код — тільки читання + виконання, дані — читання + запис)
3. **Спільний доступ** — два процеси можуть спільно використовувати один сегмент (наприклад, спільну бібліотеку), просто вказавши однакову базову адресу в своїх таблицях сегментів
4. **Контроль меж** — апаратна перевірка ліміту запобігає виходу за межі сегмента

### 7.5. Недоліки сегментації

1. **Зовнішня фрагментація** — головний недолік. Оскільки сегменти мають різні розміри, при створенні та видаленні сегментів у пам'яті утворюються «дірки» — вільні ділянки, які занадто малі для нових сегментів

```
  Зовнішня фрагментація при сегментації:

  Початковий стан:
  ┌────────────────┬──────────┬────────────────────┬──────┐
  │  Сегмент A     │ Сегмент B│    Сегмент C       │ Вільно│
  │  (10 КБ)       │ (5 КБ)  │    (15 КБ)         │(2 КБ)│
  └────────────────┴──────────┴────────────────────┴──────┘

  Після видалення B:
  ┌────────────────┬──────────┬────────────────────┬──────┐
  │  Сегмент A     │  ВІЛЬНО  │    Сегмент C       │ Вільно│
  │  (10 КБ)       │ (5 КБ)  │    (15 КБ)         │(2 КБ)│
  └────────────────┴──────────┴────────────────────┴──────┘

  Новий сегмент D (6 КБ) не вміщується в жодну "дірку"!
  Вільної пам'яті 7 КБ (5+2), але не безперервної.
```

2. **Складність керування** — потрібно підтримувати список вільних ділянок різного розміру
3. **Потреба в компактуванні** — для боротьби з фрагментацією потрібно періодично переміщувати сегменти, що дуже повільно

## 8. Сегментно-сторінкова організація

### 8.1. Ідея комбінації

**Сегментно-сторінкова організація** (Segmented Paging) поєднує переваги обох підходів:

- Від **сегментації** — логічний поділ програми на сегменти (код, дані, стек)
- Від **сторінкової організації** — кожен сегмент всередині поділяється на сторінки фіксованого розміру

Це дозволяє зберегти логічну структуру програми та захист на рівні сегментів, водночас **усунувши зовнішню фрагментацію** (оскільки всередині сегментів використовуються сторінки фіксованого розміру).

```
  Сегментно-сторінкова організація:

  Віртуальна адреса:
  ┌──────────────┬─────────────────┬──────────────┐
  │Номер сегмента│ Номер сторінки  │  Зміщення    │
  └──────┬───────┴────────┬────────┴──────────────┘
         │                │
         ▼                │
  Таблиця сегментів       │
  ┌────┬──────────────┐   │
  │ s  │ Вказівник на │   │
  │    │ таблицю      │───┘
  │    │ сторінок     │
  └────┴──────────────┘
                │
                ▼
         Таблиця сторінок
         цього сегмента
         ┌────┬────────┐
         │ p  │ Фрейм f│───> Фізична адреса
         └────┴────────┘     = f * розмір_сторінки + зміщення
```

### 8.2. Як працювала сегментно-сторінкова організація в Intel 80386

Процесори Intel 80386 та наступні x86 підтримували обидва механізми одночасно:

1. **Логічна адреса** (сегмент : зміщення) перетворювалась у **лінійну адресу** за допомогою таблиці дескрипторів сегментів
2. **Лінійна адреса** перетворювалась у **фізичну адресу** за допомогою дворівневої таблиці сторінок

```
  Intel 80386 — повний ланцюжок перетворення:

  Логічна адреса          Лінійна адреса          Фізична адреса
  ┌────────┬────────┐     ┌──────────────┐        ┌──────────────┐
  │Сегмент │Зміщення│────>│  32-бітна    │───────>│   Адреса в   │
  │(селект)│        │     │  адреса      │        │     ОЗП      │
  └────────┴────────┘     └──────────────┘        └──────────────┘
       СЕГМЕНТАЦІЯ             PAGING
    (дескриптори)         (таблиця сторінок)
```

### 8.3. Сучасний стан

У сучасних 64-бітних операційних системах (Windows, Linux, macOS) сегментація фактично **не використовується** як механізм керування пам'яттю:

- Архітектура **x86-64** (64-бітний режим) **примусово вимикає** повноцінну сегментацію: регістри сегментів CS, DS, SS, ES встановлені на базу 0 із максимальним лімітом
- Лише регістри **FS** та **GS** ще використовуються для спеціальних цілей (наприклад, для доступу до структур потоків — Thread Local Storage)
- Усе керування пам'яттю виконується через **багаторівневу сторінкову організацію** (4 або 5 рівнів)
- Причина: сторінкова організація простіша, ефективніша та не створює зовнішньої фрагментації

## 9. Порівняльна таблиця: Paging vs Segmentation

| Критерій | Сторінкова організація (Paging) | Сегментація (Segmentation) |
|----------|--------------------------------|---------------------------|
| **Розмір блоків** | Фіксований (наприклад, 4 КБ) | Змінний (визначається програмою) |
| **Хто визначає розмір** | Апаратура (процесор) | Програміст / компілятор |
| **Видимість для програміста** | Невидима (прозора) | Видима (програміст знає про сегменти) |
| **Зовнішня фрагментація** | Відсутня | Присутня (головний недолік) |
| **Внутрішня фрагментація** | Присутня (в останній сторінці) | Відсутня |
| **Захист пам'яті** | На рівні окремих сторінок | На рівні логічних сегментів |
| **Спільне використання** | Можливе, але на рівні сторінок | Зручніше — на рівні логічних одиниць |
| **Використання в сучасних ОС** | Основний механізм (Windows, Linux, macOS) | Практично не використовується (спадщина) |

## Глосарій

| Термін | Визначення |
|--------|-----------|
| **Віртуальна (логічна) адреса** | Адреса, з якою працює програма. Не відповідає реальному розташуванню даних у фізичній пам'яті |
| **Фізична адреса** | Реальна адреса комірки в оперативній пам'яті (ОЗП) |
| **MMU (Memory Management Unit)** | Апаратний блок процесора, що виконує перетворення віртуальних адрес у фізичні |
| **Сторінка (Page)** | Блок фіксованого розміру у віртуальному адресному просторі процесу (наприклад, 4 КБ) |
| **Фрейм (Frame / Кадр)** | Блок фіксованого розміру у фізичній оперативній пам'яті, рівний за розміром сторінці |
| **Таблиця сторінок (Page Table)** | Структура даних, що зберігає відповідність між номерами сторінок та номерами фреймів для кожного процесу |
| **PTE (Page Table Entry)** | Один запис у таблиці сторінок, що містить номер фрейму та службові біти |
| **Біт присутності (Present bit)** | Прапорець у PTE: 1 — сторінка в ОЗП, 0 — сторінка на диску |
| **Біт модифікації (Dirty bit)** | Прапорець у PTE: 1 — вміст сторінки було змінено, 0 — не змінено |
| **Біт захисту (Protection bits)** | Прапорці у PTE, що визначають дозволені операції: читання (R), запис (W), виконання (X) |
| **Page Fault (сторінкове переривання)** | Переривання, що виникає, коли процесор звертається до сторінки, якої немає в ОЗП (біт присутності = 0) |
| **TLB (Translation Lookaside Buffer)** | Апаратний кеш у процесорі, що зберігає нещодавні відповідності «сторінка → фрейм» для прискорення перетворення адрес |
| **TLB Hit** | Ситуація, коли потрібна відповідність знайдена в TLB (швидко) |
| **TLB Miss** | Ситуація, коли потрібної відповідності немає в TLB і потрібне звернення до таблиці сторінок (повільно) |
| **Багаторівнева таблиця сторінок** | Ієрархічна організація таблиці сторінок (2, 3, 4 рівні), що дозволяє не зберігати записи для невикористаних ділянок адресного простору |
| **Сегмент (Segment)** | Логічна частина програми змінного розміру (код, дані, стек, купа) |
| **Таблиця сегментів (Segment Table)** | Структура даних, що зберігає базову адресу та ліміт кожного сегмента |
| **Базова адреса (Base)** | Початкова фізична адреса сегмента в оперативній пам'яті |
| **Ліміт (Limit)** | Максимальний розмір сегмента в байтах; використовується для перевірки коректності зміщення |
| **Зовнішня фрагментація** | Стан пам'яті, коли вільний простір розбитий на маленькі ділянки між сегментами, недостатні для нового сегмента |
| **Внутрішня фрагментація** | Втрата простору всередині виділеного блоку (сторінки), коли дані не заповнюють його повністю |
| **Сегментно-сторінкова організація** | Комбінований підхід: програма поділяється на сегменти, а кожен сегмент — на сторінки фіксованого розміру |
| **Принцип локальності** | Властивість програм звертатися до невеликої частини адресного простору в кожен момент часу (часова та просторова локальність) |
| **Page Walk** | Процес послідовного пошуку запису в багаторівневій таблиці сторінок при TLB Miss |
