## 1. Вступ

Операційна система (ОС) — це не одна монолітна програма, а складний комплекс взаємопов'язаних компонентів, кожен з яких відповідає за свою частину роботи. Щоб зрозуміти, як працює комп'ютер, необхідно знати, з чого складається ОС, яку роль виконує кожна частина та як ці частини взаємодіють між собою.

Центральним і найважливішим компонентом будь-якої операційної системи є **ядро** (kernel). Саме воно забезпечує роботу всіх інших компонентів та є "серцем" операційної системи.

У цьому уроці ми детально розглянемо:
- з яких основних компонентів складається ОС;
- що таке ядро, які його функції та які типи ядер існують;
- як працюють системні виклики;
- як відбувається процес завантаження комп'ютера.

## 2. Структура операційної системи — основні компоненти

Операційну систему можна уявити як багатошарову конструкцію, де кожен рівень виконує свої функції та взаємодіє з іншими рівнями.

**Текстова схема загальної структури ОС:**

```
┌─────────────────────────────────────────────────┐
│              ПРИКЛАДНІ ПРОГРАМИ                  │
│     (браузер, текстовий редактор, ігри)          │
├─────────────────────────────────────────────────┤
│           СИСТЕМНІ УТИЛІТИ                       │
│   (диспетчер задач, файловий менеджер)           │
├─────────────────────────────────────────────────┤
│      БІБЛІОТЕКИ / API / СИСТЕМНІ ВИКЛИКИ         │
├─────────────────────────────────────────────────┤
│              ОБОЛОНКА (Shell)                    │
│          (CLI або GUI інтерфейс)                 │
├─────────────────────────────────────────────────┤
│               ЯДРО (Kernel)                      │
│  ┌───────────┬──────────────┬──────────────┐    │
│  │ Керування │  Керування   │  Керування   │    │
│  │ процесами │   пам'яттю   │  пристроями  │    │
│  ├───────────┼──────────────┼──────────────┤    │
│  │ Файлова   │   Драйвери   │  Планування  │    │
│  │ система   │   пристроїв  │     CPU       │    │
│  └───────────┴──────────────┴──────────────┘    │
├─────────────────────────────────────────────────┤
│           ЗАВАНТАЖУВАЧ (Bootloader)               │
├─────────────────────────────────────────────────┤
│         АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ (Hardware)          │
│     (CPU, RAM, HDD/SSD, відеокарта, мережа)      │
└─────────────────────────────────────────────────┘
```

Розглянемо кожен компонент детальніше.

### 2.1. Ядро (Kernel)

**Ядро** — це центральний і найважливіший компонент операційної системи. Це програма, яка постійно знаходиться в оперативній пам'яті комп'ютера та має повний контроль над усіма ресурсами системи: процесором, пам'яттю, пристроями вводу-виводу.

Ядро виконує такі ключові функції:
- **Керування процесами** — створення, планування та завершення процесів.
- **Керування пам'яттю** — розподіл оперативної пам'яті між програмами, реалізація віртуальної пам'яті.
- **Керування пристроями** — взаємодія з апаратним забезпеченням через драйвери.
- **Керування файлами** — організація зберігання та доступу до даних на дисках.

Ядро працює в привілейованому режимі (режимі ядра), що дозволяє йому виконувати будь-які інструкції процесора та мати доступ до всієї пам'яті.

> Детальний розгляд ядра — у розділі 3.

---

### 2.2. Оболонка (Shell)

**Оболонка** — це програма, яка забезпечує інтерфейс між користувачем та ядром операційної системи. Саме через оболонку ми "спілкуємося" з комп'ютером.

Оболонка приймає команди від користувача, перетворює їх у інструкції, зрозумілі ядру, і передає ядру для виконання. Після виконання оболонка повертає результат користувачу.

Існує два основних типи оболонок:

| Тип оболонки | Опис | Приклади |
|---|---|---|
| **CLI** (Command Line Interface) — командний рядок | Текстовий інтерфейс, де користувач вводить команди з клавіатури | Bash, PowerShell, cmd.exe, Zsh |
| **GUI** (Graphical User Interface) — графічний інтерфейс | Візуальний інтерфейс з вікнами, іконками, кнопками | Робочий стіл Windows, GNOME, KDE |

**Приклад роботи CLI:**
```
$ ls -la /home/student
drwxr-xr-x  5 student student 4096 бер  9 10:00 .
-rw-r--r--  1 student student  220 бер  9 09:30 .bashrc
drwxr-xr-x  2 student student 4096 бер  9 09:45 Documents
```

У цьому прикладі користувач ввів команду `ls -la /home/student`, оболонка Bash передала цю команду ядру, ядро виконало операцію читання каталогу і повернуло результат через оболонку на екран.

---

### 2.3. Файлова система

**Файлова система** — це компонент ОС, який відповідає за організацію зберігання, іменування та доступу до файлів і каталогів на дисках та інших носіях.

Файлова система забезпечує:
- **Структуровану організацію даних** — файли зберігаються в каталогах (папках), утворюючи деревоподібну ієрархію.
- **Іменування** — кожен файл має ім'я, за яким до нього можна звернутися.
- **Права доступу** — визначає, хто може читати, записувати або виконувати файли.
- **Журналювання** — запис змін перед їх виконанням для відновлення після збоїв (у сучасних ФС).

**Приклади файлових систем:**

| Файлова система | ОС | Особливості |
|---|---|---|
| **NTFS** | Windows | Підтримка великих файлів, журналювання, права доступу |
| **ext4** | Linux | Стандартна ФС для Linux, журналювання, швидкість |
| **APFS** | macOS | Оптимізована для SSD, шифрування |
| **FAT32** | Кросплатформна | Проста, підтримується майже всіма ОС, обмеження файлу до 4 ГБ |

---

### 2.4. Драйвери пристроїв

**Драйвер** — це спеціалізована програма, яка забезпечує взаємодію операційної системи з конкретним апаратним пристроєм (відеокартою, принтером, звуковою картою, мережевим адаптером тощо).

Кожен пристрій має свої особливості роботи, і драйвер "перекладає" стандартні команди ОС на мову конкретного пристрою.

**Як працюють драйвери:**

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Прикладна   │     │     Ядро     │     │   Апаратний  │
│  програма    │────>│   (через     │────>│   пристрій   │
│              │     │   драйвер)   │     │              │
└──────────────┘     └──────────────┘     └──────────────┘
```

**Приклад:** Коли ви натискаєте кнопку "Друк" у текстовому редакторі, програма надсилає запит ядру. Ядро використовує драйвер принтера, щоб перетворити документ у формат, зрозумілий саме вашій моделі принтера, і відправити дані на друк.

У сучасних ОС драйвери можуть бути:
- **Вбудовані** — входять до складу ОС (наприклад, драйвери стандартних USB-пристроїв).
- **Завантажувані модулі** (в Linux — LKM, Loadable Kernel Modules) — підключаються під час роботи системи без перезавантаження.
- **Від виробника** — встановлюються окремо (наприклад, драйвери NVIDIA для відеокарт).

### 2.5. Системні утиліти

**Системні утиліти** — це програми, що входять до складу ОС і призначені для обслуговування, налаштування та діагностики системи.

**Приклади системних утиліт:**

| Категорія | Windows | Linux |
|---|---|---|
| Менеджер файлів | Провідник (Explorer) | Nautilus, Dolphin, Thunar |
| Диспетчер задач | Диспетчер задач (Task Manager) | top, htop, System Monitor |
| Управління дисками | Керування дисками (Disk Management) | fdisk, gparted |
| Мережеві утиліти | ipconfig, ping | ifconfig, ip, ping |
| Архівація | Вбудований архіватор | tar, gzip |
| Редактор реєстру / конфігурації | regedit | текстові конфігураційні файли |

Системні утиліти є важливою частиною ОС, оскільки вони допомагають адміністраторам та користувачам підтримувати систему в робочому стані.

---

### 2.6. Бібліотеки, API та системні виклики

**Бібліотеки** — це набори готових функцій (підпрограм), які програмісти використовують при створенні програм, щоб не писати все "з нуля".

**API** (Application Programming Interface — інтерфейс програмування застосунків) — це набір правил та інструментів, через які програми взаємодіють з операційною системою та іншими програмами.

**Системні виклики** (System Calls) — це спеціальний механізм, через який програми звертаються до ядра ОС для виконання привілейованих операцій (робота з файлами, мережею, пам'яттю тощо).

**Схема взаємодії:**

```
┌─────────────────┐
│  Програма       │  Використовує функцію бібліотеки
│  (наприклад,    │  наприклад: fopen("file.txt")
│  текстовий      │
│  редактор)      │
└────────┬────────┘
         │
         v
┌─────────────────┐
│  Бібліотека     │  Перетворює виклик у системний виклик
│  (наприклад,    │  наприклад: open("file.txt", O_RDONLY)
│  libc / glibc)  │
└────────┬────────┘
         │
         v
┌─────────────────┐
│  Ядро ОС        │  Виконує операцію з апаратним
│  (системний     │  забезпеченням (читає файл з диску)
│  виклик)        │
└─────────────────┘
```

### 2.7. Завантажувач (Bootloader)

**Завантажувач** — це невелика програма, яка запускається першою після увімкнення комп'ютера (точніше — після ініціалізації BIOS/UEFI) і відповідає за завантаження ядра операційної системи в оперативну пам'ять.

Завантажувач виконує три основні задачі:
1. **Знайти** ядро ОС на диску.
2. **Завантажити** ядро в оперативну пам'ять.
3. **Передати** керування ядру для подальшої ініціалізації.

**Приклади завантажувачів:**

| Завантажувач | Опис |
|---|---|
| **GRUB** (GRand Unified Bootloader) | Найпоширеніший завантажувач для Linux. Підтримує мультизавантаження (вибір з декількох ОС). |
| **Windows Boot Manager** | Завантажувач для Windows. Автоматично завантажує ядро Windows. |
| **systemd-boot** | Легкий завантажувач для систем з UEFI. |
| **rEFInd** | Завантажувач для UEFI з графічним інтерфейсом, популярний для Mac та мультисистем. |

> Детальніше про процес завантаження — у розділі 5.

## 3. Ядро ОС — детальний розгляд

### 3.1. Що таке ядро та його роль

**Ядро** (англ. **kernel**) — це центральна програма операційної системи, яка постійно знаходиться в оперативній пам'яті і має повний контроль над усіма ресурсами комп'ютера.

Ядро можна порівняти з **диспетчером** великого аеропорту: воно координує роботу всіх частин системи, розподіляє ресурси, запобігає конфліктам та забезпечує безперебійну роботу.

**Роль ядра в системі:**
- Це єдина частина ОС, яка має прямий доступ до апаратного забезпечення.
- Усі інші програми (від браузера до ігор) взаємодіють з апаратурою виключно через ядро.
- Ядро ізолює програми одна від одної, запобігаючи конфліктам.
- Якщо ядро припиняє роботу — зупиняється вся система (так званий "kernel panic" у Linux або "синій екран смерті" BSOD у Windows).

---

### 3.2. Основні функції ядра

| Функція | Що робить | Приклад |
|---|---|---|
| **Керування процесами** | Створює, планує, призупиняє та завершує процеси. Розподіляє процесорний час. | Коли ви одночасно слухаєте музику і переглядаєте сайт, ядро перемикає процесор між цими задачами тисячі разів на секунду. |
| **Керування пам'яттю** | Розподіляє оперативну пам'ять між програмами. Реалізує віртуальну пам'ять. | Кожна програма "думає", що має всю пам'ять для себе, хоча насправді ядро розподіляє її між усіма. |
| **Керування пристроями** | Взаємодіє з апаратними пристроями через драйвери. | Коли ви підключаєте USB-флешку, ядро знаходить відповідний драйвер і робить пристрій доступним. |
| **Керування файловою системою** | Організує зберігання та доступ до файлів і каталогів. | Коли ви зберігаєте документ, ядро знаходить вільне місце на диску та записує дані. |
| **Обробка переривань** | Реагує на сигнали від апаратних пристроїв (натискання клавіші, прихід мережевого пакета). | Натискання клавіші на клавіатурі генерує переривання, ядро обробляє його та передає символ активній програмі. |
| **Забезпечення безпеки** | Контролює права доступу, ізолює процеси. | Звичайний користувач не може видалити системні файли, бо ядро перевіряє права доступу. |

---

### 3.3. Режим ядра vs режим користувача

Сучасні процесори підтримують щонайменше два режими виконання програм: **режим ядра** (kernel mode) та **режим користувача** (user mode). Це фундаментальний механізм захисту, без якого будь-яка програма могла б зруйнувати всю систему.

#### Режим ядра (Kernel Mode)

- Код виконується з **найвищим рівнем привілеїв**.
- Повний доступ до всієї оперативної пам'яті та всіх інструкцій процесора.
- Прямий доступ до апаратного забезпечення.
- Помилка в коді режиму ядра призводить до **краху всієї системи** (kernel panic / BSOD).
- У цьому режимі працюють: ядро ОС, драйвери пристроїв.

#### Режим користувача (User Mode)

- Код виконується з **обмеженими привілеями**.
- Кожна програма отримує свій ізольований простір пам'яті.
- Немає прямого доступу до апаратури — тільки через системні виклики до ядра.
- Помилка в одній програмі **НЕ призводить** до краху системи — падає тільки ця програма.
- У цьому режимі працюють: браузери, текстові редактори, ігри, медіаплеєри.

**Порівняльна таблиця:**

| Характеристика | Режим ядра (Kernel Mode) | Режим користувача (User Mode) |
|---|---|---|
| Рівень привілеїв | Найвищий (Ring 0) | Найнижчий (Ring 3) |
| Доступ до апаратури | Прямий, необмежений | Тільки через системні виклики |
| Доступ до пам'яті | Вся пам'ять | Тільки власний простір програми |
| Наслідки збою | Крах усієї системи | Падає лише ця програма |
| Що працює | Ядро, драйвери | Програми користувача |

**Текстова схема перемикання режимів:**

```
┌─────────────────────────────────────────────┐
│          РЕЖИМ КОРИСТУВАЧА (User Mode)       │
│                                              │
│  Програма хоче прочитати файл                │
│  --> Виклик функції read()                   │
│  --> Генерується переривання (trap)          │
│                                              │
│         │  СИСТЕМНИЙ ВИКЛИК  │                │
│         v                    v                │
├─────────────────────────────────────────────┤
│          РЕЖИМ ЯДРА (Kernel Mode)            │
│                                              │
│  Ядро перевіряє права доступу                │
│  --> Зчитує дані з диску через драйвер       │
│  --> Копіює дані в пам'ять програми           │
│  --> Повертає керування програмі              │
│                                              │
│         │  ПОВЕРНЕННЯ  │                      │
│         v              v                      │
├─────────────────────────────────────────────┤
│          РЕЖИМ КОРИСТУВАЧА (User Mode)       │
│                                              │
│  Програма продовжує роботу з отриманими     │
│  даними                                      │
└─────────────────────────────────────────────┘
```

### 3.4. Кільця захисту процесора

Сучасні процесори архітектури x86 мають чотири рівні привілеїв, які називаються **кільцями захисту** (protection rings):

```
        ┌───────────────────────────────────┐
        │        Ring 3 (User Mode)         │
        │    Прикладні програми             │
        │  ┌───────────────────────────┐    │
        │  │    Ring 2 (рідко          │    │
        │  │    використовується)      │    │
        │  │  ┌───────────────────┐    │    │
        │  │  │   Ring 1 (рідко   │    │    │
        │  │  │   використ.)      │    │    │
        │  │  │  ┌─────────────┐  │    │    │
        │  │  │  │   Ring 0    │  │    │    │
        │  │  │  │  (Kernel    │  │    │    │
        │  │  │  │   Mode)     │  │    │    │
        │  │  │  └─────────────┘  │    │    │
        │  │  └───────────────────┘    │    │
        │  └───────────────────────────┘    │
        └───────────────────────────────────┘
```

- **Ring 0** — найвищий рівень привілеїв. Тут працює ядро ОС.
- **Ring 1 та Ring 2** — проміжні рівні. У більшості сучасних ОС не використовуються.
- **Ring 3** — найнижчий рівень привілеїв. Тут працюють програми користувача.

Така архітектура забезпечує захист: програма з Ring 3 не може напряму виконувати інструкції Ring 0 та не має доступу до пам'яті ядра.

---

### 3.5. Типи ядер операційних систем

Існує кілька основних підходів до архітектури ядра. Кожен має свої переваги та недоліки.

#### 3.5.1. Монолітне ядро (Monolithic Kernel)

**Монолітне ядро** — це архітектура, в якій все ядро працює як одна велика програма в одному адресному просторі в режимі ядра.

**Що входить до монолітного ядра:**
- Керування процесами та пам'яттю
- Файлові системи
- Драйвери пристроїв
- Мережевий стек
- Безпека та контроль доступу

**Переваги:**
- Висока продуктивність — компоненти спілкуються напряму, без додаткових витрат.
- Швидкі системні виклики — немає потреби в міжпроцесній комунікації.

**Недоліки:**
- Помилка в будь-якому компоненті (наприклад, у драйвері) може привести до краху всієї системи.
- Складність підтримки та налагодження через великий обсяг коду.
- Менша модульність.

**Важливе уточнення:** Сучасні монолітні ядра (такі як Linux) підтримують **завантажувані модулі ядра** (Loadable Kernel Modules, LKM). Це дозволяє динамічно підключати та відключати драйвери та інші компоненти без перезавантаження системи, що частково компенсує недоліки монолітної архітектури.

**Приклади ОС:** Linux, FreeBSD, NetBSD, Solaris

```
┌─────────────────────────────────────────┐
│          МОНОЛІТНЕ ЯДРО                  │
│  (все працює в режимі ядра, Ring 0)      │
│                                          │
│  ┌────────┐ ┌──────────┐ ┌────────────┐ │
│  │Процеси │ │ Пам'ять  │ │ Файлова    │ │
│  │        │ │          │ │ система    │ │
│  ├────────┤ ├──────────┤ ├────────────┤ │
│  │Драйвери│ │ Мережа   │ │ Безпека    │ │
│  └────────┘ └──────────┘ └────────────┘ │
├─────────────────────────────────────────┤
│           АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ           │
└─────────────────────────────────────────┘
```

---

#### 3.5.2. Мікроядро (Microkernel)

**Мікроядро** — це архітектура, в якій в режимі ядра працює лише мінімальний набір функцій, а решта сервісів працює в режимі користувача як окремі процеси.

**Що входить до мікроядра:**
- Базове керування процесами
- Міжпроцесна комунікація (IPC — Inter-Process Communication)
- Базове керування пам'яттю
- Планування процесорного часу

**Все інше** (драйвери, файлові системи, мережевий стек) працює як **окремі сервіси** в режимі користувача.

**Переваги:**
- Висока надійність — збій драйвера або сервісу не призводить до краху всієї системи; сервіс можна перезапустити.
- Краща безпека — менше коду працює з повними привілеями.
- Легше модифікувати та тестувати окремі компоненти.

**Недоліки:**
- Нижча продуктивність через необхідність міжпроцесної комунікації (IPC) між компонентами.
- Складніша реалізація.

**Приклади ОС:** QNX (використовується в автомобілях, медичних приладах), MINIX (навчальна ОС), L4

```
┌──────────────────────────────────────────────────┐
│            РЕЖИМ КОРИСТУВАЧА (User Mode)           │
│                                                    │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ Драйвер  │ │ Файлова  │ │   Мережевий      │  │
│  │ диску    │ │ система  │ │   сервіс         │  │
│  └────┬─────┘ └────┬─────┘ └────────┬─────────┘  │
│       │  IPC       │  IPC           │  IPC        │
├───────┴────────────┴────────────────┴─────────────┤
│            МІКРОЯДРО (Kernel Mode)                  │
│  ┌─────────┐ ┌──────┐ ┌───────────────────────┐  │
│  │Планувач │ │ IPC  │ │ Базове керування      │  │
│  │  CPU    │ │      │ │ пам'яттю              │  │
│  └─────────┘ └──────┘ └───────────────────────┘  │
├────────────────────────────────────────────────────┤
│             АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                    │
└────────────────────────────────────────────────────┘
```

---

#### 3.5.3. Гібридне ядро (Hybrid Kernel)

**Гібридне ядро** — це архітектура, яка поєднує підходи монолітного та мікроядра. Структурно воно нагадує мікроядро, але деякі сервіси (наприклад, драйвери та файлові системи) працюють у режимі ядра для підвищення продуктивності.

**Ідея:** Взяти модульність мікроядра та продуктивність монолітного ядра.

**Переваги:**
- Баланс між продуктивністю та модульністю.
- Краща продуктивність, ніж у мікроядра (менше IPC).
- Краща стабільність, ніж у монолітного ядра (часткова ізоляція).

**Недоліки:**
- Складність реалізації.
- Не є "чистою" архітектурою — критики вважають гібридне ядро варіацією монолітного.

**Цікавий факт:** Лінус Торвальдс (творець Linux) назвав термін "гібридне ядро" маркетинговим ходом, оскільки вважає, що за суттю воно мало відрізняється від монолітного.

**Приклади ОС:** Windows NT (і всі сучасні Windows: 10, 11), macOS / iOS (ядро XNU), ReactOS

```
┌──────────────────────────────────────────────────┐
│            РЕЖИМ КОРИСТУВАЧА (User Mode)           │
│                                                    │
│  ┌──────────────────────────────────────────────┐ │
│  │  Деякі сервіси та підсистеми                 │ │
│  └────────────────────┬─────────────────────────┘ │
├───────────────────────┴───────────────────────────┤
│            ГІБРИДНЕ ЯДРО (Kernel Mode)              │
│                                                    │
│  ┌─────────┐ ┌──────────┐ ┌──────────────────┐   │
│  │Керування│ │ Драйвери │ │ Файлова система  │   │
│  │процесами│ │          │ │                  │   │
│  ├─────────┤ ├──────────┤ ├──────────────────┤   │
│  │ Пам'ять │ │ Мережа   │ │ IPC              │   │
│  └─────────┘ └──────────┘ └──────────────────┘   │
├────────────────────────────────────────────────────┤
│             АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                    │
└────────────────────────────────────────────────────┘
```

---

#### 3.5.4. Екзоядро (Exokernel)

**Екзоядро** — це експериментальна архітектура, в якій ядро виконує лише мінімальні функції: розподіл фізичних ресурсів (пам'яті, процесорного часу, дискових блоків) між програмами та забезпечення захисту. Усі абстракції (файлові системи, мережеві протоколи тощо) реалізуються самими програмами через спеціальні бібліотеки.

**Переваги:**
- Максимальна гнучкість — кожна програма може оптимізувати роботу з ресурсами під свої потреби.
- Потенційно найвища продуктивність для спеціалізованих задач.

**Недоліки:**
- Надзвичайна складність розробки програм.
- Потребує від програмістів глибоких знань апаратури.
- Досі залишається дослідницьким напрямком, не використовується у комерційних ОС.

**Приклади:** MIT Exokernel (Aegis, XOK) — лише дослідницькі проєкти.

---

### 3.6. Порівняльна таблиця типів ядер

| Характеристика | Монолітне | Мікроядро | Гібридне | Екзоядро |
|---|---|---|---|---|
| **Розмір ядра** | Велике | Мале | Середнє | Мінімальне |
| **Продуктивність** | Висока | Нижча (IPC) | Висока | Потенційно найвища |
| **Надійність** | Нижча (збій = крах) | Висока (ізоляція) | Середня | Залежить від програми |
| **Безпека** | Нижча (більше коду в Ring 0) | Вища (менше коду в Ring 0) | Середня | Залежить від програми |
| **Модульність** | Низька (але LKM допомагають) | Висока | Середня | Максимальна |
| **Складність розробки** | Середня | Висока | Висока | Дуже висока |
| **Де працюють драйвери** | У режимі ядра | У режимі користувача | Частково в ядрі | Через бібліотеки |
| **Приклади ОС** | Linux, FreeBSD | QNX, MINIX, L4 | Windows NT, macOS | MIT Exokernel |

### 3.7. Приклади ОС з різними типами ядер

#### Linux — монолітне ядро

Linux використовує монолітне ядро, створене Лінусом Торвальдсом у 1991 році. Усі основні компоненти (файлові системи, драйвери, мережевий стек) працюють в режимі ядра. Однак Linux підтримує **завантажувані модулі ядра (LKM)** — фрагменти коду, які можна динамічно завантажувати та вивантажувати з ядра під час роботи системи. Це дозволяє:
- Додавати підтримку нового обладнання без перекомпіляції ядра.
- Завантажувати драйвер тільки тоді, коли він потрібен, економлячи пам'ять.
- Управляти модулями командами `modprobe`, `lsmod`, `rmmod`.

Модулі ядра зберігаються в каталозі `/lib/modules/` і мають розширення `.ko` (kernel object).

#### QNX — мікроядро

QNX — це комерційна операційна система реального часу, розроблена компанією BlackBerry. Її мікроядро містить лише планувальник CPU, механізм міжпроцесної комунікації (IPC), обробку переривань та таймери. Все інше (навіть менеджер процесів та пам'яті) працює як окремі сервіси в режимі користувача.

QNX використовується в критично важливих системах: автомобілях (понад 195 мільйонів авто станом на 2022 рік), медичних приладах, промислових контролерах, залізничному транспорті.

#### MINIX — мікроядро (навчальна ОС)

MINIX була створена Ендрю Таненбаумом у 1987 році як навчальна Unix-подібна ОС. Вона демонструє принципи мікроядерної архітектури: менеджер пам'яті та файлова система працюють як окремі процеси в режимі користувача. MINIX 3 орієнтована на високу надійність і може автоматично перезапускати збійні драйвери.

#### Windows NT / macOS — гібридне ядро

**Windows NT** (основа Windows 10, 11) використовує гібридне ядро, натхненне мікроядром Mach. Драйвери та файлові системи працюють у режимі ядра для продуктивності, але архітектура зберігає модульність.

**macOS** використовує ядро **XNU** (X is Not Unix) — гібрид на основі мікроядра Mach та компонентів монолітного ядра FreeBSD. Це дозволяє поєднувати модульність з продуктивністю.

## 4. Системні виклики (System Calls)

### 4.1. Що таке системний виклик

**Системний виклик** (англ. system call, syscall) — це механізм, через який програма, що працює в режимі користувача, звертається до ядра ОС для виконання операцій, які потребують привілейованого доступу.

**Чому програми не можуть робити все самі?**

Програми працюють у режимі користувача з обмеженими правами. Вони не можуть самостійно:
- Читати або записувати файли на диск.
- Відправляти дані в мережу.
- Виділяти нові блоки пам'яті.
- Взаємодіяти з апаратними пристроями.

Для цього вони повинні **попросити ядро** зробити це за них — через системний виклик.

**Аналогія:** Уявіть банк. Ви (програма) не можете самі зайти у сховище (апаратне забезпечення). Ви підходите до каси (системний виклик) і просите касира (ядро) виконати операцію за вас.

---

### 4.2. Як працюють системні виклики

1. Програма викликає функцію бібліотеки (наприклад, `read()` у мові C).
2. Бібліотека формує системний виклик та генерує **програмне переривання** (trap/interrupt).
3. Процесор перемикається з **режиму користувача** в **режим ядра**.
4. Ядро перевіряє параметри виклику та права доступу.
5. Ядро виконує запитану операцію.
6. Результат повертається програмі.
7. Процесор перемикається назад у **режим користувача**.

---

### 4.3. Типи системних викликів

| Категорія | Призначення | Приклади (Linux/UNIX) |
|---|---|---|
| **Керування процесами** | Створення, завершення, очікування процесів | `fork()` — створити новий процес, `exec()` — запустити програму, `exit()` — завершити процес, `wait()` — дочекатися завершення дочірнього процесу |
| **Робота з файлами** | Відкриття, читання, запис, закриття файлів | `open()` — відкрити файл, `read()` — прочитати дані, `write()` — записати дані, `close()` — закрити файл |
| **Керування пристроями** | Доступ до апаратних пристроїв | `ioctl()` — управління пристроєм, `read()`/`write()` — читання/запис з пристрою |
| **Інформаційні** | Отримання інформації про систему | `getpid()` — отримати ID процесу, `time()` — отримати системний час |
| **Комунікаційні** | Обмін даними між процесами | `pipe()` — створити канал, `socket()` — створити мережеве з'єднання, `send()`/`recv()` — відправити/отримати дані |

**Практичний приклад:** Коли ви зберігаєте документ у текстовому редакторі, відбувається така послідовність системних викликів:
1. `open("document.txt", ...)` — відкрити (або створити) файл
2. `write(fd, data, size)` — записати вміст документа
3. `close(fd)` — закрити файл

Сучасні ОС мають сотні системних викликів. Наприклад, Linux має понад 300, Windows — близько 2000.

## 5. Процес завантаження операційної системи

### 5.1. Загальна схема завантаження

Коли ви натискаєте кнопку живлення комп'ютера, відбувається складний процес, який називається **завантаженням** (boot process). Ось загальна схема:

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  1. Живлення │     │  2. POST     │     │  3. BIOS /   │
│  увімкнено   │────>│  (самотест   │────>│     UEFI     │
│              │     │  апаратури)  │     │              │
└──────────────┘     └──────────────┘     └──────┬───────┘
                                                  │
                                                  v
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  6. Робочий  │     │  5. Init /   │     │ 4. Boot-     │
│  стіл або    │<────│  systemd     │<────│    loader    │
│  командний   │     │              │     │   (GRUB)     │
│  рядок       │     │              │     │    -> Kernel  │
└──────────────┘     └──────────────┘     └──────────────┘
```

---

### 5.2. Етап 1: Увімкнення живлення та POST

Після натискання кнопки живлення:

1. **Блок живлення** подає електричний струм на материнську плату.
2. Процесор починає виконувати код, записаний у мікросхемі **BIOS** або **UEFI** на материнській платі.
3. Запускається **POST** (Power-On Self-Test) — самотестування:
   - Перевіряється наявність і працездатність оперативної пам'яті (RAM).
   - Перевіряються диски (HDD/SSD).
   - Перевіряються пристрої вводу (клавіатура, миша).
   - Перевіряються інші компоненти.

Якщо POST виявляє критичну помилку (наприклад, відсутність оперативної пам'яті), комп'ютер подає **звукові сигнали** (біпи) і зупиняється. Якщо все гаразд — процес продовжується.

---

### 5.3. Етап 2: BIOS / UEFI

**BIOS** (Basic Input/Output System) та **UEFI** (Unified Extensible Firmware Interface) — це програми (прошивки), вбудовані в мікросхему на материнській платі. Вони є першим програмним забезпеченням, яке запускається при увімкненні комп'ютера.

| Характеристика | BIOS (застарілий) | UEFI (сучасний) |
|---|---|---|
| Розроблений | 1981 рік (IBM PC) | 2005+ рік |
| Інтерфейс | Текстовий | Графічний, з підтримкою миші |
| Таблиця розділів | MBR (до 2 ТБ) | GPT (до 9.4 ЗБ — зеттабайт) |
| Швидкість завантаження | Повільніше | Швидше |
| Безпека | Базова | Secure Boot (захист від шкідливого ПЗ) |
| Розрядність | 16-біт | 32/64-біт |

Після POST прошивка визначає **завантажувальний пристрій** (диск, USB-накопичувач, мережа) і передає керування **завантажувачу**.

**Для BIOS:** Зчитує перші 512 байт диска — **MBR** (Master Boot Record), де знаходиться код першого етапу завантажувача.

**Для UEFI:** Шукає на диску спеціальний розділ **ESP** (EFI System Partition) у форматі FAT32, де знаходиться файл завантажувача (наприклад, `\EFI\BOOT\BOOTX64.EFI`).

---

### 5.4. Етап 3: Завантажувач (Bootloader)

Завантажувач — це невелика програма, головне завдання якої:

1. **Знайти** файл ядра ОС на диску.
2. **Завантажити** ядро в оперативну пам'ять.
3. **Передати** керування ядру.

**GRUB** (GRand Unified Bootloader) — найпоширеніший завантажувач для Linux. Він може:
- Показувати меню вибору ОС (якщо на комп'ютері встановлено кілька систем).
- Передавати ядру параметри запуску.
- Завантажувати початковий RAM-диск (initramfs).

**Windows Boot Manager** — завантажувач для Windows, який автоматично завантажує ядро Windows (ntoskrnl.exe).

**Етапи роботи GRUB (приклад):**
```
Етап 1: Код з MBR (446 байт) --> знаходить GRUB Stage 2
Етап 2: GRUB показує меню вибору ОС
        Користувач вибирає ОС (або автоматично через таймер)
Етап 3: GRUB завантажує ядро + initramfs у пам'ять
        Передає керування ядру
```

---

### 5.5. Етап 4: Завантаження ядра

Після отримання керування від завантажувача **ядро** починає ініціалізацію:

1. **Розпаковує себе** (ядро зазвичай зберігається у стиснутому вигляді для економії місця).
2. **Ініціалізує апаратне забезпечення:** процесор, контролер пам'яті, контролер переривань.
3. **Налаштовує керування пам'яттю:** створює таблиці сторінок, активує віртуальну пам'ять.
4. **Монтує initramfs** — тимчасову файлову систему в оперативній пам'яті, яка містить мінімальний набір драйверів, необхідних для доступу до основного диска.
5. **Завантажує необхідні драйвери** та модулі ядра.
6. **Монтує кореневу файлову систему** з диска.
7. **Запускає перший процес** в режимі користувача — **init** (або **systemd** у сучасних Linux).

---

### 5.6. Етап 5: Ініціалізація системи (Init / Робочий стіл)

**Init** (або **systemd** у сучасних дистрибутивах Linux) — це перший процес, запущений ядром. Він отримує PID (Process ID) номер 1 і є "батьком" усіх інших процесів у системі.

Init/systemd виконує:
1. Перевірку файлових систем на помилки.
2. Монтування додаткових файлових систем.
3. Запуск системних сервісів (мережа, звук, друк).
4. Налаштування мережевих з'єднань.
5. Запуск графічного середовища (робочий стіл) або екрана входу.

**Повна діаграма завантаження ОС:**

```
Кнопка живлення
      │
      v
┌─────────────────────────────┐
│ POST (самотестування)        │
│ Перевірка: RAM, диски, CPU   │
└─────────────┬───────────────┘
              │
              v
┌─────────────────────────────┐
│ BIOS/UEFI                    │
│ Ініціалізація апаратури      │
│ Пошук завантажувального      │
│ пристрою                     │
└─────────────┬───────────────┘
              │
              v
┌─────────────────────────────┐
│ Завантажувач (GRUB /         │
│ Windows Boot Manager)        │
│ Завантаження ядра у RAM      │
└─────────────┬───────────────┘
              │
              v
┌─────────────────────────────┐
│ Ядро (Kernel)                │
│ Ініціалізація апаратури      │
│ Монтування файлових систем   │
│ Запуск init/systemd (PID 1)  │
└─────────────┬───────────────┘
              │
              v
┌─────────────────────────────┐
│ Init / systemd               │
│ Запуск системних сервісів    │
│ Налаштування мережі         │
│ Запуск графічного середовища │
└─────────────┬───────────────┘
              │
              v
┌─────────────────────────────┐
│ Екран входу / Робочий стіл   │
│ Система готова до роботи!    │
└─────────────────────────────┘
```

---

## 6. Підсумок

У цьому уроці ми розглянули:

1. **Структуру ОС** — операційна система складається з ядра, оболонки, файлової системи, драйверів, системних утиліт, бібліотек та завантажувача. Кожен компонент виконує свою функцію.

2. **Ядро (Kernel)** — центральний компонент ОС, який має повний контроль над апаратними ресурсами. Працює в привілейованому режимі (Ring 0).

3. **Режими роботи процесора** — режим ядра (повний доступ) та режим користувача (обмежений доступ) забезпечують безпеку системи.

4. **Типи ядер** — монолітне (Linux), мікроядро (QNX), гібридне (Windows, macOS), екзоядро (дослідницькі проєкти). Кожен тип має свої переваги та компроміси.

5. **Системні виклики** — єдиний спосіб для програм звернутися до ядра для виконання привілейованих операцій.

6. **Процес завантаження** — послідовність етапів від натискання кнопки живлення до появи робочого столу: POST -> BIOS/UEFI -> Bootloader -> Kernel -> Init -> Desktop.

## 8. Глосарій ключових термінів

| Термін | Визначення |
|---|---|
| **Ядро (Kernel)** | Центральна програма ОС, яка постійно перебуває в оперативній пам'яті та має повний контроль над усіма ресурсами комп'ютера. |
| **Оболонка (Shell)** | Програма, яка забезпечує інтерфейс між користувачем та ядром ОС; приймає команди і передає їх ядру для виконання. |
| **Файлова система** | Компонент ОС, який організовує зберігання, іменування та доступ до файлів і каталогів на носіях інформації. |
| **Драйвер пристрою** | Спеціалізована програма, яка забезпечує взаємодію ОС з конкретним апаратним пристроєм. |
| **Системна утиліта** | Програма, що входить до складу ОС і призначена для обслуговування, налаштування та діагностики системи. |
| **API (Application Programming Interface)** | Набір правил та інструментів, через які програми взаємодіють з ОС та іншими програмами. |
| **Системний виклик (System Call)** | Механізм звернення програми до ядра ОС для виконання привілейованих операцій. |
| **Завантажувач (Bootloader)** | Програма, яка завантажує ядро ОС з диска в оперативну пам'ять і передає йому керування. |
| **Режим ядра (Kernel Mode)** | Привілейований режим виконання коду з повним доступом до апаратури та пам'яті (Ring 0). |
| **Режим користувача (User Mode)** | Обмежений режим виконання коду, в якому програми не мають прямого доступу до апаратури (Ring 3). |
| **Монолітне ядро (Monolithic Kernel)** | Архітектура ядра, в якій усі компоненти ОС працюють як одна програма в режимі ядра. Приклад: Linux. |
| **Мікроядро (Microkernel)** | Архітектура ядра, в якій у режимі ядра працює лише мінімальний набір функцій, а решта — в режимі користувача. Приклад: QNX. |
| **Гібридне ядро (Hybrid Kernel)** | Архітектура, що поєднує підходи монолітного ядра та мікроядра. Приклад: Windows NT, macOS. |
| **Екзоядро (Exokernel)** | Експериментальна архітектура з мінімальним ядром, де абстракції реалізуються самими програмами. |
| **IPC (Inter-Process Communication)** | Механізм обміну даними між процесами в операційній системі. |
| **POST (Power-On Self-Test)** | Самотестування апаратури при увімкненні комп'ютера, що виконується BIOS/UEFI. |
| **BIOS (Basic Input/Output System)** | Застаріла прошивка материнської плати, яка ініціалізує апаратуру та запускає завантажувач. |
| **UEFI (Unified Extensible Firmware Interface)** | Сучасна прошивка материнської плати, яка замінила BIOS; підтримує GPT, Secure Boot, графічний інтерфейс. |
| **MBR (Master Boot Record)** | Перші 512 байт диска, що містять код завантажувача та таблицю розділів (для BIOS). |
| **GPT (GUID Partition Table)** | Сучасна схема розмітки диска для UEFI, підтримує диски понад 2 ТБ. |
| **ESP (EFI System Partition)** | Спеціальний розділ диска у форматі FAT32, де зберігаються файли завантажувача для UEFI. |
| **GRUB (GRand Unified Bootloader)** | Найпоширеніший завантажувач для Linux, підтримує мультизавантаження. |
| **LKM (Loadable Kernel Module)** | Завантажуваний модуль ядра Linux — фрагмент коду, який можна динамічно підключати до ядра без перезавантаження. |
| **initramfs** | Тимчасова файлова система в оперативній пам'яті, яка містить мінімальний набір драйверів для початкового завантаження. |
| **Init / systemd** | Перший процес (PID 1), запущений ядром; відповідає за запуск системних сервісів та ініціалізацію середовища. |
| **CLI (Command Line Interface)** | Текстовий інтерфейс командного рядка для взаємодії з ОС. |
| **GUI (Graphical User Interface)** | Графічний інтерфейс з вікнами, іконками та кнопками для взаємодії з ОС. |
| **Kernel Panic** | Критична помилка ядра в Linux/Unix, що призводить до зупинки системи. |
| **BSOD (Blue Screen of Death)** | "Синій екран смерті" — критична помилка ядра в Windows, що призводить до зупинки системи. |
| **Переривання (Interrupt)** | Сигнал від апаратного пристрою або програми, який повідомляє процесор про необхідність негайної обробки події. |
| **Віртуальна пам'ять** | Технологія, яка дозволяє програмам використовувати більше пам'яті, ніж фізично доступно, за рахунок використання дискового простору. |
