## 1. Теоретичний вступ

### 1.1. Нагадування про TCP/IP

**TCP/IP (Transmission Control Protocol / Internet Protocol)** — це набір (стек) протоколів, що є основою роботи Інтернету та більшості локальних мереж. Стек TCP/IP складається з чотирьох рівнів:

| Рівень | Назва | Приклади протоколів |
|--------|-------|---------------------|
| 4 | Прикладний | HTTP, FTP, DNS, SMTP |
| 3 | Транспортний | TCP, UDP |
| 2 | Мережевий (Інтернет) | IP, ICMP, ARP |
| 1 | Канальний (мережевого доступу) | Ethernet, Wi-Fi |

**Ключові поняття:**
- **IP-адреса** — унікальна логічна адреса пристрою в мережі (наприклад, 192.168.1.10)
- **Маска підмережі** — визначає, яка частина IP-адреси належить мережі, а яка — вузлу (наприклад, 255.255.255.0)
- **Шлюз за замовчуванням (Default Gateway)** — IP-адреса маршрутизатора, через який пристрій виходить за межі локальної мережі
- **DNS-сервер** — перетворює доменні імена (google.com) на IP-адреси (142.250.74.206)
- **MAC-адреса** — унікальна фізична адреса мережевого адаптера (наприклад, 00-1A-2B-3C-4D-5E)
- **DHCP** — протокол, що автоматично видає IP-адреси пристроям у мережі

### 1.2. Діагностичні утиліти Windows

Windows має вбудовані консольні утиліти для діагностики мережі. Усі вони запускаються з **командного рядка (cmd)**:

| Утиліта | Призначення |
|---------|-------------|
| `ipconfig` | Показує налаштування мережевих адаптерів (IP, маска, шлюз) |
| `ping` | Перевіряє доступність вузла в мережі (надсилає ICMP-пакети) |
| `tracert` | Відстежує маршрут пакета до вказаного вузла |
| `nslookup` | Запитує DNS-сервер для розпізнавання доменних імен |
| `netstat` | Показує активні мережеві з'єднання, порти, статистику |
| `arp` | Показує та керує таблицею відповідностей IP ↔ MAC |

### 1.3. Протокол ICMP та принцип роботи ping

Команда `ping` використовує протокол **ICMP (Internet Control Message Protocol)**. Вона надсилає **Echo Request** (запит луни) на вказану адресу та очікує **Echo Reply** (відповідь луни). За результатом можна визначити:

- чи доступний віддалений вузол;
- який час відповіді (затримка, latency) у мілісекундах;
- чи є втрата пакетів;
- значення **TTL (Time To Live)** — скільки маршрутизаторів може пройти пакет.

> **Важливо!** Деякі вузли та міжмережеві екрани блокують ICMP-пакети. Якщо ping не отримує відповіді, це не завжди означає, що вузол недоступний — можливо, він просто блокує ping.

## 3. Практичні завдання

### Завдання 1. ipconfig — базова інформація про мережу

Введіть команду:

```
ipconfig
```

Ви побачите для кожного мережевого адаптера:
- **IPv4-адреса** — ваша IP-адреса в локальній мережі
- **Маска підмережі** — визначає межі мережі

**Приклад виводу:**

```

   Основний шлюз . . . . . . . . . : 192.168.1.1
```

**Запишіть** у зошит: вашу IP-адресу, маску підмережі та шлюз.

### Завдання 2. ipconfig /all — повна інформація

Введіть команду:

```
ipconfig /all
```

Ця команда показує **розширену інформацію**:

| Поле | Що означає |
|------|------------|
| Ім'я вузла (Host Name) | Назва вашого комп'ютера в мережі |
| Фізична адреса (Physical Address) | MAC-адреса мережевого адаптера (6 байтів у hex) |
| DHCP увімкнено (DHCP Enabled) | Так — IP отримано автоматично; Ні — задано вручну |
| IPv4-адреса | Ваша IP-адреса |
| Маска підмережі | Маска мережі |
| Основний шлюз | Адреса маршрутизатора |
| DNS-сервери | Адреси серверів, які перетворюють імена на IP |
| DHCP-сервер | Адреса сервера, що видав вам IP |
| Термін оренди (Lease) | Час, на який видано IP-адресу |

**Запишіть** у зошит: MAC-адресу, DNS-сервери, чи увімкнений DHCP.

### Завдання 3. ping 127.0.0.1 — перевірка локального стеку TCP/IP

Введіть команду:

```
ping 127.0.0.1
```

**127.0.0.1** — це адреса **localhost** (петлевий інтерфейс, loopback). Пакет не виходить у мережу, а повертається на сам комп'ютер.

**Навіщо?** Перевірити, чи правильно встановлений і працює стек TCP/IP на вашому комп'ютері. Якщо цей ping не працює — проблема на рівні операційної системи.

**Очікуваний результат:**

```
Відповідь від 127.0.0.1: число байт=32 час<1мс TTL=128
Відповідь від 127.0.0.1: число байт=32 час<1мс TTL=128
Відповідь від 127.0.0.1: число байт=32 час<1мс TTL=128
Відповідь від 127.0.0.1: число байт=32 час<1мс TTL=128

Статистика Ping для 127.0.0.1:
    Пакетів: відправлено = 4, отримано = 4, втрачено = 0 (0% втрат)
```

**Зверніть увагу:** час відповіді < 1 мс (пакет не покидає комп'ютер), TTL = 128.

### Завдання 4. ping IP-адреси сусіднього комп'ютера

Дізнайтеся IP-адресу сусіднього комп'ютера (запитайте у сусіда або подивіться на його екрані результат `ipconfig`). Введіть:

```
ping 192.168.1.XX
```

(де XX — номер сусіднього ПК)

**Мета:** перевірити зв'язок між двома комп'ютерами в **локальній мережі**.

Якщо відповідь є — зв'язок у локальній мережі працює. Якщо відповіді немає — можливі причини:
- Брандмауер Windows блокує ICMP
- Комп'ютери в різних підмережах
- Фізична проблема з кабелем або комутатором

**Запишіть** IP сусіда та результат ping.

### Завдання 7. ping з додатковими параметрами

Спробуйте наступні варіанти команди ping:

#### 7.1. Безперервний ping: параметр `-t`

```
ping -t google.com
```

Пакети надсилаються **безперервно**, поки ви не натиснете **Ctrl + C**. Корисно для моніторингу стабільності з'єднання.

Після зупинки відображається підсумкова статистика.

#### 7.2. Задана кількість пакетів: параметр `-n`

```
ping -n 10 google.com
```

Надсилає **рівно 10** пакетів замість 4 за замовчуванням. Корисно для збору більшої статистики.

#### 7.3. Зміна розміру пакета: параметр `-l`

```
ping -l 1000 google.com
```

Надсилає пакети розміром **1000 байтів** (за замовчуванням — 32 байти). Дозволяє перевірити, як мережа справляється з більшими пакетами.

```
ping -l 65500 google.com
```

Максимальний розмір — 65527 байтів. Великі пакети можуть фрагментуватися або блокуватися.

#### 7.4. Визначення імені хосту за IP: параметр `-a`

```
ping -a 8.8.8.8
```

Команда спробує **визначити доменне ім'я** для вказаної IP-адреси (зворотне розпізнавання DNS).

**Очікуваний результат:**

```
Обмін пакетами з dns.google [8.8.8.8] з 32 байтами даних:
...
```

#### Зведена таблиця параметрів ping

| Параметр | Призначення | Приклад |
|----------|-------------|---------|
| `-t` | Безперервний ping (зупинка — Ctrl+C) | `ping -t google.com` |
| `-n <число>` | Кількість пакетів (за замовчуванням 4) | `ping -n 10 google.com` |
| `-l <розмір>` | Розмір пакета в байтах (за замовчуванням 32) | `ping -l 1000 google.com` |
| `-a` | Визначити ім'я хосту за IP | `ping -a 8.8.8.8` |
| `-i <TTL>` | Встановити значення TTL (макс. 255) | `ping -i 5 google.com` |
| `-f` | Заборонити фрагментацію пакета | `ping -f -l 1472 google.com` |
| `-w <мс>` | Тайм-аут очікування відповіді (мс) | `ping -w 1000 google.com` |

**Запишіть** результати кожного варіанту.

### Завдання 8. tracert — трасування маршруту

Введіть команду:

```
tracert google.com
```

**tracert (Trace Route)** показує маршрут, яким пакет проходить від вашого комп'ютера до вказаного вузла. Кожний рядок — це один **хоп (hop)**, тобто один маршрутизатор на шляху.

**Принцип роботи:** tracert надсилає ICMP-пакети з послідовно зростаючим значенням TTL (1, 2, 3...). Кожний маршрутизатор зменшує TTL на 1. Коли TTL стає 0, маршрутизатор повертає повідомлення "Time Exceeded", розкриваючи свою адресу.

**Приклад виводу:**

```
Трасування маршруту до google.com [142.250.74.206]:

  1     1 мс     1 мс     1 мс  192.168.1.1
  2    10 мс     9 мс    10 мс  10.0.0.1
  3    12 мс    11 мс    12 мс  72.14.215.85
  4    15 мс    14 мс    15 мс  142.250.74.206

Трасування завершено.
```

**Пояснення стовпців:**
- Номер хопу (1, 2, 3...)
- Три виміри часу відповіді (мс) — tracert робить три спроби для кожного хопу
- IP-адреса (або ім'я) маршрутизатора

**Зірочки (`* * *`)** означають, що маршрутизатор не відповів (можливо, блокує ICMP).

**Запишіть** кількість хопів та перший і останній маршрутизатори.

### Завдання 9. nslookup — запит до DNS-сервера

Введіть команду:

```
nslookup google.com
```

**nslookup (Name Server Lookup)** надсилає запит до DNS-сервера для розпізнавання доменного імені.

**Приклад виводу:**

```
Сервер:   dns.local
Address:  192.168.1.1

Не авторитетна відповідь:
Ім'я:     google.com
Addresses: 142.250.74.206
           2a00:1450:4001:812::200e
```

**Пояснення:**
- **Сервер / Address** — DNS-сервер, який обробив ваш запит
- **Не авторитетна відповідь (Non-authoritative answer)** — відповідь отримана з кешу, а не від первинного DNS-сервера домену
- **Ім'я / Addresses** — доменне ім'я та його IP-адреси (IPv4 та IPv6)

Спробуйте також:

```
nslookup ukr.net
nslookup youtube.com
```

**Запишіть** IP-адреси для кожного домену.

### Завдання 10. netstat та arp — з'єднання та ARP-таблиця

#### 10.1. netstat -an — активні з'єднання та порти

Введіть команду:

```
netstat -an
```

**netstat (Network Statistics)** показує всі активні мережеві з'єднання вашого комп'ютера.

**Параметри:**
- `-a` — показати всі з'єднання та порти, що прослуховуються
- `-n` — показувати адреси та порти в числовому вигляді (без розпізнавання імен)

**Приклад виводу:**

```
Активні підключення

  Протокол  Локальна адреса         Зовнішня адреса        Стан
  TCP       0.0.0.0:135             0.0.0.0:0              LISTENING
  TCP       192.168.1.15:49712      142.250.74.206:443     ESTABLISHED
  TCP       192.168.1.15:49720      31.13.72.36:443        TIME_WAIT
  UDP       0.0.0.0:5353            *:*
```

**Стани з'єднань:**

| Стан | Значення |
|------|----------|
| LISTENING | Порт відкритий, очікує вхідних з'єднань |
| ESTABLISHED | Активне з'єднання (дані передаються) |
| TIME_WAIT | З'єднання завершується |
| CLOSE_WAIT | Віддалена сторона закрила з'єднання |

**Відомі порти:**

| Порт | Протокол/Служба |
|------|-----------------|
| 80 | HTTP (веб-сайти) |
| 443 | HTTPS (захищені веб-сайти) |
| 53 | DNS |
| 21 | FTP |
| 25 | SMTP (пошта) |
| 3389 | Віддалений робочий стіл (RDP) |

#### 10.2. arp -a — таблиця ARP

Введіть команду:

```
arp -a
```

**ARP (Address Resolution Protocol)** відповідає за відповідність між IP-адресами та MAC-адресами в локальній мережі.

**Приклад виводу:**

```
Інтерфейс: 192.168.1.15 --- 0x7
  Адреса в Інтернеті    Фізична адреса        Тип
  192.168.1.1           60-35-c0-6b-a2-b7     динамічний
  192.168.1.20          a4-5e-60-c8-12-3f     динамічний
  192.168.1.255         ff-ff-ff-ff-ff-ff     статичний
  224.0.0.22            01-00-5e-00-00-16     статичний
```

**Пояснення:**
- **Динамічний (dynamic)** — запис створений автоматично під час зв'язку з пристроєм
- **Статичний (static)** — запис створений системою (наприклад, широкомовна адреса 255)
- **ff-ff-ff-ff-ff-ff** — широкомовна MAC-адреса (broadcast)

**Запишіть** MAC-адресу шлюзу (маршрутизатора) з ARP-таблиці.

## 4. Аналіз результатів

### 4.1. Що означає TTL (Time To Live)?

TTL — це "час життя" пакета. Кожний маршрутизатор на шляху зменшує TTL на 1. Якщо TTL стає 0 — пакет знищується (захист від зациклювання).

**Як визначити кількість хопів за TTL:**

| Початковий TTL (ОС) | Отриманий TTL | Кількість хопів |
|----------------------|---------------|-----------------|
| 128 (Windows) | 117 | 128 - 117 = 11 |
| 64 (Linux/macOS) | 56 | 64 - 56 = 8 |
| 255 (деякі маршрутизатори) | 247 | 255 - 247 = 8 |

### 4.2. Час відповіді (Round Trip Time)

| Час відповіді | Оцінка |
|---------------|--------|
| < 1 мс | Локальний хост або LAN |
| 1 — 30 мс | Відмінне з'єднання |
| 30 — 100 мс | Добре з'єднання |
| 100 — 300 мс | Задовільне з'єднання |
| > 300 мс | Погане з'єднання |
| Тайм-аут | Вузол недоступний або блокує ICMP |

### 4.3. Втрата пакетів

| Втрата пакетів | Оцінка |
|----------------|--------|
| 0% | Ідеально — з'єднання стабільне |
| 1 — 5% | Незначні проблеми, можливі затримки |
| 5 — 20% | Помітні проблеми, нестабільне з'єднання |
| > 20% | Серйозні проблеми, потрібна діагностика |
| 100% | Вузол недоступний або повністю блокує ICMP |

## 5. Діагностика мережевих проблем: алгоритм крок за кроком

Коли Інтернет не працює, використовуйте наступний алгоритм для локалізації проблеми:

```
Крок 1: ping 127.0.0.1
   ├── НЕ працює → Проблема зі стеком TCP/IP. Переінсталюйте TCP/IP.
   └── Працює → Переходимо до кроку 2

Крок 2: ping <своя IP-адреса>
   ├── НЕ працює → Проблема з мережевим адаптером або його налаштуваннями.
   └── Працює → Переходимо до кроку 3

Крок 3: ping <IP шлюзу (маршрутизатора)>
   ├── НЕ працює → Проблема в локальній мережі (кабель, комутатор, маршрутизатор).
   └── Працює → Переходимо до кроку 4

Крок 4: ping 8.8.8.8 (зовнішній IP)
   ├── НЕ працює → Проблема з підключенням до Інтернету (провайдер, маршрутизатор).
   └── Працює → Переходимо до кроку 5

Крок 5: ping google.com (доменне ім'я)
   ├── НЕ працює → Проблема з DNS. Спробуйте змінити DNS-сервер.
   └── Працює → Інтернет працює! Проблема може бути в конкретному сайті або програмі.
```

**Цей алгоритм дозволяє поетапно звузити пошук проблеми** — від самого комп'ютера до DNS-серверів.

## 6. Зведена таблиця мережевих команд CMD

| Команда | Призначення | Приклад |
|---------|-------------|---------|
| `ipconfig` | Базові мережеві налаштування (IP, маска, шлюз) | `ipconfig` |
| `ipconfig /all` | Повна інформація (MAC, DHCP, DNS, оренда) | `ipconfig /all` |
| `ipconfig /release` | Звільнити поточну IP-адресу (DHCP) | `ipconfig /release` |
| `ipconfig /renew` | Запросити нову IP-адресу від DHCP | `ipconfig /renew` |
| `ipconfig /flushdns` | Очистити кеш DNS | `ipconfig /flushdns` |
| `ipconfig /displaydns` | Показати кешовані записи DNS | `ipconfig /displaydns` |
| `ping <адреса>` | Перевірити доступність вузла | `ping 8.8.8.8` |
| `ping -t <адреса>` | Безперервний ping | `ping -t google.com` |
| `ping -n <к-ть> <адреса>` | Ping із заданою кількістю пакетів | `ping -n 20 google.com` |
| `ping -l <розмір> <адреса>` | Ping із заданим розміром пакета | `ping -l 1000 google.com` |
| `ping -a <IP>` | Визначити ім'я хосту за IP | `ping -a 8.8.8.8` |
| `tracert <адреса>` | Трасування маршруту до вузла | `tracert google.com` |
| `tracert -d <адреса>` | Трасування без розпізнавання DNS | `tracert -d google.com` |
| `nslookup <домен>` | Запит DNS (отримати IP за іменем) | `nslookup google.com` |
| `netstat -an` | Активні з'єднання та порти (числовий формат) | `netstat -an` |
| `netstat -ano` | З'єднання з PID процесів | `netstat -ano` |
| `arp -a` | Показати ARP-таблицю (IP ↔ MAC) | `arp -a` |
| `hostname` | Показати ім'я комп'ютера | `hostname` |
| `pathping <адреса>` | Комбінація ping + tracert (детальна статистика) | `pathping google.com` |
| `route print` | Показати таблицю маршрутизації | `route print` |
| `netsh wlan show profiles` | Показати збережені Wi-Fi мережі | `netsh wlan show profiles` |

## 9. Завдання для самостійної роботи

**Виконайте наступні завдання та запишіть результати:**

1. Виконайте команду `ipconfig /all` і визначте: чи ваш комп'ютер отримує IP-адресу через DHCP, чи вона задана статично? Обґрунтуйте відповідь на основі виводу команди.

2. Виконайте `ping 127.0.0.1` та `ping localhost`. Порівняйте результати. Чи є різниця? Поясніть, чому.

3. Виконайте `ping -n 20 google.com` та `ping -n 20 8.8.8.8`. Порівняйте середній час відповіді. Який ping швидший і чому?

4. Виконайте `ping -l 100 google.com`, потім `ping -l 1000 google.com`, потім `ping -l 10000 google.com`. Як змінюється час відповіді при збільшенні розміру пакета? Поясніть результат.

5. Виконайте `tracert google.com` та `tracert youtube.com`. Порівняйте кількість хопів та час проходження. Які маршрутизатори є спільними на обох маршрутах?

6. За допомогою `nslookup` визначте IP-адреси для доменів: `google.com`, `facebook.com`, `ukr.net`, `gov.ua`. Запишіть результати у таблицю. Чи мають деякі домени кілька IP-адрес? Чому?

7. Виконайте `arp -a` і знайдіть у таблиці запис для вашого шлюзу (маршрутизатора). Яка його MAC-адреса? Порівняйте її з MAC-адресою, яку отримали за допомогою `ipconfig /all`. Чи збігаються вони?

8. Виконайте `netstat -an` і знайдіть усі з'єднання зі станом ESTABLISHED. Які зовнішні IP-адреси ви бачите? Спробуйте визначити за допомогою `nslookup`, яким сайтам/сервісам вони належать.

9. Використовуючи алгоритм діагностики з розділу 5, опишіть покроково, як ви б діагностували ситуацію: "Браузер показує помилку 'Не вдається знайти сервер'. Інші сайти теж не відкриваються." Які команди ви виконаєте і в якому порядку?

10. Виконайте `ping -a 8.8.8.8` та `ping -a 1.1.1.1`. Які доменні імена відповідають цим IP-адресам? Яким компаніям належать ці DNS-сервери?

11. Виконайте `ipconfig /displaydns` і перегляньте кеш DNS. Знайдіть 5 записів та запишіть: доменне ім'я, IP-адресу, TTL запису. Потім виконайте `ipconfig /flushdns` та знову `ipconfig /displaydns`. Що змінилося і чому?

12. Виконайте `ping -f -l 1472 google.com`, потім `ping -f -l 1473 google.com`. Що відбувається? Поясніть, що таке MTU (Maximum Transmission Unit) і як параметр `-f` допомагає його визначити.

Збережіть результат у файл `report.txt` (використайте оператор `>>` для дозапису). Запустіть файл та додайте report.txt до звіту.
