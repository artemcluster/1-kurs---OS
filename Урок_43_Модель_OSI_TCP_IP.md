## 1. Вступ

Коли ви відкриваєте веб-сторінку, надсилаєте повідомлення в месенджері або завантажуєте файл — за лаштунками відбувається надзвичайно складний процес передачі даних. Ваш комп'ютер має «домовитися» з сервером: як кодувати дані, якою мережею їх надсилати, що робити, якщо частина даних загубиться дорогою, і як зібрати все докупи на іншому кінці.

Щоб цей процес був зрозумілим та організованим, його розбили на **рівні** (шари). Кожний рівень відповідає за свою частину роботи і не «знає» деталей інших рівнів. Так само, як на пошті: листоноша не читає вміст вашого листа, а ви не плануєте маршрут поштового літака.

Існують дві основні моделі, що описують мережеву взаємодію:
- **Модель OSI** (Open Systems Interconnection) — теоретична еталонна модель із 7 рівнями, розроблена Міжнародною організацією зі стандартизації (ISO) у 1984 році.
- **Стек TCP/IP** — практична модель із 4 рівнями, створена Міністерством оборони США в 1970-х роках, яка лежить в основі сучасного Інтернету.

Сьогодні ми детально розберемо обидві моделі, навчимося їх порівнювати та зрозуміємо, як дані подорожують мережею від однієї програми до іншої.

## 2. Модель OSI — 7 рівнів

Модель OSI (Open Systems Interconnection — модель взаємодії відкритих систем) — це **концептуальна модель**, яка розділяє процес мережевої взаємодії на сім рівнів. Кожен рівень виконує чітко визначену функцію і надає послуги рівню, розташованому вище за нього.

Рівні нумеруються знизу вгору: від 1 (фізичний) до 7 (прикладний).

---

### 2.1. Рівень 1 — Фізичний (Physical)

**Що робить:** Забезпечує фізичне з'єднання між пристроями та передачу необроблених бітів (нулів та одиниць) через фізичне середовище.

**Основні функції:**
- Визначає тип середовища передачі: мідний кабель, оптоволокно, радіохвилі.
- Задає електричні характеристики сигналів (рівні напруги, частоти).
- Визначає швидкість передачі даних (бітрейт).
- Керує синхронізацією бітів між відправником та отримувачем.
- Визначає тип з'єднання: симплексне (в один бік), напівдуплексне (по черзі) або повнодуплексне (одночасно в обидва боки).

**Елементи фізичного рівня:**

| Категорія | Приклади |
|---|---|
| Кабелі | Вита пара (Cat5e, Cat6, Cat6a), коаксіальний кабель, оптоволокно |
| Роз'єми | RJ-45, SC, LC (оптичні), BNC |
| Активне обладнання | Хаб (концентратор), повторювач (репітер) |
| Бездротове середовище | Радіохвилі Wi-Fi, Bluetooth, інфрачервоний порт |

**Аналогія:** Фізичний рівень — це «дорога», по якій їдуть автомобілі (дані). Він визначає тип дороги (асфальт, залізниця, повітряний маршрут), але нічого не знає про те, що перевозять автомобілі.

> **PDU (Protocol Data Unit) на цьому рівні:** Біти (bits) — послідовність 0 і 1.

---

### 2.2. Рівень 2 — Канальний (Data Link)

**Що робить:** Забезпечує надійну передачу даних між двома **безпосередньо з'єднаними** пристроями в одній локальній мережі. Перетворює потік бітів у логічні одиниці — **фрейми** (кадри).

**Основні функції:**
- **Формування фреймів (framing):** Розбиває потік бітів на логічні блоки з чіткими межами початку та кінця.
- **Фізична адресація:** Використовує MAC-адреси (Media Access Control) — унікальні 48-бітні адреси, прошиті у мережеву карту (наприклад, `00:1A:2B:3C:4D:5E`).
- **Контроль доступу до середовища:** Визначає, коли пристрій може передавати дані (щоб уникнути колізій).
- **Виявлення помилок:** Додає контрольну суму (CRC) до кожного фрейму для перевірки цілісності.

**Підрівні канального рівня:**
- **LLC (Logical Link Control)** — забезпечує взаємодію з мережевим рівнем, керує потоком даних.
- **MAC (Media Access Control)** — безпосередньо працює з фізичним середовищем, визначає правила доступу до каналу.

**Протоколи та технології:** Ethernet (IEEE 802.3), Wi-Fi (IEEE 802.11), PPP (Point-to-Point Protocol).

**Обладнання:** Комутатор (switch), мережевий міст (bridge).

**Аналогія:** Канальний рівень — це «поштовий конверт» із зворотною адресою та адресою отримувача в межах одного будинку. MAC-адреса — це номер квартири: за нею можна знайти конкретний пристрій у локальній мережі.

> **PDU:** Фрейм (frame, кадр).

---

### 2.3. Рівень 3 — Мережевий (Network)

**Що робить:** Забезпечує передачу даних між пристроями, що знаходяться у **різних мережах**. Відповідає за логічну адресацію та маршрутизацію — пошук найкращого шляху від відправника до отримувача.

**Основні функції:**
- **Логічна адресація:** Використовує IP-адреси (Internet Protocol). IPv4 — 32-бітна адреса (наприклад, `192.168.1.100`), IPv6 — 128-бітна адреса (наприклад, `2001:0db8::1`).
- **Маршрутизація (routing):** Визначення оптимального шляху для пакета через мережу маршрутизаторів.
- **Фрагментація та збірка:** Якщо пакет занадто великий для наступної мережі, він розбивається на менші частини.

**Протоколи:** IP (Internet Protocol), ICMP (ping, traceroute), ARP (перетворення IP → MAC), OSPF, BGP (протоколи маршрутизації).

**Обладнання:** Маршрутизатор (router).

**Різниця між MAC-адресою та IP-адресою:**

| Характеристика | MAC-адреса | IP-адреса |
|---|---|---|
| Рівень OSI | 2 (Канальний) | 3 (Мережевий) |
| Тип | Фізична (апаратна) | Логічна (програмна) |
| Формат | 48 біт: `AA:BB:CC:DD:EE:FF` | IPv4: 32 біт: `192.168.1.1` |
| Призначення | Ідентифікація в локальній мережі | Ідентифікація у глобальній мережі |
| Змінюється? | Ні (прошита виробником)* | Так (призначається адміністратором або DHCP) |

\* Технічно MAC-адресу можна змінити програмно, але за замовчуванням вона постійна.

**Аналогія:** IP-адреса — це поштова адреса (місто, вулиця, будинок), а MAC-адреса — це ПІБ конкретної людини. Щоб лист дійшов, потрібні обидві: адреса визначає маршрут, а ім'я — конкретного одержувача.

> **PDU:** Пакет (packet).

---

### 2.4. Рівень 4 — Транспортний (Transport)

**Що робить:** Забезпечує **наскрізну (end-to-end)** доставку даних між програмами на різних комп'ютерах. Це перший рівень, який оперує не окремими вузлами мережі, а зв'язком «програма — програма».

**Основні функції:**
- **Сегментація:** Розбиває великі повідомлення на менші частини (сегменти).
- **Керування потоком:** Регулює швидкість передачі, щоб швидкий відправник не переповнив повільного отримувача.
- **Контроль помилок:** Виявляє загублені або пошкоджені сегменти та запитує їх повторну передачу (у TCP).
- **Мультиплексування:** Дозволяє кільком програмам одночасно використовувати мережу через **порти**.

**Основні протоколи:**
- **TCP (Transmission Control Protocol)** — надійний, з встановленням з'єднання.
- **UDP (User Datagram Protocol)** — швидкий, без встановлення з'єднання.

(Детальне порівняння TCP та UDP — у розділі 5.)

**Обладнання:** Працює програмно в операційній системі (не має спеціального апаратного обладнання).

**Аналогія:** Транспортний рівень — це служба доставки. TCP — це «Нова Пошта» з номером відстеження: ви точно знатимете, що посилка дійшла, а якщо ні — її відправлять повторно. UDP — це «листівка без відстеження»: швидше і дешевше, але без гарантій.

> **PDU:** Сегмент (segment, для TCP) або Датаграма (datagram, для UDP).

---

### 2.5. Рівень 5 — Сеансовий (Session)

**Що робить:** Керує **сеансами зв'язку** між двома програмами: встановлює, підтримує та коректно завершує сеанс обміну даними.

**Основні функції:**
- **Встановлення сеансу:** Ініціювання з'єднання між двома пристроями, узгодження параметрів обміну.
- **Підтримка сеансу:** Забезпечення безперервності обміну даними; якщо зв'язок перервався — може відновити сеанс з місця розриву.
- **Завершення сеансу:** Коректне закриття з'єднання після завершення обміну.
- **Синхронізація:** Додавання контрольних точок (checkpoints) у потік даних. Якщо під час передачі великого файлу стався збій, передачу можна продовжити з останньої контрольної точки, а не починати заново.
- **Керування діалогом:** Визначає, хто може передавати дані в кожний момент часу — напівдуплексний або повнодуплексний режим.

**Протоколи та технології:** NetBIOS, PPTP (VPN-тунелювання), RPC (Remote Procedure Call).

**Аналогія:** Сеансовий рівень — це «телефонний оператор», який з'єднує два телефони, слідкує, щоб зв'язок не обірвався, і коректно відключає лінію після завершення розмови.

> **PDU:** Дані (data).

---

### 2.6. Рівень 6 — Представлення (Presentation)

**Що робить:** Відповідає за **формат та подання даних**. Забезпечує, щоб дані, надіслані однією системою, були зрозумілі для іншої, навіть якщо вони використовують різні внутрішні формати.

**Основні функції:**
- **Перетворення (кодування):** Переклад даних з одного формату в інший. Наприклад, із кодування ASCII у EBCDIC, із Unicode UTF-8 у UTF-16.
- **Шифрування / дешифрування:** Захист даних від несанкціонованого доступу під час передачі. Наприклад, TLS/SSL шифрує трафік між браузером та веб-сервером.
- **Стиснення / розпакування:** Зменшення обсягу даних для ефективнішої передачі. Наприклад, стиснення зображень (JPEG), відео (H.264/H.265), аудіо (MP3, AAC).

**Протоколи та формати:** TLS/SSL, JPEG, PNG, GIF, MPEG, MP3, ASCII, Unicode.

**Аналогія:** Рівень представлення — це «перекладач» на міжнародній конференції. Він не змінює зміст повідомлення, але перекладає його так, щоб інша сторона зрозуміла.

> **PDU:** Дані (data).

---

### 2.7. Рівень 7 — Прикладний (Application)

**Що робить:** Надає інтерфейс для взаємодії **користувацьких програм** із мережею. Це єдиний рівень, з яким безпосередньо працює кінцевий користувач (через програми).

**Основні функції:**
- Надання мережевих сервісів прикладним програмам: перегляд веб-сторінок, електронна пошта, передача файлів, служба доменних імен.
- Ідентифікація партнера по зв'язку та перевірка його доступності.
- Синхронізація обміну даними між програмами.

**Основні протоколи:**

| Протокол | Призначення | Приклад використання |
|---|---|---|
| **HTTP / HTTPS** | Передача веб-сторінок | Ви відкриваєте сайт у браузері |
| **FTP** | Передача файлів | Завантаження файлів на сервер |
| **SMTP** | Відправлення електронної пошти | Ви натискаєте «Надіслати» в Gmail |
| **POP3 / IMAP** | Отримання електронної пошти | Ваш поштовий клієнт завантажує листи |
| **DNS** | Перетворення доменного імені на IP-адресу | `google.com` → `142.250.74.206` |
| **DHCP** | Автоматичне присвоєння IP-адреси | Ваш ноутбук підключився до Wi-Fi |
| **SSH** | Безпечне віддалене підключення | Адміністратор керує сервером |
| **Telnet** | Віддалене підключення (незахищене) | Застарілий аналог SSH |

**Аналогія:** Прикладний рівень — це «віконце на пошті», куди ви приходите як клієнт: здаєте лист, отримуєте посилку, замовляєте послугу. Ви працюєте лише з цим віконцем, не знаючи, як саме лист буде доставлений.

> **PDU:** Дані (data).

### 2.8. Текстова схема моделі OSI

```
│  Рівень 7  │  Прикладний (Application)                  │  HTTP, FTP, SMTP, DNS
├─────────────────────────────────────────────────────────┤
│  Рівень 6  │  Представлення (Presentation)              │  TLS/SSL, JPEG, MPEG
├─────────────────────────────────────────────────────────┤

├─────────────────────────────────────────────────────────┤
│  Рівень 4  │  Транспортний (Transport)                  │  TCP, UDP
├─────────────────────────────────────────────────────────┤
│  Рівень 3  │  Мережевий (Network)                       │  IP, ICMP, ARP
├─────────────────────────────────────────────────────────┤
│  Рівень 2  │  Канальний (Data Link)                     │  Ethernet, Wi-Fi, PPP
├─────────────────────────────────────────────────────────┤
│  Рівень 1  │  Фізичний (Physical)                       │  Кабелі, роз'єми, сигнали
└─────────────────────────────────────────────────────────┘
         ▲                                      ▲
         │      Дані рухаються вниз при          │
         │      відправленні та вгору при         │

         ▼                                      ▼
    ВІДПРАВНИК                             ОТРИМУВАЧ
```

«**Ф**антастичний **К**іт **М**ав **Т**ри **С**маковитих **П**ончики **П**олуничних»

### 2.9. Аналогія з поштовою доставкою

Уявімо, що ви надсилаєте лист другу в інше місто. Кожен рівень OSI — це етап доставки:

| Рівень OSI | Поштова аналогія |
|---|---|
| **7. Прикладний** | Ви пишете лист рідною мовою |
| **6. Представлення** | Перекладач перекладає лист мовою, зрозумілою одержувачу, та шифрує конфіденційні дані |
| **5. Сеансовий** | Ви домовляєтеся про переписку: «Я пишу першим, потім ти відповідаєш» |
| **4. Транспортний** | Якщо лист завеликий, його розрізають на частини та нумерують сторінки (1/5, 2/5...) |
| **3. Мережевий** | На конверті пишуть поштову адресу (місто, вулиця) — пошта вирішує, яким маршрутом доставити |
| **2. Канальний** | Лист кладуть у поштову скриньку конкретного будинку (MAC-адреса — номер скриньки) |
| **1. Фізичний** | Фізична доставка: поштовий літак, потяг, автомобіль (фізичне середовище передачі) |

На боці одержувача процес відбувається у зворотному порядку: листоноша дістає лист зі скриньки → розпаковує → збирає сторінки за номерами → перекладач перекладає → друг читає.

---

## 3. Стек TCP/IP — 4 рівні

Стек протоколів **TCP/IP** (Transmission Control Protocol / Internet Protocol) — це практична модель, яка реально використовується в Інтернеті та більшості сучасних мереж. На відміну від теоретичної OSI, TCP/IP був створений з конкретною метою — забезпечити зв'язок між комп'ютерами у розподіленій мережі (прообразі Інтернету — ARPANET).

Модель TCP/IP складається з **4 рівнів**:

```
┌──────────────────────────────────────────────────┐
│  Рівень 4  │  Прикладний (Application)            │  HTTP, FTP, SMTP, DNS, SSH
├──────────────────────────────────────────────────┤
│  Рівень 3  │  Транспортний (Transport)             │  TCP, UDP
├──────────────────────────────────────────────────┤
│  Рівень 2  │  Інтернет (Internet)                  │  IP, ICMP, ARP
├──────────────────────────────────────────────────┤
│  Рівень 1  │  Мережевий доступ (Network Access)    │  Ethernet, Wi-Fi, PPP
└──────────────────────────────────────────────────┘
```

---

### 3.1. Рівень мережевого доступу (Network Access)

Об'єднує функції фізичного та канального рівнів OSI. Відповідає за фізичну передачу даних через конкретне середовище (кабель, Wi-Fi) та формування фреймів з MAC-адресами.

**Приклади технологій:** Ethernet, Wi-Fi, Token Ring, FDDI, PPP.

---

### 3.2. Рівень Інтернету (Internet)

Відповідає мережевому рівню OSI. Відповідає за логічну адресацію (IP-адреси) та маршрутизацію пакетів між різними мережами.

**Основний протокол:** IP (Internet Protocol) — версії IPv4 та IPv6.

**Допоміжні протоколи:** ICMP (діагностика, ping), ARP (визначення MAC за IP), IGMP (групове мовлення).

---

### 3.3. Транспортний рівень (Transport)

Повністю відповідає транспортному рівню OSI. Забезпечує наскрізну доставку даних між програмами.

**Протоколи:** TCP (надійний) та UDP (швидкий).

---

### 3.4. Прикладний рівень (Application)

Об'єднує функції сеансового, рівня представлення та прикладного рівнів OSI. Надає прикладним програмам доступ до мережі.

**Протоколи:** HTTP/HTTPS, FTP, SMTP, POP3, IMAP, DNS, DHCP, SSH, Telnet, SNMP.

---

## 4. Порівняння моделі OSI та стеку TCP/IP

### Таблиця відповідності рівнів

| Модель OSI (7 рівнів) | Стек TCP/IP (4 рівні) | Приклади протоколів |
|---|---|---|
| 7 — Прикладний | **Прикладний** | HTTP, FTP, SMTP, DNS |
| 6 — Представлення | **Прикладний** | TLS/SSL, JPEG, MPEG |
| 5 — Сеансовий | **Прикладний** | NetBIOS, RPC |
| 4 — Транспортний | **Транспортний** | TCP, UDP |
| 3 — Мережевий | **Інтернет** | IP, ICMP, ARP |
| 2 — Канальний | **Мережевий доступ** | Ethernet, Wi-Fi |
| 1 — Фізичний | **Мережевий доступ** | Кабелі, роз'єми, сигнали |

### Ключові відмінності

| Характеристика | Модель OSI | Стек TCP/IP |
|---|---|---|
| Кількість рівнів | 7 | 4 |
| Тип | Теоретична (еталонна) | Практична (робоча) |
| Розробник | ISO (1984) | Міноборони США (1970-ті) |
| Призначення | Опис ідеальної архітектури | Конкретна реалізація для Інтернету |
| Деталізація | Висока (кожен рівень чітко розмежований) | Нижча (деякі рівні об'єднані) |
| Використання | Навчання, проєктування, діагностика | Реальні мережі, Інтернет |
| Залежність від протоколів | Не прив'язана до конкретних протоколів | Прив'язана до протоколів TCP та IP |

> **Важливо:** На практиці модель OSI використовується як «мова спілкування» між мережевими інженерами (наприклад, «проблема на 3-му рівні» означає проблему з маршрутизацією), а стек TCP/IP — це реальна архітектура, за якою працює Інтернет.

## 5. Протокол TCP vs UDP

На транспортному рівні працюють два основних протоколи: TCP та UDP. Вони принципово відрізняються підходом до доставки даних.

---

### 5.1. TCP — Transmission Control Protocol

**TCP** — це **надійний протокол зі встановленням з'єднання (connection-oriented)**.

**Як працює TCP:**

1. **Встановлення з'єднання — «Тристороннє рукостискання» (Three-Way Handshake):**
   ```
   Клієнт  ──── SYN ────►  Сервер     (1. «Привіт, хочу з'єднатися»)
   Клієнт  ◄── SYN-ACK ──  Сервер     (2. «Привіт, згоден, підтверджую»)
   Клієнт  ──── ACK ────►  Сервер     (3. «Чудово, починаємо обмін»)
   ```

2. **Передача даних:** Кожен сегмент має порядковий номер. Отримувач підтверджує кожну отриману порцію даних (ACK). Якщо підтвердження не надійшло — сегмент надсилається повторно.

3. **Завершення з'єднання:** Обидві сторони обмінюються спеціальними повідомленнями FIN/ACK для коректного розриву з'єднання.

**Ключові властивості TCP:**
- Гарантія доставки всіх даних.
- Збереження порядку — дані прибувають у тому самому порядку, в якому були надіслані.
- Контроль потоку — регулювання швидкості передачі.
- Контроль перевантажень — зменшення навантаження при завантаженості мережі.

---

### 5.2. UDP — User Datagram Protocol

**UDP** — це **швидкий протокол без встановлення з'єднання (connectionless)**.

**Як працює UDP:**
- Дані відправляються без попереднього «рукостискання».
- Ніякого підтвердження про отримання.
- Ніякого повторного надсилання загублених даних.
- Ніякого контролю порядку — датаграми можуть прибути не в тому порядку, в якому були надіслані.

**Ключові властивості UDP:**
- Мінімальна затримка (швидкість).
- Малий заголовок (8 байт проти 20 байт у TCP).
- Підтримка широкомовної (broadcast) та групової (multicast) розсилки.
- Немає гарантії доставки.

---

### 5.3. Порівняльна таблиця TCP та UDP

| Характеристика | TCP | UDP |
|---|---|---|
| Тип з'єднання | Зі встановленням з'єднання | Без встановлення з'єднання |
| Надійність | Гарантує доставку | Не гарантує доставку |
| Порядок даних | Зберігає порядок | Не гарантує порядок |
| Контроль помилок | Є (повторна передача) | Мінімальний (лише контрольна сума) |
| Швидкість | Повільніший | Швидший |
| Розмір заголовка | 20 байт | 8 байт |
| Контроль потоку | Є | Немає |
| Broadcast / Multicast | Не підтримує | Підтримує |

---

### 5.4. Коли використовують TCP, а коли UDP

| TCP | UDP |
|---|---|
| Веб-сторінки (HTTP/HTTPS) | Онлайн-ігри |
| Електронна пошта (SMTP, IMAP) | Відеодзвінки (Zoom, Skype) |
| Передача файлів (FTP) | Потокове відео / аудіо (YouTube Live, Twitch) |
| Віддалений доступ (SSH) | DNS-запити (швидкі, короткі) |
| Банківські транзакції | VoIP-телефонія (голосовий зв'язок) |
| Завантаження оновлень | DHCP (автоматичне отримання IP) |
| Веб-сокети (чати) | IoT-датчики (відправлення показників) |

> **Чому для відеодзвінків використовують UDP?** Тому що краще пропустити один кадр відео, ніж чекати його повторної доставки і дивитись «заморожене» зображення. Для людського ока невелика втрата кадрів непомітна, а затримка — дуже дратує.

## 6. Інкапсуляція даних

### 6.1. Що таке інкапсуляція

**Інкапсуляція (encapsulation)** — це процес «загортання» даних при їх проходженні вниз по рівнях мережевої моделі. На кожному рівні до даних додається свій **заголовок (header)**, а на канальному рівні — ще й **трейлер (trailer)**. Кожен заголовок містить службову інформацію, необхідну для роботи протоколу цього рівня.

Цей процес можна порівняти з пакуванням подарунка:
1. Подарунок (дані) кладете в коробку (сегмент).
2. Коробку загортаєте в обгортковий папір (пакет).
3. Загорнуту коробку кладете у поштовий конверт (фрейм).
4. Конверт перетворюється на послідовність сигналів (біти).

---

### 6.2. PDU на кожному рівні

**PDU (Protocol Data Unit)** — це одиниця даних, характерна для кожного рівня. На кожному рівні PDU отримує свою назву:

| Рівень OSI | PDU (назва одиниці даних) | Що додається |
|---|---|---|
| 7 — Прикладний | Дані (Data) | Заголовок прикладного протоколу (HTTP, FTP...) |
| 6 — Представлення | Дані (Data) | Інформація про кодування, шифрування |
| 5 — Сеансовий | Дані (Data) | Інформація про сеанс |
| 4 — Транспортний | **Сегмент** (TCP) / **Датаграма** (UDP) | Порти відправника та отримувача, порядковий номер |
| 3 — Мережевий | **Пакет** (Packet) | IP-адреси відправника та отримувача |
| 2 — Канальний | **Фрейм** (Frame) | MAC-адреси відправника та отримувача + CRC (трейлер) |
| 1 — Фізичний | **Біти** (Bits) | Передача сигналами |

---

### 6.3. Покрокова схема інкапсуляції

```
Прикладний рівень:
┌──────────────────────────────────┐
│           ДАНІ                   │
└──────────────────────────────────┘
                 ▼
Транспортний рівень:
┌──────────┬───────────────────────┐
│ TCP/UDP  │         ДАНІ          │
│ заголовок│                       │
└──────────┴───────────────────────┘
        = СЕГМЕНТ / ДАТАГРАМА
                 ▼
Мережевий рівень:
┌───────┬──────────┬───────────────┐
│  IP   │ TCP/UDP  │     ДАНІ      │
│ загол.│ заголовок│               │
└───────┴──────────┴───────────────┘
             = ПАКЕТ
                 ▼
Канальний рівень:
┌───────┬───────┬──────────┬──────┬────────┐
│ Загол.│  IP   │ TCP/UDP  │ ДАНІ │ Трейлер│
│ фрейму│ загол.│ заголовок│      │  (CRC) │
└───────┴───────┴──────────┴──────┴────────┘
                  = ФРЕЙМ
                 ▼
Фізичний рівень:
  01101001 01010110 11001010 01110101 ...
                  = БІТИ
```

Зверніть увагу: на кожному рівні попередній PDU стає «корисним навантаженням» (payload) для наступного рівня. IP-заголовок «не знає», що всередині є TCP-заголовок — для нього це просто дані.

---

### 6.4. Деінкапсуляція

**Деінкапсуляція (de-encapsulation)** — це зворотний процес, який відбувається на стороні отримувача. Дані піднімаються по рівнях знизу вгору, і на кожному рівні знімається відповідний заголовок:

1. Фізичний рівень приймає біти та формує фрейм.
2. Канальний рівень перевіряє CRC, знімає заголовок та трейлер фрейму, передає пакет вище.
3. Мережевий рівень перевіряє IP-адресу призначення, знімає IP-заголовок, передає сегмент вище.
4. Транспортний рівень знімає TCP/UDP-заголовок, визначає порт і передає дані відповідній програмі.
5. Прикладний рівень отримує чисті дані та показує їх користувачу.

---

## 7. Порти

### 7.1. Що таке порт

**Порт** — це числовий ідентифікатор (від 0 до 65535), який вказує, якій конкретній програмі (або сервісу) на комп'ютері призначені дані.

**Навіщо потрібні порти?** IP-адреса ідентифікує комп'ютер у мережі, але на одному комп'ютері одночасно працюють десятки програм: браузер, поштовий клієнт, месенджер, онлайн-гра. Порт дозволяє ОС визначити, якій саме програмі передати отримані дані.

**Аналогія:** Якщо IP-адреса — це адреса будинку, то порт — це номер квартири. Лист (пакет даних) доставляється не просто в будинок, а в конкретну квартиру (програму).

**Зв'язка «IP-адреса : порт»** називається **сокет (socket)**:
```
192.168.1.100:443  — підключення до HTTPS-сервера
10.0.0.5:22        — підключення до SSH-сервера
```

---

### 7.2. Діапазони портів

| Діапазон | Назва | Призначення |
|---|---|---|
| **0 — 1023** | Загальновідомі (well-known) | Зарезервовані за стандартними сервісами (HTTP, FTP, DNS). Вимагають прав адміністратора. |
| **1024 — 49151** | Зареєстровані (registered) | Призначені IANA для конкретних програм (MySQL — 3306, PostgreSQL — 5432). |
| **49152 — 65535** | Динамічні / приватні (dynamic/private) | Тимчасові порти, які ОС автоматично призначає клієнтським програмам для вихідних з'єднань. |

---

### 7.3. Таблиця відомих портів

| Порт | Протокол | Транспорт | Призначення |
|---|---|---|---|
| **20** | FTP-Data | TCP | Передача даних FTP |
| **21** | FTP-Control | TCP | Керування FTP-з'єднанням |
| **22** | SSH | TCP | Безпечний віддалений доступ та передача файлів |
| **23** | Telnet | TCP | Незахищений віддалений доступ (застарілий) |
| **25** | SMTP | TCP | Відправлення електронної пошти |
| **53** | DNS | TCP + UDP | Перетворення доменних імен у IP-адреси |
| **67/68** | DHCP | UDP | Автоматичне призначення IP-адрес |
| **80** | HTTP | TCP | Передача веб-сторінок (незашифрована) |
| **110** | POP3 | TCP | Отримання пошти |
| **143** | IMAP | TCP | Отримання та управління поштою |
| **443** | HTTPS | TCP | Передача веб-сторінок (зашифрована, TLS) |
| **3389** | RDP | TCP + UDP | Віддалений робочий стіл Windows |

> **Запам'ятайте обов'язково:** 80 (HTTP), 443 (HTTPS), 21 (FTP), 22 (SSH), 25 (SMTP), 53 (DNS), 3389 (RDP) — це мінімальний набір, який має знати кожен ІТ-спеціаліст.

---

### 7.4. Як працює зв'язка IP-адреса + порт

Розглянемо приклад: ви відкриваєте сайт `google.com` у браузері.

```
Ваш комп'ютер                             Сервер Google
192.168.1.100:52431  ────────────────►  142.250.74.206:443
(ваш IP : динамічний порт)              (IP Google : порт HTTPS)
```

1. Браузер звертається до DNS (порт 53), щоб дізнатися IP-адресу `google.com`.
2. ОС автоматично призначає браузеру тимчасовий (динамічний) порт — наприклад, 52431.
3. Браузер створює TCP-з'єднання із сервером Google на порт 443 (HTTPS).
4. Сервер бачить запит від `192.168.1.100:52431` і надсилає відповідь на ту ж адресу.
5. ОС за номером порту 52431 розуміє, що відповідь призначена саме для браузера, а не для месенджера чи гри.

## 8. Підсумок

Сьогодні ми розглянули дві основні мережеві моделі:

1. **Модель OSI** — теоретичний еталон із 7 рівнями (Фізичний → Канальний → Мережевий → Транспортний → Сеансовий → Представлення → Прикладний). Використовується для навчання, проєктування та діагностики мереж.

2. **Стек TCP/IP** — практична модель із 4 рівнями (Мережевий доступ → Інтернет → Транспортний → Прикладний), на якій побудований сучасний Інтернет.

3. **TCP** — надійний протокол зі встановленням з'єднання, гарантує доставку та порядок. Використовується для веб, пошти, файлів.

4. **UDP** — швидкий протокол без встановлення з'єднання, без гарантій доставки. Використовується для відео, ігор, VoIP.

5. **Інкапсуляція** — процес «загортання» даних у заголовки на кожному рівні: Дані → Сегмент → Пакет → Фрейм → Біти.

6. **Порти** — числові ідентифікатори (0–65535), що визначають, якій програмі адресовані дані. Зв'язка IP + порт = сокет.

## 10. Глосарій ключових термінів

| Термін | Визначення |
|---|---|
| **OSI (Open Systems Interconnection)** | Еталонна 7-рівнева модель мережевої взаємодії, розроблена ISO |
| **TCP/IP** | Практична 4-рівнева модель мережевих протоколів, основа Інтернету |
| **Протокол** | Набір правил, що визначає формат і порядок обміну даними між пристроями |
| **MAC-адреса** | Унікальна 48-бітна фізична адреса мережевого адаптера (рівень 2 OSI) |
| **IP-адреса** | Логічна адреса пристрою в мережі (рівень 3 OSI). IPv4 — 32 біти, IPv6 — 128 біт |
| **TCP (Transmission Control Protocol)** | Надійний протокол транспортного рівня зі встановленням з'єднання та гарантією доставки |
| **UDP (User Datagram Protocol)** | Швидкий протокол транспортного рівня без встановлення з'єднання та без гарантії доставки |
| **Порт** | Числовий ідентифікатор (0–65535), що визначає конкретну програму або сервіс на комп'ютері |
| **Сокет (Socket)** | Зв'язка IP-адреси та порту (наприклад, 192.168.1.1:80), що однозначно ідентифікує з'єднання |
| **Фрейм (Frame, Кадр)** | PDU канального рівня; містить MAC-адреси та дані з контрольною сумою |
| **Пакет (Packet)** | PDU мережевого рівня; містить IP-адреси та дані |
| **Сегмент (Segment)** | PDU транспортного рівня (TCP); містить номери портів та порядкові номери |
| **Датаграма (Datagram)** | PDU транспортного рівня (UDP); мінімальний заголовок, без гарантій доставки |
| **PDU (Protocol Data Unit)** | Одиниця даних на кожному рівні мережевої моделі (біти, фрейми, пакети, сегменти, дані) |
| **Інкапсуляція** | Процес додавання заголовків при проходженні даних зверху вниз по рівнях моделі |
| **Деінкапсуляція** | Зворотний процес зняття заголовків при проходженні даних знизу вгору |
| **Маршрутизація (Routing)** | Процес вибору найкращого шляху для передачі пакета між мережами |
| **Three-Way Handshake** | Тристороннє рукостискання — процедура встановлення TCP-з'єднання (SYN → SYN-ACK → ACK) |
| **Ethernet** | Найпоширеніша технологія провідних локальних мереж (рівень 2 OSI) |
| **DNS (Domain Name System)** | Система, що перетворює доменні імена (google.com) на IP-адреси (142.250.74.206) |
| **HTTP / HTTPS** | Протоколи передачі веб-сторінок. HTTPS — захищена версія з шифруванням TLS |
| **TLS/SSL** | Протоколи шифрування, що забезпечують безпечну передачу даних у мережі |
| **FTP (File Transfer Protocol)** | Протокол передачі файлів між клієнтом та сервером (порти 20, 21) |
| **SSH (Secure Shell)** | Протокол безпечного віддаленого доступу до сервера (порт 22) |
| **SMTP (Simple Mail Transfer Protocol)** | Протокол відправлення електронної пошти (порт 25) |
| **RDP (Remote Desktop Protocol)** | Протокол віддаленого робочого стола Windows (порт 3389) |
| **DHCP (Dynamic Host Configuration Protocol)** | Протокол автоматичного призначення IP-адрес пристроям у мережі |
| **CRC (Cyclic Redundancy Check)** | Контрольна сума для перевірки цілісності даних у фреймі |
| **ARP (Address Resolution Protocol)** | Протокол визначення MAC-адреси за відомою IP-адресою |
