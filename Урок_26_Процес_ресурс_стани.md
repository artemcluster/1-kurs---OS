## 1. Поняття процесу. Процес vs програма

### 1.1. Що таке процес

**Процес** — це динамічний об'єкт, що описує виконання програми. Це програма, завантажена в оперативну пам'ять і виконувана процесором. Процесу виділяються системні ресурси: адресний простір, файли, пристрої введення-виведення тощо.

Процеси є **основними одиницями виконання** в операційних системах. Саме процесами ОС керує, розподіляючи між ними час процесора та інші ресурси.

### 1.2. Процес vs програма

Ці два поняття часто плутають, але між ними є принципова різниця:

| Критерій | Програма | Процес |
|----------|----------|--------|
| **Природа** | Статичний об'єкт | Динамічний об'єкт |
| **Що це** | Набір інструкцій (код), записаний на диску | Виконання цього коду в пам'яті |
| **Де зберігається** | На жорсткому диску / SSD (файл `.exe`, `.bin` тощо) | В оперативній пам'яті (RAM) |
| **Існування** | Існує постійно, поки не видалена | Існує тимчасово — від запуску до завершення |
| **Ресурси** | Не використовує ресурси (поки не запущена) | Використовує CPU, RAM, файли, пристрої |
| **Кількість** | Одна програма на диску | Може породити кілька процесів одночасно |

**Приклад:** Файл `chrome.exe` на диску — це програма. Коли ви запускаете Google Chrome, створюється процес (а часто й кілька процесів — по одному на кожну вкладку). Якщо відкрити Chrome двічі — буде два окремих процеси, хоча програма одна.

### 1.3. Одна програма — кілька процесів

```
  Програма (файл на диску)          Процеси (в оперативній пам'яті)
  ┌─────────────────────┐
  │                     │ ──запуск──> [ Процес 1 (PID 1234) ]
  │   notepad.exe       │
  │                     │ ──запуск──> [ Процес 2 (PID 5678) ]
  └─────────────────────┘
   Один файл на диску         Два незалежних процеси в RAM
```

Кожен процес має власний адресний простір, власні дані та працює незалежно від іншого, навіть якщо обидва запущені з однієї й тієї самої програми.

## 2. Контекст процесу

### 2.1. Що таке контекст процесу

**Контекст процесу** — це вся інформація, необхідна для відновлення виконання процесу після перерви. Коли ОС забирає процесор у одного процесу та передає іншому, вона має зберегти стан першого процесу, щоб потім продовжити його роботу з того самого місця.

### 2.2. Що входить до контексту

| Компонент | Призначення |
|-----------|-------------|
| **Лічильник команд (Program Counter, PC)** | Зберігає адресу наступної інструкції, яку має виконати процесор. Завдяки йому ОС знає, де процес зупинився |
| **Регістри процесора (CPU Registers)** | Значення всіх регістрів (загального призначення, індексні, стекові, регістр прапорців тощо). Це "робочі змінні" процесора |
| **Стек (Stack)** | Область пам'яті, де зберігаються адреси повернення з підпрограм, локальні змінні, параметри функцій |
| **Дані (Data)** | Глобальні та статичні змінні програми |
| **Інформація про пам'ять** | Межі адресного простору, таблиці сторінок або сегментів |

### 2.3. Перемикання контексту (Context Switch)

Коли ОС перемикає процесор з одного процесу на інший, відбувається **перемикання контексту**:

```
  Процес A (виконується)         Процес B (готовий)
       │                              │
       ▼                              │
  [ Зберегти контекст A в PCB_A ]     │
       │                              │
       └─────── CPU ──────────────────┘
                                      │
                                      ▼
                          [ Завантажити контекст B з PCB_B ]
                                      │
                                      ▼
                              Процес B (виконується)
```

Перемикання контексту займає час (мікросекунди), протягом якого процесор не виконує корисної роботи. Це накладні витрати багатозадачності.

## 3. Блок керування процесом (PCB)

### 3.1. Що таке PCB

**PCB (Process Control Block)** — блок керування процесом — це структура даних в ядрі операційної системи, яка зберігає всю інформацію про процес. Для кожного процесу в системі існує свій PCB. Сукупність усіх PCB утворює **таблицю процесів**.

PCB створюється при народженні процесу і знищується при його завершенні.

### 3.2. Структура PCB

```
  ┌──────────────────────────────────────────────┐
  │           PROCESS CONTROL BLOCK              │
  ├──────────────────────────────────────────────┤
  │  PID (ідентифікатор процесу)       : 3072   │
  │  PPID (ідентифікатор батька)       : 1024   │
  │  Стан процесу                      : Ready  │
  │  Пріоритет                         : 5      │
  ├──────────────────────────────────────────────┤
  │  Лічильник команд (PC)         : 0x004A20F8 │
  │  Регістри CPU                  : EAX, EBX...│
  │  Покажчик стеку (SP)           : 0x7FFF0040 │
  ├──────────────────────────────────────────────┤
  │  Виділена пам'ять:                           │
  │    Базова адреса               : 0x00400000 │
  │    Межа (limit)                : 0x004FFFFF │
  │    Таблиця сторінок            : (вказівник) │
  ├──────────────────────────────────────────────┤
  │  Відкриті файли                : [ fd0, fd1 ]│
  │  Поточний каталог              : /home/user  │
  ├──────────────────────────────────────────────┤
  │  Облікова інформація:                        │
  │    Час CPU (використано)       : 0.35 с     │
  │    Час створення               : 14:32:05   │
  │    Власник (UID)               : 1000       │
  └──────────────────────────────────────────────┘
```

### 3.3. Основні поля PCB

| Поле | Опис |
|------|------|
| **PID** | Унікальний числовий ідентифікатор процесу |
| **Стан процесу** | Поточний стан: новий, готовий, виконується, очікує, завершений |
| **Лічильник команд (PC)** | Адреса наступної інструкції для виконання |
| **Регістри CPU** | Збережені значення всіх регістрів процесора |
| **Пріоритет** | Числове значення, що визначає важливість процесу для планувальника |
| **Інформація про пам'ять** | Базова адреса, межа, таблиці сторінок — описують адресний простір процесу |
| **Відкриті файли** | Список дескрипторів файлів, з якими працює процес |
| **Облікова інформація** | Час CPU, час створення, власник процесу |

### 3.4. Реалізація в реальних ОС

- У **Linux** PCB реалізовано як структуру `task_struct` (у вихідному коді ядра)
- У **Windows** PCB реалізовано як структуру `EPROCESS` (у ядрі NT)

Звичайний користувач не має прямого доступу до PCB — ця структура зберігається в захищеній пам'яті ядра ОС.

## 4. Поняття ресурсу

### 4.1. Визначення

**Ресурс** — це будь-який об'єкт обчислювальної системи, який може бути розподілений між процесами. ОС виступає в ролі менеджера ресурсів: приймає запити від процесів, виділяє ресурси та забирає їх після використання.

### 4.2. Класифікація ресурсів

```
                          Ресурси
                     ┌────────┴────────┐
                Апаратні           Програмні
              (фізичні)           (логічні)
           ┌─────┼─────┐      ┌─────┼──────┐
          CPU   RAM   I/O    Файли Семафори Черги
                       │                повідомлень
                  ┌────┴────┐
                Диск  Принтер
               Мережа Клавіатура
```

### 4.3. Апаратні ресурси

| Ресурс | Опис | Приклад розподілу |
|--------|------|-------------------|
| **Процесор (CPU)** | Головний обчислювальний ресурс. Виконує інструкції процесів | Розподіл часу: кожен процес отримує квант часу (time slice) — наприклад, 10 мс |
| **Оперативна пам'ять (RAM)** | Зберігає код і дані процесів під час виконання | Кожен процес отримує свій адресний простір |
| **Пристрої введення-виведення (I/O)** | Диски, принтери, мережеві адаптери, клавіатура, монітор | Процеси стають у чергу на використання пристрою |

### 4.4. Програмні ресурси

| Ресурс | Опис |
|--------|------|
| **Файли** | Процес може відкривати, читати, записувати файли. Файл — це програмний ресурс |
| **Семафори** | Механізми синхронізації для координації роботи кількох процесів |
| **Черги повідомлень** | Засоби обміну даними між процесами |
| **Мережеві з'єднання (сокети)** | Канали зв'язку для мережевої взаємодії |

### 4.5. Типи розподілу ресурсів

Ресурси розподіляються двома основними способами:

- **Часовий розподіл (time sharing)** — процеси по черзі використовують ресурс протягом певного інтервалу часу. Приклад: процесор по черзі виконує інструкції різних процесів.

- **Просторовий розподіл (space sharing)** — ресурс ділиться на частини, кожна з яких виділяється окремому процесу. Приклад: оперативна пам'ять розділена між процесами на окремі адресні простори.

## 5. Стани процесу — базова модель

### 5.1. П'ять основних станів

Протягом свого життя процес проходить через кілька станів. Базова модель включає **п'ять станів**:

| Стан | Англійська назва | Опис |
|------|-----------------|------|
| **Новий** | New / Created | Процес щойно створений. ОС виділяє PCB, але процес ще не допущений до виконання |
| **Готовий** | Ready | Процес завантажений в оперативну пам'ять і чекає, поки планувальник призначить йому процесор |
| **Виконується** | Running | Процесор виконує інструкції цього процесу. В одноядерній системі лише один процес може бути в цьому стані |
| **Очікування** | Waiting / Blocked | Процес не може продовжити виконання, бо чекає на подію: завершення операції введення-виведення, дані від іншого процесу тощо |
| **Завершений** | Terminated / Exit | Процес завершив виконання. ОС звільняє його ресурси та видаляє PCB |

### 5.2. Діаграма станів (базова модель)

```
                        ┌─────────┐
                        │  НОВИЙ  │
                        │  (New)  │
                        └────┬────┘
                             │ допуск (admitted)
                             ▼
       час вичерпано    ┌─────────┐   призначення    ┌──────────────┐
      (time-out)  ┌─────│ ГОТОВИЙ │──────────────────>│ ВИКОНУЄТЬСЯ  │
                  │     │ (Ready) │   CPU (dispatch)  │  (Running)   │
                  │     └─────────┘                   └──┬─────┬─────┘
                  │          ▲                            │     │
                  │          │                            │     │
                  └──────────┘                            │     │
                                                         │     │
                         ┌───────────┐  запит I/O        │     │ завершення
           подія    ┌────│ ОЧІКУВАННЯ│<───────────────────┘     │ (exit)
           сталася  │    │ (Waiting) │  або подія               │
                    │    └───────────┘                           ▼
                    │         ▲                          ┌──────────────┐
                    │         │                          │  ЗАВЕРШЕНИЙ  │
                    └─────────┘                          │ (Terminated) │
                  I/O або подія                          └──────────────┘
                   завершена
```

## 6. Переходи між станами

### 6.1. Таблиця переходів

| Перехід | Причина | Приклад |
|---------|---------|---------|
| **Новий --> Готовий** | ОС допустила процес до виконання, завантажила його в пам'ять | Користувач запустив програму, ОС створила процес і поставила в чергу готових |
| **Готовий --> Виконується** | Планувальник (dispatcher) обрав цей процес і передав йому процесор | Процес був першим у черзі Ready Queue і отримав свій квант часу |
| **Виконується --> Готовий** | Квант часу вичерпано (переривання від таймера) або з'явився процес з вищим пріоритетом (витіснення) | Процес працював 10 мс, таймер спрацював — процес повернувся в чергу |
| **Виконується --> Очікування** | Процес запросив операцію, яка потребує часу (I/O, очікування даних) | Процес надіслав запит на читання файлу з диска — чекає, поки дані будуть зчитані |
| **Очікування --> Готовий** | Подія, на яку чекав процес, відбулася | Диск завершив зчитування даних — процес знову готовий до виконання |
| **Виконується --> Завершений** | Процес завершив свою роботу (нормально або аварійно) | Програма виконала останню інструкцію або була примусово зупинена користувачем |

### 6.2. Важливі зауваження

- Процес **не може** перейти зі стану "Очікування" безпосередньо у стан "Виконується". Спочатку він обов'язково потрапляє у стан "Готовий" і чекає свою чергу.

- Процес **не може** перейти зі стану "Готовий" у стан "Очікування". Заблокувати себе може лише процес, що виконується.

- Перехід "Виконується --> Готовий" називається **витісненням (preemption)** — ОС примусово забирає процесор у процесу.

## 7. Розширена модель станів (7 станів)

### 7.1. Навіщо потрібна розширена модель

У реальних системах оперативна пам'ять обмежена. Якщо занадто багато процесів перебуває в стані "Готовий" або "Очікування", пам'яті може не вистачити. Тоді ОС **вивантажує (swap out)** деякі процеси з оперативної пам'яті на диск (у файл підкачки або swap-розділ). Так з'являються два додаткових стани.

### 7.2. Сім станів процесу

| Стан | Де знаходиться | Опис |
|------|-----------------|------|
| **Новий (New)** | Вторинна пам'ять | Процес щойно створений |
| **Готовий (Ready)** | Оперативна пам'ять | Чекає на процесор |
| **Виконується (Running)** | Оперативна пам'ять | Виконується процесором |
| **Очікування (Blocked)** | Оперативна пам'ять | Чекає на подію / I/O |
| **Призупинений готовий (Suspended Ready)** | Вторинна пам'ять (диск) | Готовий, але вивантажений з RAM на диск |
| **Призупинений заблокований (Suspended Blocked)** | Вторинна пам'ять (диск) | Заблокований і вивантажений на диск |
| **Завершений (Terminated)** | -- | Виконання завершено |

### 7.3. Діаграма розширеної моделі

```
                          ┌─────────┐
                          │  НОВИЙ  │
                          └────┬────┘
                               │ допуск
                               ▼
                          ┌─────────┐     dispatch    ┌──────────────┐
              ┌───────────│ ГОТОВИЙ  │───────────────> │ ВИКОНУЄТЬСЯ  │
   timeout    │           │ (Ready)  │                 │  (Running)   │
              │           └────┬─────┘                 └──┬──┬───────┘
              │                │ ▲                        │  │  │
              └────────────────┘ │                        │  │  │
                                 │ завантаження           │  │  │exit
                      ┌──────────┘ (swap in)              │  │  │
                      │                                   │  │  ▼
              ┌───────┴────────┐            I/O запит     │  │ ┌───────────┐
              │  ПРИЗУПИНЕНИЙ  │ <────────────┐          │  │ │ЗАВЕРШЕНИЙ │
              │    ГОТОВИЙ     │              │          │  │ └───────────┘
              │(Susp. Ready)  │              │          │  │
              └───────┬────────┘              │          │  │
                      │                       │          │  │
                      │ swap out         ┌────┴─────┐   │  │
                      │ ◄────────────    │ОЧІКУВАННЯ│<──┘  │
                      │                  │(Blocked) │      │
                      │                  └────┬─────┘      │
                      │                       │ swap out   │
                      │                       ▼            │
              ┌───────┴────────┐     ┌───────────────┐     │
              │  подія сталася │     │  ПРИЗУПИНЕНИЙ  │     │
              │  ─────────────>│     │  ЗАБЛОКОВАНИЙ  │     │
              │                │     │(Susp. Blocked) │     │
              └────────────────┘     └────────────────┘     │
```

### 7.4. Додаткові переходи у розширеній моделі

| Перехід | Причина |
|---------|---------|
| **Готовий --> Призупинений готовий** | Не вистачає RAM — ОС вивантажує процес з пам'яті на диск (swap out) |
| **Призупинений готовий --> Готовий** | RAM звільнилась — ОС повертає процес з диска в пам'ять (swap in) |
| **Очікування --> Призупинений заблокований** | Не вистачає RAM — ОС вивантажує заблокований процес на диск |
| **Призупинений заблокований --> Призупинений готовий** | Подія, на яку чекав процес, відбулася (але процес залишається на диску) |

Рішення про вивантаження/завантаження процесів приймає **середньостроковий планувальник (medium-term scheduler)**.

## 8. Черги процесів

### 8.1. Що таке черги процесів

ОС організовує процеси у **черги**, використовуючи зв'язані списки PCB. Кожна черга відповідає певному стану процесів.

### 8.2. Основні черги

| Черга | Що містить | Призначення |
|-------|-----------|-------------|
| **Job Queue** (черга завдань) | Усі процеси, що існують у системі | Вхідна точка для всіх процесів |
| **Ready Queue** (черга готових) | Процеси, що завантажені в RAM і готові до виконання | Планувальник обирає з цієї черги наступний процес для CPU |
| **Device Queue** (черга пристрою / I/O Queue) | Процеси, що чекають на конкретний пристрій введення-виведення | Для кожного пристрою (диск, принтер тощо) своя окрема черга |

### 8.3. Рух процесів між чергами

```
                                  ┌───────────────┐
   Новий процес ──────────────>  │  Ready Queue   │
                                  │ [P3]-[P7]-[P1] │
                                  └───────┬────────┘
                                          │ dispatch
                                          ▼
                                  ┌───────────────┐
                                  │     C P U     │
                                  │   (Running)   │
                                  └──┬─────┬───┬──┘
                          timeout │  │     │   │ exit --> Завершений
                                  │  │     │
                 ┌────────────────┘  │     └──────────────────┐
                 │ (назад у Ready)   │ (запит I/O)            │
                 │                   ▼                        │
                 │         ┌──────────────────┐               │
                 │         │ Disk I/O Queue   │               │
                 │         │ [P4]-[P9]        │               │
                 │         └──────────────────┘               │
                 │         ┌──────────────────┐               │
                 │         │ Printer Queue    │               │
                 │         │ [P2]             │               │
                 │         └──────────────────┘               │
                 │                   │                        │
                 │                   │ I/O завершено          │
                 │                   ▼                        │
                 └───────── Ready Queue <─────────────────────┘
```

### 8.4. Типи планувальників

| Планувальник | Інша назва | Функція |
|-------------|-----------|---------|
| **Довгостроковий** | Job Scheduler | Вирішує, які процеси допустити з Job Queue до Ready Queue (контролює кількість процесів у пам'яті) |
| **Короткостроковий** | CPU Scheduler / Dispatcher | Вибирає процес з Ready Queue і передає йому процесор. Працює дуже часто (кожні кілька мілісекунд) |
| **Середньостроковий** | Swapper | Вивантажує процеси з RAM на диск (swap out) і повертає назад (swap in) для керування пам'яттю |

## 9. Ідентифікація процесів — PID та PPID

### 9.1. PID — ідентифікатор процесу

**PID (Process ID)** — це унікальне додатне ціле число, яке ОС присвоює кожному процесу при його створенні. PID дозволяє однозначно ідентифікувати процес у системі.

- PID є унікальним серед усіх запущених процесів
- Після завершення процесу його PID може бути використаний повторно
- У Linux перший процес (`init` або `systemd`) завжди має PID = 1
- У Windows процес `System` має PID = 4

### 9.2. PPID — ідентифікатор батьківського процесу

**PPID (Parent Process ID)** — це PID процесу, який створив (породив) даний процес.

Процеси утворюють **ієрархію (дерево)**: кожен процес (крім першого) має батьківський процес.

```
  systemd (PID 1)
     ├── sshd (PID 520, PPID 1)
     │     └── bash (PID 3401, PPID 520)
     │           └── python (PID 3450, PPID 3401)
     ├── apache2 (PID 780, PPID 1)
     │     ├── apache2 (PID 781, PPID 780)
     │     └── apache2 (PID 782, PPID 780)
     └── cron (PID 640, PPID 1)
```

У цьому прикладі `bash` (PID 3401) є дочірнім процесом `sshd` (PID 520), а `python` (PID 3450) — дочірнім процесом `bash`.

### 9.3. Процеси-сироти та процеси-зомбі

- **Процес-сирота (orphan)** — процес, батько якого завершився раніше. Такий процес "усиновлюється" процесом init (PID 1).
- **Процес-зомбі (zombie)** — процес, що вже завершив виконання, але його PCB ще не видалений з таблиці процесів, бо батьківський процес не зчитав код завершення.

## 10. Перегляд процесів у різних ОС

### 10.1. Windows — Диспетчер завдань

**Як відкрити:** `Ctrl + Shift + Esc` або `Ctrl + Alt + Delete` --> "Диспетчер завдань"

Вкладка **"Процеси"** показує:
- Назву процесу
- Використання CPU, пам'яті, диска, мережі
- Стан процесу

Вкладка **"Подробиці"** (Details) показує:
- PID кожного процесу
- Ім'я користувача (власника)
- Використання пам'яті
- Опис процесу

### 10.2. Windows — командний рядок (CMD)

```cmd
tasklist
```

Виводить список усіх процесів з PID та використанням пам'яті.

```cmd
tasklist /V
```

Розширена інформація: стан, ім'я користувача, час CPU.

### 10.3. Linux — команда `ps`

```bash
ps -ef
```

Виводить усі процеси з полями: UID, PID, PPID, час запуску, команда.

```bash
ps aux
```

Альтернативний формат з відображенням використання CPU та пам'яті.

### 10.4. Linux — команда `top`

```bash
top
```

Інтерактивний монітор процесів у реальному часі. Показує:
- PID, користувача, пріоритет
- Використання CPU та пам'яті
- Стан процесу (R — running, S — sleeping, Z — zombie)
- Загальне навантаження на систему

Для виходу з `top` натисніть клавішу `q`.

### 10.5. Linux — команда `pstree`

```bash
pstree -p
```

Відображає ієрархію процесів у вигляді дерева з PID кожного процесу. Наочно показує зв'язок батько-дочірній процес.

## 11. Аналогії для розуміння

### 11.1. Рецепт vs приготування страви

| Поняття ОС | Кулінарна аналогія |
|------------|-------------------|
| **Програма** | Рецепт страви, записаний у книзі. Він лежить на полиці і нічого не робить |
| **Процес** | Процес приготування: кухар читає рецепт і виконує дії (ріже, змішує, варить) |
| **CPU (процесор)** | Кухар — він може готувати лише одну страву в один момент часу |
| **Контекст** | Запам'ятати, на якому кроці рецепта зупинився кухар, які інгредієнти вже використав |
| **Перемикання контексту** | Кухар відклав одну страву (запам'ятав крок), почав готувати іншу |
| **Стан "Очікування"** | Страва в духовці — кухар чекає 30 хвилин і поки що вільний для іншої роботи |
| **Стан "Готовий"** | Усі інгредієнти готові, кухар знає, що робити далі, але зайнятий іншою стравою |
| **Ресурси** | Інгредієнти, плита, духовка, каструлі — все, що потрібно для приготування |

### 11.2. Черга в поліклініці

| Поняття ОС | Аналогія з поліклінікою |
|------------|------------------------|
| **Ready Queue** | Черга пацієнтів біля кабінету лікаря. Усі готові зайти, але лікар приймає по одному |
| **Running** | Пацієнт на прийомі у лікаря |
| **I/O Queue** | Пацієнт пішов здавати аналізи — чекає результатів |
| **Blocked --> Ready** | Аналізи готові — пацієнт знову стає в чергу до лікаря |
| **Dispatcher** | Медсестра, яка викликає наступного пацієнта з черги |

## Глосарій

| Термін | Визначення |
|--------|-----------|
| **Процес (Process)** | Динамічний об'єкт, що описує виконання програми; програма, завантажена в пам'ять і виконувана процесором |
| **Програма (Program)** | Статичний набір інструкцій (код), записаний на диску; стає процесом лише після запуску |
| **Контекст процесу (Process Context)** | Сукупність інформації (лічильник команд, регістри, стек, дані), необхідної для відновлення виконання процесу |
| **Перемикання контексту (Context Switch)** | Операція збереження стану поточного процесу та завантаження стану іншого для передачі йому процесора |
| **PCB (Process Control Block)** | Блок керування процесом — структура даних в ядрі ОС, що зберігає всю інформацію про процес |
| **Таблиця процесів (Process Table)** | Масив (список) усіх PCB в системі; зберігається в пам'яті ядра |
| **PID (Process ID)** | Унікальний числовий ідентифікатор процесу, що присвоюється ОС при створенні процесу |
| **PPID (Parent Process ID)** | Ідентифікатор батьківського процесу — PID того процесу, що породив даний |
| **Ресурс (Resource)** | Будь-який об'єкт обчислювальної системи, який може бути розподілений між процесами (CPU, RAM, файли тощо) |
| **Апаратний ресурс** | Фізичний ресурс: процесор, оперативна пам'ять, пристрої введення-виведення |
| **Програмний ресурс** | Логічний ресурс: файли, семафори, черги повідомлень, сокети |
| **Стан процесу (Process State)** | Поточний етап життєвого циклу процесу: новий, готовий, виконується, очікує, завершений |
| **Новий (New)** | Стан процесу відразу після створення; ОС виділяє PCB, але виконання ще не розпочалось |
| **Готовий (Ready)** | Процес завантажений в RAM і готовий до виконання, чекає на процесор |
| **Виконується (Running)** | Процесор виконує інструкції цього процесу |
| **Очікування (Waiting / Blocked)** | Процес не може продовжити виконання, чекає на подію (I/O, дані тощо) |
| **Завершений (Terminated)** | Процес завершив виконання; ОС звільняє ресурси та видаляє PCB |
| **Призупинений (Suspended)** | Процес вивантажений з оперативної пам'яті на диск через нестачу RAM |
| **Витіснення (Preemption)** | Примусове забирання процесора у процесу операційною системою (наприклад, після вичерпання кванту часу) |
| **Квант часу (Time Slice / Quantum)** | Максимальний інтервал часу, протягом якого процес може безперервно використовувати процесор |
| **Планувальник (Scheduler)** | Компонент ОС, що вирішує, якому процесу передати процесор і коли |
| **Диспетчер (Dispatcher)** | Компонент ОС, що безпосередньо виконує перемикання контексту та передачу керування обраному процесу |
| **Ready Queue** | Черга процесів, готових до виконання; планувальник обирає з неї наступний процес |
| **I/O Queue (Device Queue)** | Черга процесів, що чекають на завершення операції введення-виведення конкретного пристрою |
| **Swapping** | Вивантаження процесу з оперативної пам'яті на диск (swap out) та повернення назад (swap in) |
| **task_struct** | Структура даних у ядрі Linux, що реалізує PCB |
| **EPROCESS** | Структура даних у ядрі Windows, що реалізує PCB |
| **Процес-зомбі (Zombie)** | Процес, що завершив виконання, але його запис ще не видалений з таблиці процесів |
| **Процес-сирота (Orphan)** | Процес, батьківський процес якого завершився раніше |
| **Лічильник команд (Program Counter)** | Регістр процесора, що зберігає адресу наступної інструкції для виконання |
