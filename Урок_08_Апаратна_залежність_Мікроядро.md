## 1. Вступ

На попередніх уроках ми вивчили, що таке ядро операційної системи, які бувають типи ядер та які функції вони виконують. Сьогодні ми глибше розглянемо два важливих питання:

- **Чому одні ОС працюють лише на певному обладнанні, а інші — на багатьох платформах?** Це питання апаратної залежності та переносимості.
- **Як побудувати ядро ОС так, щоб воно було надійним, безпечним і легко переносимим?** Це питання мікроядерної архітектури.

Ці теми безпосередньо стосуються реальної роботи з комп'ютерами: коли ви встановлюєте Windows на новий процесор або запускаєте Linux на Raspberry Pi, ви стикаєтеся з наслідками тих рішень, які прийняли розробники ОС щодо апаратної залежності та архітектури ядра.

## 2. Апаратна залежність операційних систем

### 2.1. Що означає апаратна залежність

**Апаратна залежність** — це зв'язок між операційною системою та конкретним апаратним забезпеченням (процесором, чипсетом, контролерами), на якому вона здатна працювати.

Кожен процесор має свою **архітектуру набору команд** (ISA — Instruction Set Architecture) — набір інструкцій, які він розуміє. Наприклад:

| Архітектура | Де використовується | Приклади процесорів |
|---|---|---|
| **x86 / x86-64** | Настільні ПК, ноутбуки, сервери | Intel Core, AMD Ryzen |
| **ARM / AArch64** | Смартфони, планшети, вбудовані системи | Apple M1/M2/M3, Qualcomm Snapdragon |
| **RISC-V** | Вбудовані системи, дослідження | SiFive, Kendryte |
| **MIPS** | Маршрутизатори, ігрові консолі | MIPS R4000 |

Операційна система **повинна "розуміти" мову процесора**, тобто містити код, написаний для конкретної ISA. Програма, скомпільована для x86, не запуститься на ARM — це різні "мови" на рівні машинного коду.

**Аналогія:** Уявіть, що процесор — це людина, яка розуміє тільки одну мову. Процесор x86 "говорить" англійською, ARM — японською. ОС повинна "говорити" тією мовою, яку розуміє конкретний процесор.

Крім набору команд процесора, апаратна залежність включає:
- **Організацію пам'яті** — як процесор працює з оперативною пам'яттю (порядок байтів, розмір сторінки).
- **Систему переривань** — як апаратура повідомляє процесор про події.
- **Контролери введення-виведення** — як підключені периферійні пристрої.

---

### 2.2. HAL — рівень абстракції апаратури

**HAL (Hardware Abstraction Layer)** — це програмний шар між ядром ОС та апаратним забезпеченням, який приховує деталі конкретного обладнання та надає ядру однаковий інтерфейс для роботи з різною апаратурою.

```
┌─────────────────────────────────────────────────┐
│          ПРИКЛАДНІ ПРОГРАМИ                      │
├─────────────────────────────────────────────────┤
│          ЯДРО ОС (основна логіка)                │
│   Керування процесами, пам'яттю, файлами        │
│        (платформо-НЕЗАЛЕЖНИЙ код)                 │
├─────────────────────────────────────────────────┤
│      HAL — РІВЕНЬ АБСТРАКЦІЇ АПАРАТУРИ           │
│   Уніфікований інтерфейс для роботи             │
│   з різними процесорами та платами               │
│        (платформо-ЗАЛЕЖНИЙ код)                   │
├─────────────────────────────────────────────────┤
│          АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                    │
│     (x86, ARM, RISC-V, MIPS тощо)               │
└─────────────────────────────────────────────────┘
```

**Принцип роботи HAL:**

HAL надає набір стандартних функцій ("прочитати дані з порту", "налаштувати таймер", "обробити переривання"), які ядро викликає однаково, незалежно від того, на якому процесорі працює система. Усередині HAL ці функції реалізовані по-різному для кожної платформи, але ядро цього "не бачить".

**Аналогія:** HAL — це як перекладач-синхроніст. Доповідач (ядро) говорить однією мовою, а перекладач (HAL) перекладає для різних слухачів (процесорів) їхньою мовою.

**Приклади HAL у реальних ОС:**

| ОС | Реалізація HAL | Деталі |
|---|---|---|
| **Windows NT** | Файл `hal.dll` | Окремий HAL для кожного типу материнської плати |
| **Linux** | Каталог `arch/` у вихідному коді | Окрема папка для кожної архітектури (arch/x86, arch/arm, arch/riscv) |
| **Android** | Project Treble (з версії 8.0) | HAL відокремлює драйвери виробників від ОС |
| **NetBSD** | bus_space, bus_dma | Чиста абстракція шин введення-виведення |

### 2.3. Платформо-залежний та платформо-незалежний код

Код операційної системи поділяється на дві категорії:

**Платформо-залежний код** — частина ОС, яка написана для конкретної апаратної платформи та використовує специфічні інструкції процесора. Цей код необхідно переписувати при переході на нову архітектуру.

Що входить:
- Завантажувальний код (bootstrap).
- Код ініціалізації процесора.
- Управління таблицями сторінок пам'яті (MMU).
- Обробка апаратних переривань.
- Перемикання контексту процесів.
- Частина коду, написана на мові асемблера.

**Платформо-незалежний код** — частина ОС, яка працює однаково на будь-якій платформі та не використовує специфічних інструкцій конкретного процесора.

Що входить:
- Алгоритми планування процесів.
- Управління файловою системою.
- Мережевий стек (TCP/IP).
- Управління правами доступу та безпекою.
- Логіка управління пам'яттю (вище рівня MMU).

**Співвідношення в ядрі Linux:**

```
┌─────────────────────────────────────────────┐
│                                             │
│       Платформо-незалежний код               │
│           (приблизно 95%)                    │
│   Мова C, однаковий для всіх архітектур     │
│                                             │
├─────────────────────────────────────────────┤
│  Платформо-залежний код (~5%)               │
│  Каталог arch/: x86, ARM, RISC-V...        │
│  Частково на C, частково на Assembler       │
└─────────────────────────────────────────────┘
```

У добре спроєктованій ОС платформо-залежний код складає лише невеликий відсоток від загального обсягу. Наприклад, у ядрі Linux каталог `arch/` (де зберігається весь апаратно-залежний код) займає близько 5% від загального обсягу вихідного коду. Це дозволяє при перенесенні Linux на нову архітектуру переписати лише цей невеликий шар, залишивши 95% коду без змін.

### 2.4. Приклади апаратної залежності в реальних ОС

**Windows:**
- Спочатку працював лише на x86 (i386).
- Windows NT (1993) вже підтримував x86, MIPS, Alpha, PowerPC завдяки HAL.
- Сучасний Windows 11 офіційно підтримує x86-64 та ARM64 (наприклад, на пристроях Microsoft Surface з процесорами Qualcomm Snapdragon).

**Linux:**
- Підтримує найбільше апаратних архітектур серед усіх ОС.
- Станом на ядро версії 6.17 (2025) офіційно підтримується приблизно **22 архітектури**: x86, ARM, ARM64, RISC-V, MIPS, PowerPC, s390, SPARC, LoongArch та інші.
- Це досягається завдяки модульній структурі каталогу `arch/` та послідовному використанню мови C.

**macOS / iOS:**
- Apple здійснила два великих переходи: з PowerPC на Intel x86 (2006) та з Intel x86 на Apple Silicon ARM (2020).
- Переносимість забезпечувалася архітектурою ядра XNU, яке містить шар абстракції на базі мікроядра Mach.

## 3. Переносимість (портативність) ОС

### 3.1. Що таке переносимість і навіщо вона потрібна

**Переносимість (портативність)** операційної системи — це здатність ОС бути адаптованою для роботи на різних апаратних платформах з мінімальними змінами у вихідному коді.

**Навіщо потрібна переносимість:**

1. **Економія ресурсів.** Замість написання окремої ОС для кожного процесора, розробники переписують лише невелику частину коду.
2. **Розширення ринку.** Переносима ОС може працювати на різних пристроях: від серверів до смартфонів і вбудованих систем.
3. **Захист інвестицій.** Програми, написані для переносимої ОС, працюватимуть на нових апаратних платформах.
4. **Довговічність.** Апаратура змінюється швидко (нові процесори з'являються кожні 2-3 роки), а переносима ОС може адаптуватися до нового обладнання.

**Аналогія:** Переносимість — це як рецепт страви, записаний зрозумілою мовою. Якщо рецепт написаний універсально ("200 грам борошна, 2 яйця"), його можна приготувати на будь-якій кухні. Якщо рецепт прив'язаний до конкретної плити ("натисніть третю кнопку зліва на плиті Samsung моделі X"), він марний на іншій кухні.

---

### 3.2. Як досягається переносимість

Є три основні механізми досягнення переносимості ОС:

**1) Використання мов високого рівня (насамперед C)**

Мова C дозволяє писати код, який може бути скомпільований для будь-якого процесора за допомогою відповідного компілятора. На відміну від мови асемблера, де кожна інструкція прив'язана до конкретного процесора, код на C є абстрактним описом алгоритму.

```
Мова асемблера (x86):          Мова C (переносимий код):
  MOV EAX, [counter]             counter = counter + 1;
  ADD EAX, 1
  MOV [counter], EAX

Мова асемблера (ARM):
  LDR R0, [R1]
  ADD R0, R0, #1
  STR R0, [R1]
```

Одна й та сама операція "збільшити лічильник на 1" на асемблері виглядає по-різному для x86 та ARM, але на мові C записується однаково. Компілятор сам перекладає код C у потрібний машинний код.

**2) Використання HAL (розглянуто в розділі 2.2)**

HAL ізолює апаратно-залежний код в окремому шарі, дозволяючи основному коду ядра залишатися платформо-незалежним.

**3) Дотримання стандартів (POSIX та інші)**

Стандарти визначають єдиний інтерфейс, який мають реалізувати всі сумісні ОС. Це забезпечує переносимість не лише самої ОС, але й прикладних програм.

---

### 3.3. Історична подія: Unix переписаний на C (1973)

Одна з найважливіших подій в історії операційних систем:

- **1969-1970** — Кен Томпсон (Ken Thompson) та Денніс Рітчі (Dennis Ritchie) створили Unix у лабораторії Bell Labs. Перша версія була написана **мовою асемблера** для комп'ютера PDP-7, а потім переписана для PDP-11.
- **1972-1973** — Денніс Рітчі створив мову програмування **C** (розвиток мови B), і Unix був **переписаний на мові C**.
- Це зробило Unix **першою переносимою операційною системою** в історії.

**Чому це було революцією:**

До 1973 року кожна ОС писалася мовою асемблера під конкретний комп'ютер. Якщо з'являвся новий комп'ютер, ОС потрібно було писати заново з нуля. Переписавши Unix на C, розробники отримали можливість перенести систему на новий комп'ютер, лише написавши **компілятор C для цього комп'ютера** та невеликий шар платформо-залежного коду.

**Результат:** Unix швидко поширився на десятки різних комп'ютерів (VAX, IBM, Sun та інші), що привело до створення цілого сімейства Unix-подібних систем, включаючи сучасні Linux, macOS, FreeBSD та Android.

---

### 3.4. Стандарт POSIX

**POSIX (Portable Operating System Interface)** — це сімейство стандартів, розроблених організацією IEEE, що визначають єдиний інтерфейс операційної системи для забезпечення сумісності між різними ОС.

**Історія виникнення:**

У 1980-х роках існувало багато різних версій Unix (System V, BSD, HP-UX, AIX, SunOS), які були несумісні між собою. Програма, написана для однієї версії Unix, часто не працювала на іншій. Стандарт POSIX був створений, щоб вирішити цю проблему. Назву "POSIX" запропонував **Річард Столлман** (Richard Stallman), засновник проєкту GNU.

**Що визначає POSIX:**

| Область | Що стандартизується | Приклади |
|---|---|---|
| **API системних викликів** | Функції для роботи з процесами, файлами, пам'яттю | `fork()`, `open()`, `read()`, `write()` |
| **Командна оболонка** | Синтаксис командного інтерпретатора | Мова shell-скриптів (`/bin/sh`) |
| **Утиліти** | Набір стандартних команд | `ls`, `cp`, `grep`, `awk`, `make` |
| **Потоки (threads)** | Інтерфейс багатопотоковості | Бібліотека pthreads |

**Які ОС підтримують POSIX:**

- **Повністю сертифіковані:** macOS, HP-UX, AIX, Solaris.
- **Переважно сумісні (без сертифікації):** Linux, FreeBSD, OpenBSD, Android.
- **Часткова підтримка:** Windows (через WSL — Windows Subsystem for Linux).

**Практичне значення:** Завдяки POSIX програміст може написати програму на одній Unix-подібній системі, і вона (з мінімальними змінами або без них) буде працювати на іншій. Наприклад, скрипт, написаний для Linux, як правило, працюватиме і на macOS, тому що обидві системи дотримуються стандарту POSIX.

## 4. Мікроядерна архітектура — детальний розгляд

### 4.1. Принцип мікроядерної архітектури

**Мікроядерна архітектура** — це підхід до побудови ОС, при якому в ядрі залишається лише абсолютний мінімум функцій, а всі інші сервіси ОС працюють як окремі процеси в просторі користувача.

**Головна ідея:** "Зробити ядро якомога меншим, а все інше винести назовні."

Порівняємо з монолітним ядром, яке ми вивчали раніше:

```
МОНОЛІТНЕ ЯДРО (наприклад, Linux):
┌──────────────────────────────────────────────┐
│              ПРОСТІР КОРИСТУВАЧА              │
│     [Програма 1]  [Програма 2]  [Програма 3] │
├──────────────────────────────────────────────┤
│              РЕЖИМ ЯДРА                       │
│  ┌──────────────────────────────────────────┐│
│  │ Планування │ Пам'ять │ Файлова система  ││
│  │ Драйвери   │ Мережа  │ Безпека          ││
│  │ IPC        │ I/O     │ ... все разом     ││
│  └──────────────────────────────────────────┘│
├──────────────────────────────────────────────┤
│              АПАРАТУРА                        │
└──────────────────────────────────────────────┘

МІКРОЯДРО (наприклад, QNX):
┌──────────────────────────────────────────────┐
│              ПРОСТІР КОРИСТУВАЧА              │
│  [Програма 1]  [Програма 2]  [Програма 3]    │
│                                              │
│  [Сервер        [Сервер     [Сервер          │
│   файлової       драйверів]  мережі]         │
│   системи]                                   │
├──────────────────────────────────────────────┤
│              РЕЖИМ ЯДРА (мінімум!)            │
│  ┌──────────────────────────────────────────┐│
│  │ IPC │ Базове планування │ Базова пам'ять ││
│  └──────────────────────────────────────────┘│
├──────────────────────────────────────────────┤
│              АПАРАТУРА                        │
└──────────────────────────────────────────────┘
```

Як бачимо, у мікроядерній архітектурі ядро стає набагато меншим, а файлова система, драйвери та мережа працюють як окремі сервери в просторі користувача.

### 4.2. Що входить у мікроядро

Мікроядро містить лише **мінімально необхідний** набір функцій, без яких система не може працювати взагалі:

| Функція | Опис | Навіщо обов'язково в ядрі |
|---|---|---|
| **Міжпроцесна комунікація (IPC)** | Механізм передачі повідомлень між процесами | Усі сервери спілкуються через IPC — це "кровоносна система" мікроядра |
| **Базове планування процесів** | Розподіл часу процесора між потоками | Хтось повинен вирішувати, хто працює зараз, а хто чекає |
| **Базове управління пам'яттю** | Управління адресними просторами та таблицями сторінок | Ізоляція процесів одне від одного — основа безпеки |
| **Обробка переривань** | Прийом і маршрутизація апаратних переривань | Переривання — апаратний механізм, потребує привілейованого режиму |

**Типовий розмір мікроядра:** від 10 000 до 50 000 рядків коду.

Для порівняння: ядро Linux (монолітне) містить понад **30 мільйонів** рядків коду.

Цей принцип сформулював Йохен Лідтке (Jochen Liedtke), творець мікроядра L4: "Функція допускається всередині мікроядра лише тоді, коли її винесення назовні унеможливило б реалізацію необхідної функціональності системи."

---

### 4.3. Що виноситься в сервери користувацького простору

Усе, що не є абсолютно необхідним для роботи ядра, виноситься в окремі процеси-**сервери**, які працюють у просторі користувача:

| Сервер | Що робить | В монолітному ядрі |
|---|---|---|
| **Сервер файлової системи** | Читання/запис файлів, управління каталогами | Частина ядра |
| **Сервер драйверів пристроїв** | Взаємодія з апаратурою (диск, мережева карта, дисплей) | Частина ядра |
| **Мережевий сервер** | Реалізація TCP/IP, обробка мережевих пакетів | Частина ядра |
| **Сервер безпеки** | Перевірка прав доступу, аутентифікація | Частина ядра |
| **Сервер процесів** | Створення та завершення процесів | Частина ядра |
| **Сервер графіки** | Виведення зображення на екран | Частина ядра або модуль |

**Ключова відмінність:** Кожен сервер працює як звичайний процес з обмеженими правами. Якщо сервер "впаде" (зависне або зламається), решта системи продовжить працювати, і сервер можна перезапустити.

---

### 4.4. Механізм обміну повідомленнями (IPC)

**IPC (Inter-Process Communication)** — міжпроцесна комунікація — це головний механізм взаємодії в мікроядерній ОС. Оскільки сервіси працюють в окремих процесах, вони не можуть безпосередньо викликати функції один одного (як це робиться в монолітному ядрі). Натомість вони обмінюються **повідомленнями** через мікроядро.

**Як працює IPC — покроковий приклад:**

Уявімо, що програма хоче прочитати файл:

```
Крок 1: Програма формує повідомлення: "Прочитай файл /home/user/doc.txt"
        і відправляє його через мікроядро.

Крок 2: Мікроядро доставляє повідомлення серверу файлової системи.

Крок 3: Сервер файлової системи обробляє запит,
        відправляє повідомлення серверу драйвера диска:
        "Прочитай сектори 1024-1028 з диска"

Крок 4: Мікроядро доставляє повідомлення серверу драйвера диска.

Крок 5: Драйвер диска читає дані з апаратури.

Крок 6: Дані повертаються назад тим самим шляхом:
        Драйвер -> Файлова система -> Програма
        (кожен раз через мікроядро)
```

**Схема обміну повідомленнями:**

```
[Програма]                          [Сервер файлової системи]
    │                                        │
    │  1. Запит "прочитай файл"              │
    │──────────┐                             │
    │          ▼                             │
    │    ┌──────────┐                        │
    │    │МІКРОЯДРО │  2. Доставка запиту     │
    │    │  (IPC)   │───────────────────────► │
    │    └──────────┘                        │
    │          ▲                             │
    │          │  4. Доставка відповіді       │
    │◄─────────┘                             │
    │                  3. Відповідь "дані"    │
    │                 ◄──────────────────────│
```

**Порівняння з монолітним ядром:**

У монолітному ядрі програма робить один системний виклик, і ядро всередині себе одразу звертається до файлової системи та драйвера диска — все відбувається в одному адресному просторі, без передачі повідомлень. Це швидше, але менш безпечно.

У мікроядрі кожен крок вимагає перемикання між процесами та копіювання даних, що створює додаткові витрати часу (overhead).

### 4.5. Переваги мікроядерної архітектури

**1. Надійність (стійкість до збоїв)**

Кожен сервер ізольований у власному адресному просторі. Помилка в драйвері або файловій системі не призводить до краху всієї системи (на відміну від монолітного ядра, де помилка в будь-якому модулі може викликати "синій екран" або kernel panic).

> Приклад: У QNX, якщо мережевий драйвер зависне, ядро може автоматично його перезапустити, і користувач навіть не помітить збою. У Linux помилка в мережевому драйвері може "покласти" всю систему.

**2. Безпека**

Оскільки сервери працюють з обмеженими правами (в просторі користувача), навіть якщо зловмисник зламає один сервер, він не отримає повного контролю над системою. Мінімальний розмір ядра означає мінімальну "поверхню атаки".

**3. Модульність**

Сервери можна додавати, видаляти, замінювати та оновлювати незалежно один від одного, без перекомпіляції або перезавантаження ядра. Потрібна нова файлова система? Просто запустіть новий сервер.

**4. Переносимість**

Маленьке ядро легше перенести на нову апаратну платформу: потрібно адаптувати лише код мікроядра (10-50 тисяч рядків), а не мільйони рядків монолітного ядра.

**5. Тестованість і верифікація**

Маленький розмір мікроядра дозволяє формально верифікувати (математично довести) правильність його роботи. Наприклад, мікроядро **seL4** є першим у світі ядром ОС, для якого математично доведена відсутність помилок.

---

### 4.6. Недоліки мікроядерної архітектури

**1. Зниження продуктивності через IPC**

Головний недолік: передача повідомлень між процесами значно повільніша, ніж прямий виклик функції всередині ядра.

Кожен запит до сервісу ОС вимагає:
- Перемикання контексту з програми в ядро.
- Копіювання повідомлення.
- Перемикання контексту з ядра в сервер.
- Обробку запиту.
- Зворотний шлях (ще 2-3 перемикання контексту).

У монолітному ядрі — один системний виклик і пряма робота всередині ядра.

```
Монолітне ядро:   Програма ──> Ядро (1 перемикання) ──> Програма
                  Разом: 2 перемикання контексту

Мікроядро:        Програма ──> Ядро ──> Сервер ──> Ядро ──> Програма
                  Разом: 4+ перемикання контексту
```

**2. Складність розробки**

Розробляти систему з обміном повідомленнями складніше, ніж з прямими викликами функцій. Розробникам потрібно проєктувати протоколи повідомлень, обробляти помилки доставки тощо.

**3. Складність налагодження**

Розподілена система складніша для налагодження (debugging): помилка може виникати через взаємодію кількох серверів, і відстежити її ланцюжок непросто.

## 5. Приклади мікроядерних ОС

### MINIX

- Створена **Ендрю Таненбаумом** (Andrew Tanenbaum) у 1987 році як навчальна ОС.
- Мікроядро розміром близько 13 000 рядків коду.
- MINIX 3 (2005) — серйозна переробка з акцентом на надійність: драйвери працюють як ізольовані процеси, система може автоматично перезапускати "впавші" драйвери.
- Цікавий факт: з 2008 року MINIX працює всередині **Intel Management Engine** (IME) — окремого мікропроцесора, вбудованого в кожен процесор Intel. Тобто MINIX, ймовірно, найбільш розповсюджена ОС у світі, хоча більшість людей про це навіть не знають!

### QNX

- Розроблена компанією **BlackBerry** (раніше QNX Software Systems).
- Одна з перших **комерційно успішних** мікроядерних ОС (з 1982 року).
- **Операційна система реального часу (RTOS)** — гарантує виконання операцій у строго визначений час.
- Мікроядро QNX Neutrino: лише базове планування, IPC та обробка переривань.
- Використовується в критично важливих системах: автомобільна електроніка (понад **195 мільйонів** автомобілів), медичне обладнання, ядерні електростанції, системи управління повітряним рухом, маршрутизатори Cisco.
- Якщо драйвер "впаде", QNX автоматично його перезапустить без перезавантаження системи.

### L4 (сімейство мікроядер)

- Серія мікроядер, розроблена Йохеном Лідтке (Jochen Liedtke) з 1993 року.
- Мета: довести, що мікроядро може бути **швидким** (Лідтке вирішив проблему повільного IPC першого покоління мікроядер).
- **seL4** — сучасний нащадок L4, перше у світі ядро ОС з **формальною верифікацією** (математичний доказ правильності коду). Використовується в аерокосмічній галузі та військових системах.
- Розмір мікроядра seL4: приблизно **10 000 рядків коду** мовою C.

### GNU Hurd

- Офіційне ядро проєкту **GNU**, ініційоване Річардом Столлманом.
- Розробка почалася в **1990 році**, але до сьогодні ОС так і не досягла стабільного стану.
- Побудоване на мікроядрі **GNU Mach** (нащадок Mach від Carnegie Mellon University).
- Є класичним прикладом того, що мікроядерна архітектура складніша в реалізації. Саме затримки з GNU Hurd стали причиною, чому Лінус Торвальдс почав створювати Linux: "Якби ядро GNU було готове минулої весни, я б навіть не почав свій проєкт."

### Google Fuchsia (мікроядро Zircon)

- Сучасна ОС від **Google**, побудована "з нуля" (не на базі Linux).
- Використовує мікроядро **Zircon**, написане переважно мовою **C++** (на відміну від більшості ядер, написаних на C).
- Zircon побудований на принципі **об'єктних можливостей (capability-based security)**: програми мають доступ лише до тих ресурсів, на які їм явно надано дозвіл.
- Вперше випущена у 2021 році на пристрої Google Nest Hub.
- Остання стабільна версія — **F28**.
- Підтримує архітектури x86-64 та ARM64.
- Google також розробляє **Microfuchsia** — мінімальну версію для інтеграції з Android-пристроями.

## 6. Порівняння монолітного ядра та мікроядра

| Критерій | Монолітне ядро | Мікроядро |
|---|---|---|
| **Розмір ядра** | Великий (мільйони рядків коду) | Маленький (тисячі — десятки тисяч рядків) |
| **Де працюють сервіси ОС** | Усі в режимі ядра | Більшість у просторі користувача |
| **Продуктивність** | Висока (прямі виклики функцій) | Нижча (обмін повідомленнями через IPC) |
| **Надійність** | Помилка в будь-якому модулі може "покласти" систему | Збій сервера не руйнує систему, сервер можна перезапустити |
| **Безпека** | Велика поверхня атаки | Мала поверхня атаки (маленьке ядро) |
| **Модульність** | Обмежена (завантажувані модулі в Linux) | Висока (незалежні сервери) |
| **Переносимість** | Складніша (великий обсяг коду) | Легша (малий обсяг залежного коду) |
| **Складність розробки** | Простіша архітектура, але складніше налагодження | Складніша архітектура, розподілена система |
| **Швидкість завантаження** | Швидше (все одразу в пам'яті) | Повільніше (потрібно запустити всі сервери) |
| **Приклади ОС** | Linux, FreeBSD, класичний Unix | QNX, MINIX 3, seL4, GNU Hurd, Fuchsia |

## 7. Дебати Торвальдс vs Таненбаум (1992)

Одна з найвідоміших дискусій в історії комп'ютерних наук відбулася **29 січня 1992 року** в Usenet-групі `comp.os.minix`.

### Учасники

- **Ендрю Таненбаум** — професор Амстердамського вільного університету, автор підручників з ОС та творець MINIX. Прихильник мікроядерної архітектури.
- **Лінус Торвальдс** — на той момент 22-річний студент з Фінляндії, автор ядра Linux. Прихильник монолітної архітектури (з практичних міркувань).

### Суть дискусії

**Таненбаум** опублікував повідомлення з заголовком **"LINUX is obsolete"** ("Linux застарів"), де стверджував:
- Монолітне ядро — це "гігантський крок назад у 1970-ті роки".
- Серед фахівців, які проєктують ОС, дебати вже завершені: "мікроядра перемогли".
- Linux надто прив'язаний до архітектури x86 і не матиме майбутнього, коли ця архітектура буде замінена іншою.

**Торвальдс** відповів наступного дня:
- Визнав, що теоретично мікроядерна архітектура краща.
- Але наголосив на практичності: "Linux існує і працює зараз, а GNU Hurd (мікроядро) досі не готовий."
- Розкритикував MINIX за проблеми з багатозадачністю та обмеження файлової системи.
- Заявив: "Linux значно виграє в тому, що він доступний прямо зараз."

### Хто мав рацію?

Цікаво, що **обидві сторони виявилися частково правими**:

- **Торвальдс мав рацію** в практичному плані: Linux став домінуючою ОС для серверів, суперкомп'ютерів та Android-пристроїв. Монолітний підхід (з модулями, що завантажуються) виявився достатньо гнучким.
- **Таненбаум мав рацію** в теоретичному плані: мікроядра дійсно використовуються в критично важливих системах (QNX в автомобілях, seL4 в аерокосмічній галузі). А MINIX, як з'ясувалося у 2017 році, працює всередині кожного процесора Intel.
- **Таненбаум помилився** щодо x86: ця архітектура не зникла і залишається домінуючою на ринку ПК та серверів, хоча ARM справді набуває все більшого значення.

### Друга хвиля (2006)

У 2006 році дискусія відновилася після публікації Таненбаума. Торвальдс контраргументував: "Смішно, коли прихильники мікроядер стверджують, що їхня система 'простіша'. Це не так. Вона набагато складніша саме через бар'єри між структурами даних... Усі ваші алгоритми стають розподіленими алгоритмами."

### Значення дискусії

Ця дискусія залишається актуальною і сьогодні. Вона ілюструє фундаментальне протиріччя в інженерії: **теоретична елегантність проти практичної ефективності**. Сучасне рішення — гібридні ядра — є спробою знайти компроміс між цими двома підходами.

## 8. Гібридне ядро як компроміс

**Гібридне ядро** — це архітектура ядра, яка поєднує елементи монолітного ядра та мікроядра, намагаючись отримати переваги обох підходів.

**Принцип:** Структура нагадує мікроядро (модульність, окремі компоненти), але критично важливі сервіси працюють у режимі ядра (як у монолітному ядрі) для підвищення продуктивності.

```
ГІБРИДНЕ ЯДРО:
┌──────────────────────────────────────────────┐
│              ПРОСТІР КОРИСТУВАЧА              │
│  [Програми]  [Деякі підсистеми/сервіси]      │
├──────────────────────────────────────────────┤
│              РЕЖИМ ЯДРА                       │
│  ┌──────────────────────────────────────────┐│
│  │ Мікроядро  │ Файлова  │ Графіка │ Мережа ││
│  │ (IPC,      │ система  │         │        ││
│  │ планування,│          │         │        ││
│  │ пам'ять)   │          │         │        ││
│  └──────────────────────────────────────────┘│
├──────────────────────────────────────────────┤
│              АПАРАТУРА                        │
└──────────────────────────────────────────────┘
```

### Windows NT

- Ядро Windows NT (яке використовується у всіх версіях від Windows NT 3.1 до Windows 11) було створене під впливом мікроядра Mach з Carnegie Mellon University.
- **Мікроядерні риси:** підсистеми середовища (Win32, POSIX, OS/2) працюють як окремі процеси в просторі користувача.
- **Монолітні риси:** графічна підсистема (win32k.sys), файлові системи та більшість драйверів працюють у режимі ядра для підвищення продуктивності.
- HAL (`hal.dll`) забезпечує переносимість між різними платформами.

### macOS / iOS (ядро XNU)

- **XNU (X is Not Unix)** — ядро, розроблене компанією NeXT, а потім Apple.
- Поєднує мікроядро **Mach** (від Carnegie Mellon University) з компонентами **BSD Unix**.
- **Mach** відповідає за: управління пам'яттю, планування потоків, IPC.
- **BSD** відповідає за: файлові системи, мережевий стек, модель безпеки Unix.
- Обидва компоненти працюють в одному адресному просторі ядра (тому це не "чисте" мікроядро, а гібрид).

### Порівняння трьох підходів

| Характеристика | Монолітне | Мікроядро | Гібридне |
|---|---|---|---|
| **Продуктивність** | Найвища | Найнижча | Середня-висока |
| **Надійність** | Найнижча | Найвища | Середня |
| **Складність** | Середня | Висока | Висока |
| **Приклади** | Linux, FreeBSD | QNX, MINIX, seL4 | Windows NT, macOS XNU |

## 9. Підсумок

У цьому уроці ми розглянули три важливі теми:

1. **Апаратна залежність ОС** — кожна ОС повинна "розуміти мову" конкретного процесора. HAL (рівень абстракції апаратури) дозволяє ізолювати апаратно-залежний код і полегшити перенесення ОС на нові платформи.

2. **Переносимість ОС** — здатність ОС працювати на різних платформах з мінімальними змінами. Досягається використанням мов високого рівня (C), HAL та стандартів (POSIX). Переломним моментом став 1973 рік, коли Unix був переписаний на мові C.

3. **Мікроядерна архітектура** — підхід, при якому ядро містить лише мінімум функцій (IPC, планування, управління пам'яттю), а решта сервісів працює в просторі користувача. Це підвищує надійність та безпеку, але знижує продуктивність через IPC. Гібридні ядра (Windows NT, macOS XNU) намагаються поєднати переваги обох підходів.

Ці знання важливі для розуміння того, чому різні ОС ведуть себе по-різному, чому певні системи обираються для критично важливих завдань, і як розробники ОС балансують між продуктивністю, надійністю та переносимістю.

## 11. Глосарій ключових термінів

| Термін | Визначення |
|---|---|
| **Апаратна залежність** | Зв'язок між ОС та конкретним апаратним забезпеченням, що обмежує роботу ОС певними платформами |
| **ISA (Instruction Set Architecture)** | Архітектура набору команд — набір інструкцій, які розуміє конкретний процесор (x86, ARM, RISC-V тощо) |
| **HAL (Hardware Abstraction Layer)** | Рівень абстракції апаратури — програмний шар, що приховує деталі конкретного обладнання та надає ядру уніфікований інтерфейс |
| **Платформо-залежний код** | Частина коду ОС, яка прив'язана до конкретної апаратної платформи та потребує переписування при перенесенні |
| **Платформо-незалежний код** | Частина коду ОС, яка працює однаково на будь-якій платформі без змін |
| **Переносимість (портативність)** | Здатність ОС бути адаптованою для роботи на різних апаратних платформах з мінімальними змінами коду |
| **POSIX** | Portable Operating System Interface — сімейство стандартів IEEE, що визначають єдиний інтерфейс ОС для забезпечення сумісності |
| **Мікроядро (Microkernel)** | Архітектура ядра ОС, де в ядрі залишається лише мінімум функцій, а інші сервіси працюють у просторі користувача |
| **IPC (Inter-Process Communication)** | Міжпроцесна комунікація — механізм передачі повідомлень між процесами, основний спосіб взаємодії в мікроядерній ОС |
| **Монолітне ядро** | Архітектура ядра, де всі сервіси ОС працюють в одному адресному просторі в режимі ядра |
| **Гібридне ядро** | Архітектура ядра, що поєднує елементи монолітного ядра та мікроядра |
| **Сервер (у контексті мікроядра)** | Процес у просторі користувача, що надає певний сервіс ОС (файлова система, драйвер, мережа) |
| **Перемикання контексту** | Процес збереження стану одного процесу та відновлення стану іншого при передачі управління |
| **RTOS (Real-Time OS)** | Операційна система реального часу — ОС, що гарантує виконання операцій у строго визначений час |
| **Формальна верифікація** | Математичний доказ правильності програмного коду (відсутності помилок) |
| **Mach** | Мікроядро, розроблене в Carnegie Mellon University; є основою для macOS XNU та впливало на Windows NT |
| **Zircon** | Мікроядро Google Fuchsia OS, написане мовою C++, з моделлю безпеки на основі можливостей (capabilities) |
