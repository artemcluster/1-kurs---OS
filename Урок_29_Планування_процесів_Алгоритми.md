## 1. Навіщо потрібне планування процесів

У сучасному комп'ютері одночасно працюють десятки і сотні процесів: браузер, музичний плеєр, антивірус, оновлення системи тощо. Проте процесор (CPU) в кожну мить часу може виконувати лише **один** процес (на одному ядрі). Виникає питання: **як розподілити процесорний час між усіма процесами, щоб кожен отримав свою частку?**

Саме цим займається **планувальник процесів** (scheduler) -- спеціальний компонент ядра операційної системи. Його завдання:

- **Справедливо** розподіляти процесорний час між процесами.
- **Максимально завантажувати** CPU (щоб процесор не простоював).
- **Забезпечувати швидку відповідь** системи на дії користувача (інтерактивність).
- **Підвищувати пропускну здатність** -- виконувати якнайбільше завдань за одиницю часу.

> **Аналогія.** Уявіть чергу в магазині з одним касиром. Касир -- це CPU, покупці -- процеси. Як визначити, хто буде обслуговуватися першим? В порядку приходу? Спершу той, у кого менше товарів? Чи, можливо, пенсіонери мають пріоритет? Кожен варіант -- це окремий алгоритм планування.

## 2. Критерії ефективності планування

Щоб порівняти різні алгоритми планування, використовують такі **критерії**:

| Критерій | Опис | Бажане значення |
|---|---|---|
| **Використання CPU** (CPU Utilization) | Відсоток часу, протягом якого процесор зайнятий корисною роботою | Якнайвище (ідеал -- 100%) |
| **Пропускна здатність** (Throughput) | Кількість процесів, завершених за одиницю часу | Якнайвище |
| **Час обороту** (Turnaround Time) | Загальний час від моменту надходження процесу до його завершення | Якнайменше |
| **Час очікування** (Waiting Time) | Сумарний час, який процес провів у черзі готовності, очікуючи CPU | Якнайменше |
| **Час відгуку** (Response Time) | Час від надходження запиту до першої реакції системи | Якнайменше |

**Формули:**

- **Час обороту** = Час завершення - Час надходження
- **Час очікування** = Час обороту - Час виконання (burst time)
- **Середній час очікування** = Сума часів очікування всіх процесів / Кількість процесів

> **Важливо:** жоден алгоритм не може бути найкращим за всіма критеріями одночасно. Наприклад, алгоритм, що мінімізує середній час очікування, може бути несправедливим до деяких процесів.

## 3. Типи планування

Операційна система виконує планування на кількох рівнях:

### 3.1. Довгострокове планування (Long-term Scheduling)

- **Що робить:** вирішує, які процеси з пулу завдань допустити до виконання (завантажити в оперативну пам'ять).
- **Частота:** виконується рідко (секунди, хвилини).
- **Мета:** контролювати ступінь багатозадачності -- скільки процесів одночасно перебувають у пам'яті.
- **Приклад:** на сервері надходить 1000 запитів, але одночасно обробляється лише 50.

### 3.2. Середньострокове планування (Medium-term Scheduling)

- **Що робить:** тимчасово вивантажує процеси з пам'яті на диск (swapping) і повертає їх назад.
- **Частота:** виконується періодично.
- **Мета:** звільнити оперативну пам'ять, коли вона перевантажена.
- **Приклад:** процес, який давно не використовувався, вивантажується на диск, а коли знову потрібен -- повертається в пам'ять.

### 3.3. Короткострокове планування (Short-term Scheduling / Диспетчеризація)

- **Що робить:** вибирає, який саме процес із черги готовності отримає CPU наступним.
- **Частота:** виконується дуже часто (кожні кілька мілісекунд).
- **Мета:** ефективно розподілити процесорний час між готовими процесами.
- **Саме тут працюють алгоритми планування**, які ми розглянемо далі.

> **Диспетчер** (Dispatcher) -- це частина ОС, яка безпосередньо передає управління процесору обраному процесу. Диспетчер виконує перемикання контексту, перехід у режим користувача та стрибок на потрібну команду.

### Витісняюче vs. невитісняюче планування

- **Невитісняюче** (non-preemptive): процес утримує CPU, доки сам не звільнить його (завершиться або заблокується на вводі/виводі).
- **Витісняюче** (preemptive): ОС може примусово забрати CPU у процесу (наприклад, коли вичерпано квант часу або з'явився процес з вищим пріоритетом).

## 4. Алгоритми планування

### 4.1. FCFS -- First Come First Served

**Принцип:** процеси виконуються **в порядку їх надходження** -- хто прийшов першим, той і обслуговується першим. Реалізується за допомогою черги **FIFO** (First In, First Out).

**Характеристики:**
- Тип: **невитісняючий** (non-preemptive).
- Реалізація: найпростіша -- звичайна черга.
- Процес, що отримав CPU, виконується до повного завершення.

**Переваги:**
- Простий для розуміння та реалізації.
- Справедливий у тому сенсі, що всі обслуговуються в порядку приходу.

**Недоліки:**
- **Ефект конвою** (Convoy Effect) -- головна проблема FCFS.

> **Ефект конвою:** якщо першим у черзі стоїть процес з великим часом виконання, усі короткі процеси за ним змушені довго чекати. Це як повільна вантажівка на вузькій дорозі -- весь потік машин позаду гальмує.
>
> **Приклад:** процес P1 потребує 50 мс, а P2, P3, P4 -- по 2 мс кожен. Якщо P1 прийшов першим, то P2, P3 і P4 чекатимуть 50 мс, хоча їм потрібно лише 2 мс. Середній час очікування буде дуже високим.

- Середній час очікування зазвичай значно вищий, ніж у інших алгоритмів.
- Погано підходить для інтерактивних систем.

---

### 4.2. SJF -- Shortest Job First

**Принцип:** з черги готовності вибирається процес із **найменшим часом виконання** (burst time). Спершу виконуються короткі процеси, потім -- довші.

**Характеристики:**
- Тип: **невитісняючий** (у класичному варіанті).
- Оптимальний за критерієм мінімального середнього часу очікування (це математично доведено).

**Переваги:**
- Дає **мінімальний середній час очікування** серед усіх невитісняючих алгоритмів.
- Ефективний для пакетних систем, де час виконання відомий заздалегідь.

**Недоліки:**
- **Проблема голодування** (Starvation): довгі процеси можуть нескінченно чекати, якщо постійно надходять короткі.
- **Неможливість точного передбачення** burst time: ОС не завжди знає, скільки часу потребує процес. Використовуються евристичні методи оцінки (наприклад, експоненційне згладжування на основі попередніх burst time).

> **Аналогія:** у супермаркеті каса для покупців з невеликою кількістю товарів -- тих, хто купив 1-2 товари, пропускають першими.

---

### 4.3. SRTF -- Shortest Remaining Time First

**Принцип:** це **витісняюча версія SJF**. Якщо в чергу готовності надходить новий процес, чий час виконання **менший**, ніж залишковий час поточного процесу, то поточний процес витісняється, а новий отримує CPU.

**Характеристики:**
- Тип: **витісняючий** (preemptive).
- Дає ще менший середній час очікування, ніж SJF.

**Переваги:**
- Забезпечує мінімальний середній час очікування серед **усіх** алгоритмів планування.
- Більш гнучкий, ніж SJF, бо реагує на нові процеси.

**Недоліки:**
- Та ж **проблема голодування** довгих процесів.
- Більше **перемикань контексту** (context switches), що створює додаткові витрати.
- Складніший у реалізації -- потрібно постійно відстежувати залишковий час.

---

### 4.4. Round Robin (RR)

**Принцип:** кожному процесу виділяється **фіксований квант часу** (time quantum, зазвичай 10-100 мс). Процес виконується протягом свого кванта, а потім переривається і ставиться в кінець черги. Наступний процес отримує свій квант, і так по колу.

**Характеристики:**
- Тип: **витісняючий** (preemptive).
- Базується на FCFS, але з додаванням обмеження часу.
- Основний алгоритм для **систем розподілу часу** (time-sharing).

**Переваги:**
- **Справедливий** -- кожен процес гарантовано отримує процесорний час.
- **Добрий час відгуку** -- жоден процес не чекає довше, ніж (n-1) * квант, де n -- кількість процесів.
- Не виникає проблема голодування.

**Недоліки:**
- Середній час очікування зазвичай вищий, ніж у SJF.
- Продуктивність сильно залежить від **вибору кванта часу**:
  - **Занадто малий квант** -- надто багато перемикань контексту (великі накладні витрати).
  - **Занадто великий квант** -- алгоритм вироджується в FCFS.
  - **Оптимальний квант** -- зазвичай у 5-10 разів більший, ніж час перемикання контексту.

> **Аналогія:** діти по черзі грають на одній ігровій приставці -- кожному дають, наприклад, 5 хвилин, а потім передають контролер наступному.

**Приклад вибору кванта:**
- Квант = 1 мс, час перемикання контексту = 0.5 мс --> 33% часу витрачається на перемикання (неефективно!).
- Квант = 10 мс, час перемикання контексту = 0.5 мс --> лише 5% часу на перемикання (прийнятно).

---

### 4.5. Пріоритетне планування

**Принцип:** кожному процесу присвоюється **числовий пріоритет**. CPU отримує процес із **найвищим пріоритетом**. Процеси з однаковим пріоритетом обслуговуються за FCFS.

**Характеристики:**
- Тип: може бути **витісняючим** або **невитісняючим**.
- Пріоритет може визначатися різними факторами: важливість процесу, вимоги до пам'яті, час виконання тощо.

**Види пріоритетів:**

| Тип | Опис | Приклад |
|---|---|---|
| **Статичні** | Пріоритет задається при створенні процесу і не змінюється | Системні процеси мають вищий пріоритет, ніж користувацькі |
| **Динамічні** | Пріоритет змінюється під час виконання залежно від поведінки процесу | Процес, що довго чекає, отримує вищий пріоритет |

**Переваги:**
- Дозволяє виділяти більше ресурсів важливим процесам.
- Гнучкий механізм управління.

**Недоліки:**
- **Проблема голодування** (Starvation): процеси з низьким пріоритетом можуть нескінченно чекати.
- **Рішення -- старіння (Aging):** поступове підвищення пріоритету процесу, який довго перебуває в черзі. Наприклад, кожні 15 хвилин очікування пріоритет збільшується на 1. Рано чи пізно навіть найменш пріоритетний процес отримає CPU.

> **Приклад aging:** процес має початковий пріоритет 30 (низький). Через 15 хв очікування -- 29, через 30 хв -- 28, ... , і з часом він «доросте» до високого пріоритету і буде виконаний.

### 4.6. Багаторівневі черги (Multilevel Queue)

**Принцип:** черга готовності розділяється на **кілька окремих черг**, кожна з яких призначена для певного типу процесів. Кожна черга має свій власний алгоритм планування.

**Типова структура:**

```
Черга 1 (найвищий пріоритет): Системні процеси        --> FCFS
Черга 2: Інтерактивні процеси                          --> Round Robin
Черга 3: Процеси пакетної обробки                      --> SJF
Черга 4 (найнижчий пріоритет): Фонові процеси          --> FCFS
```

**Правила:**
- Процес потрапляє у відповідну чергу при створенні і **не може переміщуватися** між чергами.
- CPU спершу обслуговує всі процеси у черзі з найвищим пріоритетом, потім -- з нижчим, і т.д.
- Між чергами може бути додаткове планування (наприклад, 80% часу -- інтерактивним, 20% -- фоновим).

**Переваги:**
- Різні типи процесів обслуговуються алгоритмами, що найкраще підходять саме для них.

**Недоліки:**
- Негнучкий: процес назавжди прив'язаний до своєї черги.
- Процеси в нижніх чергах можуть голодувати.

---

### 4.7. Багаторівневі черги зі зворотним зв'язком (MLFQ)

**Принцип:** розширення багаторівневих черг, де процеси **можуть переміщуватися між чергами** залежно від їх поведінки. Це найскладніший, але й найадаптивніший алгоритм.

**Як працює MLFQ:**

1. **Новий процес** потрапляє в чергу з **найвищим пріоритетом** (ОС припускає, що він може бути коротким).
2. Якщо процес не завершується за відведений квант часу, він **переміщується на рівень нижче** (демонструє, що він довгий).
3. Якщо процес завершується або блокується на вводі/виводі до закінчення кванта, він **залишається на тому ж рівні** або підвищується (він інтерактивний).
4. Процеси, що довго перебувають на нижніх рівнях, **періодично підвищуються** (aging), щоб уникнути голодування.

**Приклад структури MLFQ з 3 рівнями:**

```
Рівень 1 (найвищий): квант = 8 мс,  алгоритм Round Robin
Рівень 2 (середній):  квант = 16 мс, алгоритм Round Robin
Рівень 3 (найнижчий): алгоритм FCFS
```

- Новий процес --> Рівень 1 (квант 8 мс).
- Не завершився за 8 мс --> переходить на Рівень 2 (квант 16 мс).
- Не завершився за 16 мс --> переходить на Рівень 3 (виконується за FCFS).

**Параметри MLFQ, які потрібно визначити:**
- Кількість рівнів (черг).
- Алгоритм планування для кожного рівня.
- Метод підвищення пріоритету (promotion).
- Метод зниження пріоритету (demotion).
- Правило, за яким новий процес потрапляє на певний рівень.

**Переваги:**
- **Адаптивний**: автоматично визначає тип процесу за його поведінкою.
- Не потрібно знати burst time заздалегідь (на відміну від SJF).
- Хороший баланс між часом відгуку та пропускною здатністю.
- Використовується в більшості сучасних ОС.

**Недоліки:**
- Складний у реалізації та налаштуванні.
- Можливе **обманювання планувальника** (gaming): процес може штучно звільняти CPU перед закінченням кванта, щоб залишатися на верхньому рівні.
- Можлива **інверсія пріоритетів**: процес з низьким пріоритетом утримує ресурс, потрібний процесу з високим пріоритетом.

## 6. Числові приклади з діаграмами Ганта

### Вхідні дані

Нехай маємо 4 процеси:

| Процес | Час надходження (Arrival Time) | Час виконання (Burst Time) |
|---|---|---|
| P1 | 0 | 8 |
| P2 | 1 | 4 |
| P3 | 2 | 2 |
| P4 | 3 | 1 |

---

### Приклад 1: FCFS

Процеси виконуються в порядку надходження: P1 --> P2 --> P3 --> P4.

**Діаграма Ганта:**

```
|   P1   |  P2  | P3 |P4|
0        8     12   14  15
```

**Розрахунки:**

| Процес | Час завершення | Час обороту (TAT) | Час очікування (WT) |
|---|---|---|---|
| P1 | 8  | 8 - 0 = 8  | 8 - 8 = 0  |
| P2 | 12 | 12 - 1 = 11 | 11 - 4 = 7 |
| P3 | 14 | 14 - 2 = 12 | 12 - 2 = 10 |
| P4 | 15 | 15 - 3 = 12 | 12 - 1 = 11 |

- **Середній час обороту** = (8 + 11 + 12 + 12) / 4 = **10.75 мс**
- **Середній час очікування** = (0 + 7 + 10 + 11) / 4 = **7.0 мс**

> Зверніть увагу: P3 і P4 (короткі процеси) чекали 10 і 11 мс відповідно, хоча їх виконання займає лише 2 і 1 мс. Це -- прояв ефекту конвою.

---

### Приклад 2: SJF (невитісняючий)

У момент часу 0 є лише P1 -- він починає виконуватися. Коли P1 завершується (t=8), у черзі вже є P2, P3, P4. Вибираємо найкоротший: P4(1) --> P3(2) --> P2(4).

**Діаграма Ганта:**

```
|   P1   |P4| P3 |  P2  |
0        8  9   11     15
```

**Розрахунки:**

| Процес | Час завершення | Час обороту (TAT) | Час очікування (WT) |
|---|---|---|---|
| P1 | 8  | 8 - 0 = 8  | 8 - 8 = 0  |
| P2 | 15 | 15 - 1 = 14 | 14 - 4 = 10 |
| P3 | 11 | 11 - 2 = 9  | 9 - 2 = 7  |
| P4 | 9  | 9 - 3 = 6  | 6 - 1 = 5  |

- **Середній час обороту** = (8 + 14 + 9 + 6) / 4 = **9.25 мс**
- **Середній час очікування** = (0 + 10 + 7 + 5) / 4 = **5.5 мс**

> У цьому прикладі SJF не показує повної переваги, бо P1 все одно виконується першим (він єдиний у момент 0). Але для P2, P3, P4 порядок оптимальний.

---

### Приклад 3: Round Robin (квант = 3 мс)

Процеси обслуговуються по черзі, кожен отримує не більше 3 мс за один раз.

**Покрокове виконання:**

| Час | Дія | Черга після дії |
|---|---|---|
| 0-3 | P1 виконується 3 мс (залишок: 5) | P2, P3, P4, P1 |
| 3-6 | P2 виконується 3 мс (залишок: 1) | P3, P4, P1, P2 |
| 6-8 | P3 виконується 2 мс (завершено) | P4, P1, P2 |
| 8-9 | P4 виконується 1 мс (завершено) | P1, P2 |
| 9-12 | P1 виконується 3 мс (залишок: 2) | P2, P1 |
| 12-13 | P2 виконується 1 мс (завершено) | P1 |
| 13-15 | P1 виконується 2 мс (завершено) | -- |

**Діаграма Ганта:**

```
| P1 | P2 |P3|P4| P1 |P2| P1|
0    3    6  8  9   12 13  15
```

**Розрахунки:**

| Процес | Час завершення | Час обороту (TAT) | Час очікування (WT) |
|---|---|---|---|
| P1 | 15 | 15 - 0 = 15 | 15 - 8 = 7  |
| P2 | 13 | 13 - 1 = 12 | 12 - 4 = 8  |
| P3 | 8  | 8 - 2 = 6   | 6 - 2 = 4   |
| P4 | 9  | 9 - 3 = 6   | 6 - 1 = 5   |

- **Середній час обороту** = (15 + 12 + 6 + 6) / 4 = **9.75 мс**
- **Середній час очікування** = (7 + 8 + 4 + 5) / 4 = **6.0 мс**

> Round Robin забезпечив найнижчий середній час очікування в цьому прикладі та дав кожному процесу шанс виконуватися, не допускаючи голодування.

---

### Порівняння результатів для наших прикладів

| Алгоритм | Сер. час обороту | Сер. час очікування |
|---|---|---|
| FCFS | 10.75 | 7.0 |
| SJF | 9.25 | 5.5 |
| RR (q=3) | 9.75 | 6.0 |

SJF показав найкращий середній час очікування, Round Robin -- трохи гірше, але забезпечив справедливість, а FCFS -- найгірший результат через ефект конвою.

## 7. Алгоритми планування в реальних ОС

### Linux: CFS (Completely Fair Scheduler)

**CFS** (Абсолютно справедливий планувальник) використовувався як стандартний планувальник у Linux з версії 2.6.23 (2007) до версії 6.6 (2023), де його замінив планувальник EEVDF.

**Ключові ідеї CFS:**
- **Немає фіксованих квантів часу.** Замість цього CFS намагається моделювати «ідеальний багатозадачний процесор», де кожен процес отримує рівну частку CPU.
- **Віртуальний час виконання (vruntime):** кожному процесу відстежується «віртуальний час», протягом якого він використовував CPU. Процес з **найменшим vruntime** отримує CPU наступним.
- **Червоно-чорне дерево:** процеси організовані у самобалансуюче дерево пошуку. Процес з найменшим vruntime завжди знаходиться у крайньому лівому вузлі (пошук за O(log N)).
- **Nice value:** значення nice (-20 до +19) впливає на швидкість накопичення vruntime. Процес з вищим nice (нижчий пріоритет) накопичує vruntime швидше, тому отримує менше часу CPU.

**Чому «справедливий»?** Якщо запущено 4 процеси з однаковим nice, кожен отримає приблизно 25% часу CPU -- саме так працював би ідеальний багатозадачний процесор.

### Windows: пріоритетне планування з MLFQ

Windows використовує **пріоритетне витісняюче планування з елементами MLFQ:**

- **32 рівні пріоритету** (0-31): рівні 0-15 -- звичайні процеси, 16-31 -- процеси реального часу.
- Кожному рівню відповідає **окрема черга.** Планувальник завжди обирає потік з черги найвищого непорожнього пріоритету.
- **Динамічне підвищення пріоритету** (priority boost): Windows тимчасово підвищує пріоритет процесу, який отримав ввід від клавіатури або миші (для кращої інтерактивності).
- **Зменшення пріоритету:** після виконання потоку його динамічний пріоритет поступово знижується до базового рівня.
- Використовується **фіксований квант часу** (зазвичай ~20 мс для клієнтських версій, ~120 мс для серверних).

### Порівняння

| Характеристика | Linux CFS | Windows |
|---|---|---|
| Базова модель | Справедливий розподіл | Пріоритетний з MLFQ |
| Кванти часу | Динамічні (немає фіксованих) | Фіксовані |
| Структура даних | Червоно-чорне дерево | Черги для кожного пріоритету |
| Рівні пріоритету | Nice value (-20...+19) | 32 рівні (0-31) |
| Підхід до справедливості | Математичний (vruntime) | Емпіричний (boost/decay) |

## 9. Глосарій

| Термін | Визначення |
|---|---|
| **Планувальник (Scheduler)** | Компонент ядра ОС, який визначає порядок виконання процесів на CPU |
| **Диспетчер (Dispatcher)** | Частина ОС, що безпосередньо передає управління CPU обраному процесу |
| **Час виконання (Burst Time)** | Час, необхідний процесу для виконання на CPU без переривань |
| **Час обороту (Turnaround Time)** | Загальний час від надходження процесу до його завершення |
| **Час очікування (Waiting Time)** | Час, проведений процесом у черзі готовності |
| **Час відгуку (Response Time)** | Час від надходження запиту до першої реакції системи |
| **Пропускна здатність (Throughput)** | Кількість процесів, виконаних за одиницю часу |
| **Квант часу (Time Quantum)** | Максимальний час, протягом якого процес може безперервно використовувати CPU в алгоритмі Round Robin |
| **Витісняюче планування (Preemptive)** | Планування, при якому ОС може примусово забрати CPU у процесу |
| **Невитісняюче планування (Non-preemptive)** | Планування, при якому процес утримує CPU до добровільного звільнення |
| **Ефект конвою (Convoy Effect)** | Ситуація в FCFS, коли короткі процеси довго чекають за одним довгим |
| **Голодування (Starvation)** | Ситуація, коли процес нескінченно довго не отримує CPU |
| **Старіння (Aging)** | Механізм поступового підвищення пріоритету процесу, що довго чекає |
| **Перемикання контексту (Context Switch)** | Збереження стану поточного процесу та завантаження стану наступного |
| **FIFO (First In, First Out)** | Структура даних «перший прийшов -- перший вийшов» (черга) |
| **FCFS (First Come First Served)** | Алгоритм планування: процеси виконуються в порядку надходження |
| **SJF (Shortest Job First)** | Алгоритм планування: першим виконується найкоротший процес |
| **SRTF (Shortest Remaining Time First)** | Витісняюча версія SJF |
| **RR (Round Robin)** | Алгоритм планування з циклічним розподілом квантів часу |
| **MLFQ (Multilevel Feedback Queue)** | Багаторівневі черги зі зворотним зв'язком -- адаптивний алгоритм планування |
| **CFS (Completely Fair Scheduler)** | Планувальник Linux, що забезпечує математично справедливий розподіл CPU |
| **vruntime (Virtual Runtime)** | Віртуальний час виконання процесу в CFS -- основа для вибору наступного процесу |
| **Nice value** | Числове значення (-20...+19), що визначає пріоритет процесу в Linux |
| **Інверсія пріоритетів (Priority Inversion)** | Ситуація, коли високопріоритетний процес чекає на ресурс, зайнятий низькопріоритетним |
