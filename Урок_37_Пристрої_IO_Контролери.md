## 1. Вступ

Комп'ютер без пристроїв введення-виведення (I/O, від англійського Input/Output) був би просто «чорною скринькою» — процесор міг би виконувати обчислення, але ми не змогли б ні ввести дані, ні побачити результат. Саме пристрої I/O дають нам можливість взаємодіяти з комп'ютером: набирати текст на клавіатурі, бачити зображення на моніторі, зберігати файли на диску, слухати музику через навушники.

Проте між процесором і периферійним пристроєм стоїть посередник — **контролер пристрою**. Це спеціалізована електронна схема, яка «перекладає» команди процесора мовою, зрозумілою для пристрою, і навпаки. Без контролерів CPU довелося б знати деталі роботи кожного конкретного принтера, кожної моделі клавіатури, кожного диска — що неможливо.

Сьогодні ми розглянемо:
- як класифікують пристрої введення-виведення;
- через які інтерфейси вони підключаються до комп'ютера;
- що таке контролер пристрою та як процесор з ним спілкується;
- як працює механізм переривань;
- що таке DMA — прямий доступ до пам'яті.

## 2. Класифікація пристроїв введення-виведення

### 2.1. За напрямком передачі даних

Усі периферійні пристрої можна поділити на три групи залежно від того, в якому напрямку рухаються дані:

| Тип | Опис | Приклади |
|---|---|---|
| **Пристрої введення** | Передають дані від користувача (або середовища) **до** комп'ютера | Клавіатура, миша, сканер, мікрофон, веб-камера, сенсорний екран (дотик), геймпад |
| **Пристрої виведення** | Передають дані **від** комп'ютера до користувача (або середовища) | Монітор, принтер, динаміки/навушники, проєктор |
| **Пристрої введення-виведення** | Можуть передавати дані **в обох напрямках** | Жорсткий диск (HDD), SSD, флешка, мережевий адаптер, сенсорний екран (одночасно показує і приймає дотики), модем |

> **Зверніть увагу:** сенсорний екран — це одночасно і пристрій виведення (показує зображення), і пристрій введення (реагує на дотики). Тому його можна віднести до обох категорій.

---

### 2.2. За типом доступу: символьні та блочні пристрої

Це одна з найважливіших класифікацій з точки зору операційної системи, оскільки вона визначає, як ОС організує роботу з пристроєм.

**Символьні пристрої (character devices):**
- Передають дані **потоком окремих символів (байтів)**, один за одним.
- **Немає можливості адресації** — не можна сказати «дай мені 500-й символ». Дані надходять послідовно.
- Буферизація мінімальна або відсутня.
- Приклади: клавіатура, миша, послідовний порт (COM-порт), принтер, термінал.

**Аналогія:** Символьний пристрій — це як водопровідний кран: вода (дані) тече потоком, і ви не можете «вибрати» конкретну краплю з минулого.

**Блочні пристрої (block devices):**
- Передають дані **фіксованими блоками** (зазвичай 512 байт або 4096 байт).
- Підтримують **довільний доступ** (random access) — можна прочитати будь-який блок за його адресою.
- Мають буфери та кеш для оптимізації.
- Приклади: жорсткий диск (HDD), SSD, USB-флешка, CD/DVD-привід.

**Аналогія:** Блочний пристрій — це як книга: ви можете відкрити будь-яку сторінку (блок) за номером, не читаючи все підряд від початку.

| Характеристика | Символьний пристрій | Блочний пристрій |
|---|---|---|
| Одиниця передачі | Один символ (байт) | Блок (512 / 4096 байт) |
| Довільний доступ | Ні | Так |
| Адресація | Відсутня | За номером блоку |
| Буферизація | Мінімальна | Є (з кешуванням) |
| Приклади | Клавіатура, миша, COM-порт | HDD, SSD, флешка |

> **У Linux** це чітко видно: пристрої в каталозі `/dev/` позначені літерою `c` (character) або `b` (block). Наприклад, `/dev/sda` — блочний пристрій (диск), а `/dev/ttyS0` — символьний (послідовний порт).

### 2.3. За швидкістю роботи

Швидкість передачі даних у різних пристроїв відрізняється **на багато порядків**. Це важливо для ОС, оскільки повільні та швидкі пристрої обслуговуються по-різному.

| Категорія | Пристрій | Приблизна швидкість |
|---|---|---|
| **Дуже повільні** | Клавіатура | ~100 байт/с |
| **Дуже повільні** | Миша | ~1 Кбайт/с |
| **Повільні** | Bluetooth-модуль | до 2 Мбіт/с (LE) |
| **Середні** | Мережевий адаптер (Ethernet 1G) | до 1 Гбіт/с (~125 Мбайт/с) |
| **Швидкі** | HDD (жорсткий диск) | ~100–200 Мбайт/с |
| **Швидкі** | SSD (SATA) | ~550 Мбайт/с |
| **Дуже швидкі** | SSD (NVMe PCIe 4.0) | до 7 000 Мбайт/с |
| **Надшвидкі** | SSD (NVMe PCIe 5.0) | до 14 900 Мбайт/с |
| **Надшвидкі** | Відеокарта (PCIe 5.0 x16) | до 64 Гбайт/с (в один бік) |

Як бачимо, різниця між клавіатурою (~100 байт/с) і сучасним NVMe SSD (~15 Гбайт/с) складає приблизно **150 мільйонів разів**. Ось чому ОС не може обслуговувати всі пристрої однаково — для кожної категорії потрібен свій підхід.

### 2.4. Зведена таблиця класифікації

| Пристрій | Напрямок | Тип доступу | Швидкість |
|---|---|---|---|
| Клавіатура | Введення | Символьний | Дуже повільна |
| Миша | Введення | Символьний | Дуже повільна |
| Сканер | Введення | Блочний | Середня |
| Мікрофон | Введення | Символьний | Повільна |
| Монітор | Виведення | Символьний* | Дуже швидка |
| Принтер | Виведення | Символьний | Повільна |
| Динаміки | Виведення | Символьний | Середня |
| HDD | Введення-виведення | Блочний | Швидка |
| SSD (NVMe) | Введення-виведення | Блочний | Надшвидка |
| Мережевий адаптер | Введення-виведення | Символьний** | Швидка |
| USB-флешка | Введення-виведення | Блочний | Середня |

*\* Монітор (відеоадаптер) — особливий випадок: з точки зору ОС це пристрій виведення з доступом до відеопам'яті через memory-mapped I/O.*

*\*\* Мережевий адаптер не є класичним блочним або символьним пристроєм — він працює з пакетами, але ОС часто класифікує його окремо.*

## 3. Інтерфейси підключення пристроїв

**Інтерфейс** — це стандарт з'єднання між пристроєм та комп'ютером, який визначає тип роз'єму, протокол обміну даними та електричні характеристики. Розглянемо найважливіші сучасні інтерфейси.

### 3.1. USB — Universal Serial Bus

USB — найпоширеніший інтерфейс для підключення периферійних пристроїв. Від перших версій у 1996 році до сучасних стандартів його швидкість зросла у десятки тисяч разів.

| Версія | Рік | Максимальна швидкість | Маркетингова назва |
|---|---|---|---|
| USB 1.0 | 1996 | 1,5 Мбіт/с (Low Speed) | — |
| USB 1.1 | 1998 | 12 Мбіт/с (Full Speed) | — |
| USB 2.0 | 2000 | 480 Мбіт/с (High Speed) | Hi-Speed USB |
| USB 3.0 | 2008 | 5 Гбіт/с (SuperSpeed) | SuperSpeed USB |
| USB 3.1 | 2013 | 10 Гбіт/с | SuperSpeed USB 10Gbps |
| USB 3.2 | 2017 | 20 Гбіт/с | SuperSpeed USB 20Gbps |
| USB4 v1.0 | 2019 | 40 Гбіт/с | USB4 40Gbps |
| USB4 v2.0 | 2025 | 80 Гбіт/с (симетр.) / 120 Гбіт/с (асиметр.) | USB4 80Gbps |

**Типи роз'ємів USB:**

```
  Type-A          Type-B         Type-C
 ┌────────┐     ┌──────┐      ┌────────┐
 │████████│     │ ┌──┐ │      │╔══════╗│
 │        │     │ │  │ │      │╚══════╝│
 └────────┘     │ └──┘ │      └────────┘
                └──────┘
 Прямокутний   Квадратний    Овальний,
 Найпоширеніший Для принтерів  двосторонній
```

- **Type-A** — класичний прямокутний роз'єм, який ми звикли бачити на комп'ютерах.
- **Type-B** — квадратний, використовувався переважно для принтерів та сканерів.
- **Type-C** — сучасний овальний двосторонній роз'єм (можна вставляти будь-якою стороною). Обов'язковий для USB4. Підтримує передачу даних, відео та живлення (до 240 Вт) одним кабелем.

> **Важливо:** USB Type-C — це лише **форма роз'єму**, а не версія протоколу. Кабель Type-C може працювати як на швидкості USB 2.0 (480 Мбіт/с), так і на USB4 (80 Гбіт/с). Версія визначається контролером та кабелем, а не формою штекера.

---

### 3.2. Thunderbolt

Thunderbolt — це інтерфейс, розроблений Intel спільно з Apple. Він об'єднує передачу даних, відеосигналу та живлення в одному кабелі.

| Версія | Рік | Швидкість | Роз'єм | Живлення |
|---|---|---|---|---|
| Thunderbolt 1 | 2011 | 10 Гбіт/с | Mini DisplayPort | — |
| Thunderbolt 2 | 2013 | 20 Гбіт/с | Mini DisplayPort | — |
| Thunderbolt 3 | 2015 | 40 Гбіт/с | USB Type-C | до 100 Вт |
| Thunderbolt 4 | 2020 | 40 Гбіт/с | USB Type-C | до 100 Вт |
| Thunderbolt 5 | 2024 | 80 Гбіт/с (до 120 Гбіт/с в режимі Bandwidth Boost) | USB Type-C | до 240 Вт |

Thunderbolt 5 використовує сигналізацію **PAM-3** (замість NRZ у попередніх версіях), що дозволяє передавати 1,5 біти за такт. У режимі Bandwidth Boost три з чотирьох ліній працюють на передачу, забезпечуючи до **120 Гбіт/с** в одному напрямку — цього достатньо для підключення трьох моніторів 4K при 144 Гц або двох моніторів 8K при 60 Гц.

> **Зверніть увагу:** починаючи з Thunderbolt 3, роз'єм Thunderbolt фізично ідентичний USB Type-C, і Thunderbolt-порти сумісні з USB-пристроями.

---

### 3.3. SATA

**SATA (Serial ATA)** — інтерфейс для підключення накопичувачів (жорстких дисків та SSD) всередині комп'ютера.

| Версія | Рік | Максимальна швидкість |
|---|---|---|
| SATA I (1.0) | 2003 | 1,5 Гбіт/с (~150 Мбайт/с) |
| SATA II (2.0) | 2004 | 3,0 Гбіт/с (~300 Мбайт/с) |
| SATA III (3.0) | 2009 | 6,0 Гбіт/с (~600 Мбайт/с) |

SATA III (6 Гбіт/с) залишається стандартом для підключення жорстких дисків, оскільки їхня реальна швидкість (100-200 Мбайт/с) далеко не вичерпує пропускну здатність інтерфейсу. Проте для SSD цей інтерфейс став «вузьким горлом» — швидкість SSD давно перевищила 600 Мбайт/с, тому для них використовують NVMe.

---

### 3.4. NVMe / M.2

**NVMe (Non-Volatile Memory Express)** — протокол, спеціально розроблений для швидких SSD-накопичувачів. На відміну від SATA, NVMe працює через шину **PCI Express**, що дає набагато більшу пропускну здатність.

**M.2** — це компактний форм-фактор (роз'єм на материнській платі), через який підключаються NVMe-накопичувачі. M.2 SSD мають вигляд невеличкої планки розміром приблизно 22 x 80 мм.

| Покоління шини | Макс. швидкість (x4) | Приклади SSD |
|---|---|---|
| PCIe 3.0 x4 | ~3 500 Мбайт/с | Samsung 970 EVO |
| PCIe 4.0 x4 | ~7 000 Мбайт/с | Samsung 980 PRO |
| PCIe 5.0 x4 | ~14 900 Мбайт/с | WD_BLACK SN8100, Samsung 9100 PRO |

Сучасні NVMe SSD покоління PCIe 5.0 досягають послідовної швидкості читання до **14 900 Мбайт/с** — це приблизно в **25 разів** швидше за SATA III SSD і в **75 разів** швидше за типовий жорсткий диск.

---

### 3.5. PCI Express

**PCI Express (PCIe)** — основна високошвидкісна шина всередині комп'ютера. Через неї підключаються відеокарти, мережеві адаптери, NVMe SSD, контролери USB та інші компоненти.

PCIe працює за принципом **ліній (lanes)**. Кожна лінія — це окремий двонапрямлений канал зв'язку. Пристрій може використовувати x1, x4, x8 або x16 ліній залежно від потреб.

| Покоління | Швидкість на лінію (в один бік) | x16 (в обох напрямках) | Рік |
|---|---|---|---|
| PCIe 3.0 | ~1 Гбайт/с | ~32 Гбайт/с | 2010 |
| PCIe 4.0 | ~2 Гбайт/с | ~64 Гбайт/с | 2017 |
| PCIe 5.0 | ~4 Гбайт/с | ~128 Гбайт/с | 2019 |
| PCIe 6.0 | ~8 Гбайт/с | ~256 Гбайт/с | 2022 |

Кожне нове покоління PCIe подвоює швидкість попереднього, зберігаючи при цьому **зворотну сумісність** — пристрій PCIe 3.0 працюватиме у слоті PCIe 5.0 (але на швидкості PCIe 3.0).

PCIe 6.0 використовує сигналізацію **PAM-4** (4 рівні амплітуди замість 2), що дозволяє передавати 2 біти за такт, і технологію **FLIT** (Flow Control Unit) для підвищення ефективності передачі.

---

### 3.6. Відеоінтерфейси: HDMI та DisplayPort

Ці інтерфейси призначені для передачі відеосигналу від комп'ютера до монітора або проєктора.

| Інтерфейс | Версія | Пропускна здатність | Максимальна роздільність |
|---|---|---|---|
| HDMI | 2.0 | 18 Гбіт/с | 4K @ 60 Гц |
| HDMI | 2.1 | 48 Гбіт/с | 4K @ 120 Гц, 8K @ 60 Гц |
| DisplayPort | 1.4 | 32,4 Гбіт/с | 4K @ 120 Гц (з DSC), 8K @ 30 Гц |
| DisplayPort | 2.1 | 80 Гбіт/с | 4K @ 240 Гц, 8K @ 60 Гц, 16K @ 60 Гц (з DSC) |

- **HDMI** — найпоширеніший інтерфейс для телевізорів та моніторів. Передає відео та звук одним кабелем.

## 4. Контролер пристрою

### 4.1. Що таке контролер пристрою

**Контролер пристрою** (device controller) — це електронна схема (зазвичай мікрочип або плата), яка є посередником між процесором (через системну шину) та периферійним пристроєм.

```
┌─────────┐      ┌────────────────┐      ┌──────────────┐
│         │      │                │      │              │
│   CPU   │◄────►│   КОНТРОЛЕР    │◄────►│   ПРИСТРІЙ   │
│         │ шина │   ПРИСТРОЮ     │кабель│  (принтер,   │
│         │      │                │      │   диск...)   │
└─────────┘      └────────────────┘      └──────────────┘
                  │  Регістри:    │
                  │  - стану      │
                  │  - команд     │
                  │  - даних      │
                  └───────────────┘
```

**Навіщо потрібен контролер:**
1. **Перетворення сигналів** — контролер перетворює цифрові команди CPU в електричні/оптичні сигнали, зрозумілі пристрою (наприклад, рухи головки жорсткого диска або імпульси для принтера).
2. **Буферизація** — контролер має власну пам'ять (буфер), де тимчасово зберігає дані, компенсуючи різницю у швидкості між CPU і пристроєм.
3. **Контроль помилок** — контролер перевіряє цілісність даних (наприклад, за допомогою контрольних сум).
4. **Звільнення CPU** — контролер самостійно керує деталями роботи пристрою, не навантажуючи цим процесор.

**Приклади контролерів:**
- Контролер жорсткого диска (вбудований у HDD/SSD) — керує читанням/записом секторів.
- Контролер USB (на материнській платі) — керує підключенням USB-пристроїв.
- Відеоконтролер (GPU) — спеціалізований контролер для виведення зображення на монітор.
- Контролер мережевого адаптера (NIC) — керує відправкою та прийманням мережевих пакетів.

---

### 4.2. Регістри контролера

Контролер має набір **регістрів** — невеликих комірок пам'яті, через які CPU спілкується з контролером. Є три основних типи регістрів:

| Регістр | Призначення | Напрямок | Приклад вмісту |
|---|---|---|---|
| **Регістр стану** (status register) | Повідомляє CPU про поточний стан пристрою | Контролер → CPU | «Зайнятий», «Готовий», «Помилка» |
| **Регістр команд** (command register) | CPU записує сюди команди для контролера | CPU → Контролер | «Прочитати сектор», «Записати дані», «Скинути» |
| **Регістр даних** (data register / data buffer) | Буфер для обміну даними між CPU та контролером | CPU ↔ Контролер | Байти даних для читання/запису |

**Як відбувається типова операція читання з диска:**

1. CPU записує у **регістр команд** контролера команду «прочитати сектор N».
2. Контролер починає виконувати операцію. У **регістрі стану** встановлюється прапорець «зайнятий» (busy).
3. Коли дані зчитані з диска, контролер поміщає їх у **регістр даних** (буфер).
4. Контролер знімає прапорець «зайнятий» і встановлює прапорець «готовий» (ready) у **регістрі стану**.
5. CPU читає дані з **регістру даних** контролера.

---

### 4.3. Як CPU взаємодіє з контролером: Port-Mapped I/O vs Memory-Mapped I/O

Процесор повинен мати можливість звертатися до регістрів контролера — записувати команди та читати стан. Існують два фундаментально різних способи організації такого доступу.

#### Port-Mapped I/O (ізольований ввід-вивід, PMIO)

У цьому підході регістри контролерів мають **окремий адресний простір**, відмінний від адресного простору оперативної пам'яті. Для доступу до них використовуються **спеціальні інструкції процесора**:

```
; x86 Assembler — Port-Mapped I/O
OUT 0x3F8, AL    ; записати байт з регістру AL у порт 0x3F8 (COM-порт)
IN AL, 0x60      ; прочитати байт з порту 0x60 (клавіатура) у регістр AL
```

- Адресний простір портів вводу-виводу **ізольований** від пам'яті.
- На x86 є лише 65 536 портів (адреси 0x0000–0xFFFF).
- Використовуються інструкції `IN` та `OUT`.

#### Memory-Mapped I/O (ввід-вивід через пам'ять, MMIO)

У цьому підході регістри контролерів **відображаються в загальний адресний простір пам'яті**. CPU звертається до них тими самими інструкціями, що й до звичайної оперативної пам'яті:

```
; x86 Assembler — Memory-Mapped I/O
MOV [0xB8000], AL    ; записати байт у відеопам'ять (адреса 0xB8000)
MOV AL, [0xFEE00020] ; прочитати регістр APIC
```

- Регістри пристроїв «виглядають» як комірки пам'яті.
- Звичайні інструкції роботи з пам'яттю (`MOV`, `LDR`, `STR`) використовуються для I/O.
- Адреси пристроїв «займають» частину адресного простору — їх не може використовувати RAM.

#### Порівняння двох підходів

| Характеристика | Port-Mapped I/O | Memory-Mapped I/O |
|---|---|---|
| Адресний простір | Окремий від RAM | Спільний з RAM |
| Інструкції CPU | Спеціальні (`IN`/`OUT`) | Звичайні (`MOV`, `LDR`) |
| Розмір простору | Обмежений (64К портів на x86) | Великий (весь адресний простір) |
| Швидкість | Повільніше | Швидше |
| Складність декодування | Простіше | Складніше |
| Режими адресації | Обмежені | Усі режими CPU |
| RAM простір | Не зменшує | Зменшує доступну RAM |
| Де використовується | x86 (клавіатура, COM-порт) | ARM, сучасні пристрої, відеопам'ять |

> **На практиці** сучасні системи використовують **обидва методи одночасно**. Наприклад, на x86 клавіатура та COM-порти доступні через Port-Mapped I/O (порти 0x60, 0x3F8), а відеопам'ять та сучасні контролери — через Memory-Mapped I/O.

## 5. Переривання (Interrupts)

### 5.1. Що таке переривання і навіщо воно потрібне

**Переривання (interrupt)** — це сигнал, який повідомляє процесор про те, що сталася подія, яка потребує негайної уваги.

**Проблема без переривань — опитування (polling):**

Без механізму переривань CPU змушений постійно перевіряти регістр стану кожного контролера: «Клавіатура, є нові дані? Ні. Диск, ти готовий? Ні. Мережева карта, є пакет? Ні...» — і так у нескінченному циклі. Це називається **опитування (polling)** або **програмний ввід-вивід**.

```
; Псевдокод опитування (polling) — НЕЕФЕКТИВНО!
loop:
    прочитати регістр_стану_клавіатури
    якщо є_дані → обробити
    прочитати регістр_стану_диска
    якщо готовий → забрати_дані
    прочитати регістр_стану_мережі
    якщо є_пакет → прийняти
    перейти до loop          ; знову і знову...
```

Це вкрай неефективно — CPU витрачає майже весь час на перевірки, замість того щоб виконувати корисну роботу.

**Аналогія:** Опитування — це коли ви кожні 10 секунд підходите до вхідних дверей і перевіряєте, чи не прийшов лист. Переривання — це дзвінок у двері: ви спокійно займаєтеся своїми справами, а коли листоноша прийде, він **сам** повідомить вас.

**Рішення — переривання:**

Пристрій **сам** повідомляє CPU, коли потребує уваги. CPU може спокійно виконувати інші програми, поки не отримає сигнал переривання.

---

### 5.2. Механізм обробки переривання

Ось покрокова послідовність подій при апаратному перериванні:

```
┌──────────┐    IRQ     ┌──────────────┐   сигнал   ┌──────────┐
│          │───────────►│  Контролер   │───────────►│          │
│ ПРИСТРІЙ │            │  переривань  │            │   CPU    │
│(клавіат.)│            │    (PIC)     │            │          │
└──────────┘            └──────────────┘            └──────────┘
                                                         │
                                                         ▼
                                                 1. Зберегти стан
                                                 2. Знайти ISR в IVT
                                                 3. Виконати ISR
                                                 4. Відновити стан
                                                 5. Продовжити роботу
```

**Крок 1.** Пристрій (наприклад, клавіатура) завершує операцію і надсилає сигнал свому **контролеру**.

**Крок 2.** Контролер пристрою подає сигнал на лінію **IRQ (Interrupt Request)** — лінію запиту переривання.

**Крок 3.** Сигнал надходить до **контролера переривань (PIC — Programmable Interrupt Controller)**, який визначає пріоритет і передає запит процесору.

**Крок 4.** CPU перериває виконання поточної програми і **зберігає її стан** (вміст регістрів, адресу наступної інструкції) у стек.

**Крок 5.** За номером переривання CPU знаходить адресу потрібної **процедури обробки переривання (ISR — Interrupt Service Routine)** у **таблиці переривань (IVT)**.

**Крок 6.** CPU виконує ISR — спеціальну програму, яка обробляє подію (наприклад, читає натиснуту клавішу з регістру даних контролера клавіатури).

**Крок 7.** Після завершення ISR CPU **відновлює збережений стан** зі стеку і продовжує виконання перерваної програми з того місця, де зупинився.

---

### 5.3. Апаратні та програмні переривання

| Характеристика | Апаратне переривання | Програмне переривання |
|---|---|---|
| **Джерело** | Зовнішній пристрій (клавіатура, таймер, диск) | Програма (інструкція процесора) |
| **Як виникає** | Електричний сигнал на лінії IRQ | Виконання спеціальної інструкції (наприклад, `INT` на x86) |
| **Коли виникає** | Асинхронно (будь-якого моменту) | Синхронно (у конкретному місці коду) |
| **Для чого** | Сповіщення CPU про подію пристрою | Виклик функцій ОС (системні виклики) |
| **Можна ігнорувати?** | Маскувані — так; немаскувані (NMI) — ні | Ні (це команда процесора) |
| **Приклад** | Натискання клавіші (IRQ1), тік системного таймера (IRQ0) | `INT 0x80` — системний виклик у Linux, `INT 0x21` — виклик DOS |

Окрема категорія — **виключення (exceptions)**: це переривання, що генеруються самим процесором при помилках виконання, наприклад:
- **Ділення на нуль** (Division by zero, переривання 0)
- **Помилка сторінки пам'яті** (Page fault, переривання 14) — виникає, коли програма звертається до сторінки, якої немає в оперативній пам'яті.

---

### 5.4. Таблиця переривань (Interrupt Vector Table)

**Таблиця переривань (IVT — Interrupt Vector Table)** — це масив адрес у пам'яті, де кожному номеру переривання (від 0 до 255) відповідає адреса обробника (ISR).

У реальному режимі x86 IVT займає перші 1024 байти оперативної пам'яті (адреси 0x00000–0x003FF). Кожен запис — **4 байти** (сегмент + зміщення адреси ISR), усього **256 записів**.

```
Таблиця переривань (IVT) — спрощена схема:

Номер переривання    Адреса ISR
┌──────────────────────────────────────┐
│  0  →  Обробник ділення на нуль      │  ← Виключення CPU
│  1  →  Обробник покрокового режиму   │  ← Виключення CPU
│  2  →  NMI (немаскуване переривання) │  ← Апаратне
│  ...                                 │
│  8  →  Обробник системного таймера   │  ← IRQ0 (таймер)
│  9  →  Обробник клавіатури           │  ← IRQ1 (клавіатура)
│ 10  →  Каскад (до другого PIC)       │  ← IRQ2
│ ...                                  │
│ 14  →  Обробник жорсткого диска      │  ← IRQ6
│ ...                                  │
│ 0x80 → Системний виклик Linux        │  ← Програмне
│ ...                                  │
│ 255 →  (останній вектор)             │
└──────────────────────────────────────┘
```

У захищеному режимі x86 замість IVT використовується **IDT (Interrupt Descriptor Table)** — вона виконує ту саму функцію, але має гнучкішу структуру й підтримує механізми захисту.

---

### 5.5. Контролер переривань (PIC)

**PIC (Programmable Interrupt Controller)** — спеціальний чип, який керує апаратними перериваннями. Він збирає сигнали IRQ від різних пристроїв, визначає їхній **пріоритет** і передає запит процесору по черзі.

У класичній архітектурі IBM PC використовується **каскад із двох контролерів Intel 8259A**:
- **Головний (Master) PIC** — обробляє IRQ0–IRQ7.
- **Підлеглий (Slave) PIC** — обробляє IRQ8–IRQ15.
- IRQ2 використовується для з'єднання (каскадування) двох PIC.

Класичний розподіл IRQ:

| IRQ | Пристрій |
|---|---|
| IRQ0 | Системний таймер |
| IRQ1 | Клавіатура |
| IRQ2 | Каскад до Slave PIC |
| IRQ3 | COM2 (послідовний порт 2) |
| IRQ4 | COM1 (послідовний порт 1) |
| IRQ5 | LPT2 або звукова карта |
| IRQ6 | Контролер гнучких дисків |
| IRQ7 | LPT1 (паралельний порт) |
| IRQ8 | Годинник реального часу (RTC) |
| IRQ12 | PS/2 миша |
| IRQ13 | Математичний співпроцесор |
| IRQ14 | Первинний канал IDE (HDD) |
| IRQ15 | Вторинний канал IDE |

> **У сучасних системах** замість PIC 8259A використовується **APIC (Advanced Programmable Interrupt Controller)**, який підтримує більше переривань, багатопроцесорні системи та гнучкішу маршрутизацію переривань.

---

## 6. DMA — прямий доступ до пам'яті

### 6.1. Проблема: навіщо потрібен DMA

Розглянемо, що відбувається при копіюванні файлу з диска в оперативну пам'ять **без DMA** (програмний ввід-вивід):

1. CPU надсилає команду контролеру диска: «Прочитай сектор N».
2. Контролер зчитує дані з диска у свій буфер.
3. CPU **побайтово** (або послівно) копіює дані з буфера контролера в оперативну пам'ять.
4. Повтор для кожного сектору.

**Проблема:** Під час кроку 3 процесор повністю зайнятий рутинною операцією копіювання байтів — він не може виконувати жодних інших обчислень. Для великих файлів це може тривати значний час.

```
Без DMA:

CPU:    [Команда]──[Чекай]──[Копіюй байт 1]──[Копіюй байт 2]──...──[Копіюй байт N]──[Далі]
         ▲                   ▲                                        ▲
         │                   └── CPU повністю зайнятий копіюванням ────┘
         │
     Корисна робота          Марнування ресурсів CPU!
```

**Аналогія:** Уявіть, що директор компанії (CPU) особисто переносить кожну коробку з вантажівки на склад. Він не може керувати компанією, поки переносить коробки. Набагато ефективніше найняти вантажника (DMA-контролер), який сам розвантажить вантажівку, а директор тим часом займатиметься важливішими справами.

---

### 6.2. Як працює DMA

**DMA (Direct Memory Access)** — це механізм, що дозволяє периферійним пристроям передавати дані безпосередньо в оперативну пам'ять (або з неї), **минаючи процесор**.

Для цього використовується спеціальний чип — **DMA-контролер (DMAC)**.

**Покрокова схема роботи DMA:**

```
                        ┌──────────────────┐
                        │  DMA-контролер   │
                        │  ┌────────────┐  │
                        │  │ Адреса RAM │  │
                        │  │ Кількість  │  │
                        │  │ Напрямок   │  │
                        │  └────────────┘  │
                        └───────┬──────────┘
                                │
    ┌─────────┐     шина    ┌───▼───┐     шина    ┌─────────┐
    │   CPU   │◄───────────►│  RAM  │◄───────────►│  Диск   │
    └─────────┘             └───────┘             └─────────┘
```

**Крок 1 — Ініціалізація.** CPU програмує DMA-контролер, записуючи в його регістри:
- **Адресу в RAM**, куди (або звідки) треба копіювати дані.
- **Кількість байтів** для передачі.
- **Напрямок** передачі (пристрій → RAM або RAM → пристрій).

**Крок 2 — Передача.** DMA-контролер запитує доступ до системної шини (bus request). Коли шина звільняється, CPU тимчасово відключається від неї (bus grant), і DMA-контролер **самостійно** передає дані між пристроєм та RAM.

**Крок 3 — Завершення.** Після завершення передачі DMA-контролер звільняє шину та надсилає CPU **переривання**, повідомляючи: «Передачу завершено, дані в пам'яті».

```
З DMA:

CPU:    [Програмуй DMA]──[Виконуй іншу програму]──────────────[Переривання! Дані готові]
DMA:                      [Копіюй байт 1]──[байт 2]──...──[N]──[Готово!]

CPU вільний для корисної роботи!
```

**Регістри DMA-контролера:**

| Регістр | Призначення |
|---|---|
| Регістр адреси пам'яті | Адреса в RAM для читання/запису |
| Регістр лічильника слів | Кількість одиниць даних для передачі |
| Регістр управління | Напрямок передачі, режим роботи, старт/стоп |

---

### 6.3. Режими передачі DMA

DMA-контролер може працювати у трьох режимах:

| Режим | Опис | Переваги | Недоліки |
|---|---|---|---|
| **Пакетний (Burst)** | DMA захоплює шину і передає весь блок даних за один раз, не віддаючи шину, поки не закінчить | Максимальна швидкість передачі | CPU повністю заблокований на час передачі |
| **Покрадений цикл (Cycle Stealing)** | DMA передає одне слово, потім віддає шину CPU, потім знову забирає для наступного слова | CPU може працювати між передачами | Повільніша загальна передача |
| **Прозорий (Transparent)** | DMA використовує шину лише тоді, коли CPU її не використовує (у «порожніх» тактах) | CPU не сповільнюється взагалі | Найповільніша передача, складна реалізація |

**Де використовується DMA:** практично в усіх сучасних пристроях, які передають великі обсяги даних — дискові контролери, мережеві адаптери, звукові карти, відеоконтролери, контролери USB.

> **Порівняння ефективності:**
> | Метод | Роль CPU | Ефективність |
> |---|---|---|
> | Програмний I/O (polling) | CPU копіює кожен байт | Дуже низька |
> | I/O з перериваннями | CPU копіює дані, але не чекає готовності | Середня |
> | DMA | CPU лише ініціалізує, все копіює DMAC | Висока |

## 7. Підсумок

На цьому уроці ми розглянули підсистему введення-виведення — один з найважливіших компонентів взаємодії комп'ютера з зовнішнім світом:

1. **Пристрої I/O** класифікуються за напрямком (введення / виведення / обидва), типом доступу (символьні / блочні) та швидкістю (від 100 байт/с у клавіатури до 15 Гбайт/с у NVMe SSD).

2. **Інтерфейси** визначають, як пристрій фізично та логічно підключається до комп'ютера. Сучасна тенденція — універсальний роз'єм USB Type-C, через який можна передавати дані (до 80–120 Гбіт/с), відео та живлення (до 240 Вт).

3. **Контролер пристрою** — посередник між CPU і пристроєм, який має регістри стану, команд та даних. CPU може звертатися до контролера через Port-Mapped I/O (окремий адресний простір, інструкції `IN`/`OUT`) або Memory-Mapped I/O (спільний з RAM адресний простір, звичайні інструкції).

4. **Переривання** — механізм, що дозволяє пристроям самим повідомляти CPU про події, замість того щоб CPU постійно опитував кожен пристрій. Ланцюжок: пристрій → контролер → IRQ → PIC → CPU → ISR (через IVT).

5. **DMA** звільняє CPU від рутинного копіювання даних: DMA-контролер самостійно передає блоки даних між пристроєм та RAM, а CPU лише ініціалізує передачу та отримує переривання по завершенню.

## 9. Глосарій ключових термінів

| Термін | Визначення |
|---|---|
| **I/O (Input/Output)** | Введення-виведення — процес обміну даними між процесором та периферійними пристроями |
| **Символьний пристрій** | Пристрій, що передає дані потоком окремих байтів без можливості довільного доступу (клавіатура, миша) |
| **Блочний пристрій** | Пристрій, що передає дані фіксованими блоками з можливістю довільного доступу за адресою (HDD, SSD) |
| **Інтерфейс** | Стандарт фізичного та логічного з'єднання між пристроєм та комп'ютером (USB, SATA, PCIe) |
| **USB (Universal Serial Bus)** | Універсальна послідовна шина для підключення периферійних пристроїв |
| **NVMe (Non-Volatile Memory Express)** | Протокол для високошвидкісних SSD, що працює через шину PCIe |
| **PCI Express (PCIe)** | Високошвидкісна послідовна шина всередині комп'ютера для підключення компонентів |
| **SATA (Serial ATA)** | Інтерфейс для підключення жорстких дисків та SSD всередині комп'ютера |
| **Thunderbolt** | Високошвидкісний інтерфейс від Intel, що об'єднує дані, відео та живлення |
| **Контролер пристрою** | Електронна схема-посередник між CPU (через шину) та периферійним пристроєм |
| **Регістр стану** | Регістр контролера, що повідомляє CPU про поточний стан пристрою (готовий, зайнятий, помилка) |
| **Регістр команд** | Регістр контролера, куди CPU записує команди для виконання |
| **Регістр даних** | Буфер контролера для обміну даними між CPU та пристроєм |
| **Port-Mapped I/O (PMIO)** | Метод доступу до контролерів через окремий адресний простір із спеціальними інструкціями IN/OUT |
| **Memory-Mapped I/O (MMIO)** | Метод доступу до контролерів через загальний адресний простір пам'яті зі звичайними інструкціями |
| **Переривання (Interrupt)** | Сигнал, що змушує CPU перервати виконання поточної програми для обробки події |
| **IRQ (Interrupt Request)** | Лінія запиту апаратного переривання від пристрою до контролера переривань |
| **ISR (Interrupt Service Routine)** | Процедура обробки переривання — програма, яку виконує CPU у відповідь на переривання |
| **IVT (Interrupt Vector Table)** | Таблиця векторів переривань — масив адрес ISR для кожного номера переривання |
| **PIC (Programmable Interrupt Controller)** | Програмований контролер переривань — чип, що керує пріоритетами та маршрутизацією IRQ |
| **APIC (Advanced PIC)** | Розширений контролер переривань для багатопроцесорних систем |
| **Polling (опитування)** | Метод, при якому CPU циклічно перевіряє стан кожного пристрою — неефективна альтернатива перериванням |
| **NMI (Non-Maskable Interrupt)** | Немаскуване переривання — переривання, яке CPU не може ігнорувати (критичні апаратні помилки) |
| **DMA (Direct Memory Access)** | Прямий доступ до пам'яті — механізм передачі даних між пристроєм та RAM без участі CPU |
| **DMAC (DMA Controller)** | Контролер DMA — спеціалізований чип, що керує передачею даних у режимі DMA |
| **Bus Arbitration** | Арбітраж шини — процес визначення, хто (CPU чи DMAC) отримає доступ до системної шини |
| **Burst Mode** | Пакетний режим DMA — DMAC захоплює шину на весь час передачі блоку даних |
| **Cycle Stealing** | Режим покраденого циклу DMA — DMAC передає одне слово за раз, чергуючись із CPU |
