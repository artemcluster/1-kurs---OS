## 1. Вступ

У попередніх уроках ми дізналися, що операційна система складається з багатьох компонентів: ядра, оболонки, файлової системи, драйверів. Але як саме ці компоненти організовані між собою? За яким принципом вони взаємодіють? Чи всі ОС побудовані однаково?

Відповідь на ці питання дає **архітектура операційної системи** — тобто спосіб внутрішньої організації та взаємодії компонентів ОС. Різні архітектурні підходи мають різні переваги та недоліки, і саме від вибору архітектури залежить продуктивність, надійність та безпека операційної системи.

У цьому уроці ми розглянемо:
- що таке архітектура ОС і навіщо її вивчати;
- монолітну архітектуру — найпростіший підхід;
- багатошарову (рівневу) архітектуру — ідею Едсгера Дейкстри;
- мікроядерну архітектуру (короткий огляд перед уроком 8);
- клієнт-серверну модель в ОС;
- модульну архітектуру та завантажувані модулі ядра;
- віртуальні машини як архітектурний підхід;
- як побудовані реальні ОС: Windows, Linux та macOS.

## 2. Поняття архітектури ОС

### 2.1. Що таке архітектура ОС

**Архітектура операційної системи** — це загальна структурна організація ОС, яка визначає:
- з яких компонентів (модулів, підсистем) складається ОС;
- як ці компоненти взаємодіють між собою;
- які компоненти працюють у режимі ядра, а які — у режимі користувача;
- як відбувається звернення до апаратного забезпечення.

Архітектуру ОС можна порівняти з планом будівлі: два будинки можуть мати однакову кількість кімнат, але бути організовані зовсім по-різному — одноповерховий будинок (монолітна архітектура) чи багатоповерховий з чітким поділом на поверхи (багатошарова архітектура).

### 2.2. Навіщо вивчати архітектуру ОС

Розуміння архітектури ОС важливе з кількох причин:

1. **Для IT-спеціалістів:** Знання архітектури допомагає зрозуміти, чому система поводиться певним чином, де шукати причини збоїв, як оптимізувати роботу.
2. **Для адміністраторів:** Різні архітектури вимагають різних підходів до налаштування та обслуговування. Наприклад, модульна архітектура Linux дозволяє завантажувати драйвери "на ходу".
3. **Для розробників:** Знання архітектури допомагає писати ефективніші програми, що правильно взаємодіють з ОС.
4. **Для вибору ОС:** Для різних задач потрібні ОС з різною архітектурою. Для автомобілів — надійна мікроядерна QNX, для серверів — ефективний монолітний Linux.

## 3. Монолітна архітектура

### 3.1. Принцип побудови

**Монолітна архітектура** — це найпростіший і найстаріший підхід до побудови ОС, при якому **вся операційна система працює як одна велика програма в єдиному адресному просторі** в режимі ядра (Ring 0).

У монолітній архітектурі всі компоненти ОС — керування процесами, керування пам'яттю, файлова система, драйвери пристроїв, мережевий стек — скомпільовані в один великий виконуваний файл. Усі частини системи можуть безпосередньо викликати функції одна одної, без будь-яких обмежень.

**Аналогія:** Уявіть велику кімнату, в якій працюють усі працівники компанії: бухгалтерія, відділ продажів, IT-відділ, керівництво. Вони можуть спілкуватися миттєво (висока продуктивність), але якщо один працівник створить проблему — це вплине на всіх (низька ізоляція).

**Текстова схема монолітної архітектури:**

```
┌─────────────────────────────────────────────────────┐
│              ПРОГРАМИ КОРИСТУВАЧА                     │
│         (браузер, текстовий редактор)                 │
├─────────────────────────────────────────────────────┤
│                                                       │
│           МОНОЛІТНЕ ЯДРО (Ring 0)                     │
│  Все працює в одному адресному просторі:             │
│                                                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐   │
│  │Керування │  │Керування │  │Файлова система   │   │
│  │процесами │  │ пам'яттю │  │(NTFS, ext4, FAT) │   │
│  ├──────────┤  ├──────────┤  ├──────────────────┤   │
│  │ Драйвери │  │Мережевий │  │   Безпека та     │   │
│  │пристроїв │  │  стек    │  │   контроль       │   │
│  └──────────┘  └──────────┘  └──────────────────┘   │
│                                                       │
│  Усі компоненти можуть напряму викликати один одного │
│                                                       │
├─────────────────────────────────────────────────────┤
│             АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                      │
│          (CPU, RAM, диски, пристрої)                  │
└─────────────────────────────────────────────────────┘
```

### 3.2. Переваги та недоліки

**Переваги монолітної архітектури:**

| Перевага | Пояснення |
|----------|-----------|
| **Висока продуктивність** | Компоненти спілкуються напряму через виклики функцій без додаткових витрат. Немає потреби передавати повідомлення між процесами. |
| **Швидкі системні виклики** | Перехід від програми до ядра і назад відбувається швидко, без складних маршрутизацій. |
| **Простота реалізації** | Усі компоненти в одному місці — простіше розробляти та компілювати. |

**Недоліки монолітної архітектури:**

| Недолік | Пояснення |
|---------|-----------|
| **Низька надійність** | Помилка в будь-якому компоненті (навіть у драйвері малозначного пристрою) може привести до краху всієї системи. |
| **Складність підтримки** | Великий обсяг коду в одному блоці важко підтримувати, тестувати та відлагоджувати. |
| **Складність модифікації** | Додавання нових функцій або зміна існуючих потребує перекомпіляції всього ядра. |
| **Проблеми з безпекою** | Увесь код працює з максимальними привілеями (Ring 0), тому уразливість в одному компоненті загрожує всій системі. |

### 3.3. Приклади ОС з монолітною архітектурою

**MS-DOS (1981)** — класичний приклад монолітної ОС. Вся система працювала в одному адресному просторі, не мала розділення на режим ядра та режим користувача. Будь-яка програма мала повний доступ до всієї пам'яті та апаратури, що робило систему простою, але вкрай ненадійною.

**Ранні версії UNIX (1970-ті)** — перші реалізації UNIX також були монолітними. Усі компоненти (файлова система, керування процесами, драйвери) працювали в єдиному ядрі. Проте UNIX заклав важливі архітектурні принципи: розділення на режим ядра та режим користувача, ієрархію файлової системи, концепцію "все є файлом".

## 4. Багатошарова (рівнева) архітектура

### 4.1. Ідея Дейкстри — система THE (1968)

**Едсгер Дейкстра** (Edsger W. Dijkstra) — нідерландський вчений, один із засновників структурного програмування та піонер операційних систем. У 1968 році він опублікував у журналі Communications of the ACM статтю "The Structure of the 'THE'-Multiprogramming System", яка стала революційною для проєктування ОС.

**THE** (скорочення від "Technische Hogeschool Eindhoven" — Технологічний університет Ейндговена) — це операційна система, розроблена командою Дейкстри для комп'ютера Electrologica X8.

**Головна ідея Дейкстри:** організувати ОС у вигляді **суворої ієрархії рівнів (шарів)**, де кожен рівень може використовувати **лише послуги нижчого рівня** та надає послуги верхньому рівню. Вищий рівень ніколи не звертається "вниз через голову" — тільки до безпосередньо нижчого рівня.

**Система THE мала 6 рівнів (0-5):**

```
┌─────────────────────────────────────────────────────┐
│  Рівень 5:  КОРИСТУВАЧ (ОПЕРАТОР)                    │
│             "не реалізований нами" — писав Дейкстра  │
├─────────────────────────────────────────────────────┤
│  Рівень 4:  ПРОГРАМИ КОРИСТУВАЧА                     │
│             Компіляція, виконання, друк               │
├─────────────────────────────────────────────────────┤
│  Рівень 3:  КЕРУВАННЯ ПРИСТРОЯМИ ВВОДУ-ВИВОДУ       │
│             Управління периферійними пристроями       │
│             та буферизація даних                      │
├─────────────────────────────────────────────────────┤
│  Рівень 2:  КОМУНІКАЦІЯ З КОНСОЛЛЮ ОПЕРАТОРА        │
│             Зв'язок між ОС та консоллю управління    │
├─────────────────────────────────────────────────────┤
│  Рівень 1:  КЕРУВАННЯ ПАМ'ЯТТЮ                      │
│             Програмна віртуальна пам'ять              │
│             (сторінкова організація)                  │
├─────────────────────────────────────────────────────┤
│  Рівень 0:  РОЗПОДІЛ ПРОЦЕСОРА                       │
│             Планування процесів, перемикання          │
│             контексту, обробка переривань таймера     │
├─────────────────────────────────────────────────────┤
│             АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                      │
│             (Electrologica X8)                        │
└─────────────────────────────────────────────────────┘
```

**Ключові новаторства системи THE:**
- **Перший приклад програмної віртуальної пам'яті** — комп'ютер X8 не мав апаратної підтримки пам'яті, тому Дейкстра реалізував її програмно.
- **Семафори** (P та V операції) — механізм синхронізації процесів, винайдений Дейкстрою, який використовується дотепер.
- **Формальна верифікація** — Дейкстра математично довів коректність кожного рівня, потім з'єднав ці доведення в доказ коректності всієї системи.

### 4.2. Принцип побудови багатошарової архітектури

Багатошарова (рівнева) архітектура базується на одному ключовому правилі:

> **Кожен рівень може використовувати ЛИШЕ послуги безпосередньо нижчого рівня і надає послуги безпосередньо верхньому рівню.**

Це означає:
- Рівень 3 може викликати функції рівня 2, але **не може** звернутися напряму до рівня 0 або рівня 1.
- Рівень 2 не знає про існування рівня 4 — він лише обслуговує запити від рівня 3.
- Кожен рівень "ховає" деталі своєї реалізації від верхніх рівнів.

**Аналогія:** Уявіть багатоповерхову будівлю. Щоб дістатися з 5-го поверху на 1-й, ви повинні пройти через кожен проміжний поверх — ліфт зупиняється на кожному рівні. Ви не можете "перестрибнути" через кілька поверхів.

**Текстова схема принципу роботи:**

```
     Запит від програми
            │
            v
    ┌───────────────┐
    │   Рівень N    │  Приймає запит, обробляє те,
    │               │  що може, передає далі вниз
    └───────┬───────┘
            │  Виклик послуг рівня N-1
            v
    ┌───────────────┐
    │  Рівень N-1   │  Обробляє свою частину,
    │               │  передає далі вниз
    └───────┬───────┘
            │  Виклик послуг рівня N-2
            v
    ┌───────────────┐
    │  Рівень N-2   │  ...і так далі до
    │               │  найнижчого рівня
    └───────┬───────┘
            │
            v
    ┌───────────────┐
    │  Апаратура    │  Виконання фізичної операції
    └───────────────┘
```

### 4.3. Типові рівні багатошарової ОС

Хоча кількість і назви рівнів можуть відрізнятися в різних ОС, типова узагальнена модель багатошарової архітектури виглядає так:

```
┌─────────────────────────────────────────────────────┐
│                                                       │
│  РІВЕНЬ 6:  ІНТЕРФЕЙС КОРИСТУВАЧА                    │
│  Графічний робочий стіл (GUI) або                    │
│  командний рядок (CLI)                                │
│                                                       │
├─────────────────────────────────────────────────────┤
│                                                       │
│  РІВЕНЬ 5:  ФАЙЛОВА СИСТЕМА                          │
│  Організація зберігання файлів та каталогів           │
│  (NTFS, ext4, APFS, FAT32)                           │
│                                                       │
├─────────────────────────────────────────────────────┤
│                                                       │
│  РІВЕНЬ 4:  КЕРУВАННЯ ПРИСТРОЯМИ (ДРАЙВЕРИ)         │
│  Взаємодія з периферійними пристроями:               │
│  диски, мережа, відео, звук, USB                     │
│                                                       │
├─────────────────────────────────────────────────────┤
│                                                       │
│  РІВЕНЬ 3:  КЕРУВАННЯ ПРОЦЕСАМИ                      │
│  Створення, планування, синхронізація                 │
│  та завершення процесів. Розподіл часу CPU.          │
│                                                       │
├─────────────────────────────────────────────────────┤
│                                                       │
│  РІВЕНЬ 2:  КЕРУВАННЯ ПАМ'ЯТТЮ                      │
│  Розподіл оперативної пам'яті, віртуальна            │
│  пам'ять, захист пам'яті між процесами               │
│                                                       │
├─────────────────────────────────────────────────────┤
│                                                       │
│  РІВЕНЬ 1:  БАЗОВІ МЕХАНІЗМИ ЯДРА                    │
│  Обробка переривань, перемикання контексту,           │
│  базові примітиви синхронізації                       │
│                                                       │
├─────────────────────────────────────────────────────┤
│                                                       │
│  РІВЕНЬ 0:  АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                    │
│  Процесор (CPU), оперативна пам'ять (RAM),           │
│  диски (HDD/SSD), мережевий адаптер,                 │
│  відеокарта, контролери                               │
│                                                       │
└─────────────────────────────────────────────────────┘
```

**Як це працює на прикладі:**

Уявімо, що користувач хоче відкрити файл з диска:

1. **Рівень 6 (Інтерфейс):** Користувач натискає на файл у графічному провіднику.
2. **Рівень 5 (Файлова система):** Провідник звертається до файлової системи, яка знаходить, де саме на диску зберігається цей файл.
3. **Рівень 4 (Керування пристроями):** Файлова система звертається до драйвера диска з проханням прочитати потрібні блоки даних.
4. **Рівень 3 (Керування процесами):** Процес, що виконує цю операцію, може бути призупинений, поки дані зчитуються з диска, і процесор передається іншому процесу.
5. **Рівень 2 (Керування пам'яттю):** Зчитані дані розміщуються в оперативній пам'яті, виділеній для цього процесу.
6. **Рівень 1 (Базові механізми):** Переривання від диска повідомляє, що дані готові.
7. **Рівень 0 (Апаратура):** Фізичний диск зчитує дані.

### 4.4. Переваги багатошарової архітектури

| Перевага | Пояснення |
|----------|-----------|
| **Модульність** | Кожен рівень — це окремий модуль з чітко визначеними функціями та інтерфейсами. Зміна одного рівня не впливає на інші, якщо інтерфейс залишається незмінним. |
| **Простота відлагодження та тестування** | Рівні можна розробляти та тестувати послідовно, знизу вгору. Спочатку перевіряється рівень 0, потім 1 (який використовує вже перевірений рівень 0), і так далі. |
| **Можливість заміни рівнів** | Окремий рівень можна замінити іншою реалізацією, не чіпаючи решту системи. Наприклад, можна замінити файлову систему NTFS на ext4, не змінюючи драйвери пристроїв. |
| **Абстракція** | Кожен рівень ховає складність нижчих рівнів. Файлова система не повинна знати, як саме працює контролер диска — це турбота рівня нижче. |
| **Верифікованість** | Як показав Дейкстра, можна математично доводити коректність кожного рівня окремо, а потім коректність усієї системи. |

### 4.5. Недоліки багатошарової архітектури

| Недолік | Пояснення |
|---------|-----------|
| **Складність визначення рівнів** | Дуже важко правильно розподілити функції між рівнями. Деякі компоненти логічно залежать одразу від кількох рівнів, і незрозуміло, де їх розмістити. |
| **Зниження продуктивності** | Кожен запит повинен пройти через усі проміжні рівні, навіть якщо деякі з них не виконують корисної роботи для цього конкретного запиту. Кожен перехід між рівнями — це додатковий час. |
| **Негнучкість** | Суворе правило "тільки до сусіднього рівня" іноді є надмірно обмежувальним. У реальних системах компонентам часто потрібен доступ до функцій через кілька рівнів. |
| **Складність реалізації** | Побудувати "ідеальну" багатошарову систему на практиці дуже складно — реальні залежності між компонентами часто не вкладаються у лінійну ієрархію. |

**Важливо:** Через ці недоліки жодна сучасна комерційна ОС не використовує "чисту" багатошарову архітектуру. Натомість сучасні ОС запозичують ідею шарів як **принцип організації**, але дозволяють певні "обхідні шляхи" для підвищення продуктивності.

## 5. Мікроядерна архітектура — короткий огляд

**Мікроядерна архітектура** — це підхід, при якому в режимі ядра (Ring 0) працює лише **мінімальний набір функцій** (мікроядро), а всі інші сервіси ОС (драйвери, файлові системи, мережевий стек) працюють у режимі користувача як окремі процеси.

**Що входить до мікроядра:**
- Базове керування процесами (планувальник CPU)
- Міжпроцесна комунікація (IPC — Inter-Process Communication)
- Базове керування пам'яттю
- Обробка переривань

**Що працює як окремі сервіси в режимі користувача:**
- Драйвери пристроїв
- Файлові системи
- Мережевий стек
- Системи безпеки

**Текстова схема мікроядерної архітектури:**

```
┌─────────────────────────────────────────────────────┐
│            РЕЖИМ КОРИСТУВАЧА (User Mode)              │
│                                                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐   │
│  │ Драйвер  │  │ Файлова  │  │   Мережевий      │   │
│  │ диска    │  │ система  │  │   сервіс         │   │
│  └────┬─────┘  └────┬─────┘  └────────┬─────────┘   │
│       │  IPC        │  IPC            │  IPC         │
├───────┴─────────────┴─────────────────┴──────────────┤
│             МІКРОЯДРО (Kernel Mode)                    │
│  ┌──────────┐  ┌───────┐  ┌────────────────────┐    │
│  │Планувач  │  │  IPC  │  │Базове керування    │    │
│  │  CPU     │  │       │  │   пам'яттю         │    │
│  └──────────┘  └───────┘  └────────────────────┘    │
├─────────────────────────────────────────────────────┤
│             АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                      │
└─────────────────────────────────────────────────────┘
```

**Ключова відмінність від монолітної архітектури:** Якщо в монолітній системі збій драйвера "ронить" всю ОС, то в мікроядерній системі збійний сервіс можна просто перезапустити, не перезавантажуючи систему.

**Приклади:** QNX (автомобілі, медичне обладнання), MINIX, L4.

> **Детально мікроядерну архітектуру ми розглянемо на наступному уроці (Урок 8).**

## 6. Клієнт-серверна модель в ОС

### 6.1. Принцип роботи

**Клієнт-серверна модель** в операційних системах — це архітектурний підхід, при якому компоненти ОС розділяються на дві категорії:

- **Сервери** — процеси, які надають певні послуги (файловий сервер, мережевий сервер, сервер друку).
- **Клієнти** — процеси, які запитують ці послуги (прикладні програми або інші сервери).

Взаємодія між клієнтами та серверами відбувається через **обмін повідомленнями** (message passing): клієнт відправляє запит (request), сервер обробляє його і повертає відповідь (response).

**Текстова схема клієнт-серверної моделі в ОС:**

```
┌─────────────────────────────────────────────────────┐
│              РЕЖИМ КОРИСТУВАЧА                        │
│                                                       │
│  ┌──────────┐   ┌────────────┐   ┌──────────────┐   │
│  │ Клієнт 1 │   │  Клієнт 2  │   │  Клієнт 3    │   │
│  │(браузер) │   │(редактор)  │   │(медіаплеєр)  │   │
│  └────┬─────┘   └─────┬──────┘   └──────┬───────┘   │
│       │ запит         │ запит           │ запит      │
│       v               v                 v            │
│  ┌──────────┐   ┌────────────┐   ┌──────────────┐   │
│  │ Файловий │   │ Мережевий  │   │   Сервер     │   │
│  │ сервер   │   │  сервер    │   │   друку      │   │
│  └────┬─────┘   └─────┬──────┘   └──────┬───────┘   │
│       │               │                 │            │
├───────┴───────────────┴─────────────────┴────────────┤
│                    МІКРОЯДРО                           │
│        (передача повідомлень, базові функції)         │
├─────────────────────────────────────────────────────┤
│              АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                     │
└─────────────────────────────────────────────────────┘
```

### 6.2. Зв'язок з мікроядерною архітектурою

Клієнт-серверна модель тісно пов'язана з мікроядерною архітектурою:
- **Мікроядро** забезпечує лише передачу повідомлень між клієнтами та серверами.
- **Сервери** (файлова система, мережевий стек, драйвери) працюють у режимі користувача.
- **Клієнти** (прикладні програми) звертаються до серверів через мікроядро.

По суті, клієнт-серверна модель — це спосіб організації взаємодії компонентів у мікроядерній ОС.

### 6.3. Переваги та недоліки

**Переваги:**

| Перевага | Пояснення |
|----------|-----------|
| **Модульність** | Сервери можна додавати, видаляти та оновлювати незалежно один від одного. |
| **Надійність** | Збій одного сервера не призводить до краху системи — його можна перезапустити. |
| **Безпека** | Кожен сервер має обмежені привілеї та доступ лише до своїх ресурсів. |
| **Масштабованість** | Модель природно розширюється на мережеві ОС — сервери можуть працювати на різних комп'ютерах у мережі. |

**Недоліки:**

| Недолік | Пояснення |
|---------|-----------|
| **Зниження продуктивності** | Передача повідомлень через мікроядро значно повільніша, ніж прямий виклик функції в монолітному ядрі. |
| **Складність реалізації** | Необхідно ретельно проєктувати протоколи обміну повідомленнями та забезпечувати їхню надійність. |

## 7. Модульна архітектура

### 7.1. Що таке завантажувані модулі ядра

**Модульна архітектура** — це підхід, при якому ядро ОС складається з **базової частини** (основного ядра) та набору **завантажуваних модулів** (Loadable Kernel Modules, LKM), які можна динамічно додавати та вилучати під час роботи системи без перезавантаження.

**Завантажуваний модуль ядра (LKM)** — це фрагмент коду, який розширює можливості працюючого ядра. Модулі завантажуються в ядро за потребою і вивантажуються, коли більше не потрібні.

**Що зазвичай реалізується як модулі:**
- Драйвери пристроїв (відеокарти, мережеві адаптери, USB-пристрої)
- Підтримка файлових систем (NTFS, FAT32, ext4)
- Мережеві протоколи
- Криптографічні алгоритми

**Аналогія:** Уявіть конструктор LEGO. Базова платформа (основне ядро) завжди на місці, а додаткові деталі (модулі) ви приєднуєте або від'єднуєте за потребою: потрібен Wi-Fi — підключаєте модуль Wi-Fi; підключили принтер — завантажується модуль принтера; відключили принтер — модуль вивантажується, звільняючи пам'ять.

**Текстова схема модульної архітектури:**

```
┌─────────────────────────────────────────────────────┐
│              ПРОГРАМИ КОРИСТУВАЧА                     │
├─────────────────────────────────────────────────────┤
│                                                       │
│  ЯДРО (базова частина) + ЗАВАНТАЖУВАНІ МОДУЛІ        │
│                                                       │
│  ┌──────────────────────────────────────────┐        │
│  │         ОСНОВНЕ ЯДРО (постійне)          │        │
│  │  Планувальник, пам'ять, IPC, безпека     │        │
│  └──┬──────┬──────┬──────┬──────┬───────────┘        │
│     │      │      │      │      │                     │
│  ┌──┴──┐┌──┴──┐┌──┴──┐┌──┴──┐┌──┴──┐                │
│  │Модуль││Модуль││Модуль││Модуль││Модуль│  <-- можна  │
│  │Wi-Fi ││NTFS  ││USB   ││Звук  ││GPU   │  завантажу- │
│  │.ko   ││.ko   ││.ko   ││.ko   ││.ko   │  вати та    │
│  └──────┘└──────┘└──────┘└──────┘└──────┘  вивантажу-  │
│                                             вати без   │
│                                             перезаван- │
│                                             таження    │
├─────────────────────────────────────────────────────┤
│             АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                      │
└─────────────────────────────────────────────────────┘
```

### 7.2. Як працюють LKM у Linux

Linux — найяскравіший приклад модульної архітектури. Хоча ядро Linux є **монолітним** (усі модулі працюють у режимі ядра, Ring 0), воно підтримує **динамічне завантаження модулів** з 1995 року (Linux 1.2).

**Основні команди для роботи з модулями:**

| Команда | Призначення | Приклад |
|---------|-------------|---------|
| `lsmod` | Показати список завантажених модулів | `lsmod` — виведе всі активні модулі |
| `modprobe` | Завантажити модуль (з урахуванням залежностей) | `modprobe usb_storage` — завантажити драйвер USB-накопичувачів |
| `rmmod` | Вивантажити модуль | `rmmod usb_storage` — вивантажити драйвер |
| `modinfo` | Показати інформацію про модуль | `modinfo e1000` — інформація про мережевий драйвер |

**Де зберігаються модулі:**
- Каталог: `/lib/modules/<версія_ядра>/`
- Розширення файлів: `.ko` (kernel object — об'єкт ядра)

**Важливо:** Хоча модулі завантажуються динамічно, вони працюють **в режимі ядра** (Ring 0), тобто мають повний доступ до апаратури та пам'яті. Це відрізняє модульну архітектуру Linux від мікроядерної, де сервіси працюють у режимі користувача.

### 7.3. Переваги модульної архітектури

| Перевага | Пояснення |
|----------|-----------|
| **Не потрібна перекомпіляція ядра** | Нові драйвери та функції додаються без перебудови всього ядра. |
| **Економія пам'яті** | Модулі завантажуються лише тоді, коли вони потрібні, та вивантажуються, коли стають непотрібними. |
| **Швидке оновлення** | Оновити драйвер можна без перезавантаження системи — вивантажити старий модуль і завантажити новий. |
| **Зручність розробки** | Розробники драйверів можуть тестувати свій код без повної перебудови та перезавантаження ядра. |
| **Висока продуктивність** | На відміну від мікроядра, модулі працюють у режимі ядра — немає витрат на IPC (міжпроцесну комунікацію). |

## 8. Віртуальні машини як архітектурний підхід

### 8.1. Що таке віртуалізація та гіпервізор

**Віртуалізація** — це технологія, яка дозволяє на одному фізичному комп'ютері одночасно запускати **кілька ізольованих операційних систем**, кожна з яких "думає", що має у своєму розпорядженні окремий фізичний комп'ютер.

**Гіпервізор** (hypervisor), або **монітор віртуальних машин** (VMM — Virtual Machine Monitor), — це програмне забезпечення, яке створює та керує віртуальними машинами. Гіпервізор розподіляє ресурси фізичного комп'ютера (процесор, пам'ять, диск, мережу) між віртуальними машинами.

**Віртуальна машина (ВМ)** — це програмна емуляція фізичного комп'ютера, яка має свій "віртуальний" процесор, пам'ять, диск та мережу.

**Аналогія:** Уявіть великий будинок (фізичний комп'ютер), який розділений на кілька квартир (віртуальних машин). Кожна квартира має свою кухню, ванну та кімнати. Мешканці кожної квартири (операційні системи) живуть незалежно і не заважають сусідам. Керуючий будинком (гіпервізор) розподіляє спільні ресурси: воду, електрику, опалення.

### 8.2. Типи гіпервізорів

Існують два основних типи гіпервізорів:

**Тип 1 — "Bare-metal" (на голому залізі):**

Гіпервізор працює **безпосередньо на апаратному забезпеченні**, без проміжної операційної системи. Це забезпечує найвищу продуктивність та безпеку.

```
┌───────────────┐  ┌───────────────┐  ┌───────────────┐
│ Віртуальна    │  │ Віртуальна    │  │ Віртуальна    │
│ машина 1      │  │ машина 2      │  │ машина 3      │
│ (Windows)     │  │ (Linux)       │  │ (macOS)       │
├───────────────┴──┴───────────────┴──┴───────────────┤
│         ГІПЕРВІЗОР ТИП 1 (Bare-metal)                │
│    (VMware ESXi, Microsoft Hyper-V, KVM, Xen)       │
├─────────────────────────────────────────────────────┤
│              АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                     │
└─────────────────────────────────────────────────────┘
```

**Приклади:** VMware ESXi, Microsoft Hyper-V, KVM (вбудований у Linux), Xen.
**Використання:** сервери, хмарні платформи (AWS, Azure, Google Cloud), дата-центри.

**Тип 2 — "Hosted" (на базі ОС):**

Гіпервізор працює **як звичайна програма поверх існуючої ОС**. Простіший у використанні, але менш продуктивний через додатковий рівень — ОС-хост.

```
┌───────────────┐  ┌───────────────┐
│ Віртуальна    │  │ Віртуальна    │
│ машина 1      │  │ машина 2      │
│ (Ubuntu)      │  │ (Windows)     │
├───────────────┴──┴───────────────┤
│    ГІПЕРВІЗОР ТИП 2 (Hosted)     │
│ (VirtualBox, VMware Workstation) │
├─────────────────────────────────┤
│  ОСНОВНА ОС (Windows / macOS)    │
├─────────────────────────────────┤
│     АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ         │
└─────────────────────────────────┘
```

**Приклади:** Oracle VirtualBox (безкоштовний), VMware Workstation, Parallels Desktop (macOS).
**Використання:** розробка та тестування ПЗ, навчання, запуск інших ОС на домашньому комп'ютері.

**Порівняння типів гіпервізорів:**

| Характеристика | Тип 1 (Bare-metal) | Тип 2 (Hosted) |
|---|---|---|
| Працює на | Безпосередньо на апаратурі | Поверх існуючої ОС |
| Продуктивність | Висока | Нижча (через ОС-хост) |
| Безпека | Вища (ізоляція) | Залежить від ОС-хоста |
| Використання | Сервери, хмари, дата-центри | Домашнє використання, навчання |
| Приклади | VMware ESXi, Hyper-V, KVM | VirtualBox, VMware Workstation |

### 8.3. Віртуалізація як архітектурний підхід в ОС

Віртуалізація — це не просто зручний інструмент, а повноцінний **архітектурний підхід** до проєктування систем:

1. **Ізоляція:** Кожна ВМ повністю ізольована від інших. Якщо одна ВМ "зависне" або буде заражена вірусом — інші продовжать працювати нормально.
2. **Абстракція апаратури:** Гіпервізор приховує деталі фізичного обладнання. ОС у ВМ працює зі стандартним "віртуальним" обладнанням, що спрощує переносимість.
3. **Консолідація серверів:** Замість 10 фізичних серверів можна використовувати 1 потужний сервер з 10 ВМ — це економить обладнання, електроенергію та місце.
4. **Хмарні обчислення:** AWS, Azure, Google Cloud побудовані на віртуалізації. Коли ви "орендуєте сервер" у хмарі — ви отримуєте віртуальну машину.

## 10. Реальні приклади архітектур у сучасних ОС

### 10.1. Windows NT — гібридна архітектура

**Windows NT** (New Technology) — це сімейство ОС Microsoft, яке лежить в основі всіх сучасних версій Windows: XP, Vista, 7, 8, 10, 11, а також Windows Server.

Windows NT має **гібридну архітектуру**, яка поєднує елементи монолітного, багатошарового та мікроядерного підходів.

**Структура Windows NT:**

```
┌─────────────────────────────────────────────────────┐
│               РЕЖИМ КОРИСТУВАЧА                       │
│                                                       │
│  ┌─────────────────────────────────────────────┐     │
│  │  Підсистеми середовища:                      │     │
│  │  Win32, POSIX, OS/2                          │     │
│  ├─────────────────────────────────────────────┤     │
│  │  Інтегральні підсистеми:                     │     │
│  │  Безпека, Сервер робочих станцій             │     │
│  └─────────────────────────────────────────────┘     │
├─────────────────────────────────────────────────────┤
│               РЕЖИМ ЯДРА                              │
│                                                       │
│  ┌─────────────────────────────────────────────┐     │
│  │  Executive (Виконавча підсистема):           │     │
│  │  Менеджер вводу-виводу, менеджер об'єктів,   │     │
│  │  менеджер процесів, менеджер пам'яті,        │     │
│  │  менеджер безпеки, менеджер кешу             │     │
│  ├─────────────────────────────────────────────┤     │
│  │  Ядро (Kernel):                              │     │
│  │  Планувальник потоків, обробка переривань,    │     │
│  │  синхронізація, диспетчеризація              │     │
│  ├─────────────────────────────────────────────┤     │
│  │  Драйвери пристроїв (Kernel-Mode Drivers)   │     │
│  ├─────────────────────────────────────────────┤     │
│  │  HAL (Hardware Abstraction Layer):           │     │
│  │  Рівень абстракції апаратури                 │     │
│  │  (приховує відмінності між платформами)      │     │
│  └─────────────────────────────────────────────┘     │
├─────────────────────────────────────────────────────┤
│              АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                     │
└─────────────────────────────────────────────────────┘
```

**Ключові особливості Windows NT:**

- **HAL (Hardware Abstraction Layer)** — рівень абстракції апаратури. Приховує відмінності між різними платформами (x86, ARM), дозволяючи решті ядра працювати однаково на різному обладнанні. Це елемент багатошарового підходу.
- **Executive** — виконавча підсистема, яка містить менеджери різних ресурсів (пам'ять, процеси, ввід-вивід). Це елемент модульного підходу.
- **Підсистеми середовища** працюють у режимі користувача — це елемент мікроядерного підходу (натхненний Mach).
- Водночас драйвери працюють у режимі ядра для продуктивності — це елемент монолітного підходу.

**Чому "гібрид":** Windows NT не є ні чисто монолітною, ні чисто мікроядерною. Вона запозичує ідеї з різних підходів, створюючи баланс між продуктивністю та модульністю.

### 10.2. Linux — монолітне ядро з модулями

**Linux** — це операційна система з **монолітним ядром**, створена Лінусом Торвальдсом у 1991 році. Усі основні компоненти (керування процесами, пам'яттю, файлові системи, драйвери, мережевий стек) працюють в єдиному адресному просторі в режимі ядра.

Проте сучасне ядро Linux активно використовує **завантажувані модулі ядра (LKM)**, що робить його значно гнучкішим за класичне монолітне ядро.

```
┌─────────────────────────────────────────────────────┐
│              ПРОГРАМИ КОРИСТУВАЧА                     │
│           (браузер, термінал, офіс)                  │
├─────────────────────────────────────────────────────┤
│  Бібліотеки (glibc) / Системні виклики               │
├─────────────────────────────────────────────────────┤
│                                                       │
│         МОНОЛІТНЕ ЯДРО LINUX (Ring 0)                │
│                                                       │
│  ┌──────────────────────────────────────────┐        │
│  │    Основна частина ядра (постійна):       │        │
│  │    Планувальник, керування пам'яттю,      │        │
│  │    мережевий стек, VFS                    │        │
│  └──┬──────┬──────┬──────┬──────┬───────────┘        │
│  ┌──┴──┐┌──┴──┐┌──┴──┐┌──┴──┐┌──┴──┐                │
│  │ext4 ││NTFS ││e1000││WiFi ││USB  │ Завантажувані   │
│  │.ko  ││.ko  ││.ko  ││.ko  ││.ko  │ модулі (LKM)   │
│  └─────┘└─────┘└─────┘└─────┘└─────┘                 │
│                                                       │
├─────────────────────────────────────────────────────┤
│             АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                      │
└─────────────────────────────────────────────────────┘
```

**Ключові факти про Linux:**
- Ядро Linux має понад **36 мільйонів рядків** коду.
- Підтримує тисячі завантажуваних модулів для різних пристроїв та файлових систем.
- Це найпоширеніша серверна ОС у світі (понад 90% серверів в Інтернеті працюють на Linux).
- Android (ОС для смартфонів) базується на ядрі Linux.

### 10.3. macOS — гібрид XNU

**macOS** (раніше Mac OS X) використовує ядро **XNU** (X is Not Unix — "X — це не Unix"), яке є **гібридним** ядром, що поєднує три основних компоненти:

```
┌─────────────────────────────────────────────────────┐
│             ПРОГРАМИ КОРИСТУВАЧА                      │
│          (Safari, Finder, Terminal)                   │
├─────────────────────────────────────────────────────┤
│             Frameworks (Cocoa, Carbon)                │
├─────────────────────────────────────────────────────┤
│                                                       │
│              ЯДРО XNU (гібрид)                        │
│                                                       │
│  ┌────────────────────────────────────────────┐      │
│  │  BSD-підсистема:                            │      │
│  │  POSIX API, мережевий стек, файлові         │      │
│  │  системи (APFS, HFS+), безпека, процеси    │      │
│  ├────────────────────────────────────────────┤      │
│  │  Мікроядро Mach:                            │      │
│  │  Керування пам'яттю, планування потоків,    │      │
│  │  IPC (Mach ports), абстракція задач         │      │
│  ├────────────────────────────────────────────┤      │
│  │  IOKit (фреймворк драйверів):              │      │
│  │  Об'єктно-орієнтований C++ фреймворк       │      │
│  │  для написання драйверів пристроїв          │      │
│  └────────────────────────────────────────────┘      │
│                                                       │
├─────────────────────────────────────────────────────┤
│             АПАРАТНЕ ЗАБЕЗПЕЧЕННЯ                      │
│        (Apple Silicon M1/M2/M3/M4 або Intel)         │
└─────────────────────────────────────────────────────┘
```

**Три компоненти XNU:**

1. **Mach** (мікроядро від Carnegie Mellon University) — забезпечує низькорівневі функції: керування пам'яттю, планування потоків, IPC через Mach ports, абстракцію задач (tasks) та потоків (threads).

2. **BSD** (підсистема від FreeBSD) — надає POSIX-сумісний API, мережевий стек, файлові системи (APFS, HFS+), модель процесів UNIX, систему безпеки (права доступу, користувачі та групи).

3. **IOKit** — об'єктно-орієнтований фреймворк для написання драйверів пристроїв, написаний на обмеженій підмножині C++. Підтримує гаряче підключення пристроїв (hot-plug) та автоматичну конфігурацію.

**Чому XNU — це гібрид:** XNU бере **структуру мікроядра** від Mach (розділення на задачі, повідомлення, порти), але реалізує більшість сервісів **в режимі ядра** (як монолітне ядро) для підвищення продуктивності. BSD-підсистема працює "поверх" Mach, але в тому ж адресному просторі.

**Цікавий факт:** macOS отримала офіційну сертифікацію UNIX у 2007 році (Mac OS X 10.5 Leopard), тоді як Linux, попри свою Unix-подібність, формально не є сертифікованим UNIX.

## 11. Підсумок

У цьому уроці ми розглянули основні архітектурні підходи до побудови операційних систем:

1. **Монолітна архітектура** — найпростіший підхід, де вся ОС працює як одна програма в режимі ядра. Висока продуктивність, але низька надійність. Приклади: ранні UNIX, MS-DOS.

2. **Багатошарова (рівнева) архітектура** — ідея Дейкстри (система THE, 1968), де ОС організована як ієрархія рівнів. Кожен рівень використовує лише послуги нижчого рівня. Забезпечує модульність та простоту тестування, але знижує продуктивність через проходження всіх рівнів.

3. **Мікроядерна архітектура** — мінімальне ядро з сервісами в режимі користувача. Висока надійність та безпека, але нижча продуктивність. Приклади: QNX, MINIX, L4.

4. **Клієнт-серверна модель** — сервіси ОС працюють як сервери, програми — як клієнти, взаємодія через повідомлення. Тісно пов'язана з мікроядерною архітектурою.

5. **Модульна архітектура** — основне ядро плюс завантажувані модулі (LKM). Поєднує продуктивність монолітного ядра з гнучкістю модулів. Приклад: Linux.

6. **Віртуальні машини** — гіпервізор створює ізольовані середовища для кількох ОС на одному комп'ютері. Два типи: bare-metal (ESXi, Hyper-V) та hosted (VirtualBox).

7. **Реальні ОС** використовують **гібридні** підходи: Windows NT поєднує елементи монолітної, багатошарової та мікроядерної архітектур; Linux — монолітне ядро з модулями; macOS (XNU) — гібрид мікроядра Mach та монолітної BSD-підсистеми.

**Головний висновок:** Не існує "ідеальної" архітектури ОС. Кожен підхід має свої переваги та недоліки, і сучасні ОС комбінують різні підходи для досягнення оптимального балансу між продуктивністю, надійністю, безпекою та зручністю.

| Термін | Визначення |
|--------|------------|
| **Архітектура ОС** | Загальна структурна організація операційної системи, що визначає склад компонентів, спосіб їх взаємодії та розподіл між режимами ядра і користувача. |
| **Монолітна архітектура** | Архітектура ОС, при якій уся система працює як одна програма в єдиному адресному просторі в режимі ядра. Приклади: ранні UNIX, MS-DOS. |
| **Багатошарова (рівнева) архітектура** | Архітектура ОС, організована як ієрархія рівнів, де кожен рівень використовує лише послуги безпосередньо нижчого рівня. Запропонована Дейкстрою (1968). |
| **Система THE** | Операційна система, розроблена Едсгером Дейкстрою у 1968 році в Технологічному університеті Ейндговена (Нідерланди). Перший приклад багатошарової архітектури ОС. |
| **Едсгер Дейкстра (Edsger W. Dijkstra)** | Нідерландський вчений (1930-2002), один із засновників структурного програмування, автор багатошарової архітектури ОС, семафорів та алгоритму найкоротшого шляху. |
| **Мікроядерна архітектура** | Архітектура ОС, при якій в режимі ядра працює лише мінімальний набір функцій (мікроядро), а всі інші сервіси — в режимі користувача. Приклади: QNX, MINIX. |
| **Клієнт-серверна модель** | Архітектурний підхід, при якому компоненти ОС розділені на сервери (надають послуги) та клієнти (запитують послуги), які взаємодіють через обмін повідомленнями. |
| **Модульна архітектура** | Архітектура ОС, при якій ядро складається з основної частини та набору завантажуваних модулів, що підключаються динамічно без перезавантаження. |
| **LKM (Loadable Kernel Module)** | Завантажуваний модуль ядра — фрагмент коду (зазвичай драйвер), який можна динамічно завантажити в ядро Linux або вивантажити з нього без перезавантаження системи. |
| **Віртуальна машина (ВМ)** | Програмна емуляція фізичного комп'ютера, яка має свій "віртуальний" процесор, пам'ять та диск і дозволяє запускати окрему ОС. |
| **Гіпервізор (Hypervisor)** | Програмне забезпечення (монітор віртуальних машин), яке створює та керує віртуальними машинами, розподіляючи ресурси фізичного комп'ютера між ними. |
| **Гіпервізор Типу 1 (Bare-metal)** | Гіпервізор, що працює безпосередньо на апаратному забезпеченні без проміжної ОС. Приклади: VMware ESXi, Hyper-V, KVM. |
| **Гіпервізор Типу 2 (Hosted)** | Гіпервізор, що працює як програма поверх існуючої ОС. Приклади: VirtualBox, VMware Workstation, Parallels Desktop. |
| **Гібридна архітектура** | Архітектура ОС, яка поєднує елементи різних підходів (монолітного, мікроядерного, багатошарового) для досягнення балансу між продуктивністю та модульністю. |
| **Windows NT** | Сімейство ОС Microsoft з гібридною архітектурою, що є основою всіх сучасних версій Windows (XP, 7, 10, 11). |
| **HAL (Hardware Abstraction Layer)** | Рівень абстракції апаратури у Windows NT — компонент, який приховує відмінності між апаратними платформами та надає ядру уніфікований інтерфейс. |
| **Executive (Windows NT)** | Виконавча підсистема Windows NT, яка містить менеджери ресурсів: менеджер пам'яті, менеджер процесів, менеджер вводу-виводу та інші. |
| **XNU (X is Not Unix)** | Гібридне ядро macOS, що поєднує мікроядро Mach, BSD-підсистему та фреймворк драйверів IOKit. Є основою для macOS, iOS, iPadOS, watchOS. |
| **Mach** | Мікроядро, розроблене в Carnegie Mellon University, яке є основою ядра XNU в macOS. Забезпечує керування пам'яттю, IPC та планування потоків. |
| **IOKit** | Об'єктно-орієнтований фреймворк для написання драйверів пристроїв у macOS, написаний на обмеженій підмножині C++. |
| **IPC (Inter-Process Communication)** | Міжпроцесна комунікація — механізм обміну даними між процесами. У мікроядерних ОС — основний спосіб взаємодії сервісів з ядром і між собою. |
| **Message Passing (передача повідомлень)** | Механізм комунікації, при якому процеси обмінюються даними через повідомлення, а не через спільну пам'ять. Використовується у клієнт-серверній та мікроядерній моделях. |
| **Семафор** | Механізм синхронізації процесів, винайдений Дейкстрою. Використовує операції P (зайняти) та V (звільнити) для координації доступу до спільних ресурсів. |
| **Віртуалізація** | Технологія, що дозволяє створювати віртуальні версії апаратних ресурсів (процесор, пам'ять, диск) для одночасного запуску кількох ОС на одному фізичному комп'ютері. |
| **KVM (Kernel-based Virtual Machine)** | Гіпервізор Типу 1, вбудований безпосередньо в ядро Linux. Перетворює Linux на повноцінну платформу віртуалізації. |
| **modprobe** | Команда Linux для завантаження модулів ядра з автоматичним вирішенням залежностей. |
| **lsmod** | Команда Linux для перегляду списку поточних завантажених модулів ядра. |
| **Консолідація серверів** | Практика об'єднання кількох фізичних серверів у віртуальні машини на одному потужному сервері для економії ресурсів. |
