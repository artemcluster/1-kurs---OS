## 1. Проблема нестачі оперативної пам'яті

### 1.1. Суть проблеми

Оперативна пам'ять (RAM) — це швидка, але **обмежена** за обсягом пам'ять, у якій зберігаються дані програм, що виконуються. Коли користувач запускає одночасно браузер з десятками вкладок, текстовий редактор, середовище розробки та відеоплеєр, сумарний обсяг даних може перевищити фізично встановлену RAM.

**Типова ситуація:**

| Програма | Потреба в RAM |
|----------|---------------|
| Операційна система (Windows 11) | 2-4 ГБ |
| Браузер Chrome (20 вкладок) | 2-4 ГБ |
| Середовище розробки (VS Code) | 0,5-1 ГБ |
| Відеоредактор | 2-8 ГБ |
| Фонові процеси | 1-2 ГБ |
| **Разом** | **7,5-19 ГБ** |

Якщо у комп'ютері встановлено лише 8 ГБ RAM, а програми потребують 12 ГБ, виникає дефіцит у 4 ГБ.

### 1.2. Що станеться без віртуальної пам'яті?

Без механізму віртуальної пам'яті операційна система мала б лише два варіанти:

1. **Відмовити в запуску нових програм** — з'явиться повідомлення "Недостатньо пам'яті"
2. **Аварійно завершити вже запущені програми** — втрата незбережених даних

Обидва варіанти неприйнятні для користувача. Саме тому було створено механізм **віртуальної пам'яті**.

## 2. Віртуальна пам'ять — концепція

### 2.1. Що таке віртуальна пам'ять

**Віртуальна пам'ять** — це технологія управління пам'яттю, яка створює для кожного процесу **ілюзію великого безперервного адресного простору**, навіть якщо фізичної RAM недостатньо.

Кожен процес "думає", що має у своєму розпорядженні весь адресний простір (наприклад, до 128 ТБ у 64-бітній Windows), хоча реальної оперативної пам'яті може бути лише 8 або 16 ГБ.

### 2.2. Як це працює: загальна ідея

Основний принцип простий: **частина даних зберігається в RAM (швидко), а частина — на диску (повільніше)**. Операційна система автоматично переміщує дані між RAM і диском залежно від того, що потрібно процесору в даний момент.

```
                  Віртуальна пам'ять
    ┌──────────────────────────────────────────────┐
    │                                              │
    │   Адресний простір процесу (великий)         │
    │                                              │
    └────────────────┬─────────────────────────────┘
                     │
           ┌─────────┴──────────┐
           │                    │
           ▼                    ▼
    ┌──────────────┐    ┌──────────────────┐
    │   RAM (ОЗП)  │    │ Диск (HDD/SSD)   │
    │              │    │                  │
    │ Активні дані │    │ Файл підкачки    │
    │ (швидко)     │    │ (повільніше)     │
    │              │    │                  │
    │  8-32 ГБ     │    │ pagefile.sys /   │
    │              │    │ swap partition   │
    └──────────────┘    └──────────────────┘
```

### 2.3. Адресний простір: віртуальні та фізичні адреси

Процесор працює з **віртуальними адресами**, а не з фізичними. Спеціальний апаратний компонент — **MMU (Memory Management Unit)** — автоматично перетворює віртуальну адресу у фізичну під час кожного звернення до пам'яті.

```
    Процес А                Процес Б
    ┌──────────┐            ┌──────────┐
    │Адреса 0  │            │Адреса 0  │   <-- Обидва "бачать" адресу 0,
    │Адреса 1  │            │Адреса 1  │       але це різні фізичні місця
    │  ...     │            │  ...     │
    │Адреса N  │            │Адреса N  │
    └────┬─────┘            └────┬─────┘
         │                       │
         └───────┬───────────────┘
                 │
            ┌────▼────┐
            │  MMU    │   <-- Перетворює віртуальну адресу у фізичну
            └────┬────┘
                 │
         ┌───────┴───────────────┐
         ▼                       ▼
    ┌──────────┐           ┌──────────┐
    │ RAM:     │           │ RAM:     │
    │ Фіз. 100│           │ Фіз. 500│
    └──────────┘           └──────────┘
```

Це забезпечує **ізоляцію процесів**: кожен процес працює у своєму віртуальному адресному просторі та не може випадково або навмисно змінити дані іншого процесу.

## 3. Принцип локальності

### 3.1. Чому віртуальна пам'ять працює ефективно

На перший погляд, зберігання частини даних на повільному диску має критично сповільнити роботу. Але на практиці це не так, завдяки **принципу локальності** (locality of reference) — фундаментальній властивості поведінки програм.

Суть: програми в будь-який момент часу активно використовують лише **невелику частину** свого адресного простору. Решта даних "лежить без діла" і може спокійно зберігатися на диску.

### 3.2. Часова локальність (temporal locality)

**Часова локальність** означає: якщо до певної адреси пам'яті звернулися зараз, з високою ймовірністю до неї звернуться знову **найближчим часом**.

**Приклади:**
- Змінна-лічильник у циклі `for (i = 0; i < 1000; i++)` — до неї звертаються 1000 разів поспіль
- Часто використовувані глобальні змінні програми
- Код функції, що викликається багаторазово

```
    Часова локальність:

    Час:    t1    t2    t3    t4    t5    t6
    Адреса: [A]   [B]   [A]   [A]   [C]   [A]
                         ↑     ↑           ↑
                  Адреса A використовується повторно
```

### 3.3. Просторова локальність (spatial locality)

**Просторова локальність** означає: якщо до певної адреси звернулися, то найближчим часом звернуться до **сусідніх адрес**.

**Приклади:**
- Послідовне читання елементів масиву: `arr[0]`, `arr[1]`, `arr[2]`...
- Виконання інструкцій програми одна за одною (інструкції зберігаються послідовно в пам'яті)
- Читання полів структури даних

```
    Просторова локальність:

    Пам'ять: [  ][  ][ A][ B][ C][ D][  ][  ][  ]
                       ↑   ↑   ↑   ↑
              Якщо звернулися до A, скоро
              звернуться до B, C, D (сусідні)
```

### 3.4. Робочий набір (Working Set)

На основі принципу локальності сформовано поняття **робочого набору** — це множина сторінок пам'яті, які процес активно використовує протягом певного проміжку часу.

Якщо операційна система тримає робочий набір кожного процесу в RAM, кількість звернень до диска буде мінімальною, і система працюватиме швидко.

## 4. Файл підкачки (swap file / page file)

### 4.1. Що таке файл підкачки

**Файл підкачки** — це спеціальний файл (або розділ) на диску, який операційна система використовує як розширення оперативної пам'яті. Коли RAM заповнена, ОС переміщує частину даних з RAM у файл підкачки, звільняючи місце для активних програм.

### 4.2. Windows: pagefile.sys

У Windows файл підкачки називається **pagefile.sys** і зазвичай розташований у кореневому каталозі системного диска (`C:\pagefile.sys`). Це прихований системний файл.

**Налаштування pagefile.sys у Windows 10/11:**

1. Відкрити **Пошук** та ввести "Перегляд додаткових параметрів системи" ("View advanced system settings")
2. У розділі **Швидкодія** натиснути **Параметри** (Settings)
3. Перейти на вкладку **Додатково** (Advanced)
4. У розділі **Віртуальна пам'ять** натиснути **Змінити** (Change)
5. Зняти позначку "Автоматично вибирати обсяг файлу підкачки" для ручного налаштування

**Параметри налаштування:**

| Параметр | Опис |
|----------|------|
| **Автоматичне управління** | Windows сам визначає розмір (рекомендовано для більшості користувачів) |
| **Нестандартний розмір** | Користувач задає початковий та максимальний розмір |
| **Розмір за вибором системи** | Windows визначає розмір на основі поточних потреб |
| **Без файлу підкачки** | Вимкнено (не рекомендується!) |

**Порада:** якщо задаєте розмір вручну, встановіть **однакові** початковий та максимальний розміри (наприклад, 4096 МБ / 4096 МБ). Це запобігає фрагментації файлу підкачки та забезпечує стабільнішу роботу.

### 4.3. Linux: swap partition та swap file

У Linux підкачка реалізується двома способами:

**Swap-розділ (swap partition):**
- Окремий розділ диска, відформатований спеціально для підкачки
- Традиційний спосіб, використовується десятиліттями
- Створюється при встановленні системи
- Перевага: трохи вища продуктивність, передбачувана поведінка
- Недолік: складно змінити розмір після створення

**Swap-файл (swap file):**
- Звичайний файл у файловій системі, налаштований як простір підкачки
- Сучасний спосіб (за замовчуванням в Ubuntu, Fedora та інших дистрибутивах)
- Легко створити, змінити розмір або видалити
- Продуктивність на сучасних ядрах Linux (починаючи з 2.6) практично не відрізняється від swap-розділу

**Основні команди Linux для роботи зі swap:**

```bash
# Переглянути поточний swap
swapon --show

# Створити swap-файл розміром 4 ГБ
sudo fallocate -l 4G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# Параметр swappiness (наскільки агресивно ОС використовує swap)
cat /proc/sys/vm/swappiness
# Значення 0-100: менше = рідше свопить, більше = частіше
# Типово: 60, для десктопу рекомендують: 10-20
```

### 4.4. Рекомендований розмір файлу підкачки

Старе правило "2x RAM" було актуальним, коли комп'ютери мали 64-256 МБ оперативної пам'яті. Сучасні рекомендації залежать від обсягу RAM та задач:

| Обсяг RAM | Рекомендований swap (без гібернації) | Зі сплячим режимом (гібернація) |
|-----------|--------------------------------------|----------------------------------|
| 2 ГБ | 4 ГБ (2x) | 6 ГБ |
| 4 ГБ | 4-6 ГБ (1-1,5x) | 8 ГБ |
| 8 ГБ | 4-8 ГБ (0,5-1x) | 12 ГБ |
| 16 ГБ | 4-8 ГБ (0,25-0,5x) | 20 ГБ |
| 32 ГБ | 4-8 ГБ (0,125-0,25x) | 36 ГБ |
| 64 ГБ | 4-8 ГБ | 68 ГБ |

**Загальні правила:**
- **Менше 8 ГБ RAM** — swap 1x-2x від RAM
- **8-16 ГБ RAM** — swap 0,5x-1x від RAM (4-16 ГБ)
- **Понад 16 ГБ RAM** — swap 4-8 ГБ зазвичай достатньо
- **Для гібернації** — swap має бути не менше обсягу RAM (вміст RAM зберігається на диск)
- Windows за замовчуванням встановлює початковий розмір **1,5x RAM** з максимумом до **3x RAM**

## 5. Механізм роботи віртуальної пам'яті

### 5.1. Сторінки (pages) та фрейми (frames)

Віртуальна пам'ять поділяє весь адресний простір на блоки однакового розміру — **сторінки** (pages).

- **Сторінка (page)** — блок віртуальної пам'яті фіксованого розміру
- **Фрейм (frame, page frame)** — блок фізичної пам'яті (RAM) такого ж розміру
- **Типовий розмір** — **4 КБ** (4096 байт) у більшості сучасних ОС

```
    Віртуальна пам'ять              Фізична пам'ять (RAM)
    ┌─────────────────┐             ┌─────────────────┐
    │  Сторінка 0     │────────────→│  Фрейм 3        │
    ├─────────────────┤             ├─────────────────┤
    │  Сторінка 1     │──┐          │  Фрейм 0        │
    ├─────────────────┤  │          ├─────────────────┤
    │  Сторінка 2     │──┼─────────→│  Фрейм 1        │
    ├─────────────────┤  │          ├─────────────────┤
    │  Сторінка 3     │  └─────────→│  Фрейм 2        │
    ├─────────────────┤             └─────────────────┘
    │  Сторінка 4     │─ ─ ─ ─ ─ ─ ─ ─ ┐
    ├─────────────────┤                  │
    │  Сторінка 5     │─ ─ ─ ─ ─ ┐      │    Диск
    └─────────────────┘           │      │  ┌─────────┐
                               На диску  ├─→│ Swap    │
                                  └──────┘  └─────────┘
```

**Таблиця сторінок (Page Table)** — структура даних, яку ОС створює для кожного процесу. Вона зберігає відповідність між номером віртуальної сторінки та номером фізичного фрейму (або вказує, що сторінка зараз на диску).

### 5.2. Page fault (промах сторінки)

**Page fault** — це подія, яка виникає, коли процесор намагається звернутися до сторінки, якої **немає у фізичній пам'яті** (вона на диску або ще ніколи не завантажувалась).

**Що відбувається при page fault:**

```
    1. Процес звертається до адреси
       ┌─────────────────┐
       │ MOV EAX, [addr] │
       └────────┬────────┘
                │
    2. MMU перевіряє таблицю сторінок
                │
       ┌────────▼────────┐
       │ Сторінка в RAM? │
       └────────┬────────┘
          Ні!   │
                │
    3. Генерується переривання PAGE FAULT
                │
       ┌────────▼────────────────────────────────┐
       │ Обробник переривання ОС:                │
       │  a) Знайти сторінку на диску (у swap)    │
       │  b) Знайти вільний фрейм у RAM          │
       │     (або звільнити зайнятий)             │
       │  c) Завантажити сторінку з диска в RAM   │
       │  d) Оновити таблицю сторінок             │
       │  e) Повернути керування процесу          │
       └─────────────────────────────────────────┘
                │
    4. Процес продовжує роботу (не знає, що був page fault)
```

Page fault — це **нормальна** подія; вона відбувається регулярно. Проблема виникає лише тоді, коли page fault-ів стає **занадто багато**.

### 5.3. Swapping (підкачка)

**Swapping** (підкачка) — це процес переміщення сторінок між RAM та диском:

- **Swap out (вивантаження)** — сторінка переміщується з RAM на диск, щоб звільнити фрейм
- **Swap in (завантаження)** — сторінка завантажується з диска в RAM, коли вона знову потрібна

```
    Swap Out (вивантаження):     Swap In (завантаження):

    RAM         Диск             RAM         Диск
    ┌────┐    ┌────────┐         ┌────┐    ┌────────┐
    │ P5 │═══→│ P5     │         │    │←═══│ P5     │
    │    │    │        │         │ P5 │    │        │
    └────┘    └────────┘         └────┘    └────────┘
    Фрейм                       Фрейм
    звільнено                   зайнято
```

ОС обирає для вивантаження ті сторінки, які **найменш ймовірно** знадобляться найближчим часом (на основі алгоритмів заміщення).

### 5.4. Thrashing (пробуксовка)

**Thrashing** (пробуксовка) — це критичний стан системи, коли вона витрачає **більше часу на переміщення сторінок між RAM та диском**, ніж на корисну роботу.

**Як це відбувається:**

1. Запущено забагато програм, RAM перевантажена
2. ОС вивантажує сторінки процесу А, щоб завантажити сторінки процесу Б
3. Процес А знову потребує свої сторінки --> page fault --> вивантажуються сторінки процесу Б
4. Процес Б знову потребує свої сторінки --> page fault --> і так по колу
5. Диск працює на 100%, процесор простоює, система "зависає"

```
    Нормальна робота:          Thrashing:

    Корисна    ████████████    Корисна    ██░░░░░░░░░░
    робота     ████████████    робота     ░░░░░░░░░░░░

    Підкачка   ██              Підкачка   ████████████
    (swap)                     (swap)     ████████████

    Продуктивність: висока     Продуктивність: мінімальна
```

**Ознаки thrashing:**
- Комп'ютер різко сповільнюється
- Індикатор диска горить постійно
- У Диспетчері завдань: диск завантажений на 100%, але CPU -- на 5-20%
- Програми "зависають" на десятки секунд при перемиканні між ними

**Як боротися з thrashing:**
- Закрити частину програм, щоб зменшити навантаження на RAM
- Збільшити обсяг фізичної RAM (найефективніший спосіб)
- Використовувати SSD замість HDD для файлу підкачки
- Налаштувати пріоритети процесів

## 6. Алгоритми заміщення сторінок

Коли всі фрейми RAM зайняті й потрібно завантажити нову сторінку, ОС повинна вирішити, **яку сторінку вивантажити** на диск. Цю задачу розв'язують алгоритми заміщення сторінок.

### 6.1. FIFO (First In, First Out — перший прийшов, перший вийшов)

**Принцип:** вивантажується сторінка, яка була завантажена в RAM **найдавніше** (перша в черзі).

```
    Черга сторінок у RAM:

    Завантажена першою                    Завантажена останньою
    ┌────┐  ┌────┐  ┌────┐  ┌────┐
    │ P1 │──│ P3 │──│ P7 │──│ P5 │
    └────┘  └────┘  └────┘  └────┘
      ↑                        ↑
    Буде                   Щойно
    вивантажена            завантажена

    При page fault: P1 вивантажується, нова сторінка стає в кінець черги
```

**Переваги:**
- Найпростіший у реалізації
- Мінімальні накладні витрати

**Недоліки:**
- Не враховує, наскільки часто сторінка використовується
- Може вивантажити сторінку, яка постійно потрібна (наприклад, код головного циклу програми)
- Підвладний **аномалії Беладі** — при збільшенні кількості фреймів кількість page fault може зрости (парадокс!)

### 6.2. LRU (Least Recently Used — найменш нещодавно використана)

**Принцип:** вивантажується сторінка, до якої **найдовше не зверталися**.

```
    Сторінки в RAM з мітками часу останнього звернення:

    ┌──────────────────────────────────┐
    │  P1: останнє звернення t=100     │  <-- найдавніше використана
    │  P3: останнє звернення t=350     │
    │  P7: останнє звернення t=280     │
    │  P5: останнє звернення t=490     │  <-- щойно використана
    └──────────────────────────────────┘

    При page fault: P1 вивантажується (t=100 — найстаріша мітка)
```

**Переваги:**
- Добре відповідає принципу часової локальності
- Зазвичай дає значно менше page fault, ніж FIFO
- Не підвладний аномалії Беладі

**Недоліки:**
- Складний у реалізації: потрібно відстежувати час останнього звернення до кожної сторінки
- Високі накладні витрати (оновлення мітки при кожному зверненні)
- На практиці використовують наближення LRU (див. алгоритм годинника)

### 6.3. Оптимальний алгоритм (алгоритм Беладі)

**Принцип:** вивантажується сторінка, яка **не знадобиться найдовше** в майбутньому.

```
    Сторінки в RAM:

    ┌──────────────────────────────────────────┐
    │  P1: наступне звернення через 50 тактів  │
    │  P3: наступне звернення через 200 тактів │  <-- не потрібна найдовше
    │  P7: наступне звернення через 10 тактів  │
    │  P5: наступне звернення через 80 тактів  │
    └──────────────────────────────────────────┘

    При page fault: P3 вивантажується (не знадобиться найдовше)
```

**Переваги:**
- Теоретично ідеальний — дає мінімальну кількість page fault

**Недоліки:**
- **Неможливий для реалізації на практиці!** Ми не можемо передбачити майбутнє
- Використовується лише як **еталон** для порівняння ефективності інших алгоритмів

### 6.4. Алгоритм годинника (Clock Algorithm / Second Chance)

**Принцип:** наближення до LRU з набагато меншими накладними витратами. Сторінки розташовані по колу (як цифри на годиннику), кожна має **біт звернення R** (reference bit).

```
    Уявний "годинник" зі сторінками:

              P1 (R=0)
             /        \
        P6 (R=1)    P2 (R=1)
           |     ↑     |
           |   стрілка |
        P5 (R=0)    P3 (R=1)
             \        /
              P4 (R=0)

    Алгоритм при page fault:
    1. Стрілка вказує на P2 (R=1)
       → R=1, значить сторінка нещодавно використовувалась
       → Даємо "другий шанс": встановлюємо R=0
       → Рухаємо стрілку далі

    2. Стрілка вказує на P3 (R=1)
       → R=1 → встановлюємо R=0, рухаємо далі

    3. Стрілка вказує на P4 (R=0)
       → R=0, значить сторінка давно не використовувалась
       → ВИВАНТАЖУЄМО P4, на її місце завантажуємо нову сторінку
       → Стрілка рухається далі
```

**Як працює біт R:**
- Апаратно встановлюється в `1` при кожному зверненні до сторінки
- Алгоритм годинника скидає його в `0`, коли проходить повз сторінку
- Якщо при наступному проході R все ще `0` — значить, сторінку давно не чіпали

**Переваги:**
- Ефективність близька до LRU
- Набагато простіший і швидший у реалізації, ніж "справжній" LRU
- Використовується в реальних ОС (Linux, Windows)

**Недоліки:**
- Менш точний, ніж справжній LRU

### 6.5. Порівняння алгоритмів

| Алгоритм | Ефективність | Складність реалізації | Використання на практиці |
|----------|-------------|----------------------|--------------------------|
| **FIFO** | Низька | Дуже проста | Навчальні системи |
| **LRU** | Висока | Складна | Рідко в чистому вигляді |
| **Оптимальний** | Ідеальна | Неможлива | Лише як теоретичний еталон |
| **Годинник** | Близька до LRU | Помірна | Linux, Windows (з модифікаціями) |

## 7. Переваги та недоліки віртуальної пам'яті

### 7.1. Переваги

| Перевага | Пояснення |
|----------|-----------|
| **Збільшення доступної пам'яті** | Програми можуть використовувати більше пам'яті, ніж фізично встановлено RAM |
| **Ізоляція процесів** | Кожен процес працює у власному віртуальному адресному просторі і не може пошкодити дані іншого процесу |
| **Спрощення програмування** | Програміст може писати код так, ніби має необмежену кількість пам'яті, не турбуючись про фізичний обсяг RAM |
| **Ефективне використання RAM** | В RAM зберігаються лише ті дані, що реально використовуються зараз |
| **Підтримка багатозадачності** | Можна запускати більше програм одночасно |
| **Захист пам'яті** | ОС може обмежувати доступ до певних ділянок пам'яті (тільки читання, тільки виконання тощо) |
| **Спільне використання пам'яті** | Кілька процесів можуть спільно використовувати одну фізичну сторінку (наприклад, код спільних бібліотек) |

### 7.2. Недоліки

| Недолік | Пояснення |
|---------|-----------|
| **Зниження швидкості при підкачці** | Доступ до диска в 1000-100000 разів повільніший, ніж до RAM |
| **Використання дискового простору** | Файл підкачки займає місце на диску (зазвичай кілька ГБ) |
| **Додаткове навантаження на ОС** | Управління таблицями сторінок, обробка page fault потребують процесорного часу |
| **Ризик thrashing** | При нестачі RAM система може стати практично непрацездатною |
| **Зношування SSD** | Активна підкачка створює додаткові операції запису, що скорочує ресурс SSD |
| **Складність реалізації** | Підтримка віртуальної пам'яті потребує спеціального апаратного забезпечення (MMU, TLB) |

## 8. Вплив SSD vs HDD на продуктивність підкачки

### 8.1. Ключові відмінності

Продуктивність файлу підкачки безпосередньо залежить від швидкості диска, на якому він розташований.

| Характеристика | HDD | SSD |
|----------------|-----|-----|
| **Час довільного доступу** | 5-15 мс | 0,05-0,1 мс |
| **Швидкість послідовного читання** | 80-200 МБ/с | 500-7000 МБ/с |
| **Швидкість довільного читання (IOPS)** | 75-200 | 10 000-1 000 000 |
| **Затримка при page fault** | Відчутна (десятки мс) | Мінімальна (долі мс) |
| **Вплив підкачки на загальну швидкість** | Значний | Помірний |

### 8.2. Практичні наслідки

**HDD як основа підкачки:**
- Підкачка помітно сповільнює роботу навіть при невеликому обсязі
- Thrashing на HDD — це практично повне "зависання" системи
- Одночасна робота з файлами та підкачка конкурують за пропускну здатність одного диска

**SSD як основа підкачки:**
- Підкачка працює в 10-200 разів швидше, ніж на HDD
- Навіть при активному swapping система залишається відносно чуйною
- Page fault обробляється за мікросекунди замість мілісекунд

```
    Порівняння часу обробки page fault:

    HDD:  |████████████████████████████████████████| ~10 мс
    SSD:  |██|                                       ~0,1 мс
    RAM:  |·|                                        ~0,0001 мс (100 нс)
```

### 8.3. Рекомендації

- **Завжди розміщуйте файл підкачки на SSD**, якщо є вибір
- Сучасні SSD з технологіями TRIM та wear leveling добре витримують навантаження підкачки
- Якщо в системі і SSD, і HDD — перемістіть pagefile.sys на SSD
- **Найкращий варіант** — збільшити обсяг RAM, щоб мінімізувати підкачку

## 9. Моніторинг використання пам'яті

### 9.1. Диспетчер завдань Windows (Task Manager)

Відкрийте Диспетчер завдань (`Ctrl + Shift + Esc`) та перейдіть на вкладку **Продуктивність** (Performance) --> **Пам'ять** (Memory).

**Основні показники:**

| Показник | Опис |
|----------|------|
| **Використовується (In use)** | Обсяг RAM, зайнятий процесами та ОС |
| **Доступно (Available)** | Обсяг RAM, який можна негайно виділити (включаючи кеш, який можна звільнити) |
| **Виділено (Committed)** | Загальний обсяг віртуальної пам'яті, зарезервованої для всіх процесів (RAM + swap). Формат: "використано / максимум" |
| **Кешовано (Cached)** | Дані, збережені в RAM для швидкого повторного доступу |
| **Вивантажуваний пул (Paged pool)** | Частина пам'яті ядра ОС, яку можна вивантажити на диск |
| **Невивантажуваний пул (Non-paged pool)** | Частина пам'яті ядра ОС, яка завжди залишається в RAM |

```
    Продуктивність --> Пам'ять:
    ┌─────────────────────────────────────────────┐
    │  Пам'ять                                    │
    │                                             │
    │  Використовується:  6,2 ГБ                  │
    │  Доступно:          9,5 ГБ                  │
    │  Виділено:          8,1 / 23,7 ГБ           │
    │  Кешовано:          4,8 ГБ                  │
    │  Вивантажуваний пул:    382 МБ              │
    │  Невивантажуваний пул:  256 МБ              │
    │                                             │
    │  Швидкість: 3200 МГц    Слотів: 2 з 4       │
    └─────────────────────────────────────────────┘
```

**Як читати показник "Виділено" (Committed):**
- Формат: `8,1 / 23,7 ГБ` означає "використано 8,1 ГБ з максимально доступних 23,7 ГБ"
- Максимум = RAM + розмір файлу підкачки (наприклад, 16 ГБ RAM + 7,7 ГБ swap = 23,7 ГБ)
- Якщо використана частина наближається до максимуму — система може вичерпати пам'ять

### 9.2. Монітор ресурсів Windows (Resource Monitor)

Більш детальну інформацію надає **Монітор ресурсів** (`resmon.exe`):

- Вкладка **Пам'ять** показує розподіл для кожного процесу окремо
- Стовпець **Hard Faults/sec** — кількість page fault на секунду (звернення до диска)
- Якщо Hard Faults/sec постійно високий (сотні й більше) — ознака thrashing

### 9.3. Linux: команди моніторингу

```bash
# Загальна інформація про пам'ять
free -h

# Приклад виводу:
#               total   used    free   shared  buff/cache  available
# Mem:           15Gi   6,2Gi   2,1Gi   512Mi    7,1Gi      8,5Gi
# Swap:          4,0Gi  256Mi   3,7Gi

# Детальна статистика віртуальної пам'яті
vmstat 1
# Стовпці si (swap in) та so (swap out) показують активність підкачки

# Топ процесів за використанням пам'яті
top -o %MEM
```

## Глосарій

| Термін | Визначення |
|--------|-----------|
| **Віртуальна пам'ять (Virtual Memory)** | Технологія управління пам'яттю, що створює для кожного процесу ілюзію великого безперервного адресного простору, використовуючи RAM та дисковий простір |
| **Файл підкачки (Page File / Swap File)** | Спеціальний файл або розділ на диску, який ОС використовує як розширення оперативної пам'яті для зберігання тимчасово неактивних даних |
| **pagefile.sys** | Назва файлу підкачки в операційній системі Windows, розташований у кореневому каталозі системного диска |
| **Swap partition** | Окремий розділ диска в Linux, відформатований спеціально для підкачки |
| **Swap file** | Звичайний файл у файловій системі Linux, налаштований як простір підкачки |
| **Сторінка (Page)** | Блок віртуальної пам'яті фіксованого розміру (типово 4 КБ), який є одиницею переміщення між RAM та диском |
| **Фрейм (Frame / Page Frame)** | Блок фізичної пам'яті (RAM) такого ж розміру, як сторінка, у який завантажується сторінка з віртуальної пам'яті |
| **Таблиця сторінок (Page Table)** | Структура даних, що зберігає відповідність між віртуальними сторінками та фізичними фреймами для кожного процесу |
| **MMU (Memory Management Unit)** | Апаратний компонент процесора, що автоматично перетворює віртуальні адреси у фізичні |
| **TLB (Translation Lookaside Buffer)** | Швидкий апаратний кеш, що зберігає нещодавно використані записи таблиці сторінок для прискорення трансляції адрес |
| **Page fault (промах сторінки)** | Подія, що виникає при зверненні до сторінки, якої немає у фізичній пам'яті, та ініціює завантаження сторінки з диска |
| **Swapping (підкачка)** | Процес переміщення сторінок пам'яті між RAM та файлом підкачки на диску |
| **Swap out (вивантаження)** | Переміщення сторінки з RAM на диск для звільнення фрейму |
| **Swap in (завантаження)** | Переміщення сторінки з диска в RAM при page fault |
| **Thrashing (пробуксовка)** | Критичний стан, коли система витрачає основну частину часу на підкачку сторінок, а не на корисну роботу процесів |
| **Часова локальність (Temporal Locality)** | Властивість програм: якщо до адреси звернулися, ймовірно до неї звернуться знову найближчим часом |
| **Просторова локальність (Spatial Locality)** | Властивість програм: якщо до адреси звернулися, ймовірно найближчим часом звернуться до сусідніх адрес |
| **Робочий набір (Working Set)** | Множина сторінок пам'яті, що активно використовуються процесом у поточний період часу |
| **FIFO (First In, First Out)** | Алгоритм заміщення сторінок, що вивантажує найстарішу (першу завантажену) сторінку |
| **LRU (Least Recently Used)** | Алгоритм заміщення сторінок, що вивантажує сторінку, до якої найдовше не зверталися |
| **Оптимальний алгоритм (Беладі)** | Теоретично ідеальний алгоритм, що вивантажує сторінку, яка не знадобиться найдовше; нереалізовний на практиці |
| **Алгоритм годинника (Clock)** | Наближення до LRU з використанням біта звернення та циклічного обходу сторінок |
| **Біт звернення (Reference bit, R-bit)** | Прапорець у таблиці сторінок, що автоматично встановлюється апаратно при зверненні до сторінки |
| **Аномалія Беладі** | Парадоксальне явище в алгоритмі FIFO: збільшення кількості фреймів може призвести до зростання кількості page fault |
| **Swappiness** | Параметр ядра Linux (0-100), що визначає, наскільки агресивно система використовує swap; менше значення = рідше свопить |
| **Committed Memory (виділена пам'ять)** | Загальний обсяг віртуальної пам'яті, зарезервованої ОС для всіх процесів (сума RAM та swap) |
| **Paged Pool (вивантажуваний пул)** | Область пам'яті ядра Windows, дані з якої можуть бути вивантажені у файл підкачки |
| **Non-paged Pool (невивантажуваний пул)** | Область пам'яті ядра Windows, дані з якої завжди залишаються в RAM і ніколи не вивантажуються |
