## 1. Облікові записи користувачів (User Accounts)

**Обліковий запис користувача** (User Account) — це набір даних, що ідентифікує конкретного користувача в операційній системі та визначає його права, дозволи й персональні налаштування.

### Навіщо потрібні облікові записи?

- **Ідентифікація** — система розрізняє, хто саме працює за комп'ютером.
- **Автентифікація** — перевірка справжності користувача (пароль, біометрія).
- **Авторизація** — визначення, що саме дозволено робити цьому користувачу.
- **Персоналізація** — кожен користувач має власні налаштування робочого столу, тему, закладки браузера тощо.
- **Безпека** — розмежування доступу до файлів і програм між різними користувачами.
- **Аудит** — система може вести журнал дій кожного користувача.

### Три базових поняття безпеки

| Поняття | Опис | Приклад |
|---------|------|---------|
| **Ідентифікація** | Називаю себе | Ввожу логін `petro` |
| **Автентифікація** | Доводжу, що я — це я | Ввожу пароль або сканую палець |
| **Авторизація** | Система визначає мої права | Стандартний користувач не може встановлювати програми |

## 2. Типи облікових записів у Windows

### 2.1. Адміністратор (Administrator)

- Має **повний контроль** над системою.
- Може встановлювати та видаляти програми, змінювати системні налаштування, створювати та видаляти інші облікові записи, змінювати їхні типи.
- При спробі виконати дію, що потребує підвищених прав, отримує запит UAC на підтвердження.
- За замовчуванням під час встановлення Windows створюється один обліковий запис адміністратора.

> **Важливо!** У Windows існує також вбудований обліковий запис `Administrator` (прихований), який за замовчуванням **вимкнений**. Він є мішенню для зловмисників, тому його рекомендується залишати вимкненим.

### 2.2. Стандартний користувач (Standard User)

- Може використовувати більшість програм і змінювати налаштування, що **не впливають на інших користувачів** і безпеку системи.
- **Не може** встановлювати або видаляти програми, змінювати системні параметри, створювати нових користувачів.
- При спробі виконати адміністративну дію отримує запит UAC із проханням ввести пароль адміністратора.
- Усі нові облікові записи у Windows створюються як **стандартні за замовчуванням**.

> **Рекомендація з безпеки:** Для щоденної роботи завжди використовуйте стандартний обліковий запис. Це значно зменшує ризик зараження шкідливим ПЗ та випадкових змін системи.

### 2.3. Гість (Guest)

- Призначений для тимчасового доступу до комп'ютера.
- Має **мінімальні права** — може лише запускати програми та переглядати інтернет.
- Не може змінювати налаштування системи, встановлювати програми, переглядати файли інших користувачів.
- У **Windows 10/11** окремий обліковий запис гостя **вимкнений за замовчуванням** і фактично прибраний з інтерфейсу налаштувань. Замість нього можна створити стандартний обліковий запис з обмеженими правами.

### Порівняльна таблиця

| Можливість | Адміністратор | Стандартний | Гість |
|------------|:---:|:---:|:---:|
| Встановлення програм | + | - | - |
| Зміна системних налаштувань | + | - | - |
| Створення/видалення користувачів | + | - | - |
| Запуск встановлених програм | + | + | + |
| Зміна власних налаштувань (шпалери, тема) | + | + | +/- |
| Доступ до інтернету | + | + | + |
| Перегляд файлів інших користувачів | + | - | - |

## 3. Локальний обліковий запис vs обліковий запис Microsoft

У Windows 10/11 при створенні облікового запису можна обрати один із двох варіантів:

### Локальний обліковий запис

- Існує **тільки на цьому комп'ютері**.
- Не потребує підключення до інтернету.
- Налаштування, файли, програми зберігаються лише локально.
- Для входу використовується логін і пароль, задані при створенні.

### Обліковий запис Microsoft

- Прив'язаний до електронної пошти (наприклад, `user@outlook.com`).
- Дозволяє **синхронізацію** налаштувань, тем, паролів Wi-Fi, закладок Edge між кількома пристроями.
- Надає доступ до хмарних сервісів: OneDrive, Microsoft Store, Microsoft 365.
- Потребує підключення до інтернету (хоча після першого входу можна працювати офлайн).
- Можна відновити доступ через електронну пошту або телефон у разі забутого пароля.

### Порівняння

| Критерій | Локальний | Microsoft |
|----------|:---------:|:---------:|
| Інтернет для входу | Не потрібен | Потрібен (перший раз) |
| Синхронізація між пристроями | - | + |
| Доступ до Microsoft Store | Обмежений | Повний |
| OneDrive інтеграція | Вручну | Автоматично |
| Відновлення пароля | Лише локально (підказка/диск) | Через email/телефон |
| Конфіденційність | Вища | Нижча (дані на серверах MS) |

## 4. UAC — Контроль облікових записів користувачів

### Що таке UAC?

**UAC (User Account Control)** — це механізм безпеки Windows, який запобігає несанкціонованим змінам операційної системи. Коли програма або дія потребує прав адміністратора, UAC показує діалогове вікно з проханням підтвердити або відхилити цю дію.

### Як працює UAC?

Коли адміністратор входить у систему, Windows створює **два маркери доступу**:

1. **Стандартний маркер** — використовується для повсякденної роботи (браузер, документи, пошта).
2. **Адміністративний маркер** — активується лише при потребі підвищених прав.

Завдяки цьому навіть адміністратор за замовчуванням працює з правами стандартного користувача, що зменшує ризик зараження шкідливим ПЗ.

Для **стандартного користувача**: UAC вимагає ввести пароль адміністратора.

Для **адміністратора**: UAC показує запит на підтвердження (кнопки "Так" / "Ні").

### 4 рівні UAC

UAC має повзунок із чотирма рівнями сповіщень:

| Рівень | Назва | Опис | Безпека |
|:------:|-------|------|:-------:|
| 1 | **Завжди сповіщати** | Сповіщає при будь-яких змінах (програми та користувач) | Максимальна |
| 2 | **Сповіщати про зміни програм** (за замовчуванням) | Сповіщає тільки коли програми намагаються внести зміни | Висока |
| 3 | **Без затемнення робочого столу** | Як рівень 2, але без захищеного робочого столу (Secure Desktop) | Середня |
| 4 | **Ніколи не сповіщати** | Вимикає всі запити UAC | Мінімальна |

> **Увага!** Рівень 4 ("Ніколи не сповіщати") фактично вимикає захист UAC. Це **не рекомендується**, оскільки будь-яка програма зможе вносити зміни в систему без вашого відома.

### Як змінити рівень UAC

**Панель керування** > **Облікові записи користувачів** > **Змінити параметри контролю облікових записів** > пересунути повзунок > **OK**.

## 5. Профіль користувача

Кожен обліковий запис має свій **профіль** — набір папок та файлів налаштувань, що створюється автоматично при першому вході користувача в систему.

### Розташування профілю

```
C:\Users\<ім'я_користувача>\
```

Наприклад: `C:\Users\Petro\`

### Стандартні папки профілю

| Папка | Призначення |
|-------|-------------|
| `Desktop` (Робочий стіл) | Ярлики та файли на робочому столі |
| `Documents` (Документи) | Документи користувача |
| `Downloads` (Завантаження) | Завантажені файли з інтернету |
| `Pictures` (Зображення) | Фотографії та картинки |
| `Music` (Музика) | Музичні файли |
| `Videos` (Відео) | Відеофайли |
| `AppData` (прихована) | Налаштування програм, кеш, тимчасові файли |

### Папка AppData

Прихована папка `AppData` містить три підпапки:

- **Local** — локальні дані програм (кеш, тимчасові файли).
- **LocalLow** — дані програм із низьким рівнем довіри (наприклад, браузерні додатки).
- **Roaming** — дані, що можуть синхронізуватися між комп'ютерами в домені (налаштування програм).

### Спеціальні профілі

- `C:\Users\Default` — шаблон профілю; копіюється при створенні нового користувача.
- `C:\Users\Public` — загальна папка, доступна всім користувачам комп'ютера.

## 6. Паролі

### 6.1. Вимоги до надійного пароля

Надійний пароль є першою лінією захисту облікового запису. Згідно з сучасними рекомендаціями NIST (Національний інститут стандартів і технологій США, оновлення 2024 р.):

**Основні принципи:**

| Критерій | Рекомендація |
|----------|-------------|
| **Довжина** | Мінімум 8 символів, рекомендовано 15+ символів |
| **Максимальна довжина** | Система повинна дозволяти до 64 символів |
| **Складність** | Сучасні рекомендації НЕ вимагають обов'язкового використання спецсимволів; довжина важливіша за складність |
| **Унікальність** | Кожен сервіс — окремий пароль |
| **Парольна фраза** | Кілька випадкових слів ("зелений-вовк-літає-швидко") — найкращий варіант |

**Що РОБИТИ:**

- Використовувати довгі паролі або парольні фрази (passphrase).
- Використовувати менеджер паролів (KeePass, Bitwarden, 1Password).
- Перевіряти, чи не скомпрометований пароль (haveibeenpwned.com).
- Увімкнути двофакторну автентифікацію (2FA), де це можливо.

**Чого НЕ робити:**

- Не використовувати особисту інформацію (дата народження, ім'я, прізвище).
- Не використовувати один пароль для різних сервісів.
- Не записувати паролі на стікерах біля монітора.
- Не передавати пароль іншим людям.
- Не використовувати словникові слова без модифікації.

### 6.2. Приклади паролів

| Пароль | Надійність | Чому? |
|--------|:----------:|-------|
| `123456` | Дуже слабкий | Найпоширеніший пароль у світі |
| `password` | Дуже слабкий | Словникове слово |
| `Petro2009` | Слабкий | Ім'я + рік народження |
| `Kf8$mN2!` | Середній | Короткий (8 символів), хоча й складний |
| `correct-horse-battery-staple` | Сильний | Довга парольна фраза з 4 випадкових слів |
| `Dk9$vLm2#nRt5Wp!` | Дуже сильний | Довгий, випадковий, різні типи символів |

### 6.3. Політика паролів у Windows

Адміністратор може налаштувати **політику паролів** через **Локальну політику безпеки** (`secpol.msc`) або **Групову політику** (`gpedit.msc`):

| Параметр | Опис | Типове значення |
|----------|------|:---------------:|
| **Мінімальна довжина пароля** | Найменша кількість символів | 8 символів |
| **Вимоги до складності** | Великі + малі літери + цифри + спецсимволи | Увімкнено |
| **Максимальний термін дії** | Через скільки днів пароль треба змінити | 42 дні (або без обмеження) |
| **Мінімальний термін дії** | Мінімум днів до можливості зміни пароля | 1 день |
| **Журнал паролів** | Скільки попередніх паролів система пам'ятає (заборона повторного використання) | 24 паролі |
| **Поріг блокування** | Після скількох невдалих спроб входу обліковий запис блокується | 5 спроб |

> **Цікавий факт:** Сучасні рекомендації NIST (2024) стверджують, що примусова регулярна зміна пароля (наприклад, кожні 90 днів) **погіршує безпеку**, оскільки користувачі починають використовувати передбачувані модифікації (Password1, Password2, Password3...). Зміну пароля рекомендують лише при підозрі на компрометацію.

## 7. Хешування паролів — як ОС зберігає паролі

### Головний принцип

Операційна система **ніколи не зберігає пароль у відкритому вигляді** (plain text). Замість цього зберігається **хеш** пароля.

### Що таке хеш?

**Хеш (hash)** — це результат математичної функції, яка перетворює вхідні дані будь-якої довжини у рядок фіксованої довжини. Хеш-функція є **односторонньою** — з хешу неможливо відновити оригінальний пароль.

```
Пароль: "MyPassword123"
    |
    v  [хеш-функція]
    |
Хеш: "a1b2c3d4e5f6..."  (фіксована довжина)
```

**Властивості хеш-функції:**

- Однакові вхідні дані завжди дають однаковий хеш.
- Навіть мінімальна зміна вхідних даних повністю змінює хеш (лавинний ефект).
- З хешу неможливо відновити оригінальні дані.
- Різні вхідні дані (теоретично) дають різні хеші.

### Процес автентифікації

1. Користувач вводить пароль.
2. Система обчислює хеш введеного пароля.
3. Порівнює обчислений хеш зі збереженим хешем.
4. Якщо хеші збігаються — доступ дозволено. Якщо ні — відмова.

```
Вхід користувача:
  Введений пароль: "MyPassword123"
       |
       v  [хеш-функція]
       |
  Обчислений хеш: "a1b2c3d4e5f6..."
       |
       v  [порівняння]
       |
  Збережений хеш: "a1b2c3d4e5f6..."  <-- з бази даних
       |
  Результат: ЗБІГ --> Доступ дозволено
```

### Хешування паролів у Windows

- Паролі локальних облікових записів зберігаються у файлі **SAM** (Security Account Manager): `C:\Windows\System32\config\SAM`.
- Файл SAM заблокований операційною системою під час роботи Windows — його неможливо скопіювати або прочитати безпосередньо.
- Використовується алгоритм хешування **NTLM** (NT LAN Manager), заснований на MD4.
- Починаючи з Windows 10 версії 1607, хеші у SAM додатково шифруються алгоритмом **AES-256** (замість старого RC4).
- Застарілий алгоритм **LM Hash** (LAN Manager) вимкнений за замовчуванням починаючи з Windows Vista через критичні вразливості.

> **Особливість NTLM:** Хеші NTLM **не використовують "сіль" (salt)** — тобто однаковий пароль завжди генерує однаковий хеш. Це робить їх вразливими до атак за допомогою "райдужних таблиць" (rainbow tables). Для захисту використовуються додаткові механізми: блокування облікового запису, Credential Guard тощо.

### Хешування паролів у Linux

- Хеші паролів зберігаються у файлі `/etc/shadow`, доступному лише користувачу **root**.
- Linux використовує **сіль (salt)** — випадковий рядок, що додається до пароля перед хешуванням, щоб однакові паролі мали різні хеші.
- Сучасні дистрибутиви Linux використовують алгоритм **SHA-512** (або yescrypt у новіших версіях).

## 8. Двофакторна автентифікація (2FA) та Windows Hello

### Що таке 2FA?

**Двофакторна автентифікація (2FA)** — це метод захисту, що вимагає **два різних фактори** для підтвердження особи:

| Фактор | Опис | Приклад |
|--------|------|---------|
| **Те, що ви знаєте** | Секретна інформація | Пароль, PIN-код |
| **Те, що ви маєте** | Фізичний пристрій | Телефон, USB-ключ, смарт-карта |
| **Те, що ви є** | Біометричні дані | Відбиток пальця, обличчя, райдужна оболонка ока |

2FA використовує **комбінацію двох** із цих факторів. Наприклад: пароль (знаєте) + код з телефону (маєте).

### Windows Hello

**Windows Hello** — технологія Microsoft для входу в Windows 10/11 без традиційного пароля, з використанням біометрії або PIN-коду.

**Методи автентифікації Windows Hello:**

| Метод | Опис | Вимоги |
|-------|------|--------|
| **Розпізнавання обличчя** | Інфрачервона камера сканує обличчя | IR-камера (не звичайна веб-камера) |
| **Відбиток пальця** | Сканер зчитує унікальний малюнок пальця | Вбудований або USB-сканер |
| **PIN-код** | Цифровий або буквено-цифровий код | Жодних додаткових пристроїв |

**Чому PIN безпечніший за пароль?**

- PIN-код прив'язаний **лише до конкретного пристрою** і не передається мережею.
- Навіть якщо зловмисник дізнається PIN, він не зможе використати його на іншому комп'ютері.
- Пароль Microsoft-акаунта передається на сервер для перевірки; PIN перевіряється локально.

**Безпека біометричних даних:**

- Біометричні дані зберігаються **тільки локально** на пристрої.
- Вони **не передаються** на сервери Microsoft.
- Захищені за допомогою **TPM** (Trusted Platform Module) — апаратного модуля безпеки.
- Windows Hello використовує IR-камери та технології Anti-Spoofing для захисту від підробки (фото, маски).

## 9. Групи користувачів у Windows

### Навіщо потрібні групи?

**Група** — це набір облікових записів, яким можна призначити дозволи **одночасно**, замість налаштування прав кожного користувача окремо. Це значно спрощує адміністрування, особливо в організаціях з великою кількістю користувачів.

**Принцип:** замість того, щоб надавати доступ до папки кожному з 50 бухгалтерів окремо, ви додаєте їх у групу "Бухгалтерія" і надаєте доступ цій групі.

### Вбудовані групи Windows

| Група | Опис |
|-------|------|
| **Administrators** (Адміністратори) | Повний контроль над системою. Можуть змінювати будь-які налаштування, встановлювати ПЗ, керувати іншими користувачами |
| **Users** (Користувачі) | Стандартні користувачі. Можуть запускати програми, змінювати власні налаштування, але не можуть змінювати системні параметри |
| **Guests** (Гості) | Мінімальний набір прав. Тимчасовий доступ без можливості змінювати систему |
| **Power Users** (Досвідчені користувачі) | Застаріла група (legacy). У сучасних Windows (Vista і новіших) фактично не надає додаткових прав порівняно з Users. Існує для зворотної сумісності |
| **Remote Desktop Users** (Користувачі віддаленого робочого столу) | Дозволяє членам підключатися до комп'ютера через протокол RDP (Remote Desktop Protocol). Не надає адміністративних прав |
| **Backup Operators** (Оператори резервного копіювання) | Можуть створювати та відновлювати резервні копії файлів, навіть тих, до яких не мають прямого доступу |
| **Network Configuration Operators** | Можуть змінювати параметри мережевих підключень (IP-адреса, DNS тощо) |

### Принцип найменших привілеїв (Least Privilege)

Кожному користувачу та групі слід надавати **лише ті права, які необхідні** для виконання їхніх завдань. Це мінімізує потенційну шкоду від помилок або зловмисних дій.

## 11. Управління обліковими записами через CMD

### Основні команди

#### `whoami` — хто я?

```cmd
C:\> whoami
desktop-abc\petro

C:\> whoami /groups
```

Показує ім'я поточного користувача та (з ключем `/groups`) групи, до яких він належить.

#### `net user` — управління користувачами

```cmd
:: Показати список усіх користувачів
net user

:: Детальна інформація про користувача
net user Petro

:: Створити нового користувача
net user Olena Pa$$w0rd123 /add

:: Видалити користувача
net user Olena /delete

:: Змінити пароль користувача
net user Petro NewP@ssword

:: Вимкнути обліковий запис
net user Olena /active:no

:: Увімкнути обліковий запис
net user Olena /active:yes
```

#### `net localgroup` — управління групами

```cmd
:: Показати список усіх локальних груп
net localgroup

:: Показати членів групи Administrators
net localgroup Administrators

:: Додати користувача до групи
net localgroup Administrators Olena /add

:: Видалити користувача з групи
net localgroup Administrators Olena /delete
```

> **Примітка:** Команди `net user ... /add`, `net user ... /delete` та `net localgroup ... /add` потребують запуску командного рядка **від імені адміністратора**.

## 12. Управління користувачами в Linux

### Ключові файли

#### `/etc/passwd` — інформація про користувачів

Кожен рядок містить інформацію про одного користувача у форматі:

```
ім'я:x:UID:GID:коментар:домашня_папка:оболонка
```

Приклад:

```
petro:x:1001:1001:Petro Ivanov:/home/petro:/bin/bash
```

| Поле | Значення | Опис |
|------|----------|------|
| `petro` | Ім'я | Логін користувача |
| `x` | Пароль | Позначка, що пароль зберігається в `/etc/shadow` |
| `1001` | UID | Унікальний ідентифікатор користувача |
| `1001` | GID | Ідентифікатор основної групи |
| `Petro Ivanov` | Коментар | Повне ім'я або опис |
| `/home/petro` | Домашня папка | Шлях до домашнього каталогу |
| `/bin/bash` | Оболонка | Командна оболонка за замовчуванням |

**Спеціальні UID:**
- `0` — root (суперкористувач)
- `1-999` — системні облікові записи (служби, демони)
- `1000+` — звичайні користувачі

#### `/etc/shadow` — хеші паролів

Файл доступний **тільки root**. Містить хеші паролів та параметри старіння:

```
petro:$6$salt$hash...:19500:0:99999:7:::
```

| Поле | Опис |
|------|------|
| `$6$` | Алгоритм хешування (6 = SHA-512) |
| `salt` | Випадкова "сіль" |
| `hash...` | Хеш пароля |
| `19500` | Дата останньої зміни (дні від 01.01.1970) |
| `0` | Мінімальний термін між змінами |
| `99999` | Максимальний термін дії пароля |
| `7` | За скільки днів попереджати про закінчення |

#### `/etc/group` — інформація про групи

```
developers:x:1002:petro,olena
```

| Поле | Опис |
|------|------|
| `developers` | Назва групи |
| `x` | Пароль групи (зазвичай не використовується) |
| `1002` | GID (ідентифікатор групи) |
| `petro,olena` | Список членів групи |

### Основні команди Linux

#### Управління користувачами

```bash
# Створити користувача з домашньою папкою
sudo useradd -m -s /bin/bash petro

# Встановити пароль
sudo passwd petro

# Змінити оболонку користувача
sudo usermod -s /bin/zsh petro

# Додати користувача до групи (ВАЖЛИВО: -aG, а не просто -G!)
sudo usermod -aG sudo petro

# Заблокувати обліковий запис
sudo usermod -L petro

# Розблокувати обліковий запис
sudo usermod -U petro

# Видалити користувача разом із домашньою папкою
sudo userdel -r petro
```

> **Увага!** Команда `usermod -G` (без `-a`) **видалить** користувача з усіх інших додаткових груп! Завжди використовуйте `-aG` для додавання до групи.

#### Управління групами

```bash
# Створити нову групу
sudo groupadd developers

# Додати користувача до групи
sudo usermod -aG developers petro

# Показати групи користувача
groups petro

# Видалити групу
sudo groupdel developers
```

#### Команда `sudo`

```bash
# Виконати команду від імені root
sudo apt update

# Перейти в режим root
sudo -i

# Перевірити, хто я
whoami
```

Файл `/etc/sudoers` визначає, які користувачі та групи можуть виконувати команди з правами root. Редагується **тільки** через команду `visudo`.

### Порівняння Windows та Linux

| Дія | Windows (CMD) | Linux (Terminal) |
|-----|---------------|------------------|
| Хто я? | `whoami` | `whoami` |
| Список користувачів | `net user` | `cat /etc/passwd` |
| Інформація про користувача | `net user Petro` | `id petro` |
| Створити користувача | `net user Petro P@ss /add` | `sudo useradd -m petro` |
| Встановити пароль | `net user Petro NewP@ss` | `sudo passwd petro` |
| Видалити користувача | `net user Petro /delete` | `sudo userdel -r petro` |
| Список груп | `net localgroup` | `cat /etc/group` |
| Додати до групи | `net localgroup Admins Petro /add` | `sudo usermod -aG sudo petro` |

## 14. Глосарій

| Термін | Визначення |
|--------|------------|
| **Автентифікація (Authentication)** | Процес перевірки справжності користувача (підтвердження особи) |
| **Авторизація (Authorization)** | Процес надання або відмови у доступі до ресурсів на основі прав користувача |
| **Біометрія (Biometrics)** | Технологія ідентифікації на основі унікальних фізичних характеристик (відбиток пальця, обличчя, райдужна оболонка ока) |
| **Група користувачів (User Group)** | Набір облікових записів, об'єднаних для спрощення управління правами доступу |
| **Двофакторна автентифікація (2FA)** | Метод автентифікації, що вимагає два різних фактори підтвердження особи |
| **Домашня папка (Home Directory)** | Особистий каталог користувача для зберігання файлів і налаштувань |
| **Ідентифікація (Identification)** | Процес повідомлення системі свого імені (логіну) |
| **Менеджер паролів (Password Manager)** | Програма для безпечного зберігання та генерації унікальних паролів |
| **Обліковий запис (User Account)** | Набір даних, що ідентифікує користувача в ОС та визначає його права |
| **Парольна фраза (Passphrase)** | Послідовність кількох випадкових слів, що використовується як пароль |
| **Політика паролів (Password Policy)** | Набір правил, що визначають вимоги до створення та використання паролів |
| **Профіль користувача (User Profile)** | Набір персональних налаштувань та папок, пов'язаних з обліковим записом |
| **Сіль (Salt)** | Випадковий рядок, що додається до пароля перед хешуванням для унікальності хешу |
| **Суперкористувач (root)** | Обліковий запис з необмеженими правами в Linux (аналог Адміністратора) |
| **Хеш (Hash)** | Результат односторонньої математичної функції, що перетворює дані у рядок фіксованої довжини |
| **Хеш-функція (Hash Function)** | Одностороння математична функція, з результату якої неможливо відновити вхідні дані |
| **GID (Group ID)** | Унікальний числовий ідентифікатор групи в Linux |
| **NTLM** | Алгоритм хешування паролів у Windows (NT LAN Manager) |
| **SAM (Security Account Manager)** | Файл-база даних Windows, що зберігає хеші паролів локальних облікових записів |
| **TPM (Trusted Platform Module)** | Апаратний модуль безпеки для зберігання криптографічних ключів та біометричних даних |
| **UAC (User Account Control)** | Механізм безпеки Windows, що запобігає несанкціонованим змінам системи |
| **UID (User ID)** | Унікальний числовий ідентифікатор користувача в Linux |
| **Windows Hello** | Технологія Microsoft для входу в Windows за допомогою біометрії або PIN-коду |
