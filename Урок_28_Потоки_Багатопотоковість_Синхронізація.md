## 1. Вступ

У попередніх уроках ми детально розглянули процеси — програми, що виконуються в операційній системі. Кожен процес має власний адресний простір, ресурси та стан. Але чи може один процес виконувати кілька завдань одночасно? Наприклад, як браузер одночасно завантажує сторінку, відтворює відео та реагує на натискання кнопок?

Відповідь на це питання дають **потоки** (threads) — менші одиниці виконання всередині процесу. Саме потоки дозволяють сучасним програмам бути швидкими та чуйними.

У цьому уроці ми дізнаємося:
- що таке потік і чим він відрізняється від процесу;
- які бувають типи потоків та моделі їх реалізації;
- як працює багатопотоковість на одному та багатьох ядрах;
- що таке гіперпотоковість (Hyper-Threading);
- які проблеми виникають при паралельній роботі потоків;
- як ці проблеми вирішує синхронізація.

## 2. Поняття потоку (thread)

### 2.1. Що таке потік

**Потік** (англ. thread — "нитка") — це найменша одиниця виконання, яку операційна система може планувати (розподіляти процесорний час). Потік існує всередині процесу і виконує частину його коду.

Кожен процес має щонайменше один потік — **головний потік** (main thread). Але процес може створити додаткові потоки для виконання паралельних завдань.

Якщо **процес** — це "програма, що виконується", то **потік** — це "конкретне завдання всередині цієї програми".

```
┌──────────────────────────────────────────┐
│              ПРОЦЕС                      │
│  ┌──────────┐  ┌──────────┐  ┌────────┐  │
│  │ Потік 1  │  │ Потік 2  │  │Потік 3 │  │
│  │(головний)│  │(завант.) │  │(рендер)│  │
│  └──────────┘  └──────────┘  └────────┘  │
│                                          │
│  [ Спільна пам'ять процесу ]             │
│  [ Спільні файли та ресурси ]            │
└──────────────────────────────────────────┘
```

### 2.2. Потік як "легкий процес"

Потік часто називають **"легким процесом"** (lightweight process), і ось чому:

| Що має процес | Що має потік |
|---|---|
| Власний адресний простір | Використовує адресний простір процесу |
| Власні відкриті файли | Спільні файли з іншими потоками |
| Власні змінні оточення | Спільне оточення |
| Власний лічильник команд | **Власний** лічильник команд |
| Власний стек | **Власний** стек |
| Власні регістри CPU | **Власні** регістри CPU |

Таким чином, у потоку є власний лише мінімум: лічильник команд (який вказує, яку інструкцію виконувати далі), стек (для локальних змінних) та набір регістрів процесора. Все решта — спільне з іншими потоками того ж процесу.

Саме тому потоки значно "дешевші" у створенні та перемиканні, ніж процеси — не потрібно копіювати весь адресний простір.

### 2.3. Для чого потрібні потоки

Потоки потрібні для **паралелізму всередині програми** — можливості виконувати кілька завдань одночасно в межах одного процесу.

**Приклади з повсякденного життя:**

1. **Веб-браузер** (Chrome, Firefox):
   - Потік 1: завантажує HTML-код сторінки з Інтернету
   - Потік 2: відображає (рендерить) вже завантажені елементи
   - Потік 3: виконує JavaScript-скрипти
   - Потік 4: відтворює відео або анімацію
   - Потік 5: реагує на дії користувача (клік, прокрутка)

2. **Текстовий редактор** (Microsoft Word):
   - Потік 1: обробляє натискання клавіш
   - Потік 2: перевіряє орфографію у фоновому режимі
   - Потік 3: автоматично зберігає документ кожні кілька хвилин
   - Потік 4: форматує текст

3. **Мобільний банківський додаток:**
   - Потік 1: відображає інтерфейс
   - Потік 2: завантажує баланс рахунку з сервера
   - Потік 3: обробляє дотики та жести
   - Потік 4: завантажує зображення та іконки

**Без потоків** кожна з цих операцій виконувалася б послідовно: спочатку завантажити сторінку, потім показати, потім виконати скрипти. Програма б "зависала" на час кожної операції.

## 3. Порівняння процесу та потоку

### 3.1. Ключові відмінності

Розглянемо основні відмінності між процесом та потоком:

**Адресний простір:**
- Процес має **власний** ізольований адресний простір. Один процес не може напряму звернутися до пам'яті іншого процесу.
- Потоки одного процесу **ділять** спільний адресний простір. Будь-який потік може прочитати або змінити будь-яку змінну процесу.

**Вартість створення та перемикання:**
- Створення нового процесу — **дорога** операція: потрібно виділити пам'ять, скопіювати дані, створити структури даних ядра.
- Створення нового потоку — **дешева** операція: потрібно лише виділити стек та зберегти набір регістрів.

**Комунікація:**
- Між процесами — через спеціальні механізми **міжпроцесної взаємодії** (IPC): канали, черги повідомлень, спільна пам'ять.
- Між потоками — **напряму** через спільні змінні (але це створює проблеми, про які ми поговоримо пізніше).

**Ізоляція та надійність:**
- Якщо один процес аварійно завершиться — інші процеси продовжать працювати.
- Якщо один потік викличе критичну помилку — **весь процес** (і всі його потоки) завершиться аварійно.

### 3.2. Порівняльна таблиця

| Характеристика | Процес | Потік |
|---|---|---|
| Адресний простір | Власний, ізольований | Спільний з іншими потоками процесу |
| Вартість створення | Висока | Низька |
| Вартість перемикання контексту | Висока (зміна адресного простору) | Низька (зміна лише стеку та регістрів) |
| Комунікація | Через IPC (канали, черги) | Напряму через спільну пам'ять |
| Ізоляція | Сильна (крах одного не впливає на інші) | Слабка (крах одного = крах усіх) |
| Споживання пам'яті | Більше (власний адресний простір) | Менше (спільний адресний простір) |
| Паралелізм | Між різними програмами | Всередині однієї програми |
| Приклад | Chrome (кожна вкладка = процес) | Word (перевірка орфографії + введення тексту) |

### 3.3. Аналогія для запам'ятовування

**Процеси — це окремі квартири** в багатоквартирному будинку:
- Кожна квартира має власні стіни, двері, ключі.
- Мешканці однієї квартири не можуть напряму зайти в іншу.
- Щоб передати щось сусідам, потрібно вийти в коридор (IPC).
- Якщо в одній квартирі стався потоп — інші квартири не постраждають.

**Потоки — це люди всередині однієї квартири:**
- Вони ділять спільну кухню, ванну, вітальню (спільна пам'ять).
- Можуть спілкуватися напряму.
- Але якщо хтось влаштує пожежу — постраждають усі мешканці.
- Потрібні правила: хто коли використовує ванну (синхронізація).

## 4. Типи потоків

### 4.1. Потоки рівня користувача (User-Level Threads)

**Потоки рівня користувача** (ULT) — це потоки, які створюються та керуються бібліотекою потоків у просторі користувача, **без участі ядра ОС**. Ядро навіть не знає про їх існування.

**Як це працює:**
- Програма використовує спеціальну бібліотеку (наприклад, Green Threads у Java або потоки в ранніх версіях Python).
- Бібліотека сама планує, який потік коли виконуватиметься.
- Для ядра ОС весь процес виглядає як один потік.

**Переваги ULT:**
- Дуже швидке створення та перемикання (не потрібно переходити в режим ядра).
- Портативність — працюють на будь-якій ОС, навіть якщо та не підтримує потоки.
- Гнучкість у плануванні — програма сама визначає алгоритм.

**Недоліки ULT:**
- **Немає справжнього паралелізму** на багатоядерних процесорах — ядро ОС бачить лише один потік і виділяє лише одне ядро.
- Якщо один ULT виконує блокуючу операцію (наприклад, читання з диска) — **всі потоки процесу блокуються**, бо ядро зупиняє весь процес.

### 4.2. Потоки рівня ядра (Kernel-Level Threads)

**Потоки рівня ядра** (KLT) — це потоки, які створюються та керуються безпосередньо **ядром операційної системи**. Ядро знає про кожен потік і може планувати їх незалежно.

**Як це працює:**
- Програма звертається до ОС через системні виклики для створення потоків.
- Ядро створює потік, відстежує його стан, планує його на процесорі.
- Кожен потік може бути призначений на окреме ядро процесора.

**Переваги KLT:**
- **Справжній паралелізм** — різні потоки можуть одночасно виконуватися на різних ядрах.
- Блокування одного потоку **не блокує** інші потоки.
- Ядро може ефективно розподіляти потоки між ядрами процесора.

**Недоліки KLT:**
- Створення та перемикання потоків **повільніше** (потрібен перехід у режим ядра).
- Споживають **більше ресурсів** (пам'ять для структур даних ядра).

**Приклади:** Windows Threads, POSIX Threads (pthreads) у Linux, потоки в macOS.

### 4.3. Моделі відображення потоків

Оскільки існують два типи потоків (ULT та KLT), виникає питання: як вони пов'язані між собою? Існує три моделі відображення:

#### Модель N:1 (багато-до-одного)

```
  ULT 1 ──┐
  ULT 2 ──┼──► KLT 1
  ULT 3 ──┘
```

- **Багато** потоків рівня користувача відображаються на **один** потік ядра.
- Планування виконує бібліотека в просторі користувача.
- Неможливий справжній паралелізм — усі ULT виконуються на одному ядрі CPU.
- Блокування одного ULT блокує всі.
- **Приклад:** Green Threads у ранній Java, GNU Portable Threads.

#### Модель 1:1 (один-до-одного)

```
  ULT 1 ────► KLT 1
  ULT 2 ────► KLT 2
  ULT 3 ────► KLT 3
```

- **Кожен** потік рівня користувача відображається на **один** потік ядра.
- Повний паралелізм — потоки можуть виконуватися на різних ядрах.
- Блокування одного потоку не впливає на інші.
- Але кожен потік потребує ресурсів ядра — є обмеження на кількість.
- **Приклад:** Windows, Linux (NPTL), macOS — найпоширеніша модель у сучасних ОС.

#### Модель M:N (багато-до-багатьох)

```
  ULT 1 ──┐       ┌──► KLT 1
  ULT 2 ──┼── M:N ┤
  ULT 3 ──┤       └──► KLT 2
  ULT 4 ──┘
```

- **M** потоків рівня користувача відображаються на **N** потоків ядра (де N <= M).
- Компроміс між двома попередніми моделями.
- Дозволяє справжній паралелізм, при цьому кількість потоків ядра може бути меншою.
- Складна реалізація — потребує координації між планувальниками.
- **Приклад:** Solaris (ранні версії), Go (горутини), Erlang (легкі процеси).

## 5. Багатопотоковість (Multithreading)

### 5.1. Що таке багатопотоковість

**Багатопотоковість** (multithreading) — це здатність операційної системи або програми підтримувати кілька потоків виконання всередині одного процесу. Кожен потік виконує свою частину роботи, і всі потоки ділять ресурси процесу.

Багатопотоковість є ключовою концепцією сучасного програмування та операційних систем. Майже всі сучасні програми — від ігор до офісних пакетів — використовують множину потоків.

### 5.2. Багатопотоковість на одному ядрі

На процесорі з **одним ядром** справжній паралелізм неможливий — ядро може виконувати лише одну інструкцію за раз. Однак ОС створює **ілюзію паралельності** за допомогою **конкурентного виконання** (concurrency):

```
Час ──────────────────────────────────────────►

Потік A: ████░░░░████░░░░████░░░░
Потік B: ░░░░████░░░░████░░░░████
Потік C: ░░░░░░░░░░░░░░░░████░░░░

████ = виконується на CPU
░░░░ = очікує своєї черги
```

**Як це працює:**
1. Планувальник ОС виділяє кожному потоку короткий проміжок часу — **квант** (time slice), зазвичай 10-20 мілісекунд.
2. Потік виконується протягом свого кванту.
3. Планувальник зупиняє поточний потік і перемикається на наступний (**перемикання контексту**).
4. Цикл повторюється так швидко, що для користувача це виглядає як одночасне виконання.

**Аналогія:** Один кухар, який готує три страви. Він не може готувати всі три одночасно, але постійно перемикається: помішав суп, нарізав салат, перевернув м'ясо, знову помішав суп. Для того, хто спостерігає ззовні, все "готується одночасно".

**Це не паралелізм, а конкурентність (concurrency):**
- **Конкурентність** — кілька завдань просуваються вперед за один період часу (але не обов'язково одночасно).
- **Паралелізм** — кілька завдань виконуються буквально одночасно.

### 5.3. Багатопотоковість на багатоядерному процесорі

На процесорі з **кількома ядрами** можливий справжній **паралелізм** — кілька потоків виконуються одночасно, кожен на своєму ядрі:

```
Ядро 1: Потік A ████████████████████████████
Ядро 2: Потік B ████████████████████████████
Ядро 3: Потік C ████████████████████████████
Ядро 4: Потік D ████████████████████████████
```

**Приклад:** 4-ядерний процесор може одночасно:
- Ядро 1: обробляти натискання клавіш у текстовому редакторі
- Ядро 2: перевіряти орфографію
- Ядро 3: завантажувати файл з Інтернету
- Ядро 4: відтворювати музику

Якщо потоків **більше, ніж ядер**, ОС комбінує обидва підходи: частина потоків виконується паралельно (на різних ядрах), а на кожному ядрі потоки чергуються (конкурентність).

```
Ядро 1: Потік A ████░░░░████  Потік C ░░░░████░░░░
Ядро 2: Потік B ████░░░░████  Потік D ░░░░████░░░░
```

## 6. Гіперпотоковість (Hyper-Threading)

### 6.1. Що таке Hyper-Threading

**Гіперпотоковість** (Hyper-Threading Technology, HTT) — це технологія компанії Intel, яка є реалізацією **одночасної багатопотоковості** (Simultaneous Multithreading, SMT). Вона дозволяє одному фізичному ядру процесора виконувати два потоки одночасно.

Простіше кажучи, Hyper-Threading "подвоює" кількість ядер з точки зору операційної системи: процесор із 4 фізичними ядрами представляється системі як 8 **логічних** ядер.

### 6.2. Як працює гіперпотоковість

Всередині кожного фізичного ядра є багато функціональних блоків: арифметико-логічний пристрій (АЛП), блок обчислень з рухомою комою, блок завантаження/збереження тощо. При виконанні одного потоку частина цих блоків часто **простоює** — наприклад, поки потік чекає даних з пам'яті.

Hyper-Threading використовує ці "простої": поки один логічний потік чекає, другий логічний потік може зайняти вільні блоки.

```
Без Hyper-Threading:
┌─────────────────────────────────────────┐
│ Фізичне ядро                            │
│ [АЛП] [FPU] [Завант.] [Збереж.]         │
│  ████   ░░░   ████      ░░░░            │
│  (Потік A використовує лише частину)     │
└─────────────────────────────────────────┘

З Hyper-Threading:
┌─────────────────────────────────────────┐
│ Фізичне ядро (2 логічні ядра)           │
│ [АЛП] [FPU] [Завант.] [Збереж.]         │
│  ████   ████   ████      ████           │
│  (A)    (B)    (A)       (B)            │
│  Потік A та B ділять ресурси            │
└─────────────────────────────────────────┘
```

**Ключовий момент:** Hyper-Threading не подвоює продуктивність. Два логічні ядра ділять одні й ті ж фізичні ресурси. Intel зазначає, що перший варіант реалізації HTT використовував лише на 5% більше площі кристала, але давав приріст продуктивності **15-30%** залежно від навантаження.

### 6.3. Приклад і продуктивність

**Приклад:** Процесор Intel Core i7 з 8 фізичними ядрами та Hyper-Threading:
- Фізичних ядер: 8
- Логічних ядер (потоків): 16
- ОС "бачить" 16 процесорів

**Де можна побачити:** У Windows відкрийте Диспетчер пристроїв (Device Manager) і розгорніть "Процесори". Якщо у вас 4-ядерний процесор з HT, ви побачите 8 записів.

**Як перевірити в Диспетчері завдань:**
- Вкладка "Продуктивність" (Performance)
- Пункт "ЦП" (CPU)
- Рядок "Логічні процесори" (Logical processors) vs "Ядра" (Cores)

**Цікавий факт:** Intel тимчасово прибрала Hyper-Threading з деяких моделей процесорів (Arrow Lake, Lunar Lake), але оголосила про повернення технології у серверних процесорах Xeon 7000 "Diamond Rapids".

## 7. Проблеми паралельного доступу

Коли кілька потоків працюють зі **спільними даними**, виникають серйозні проблеми. Дві найважливіші з них — **гонка даних** та **взаємне блокування**.

### 7.1. Гонка даних (Race Condition)

**Гонка даних** (race condition) — це ситуація, коли результат роботи програми залежить від того, в якому порядку виконуються потоки. Якщо порядок зміниться — результат буде іншим (і, можливо, неправильним).

**Класичний приклад — банківський рахунок:**

Уявімо, що на рахунку 1000 грн. Два потоки одночасно намагаються зняти гроші:

```
Потік A: зняти 700 грн          Потік B: зняти 500 грн

Крок 1 (Потік A): Прочитати баланс → 1000 грн
Крок 2 (Потік B): Прочитати баланс → 1000 грн
Крок 3 (Потік A): 1000 - 700 = 300, записати 300 грн
Крок 4 (Потік B): 1000 - 500 = 500, записати 500 грн

Результат: на рахунку 500 грн (а знято 1200 грн!)
Правильний результат: операцію B слід відхилити (недостатньо коштів)
```

**Проблема:** Обидва потоки прочитали **старе** значення балансу, перш ніж будь-який з них встиг записати нове. Кожен потік "не знав", що інший теж працює з цим рахунком.

**Ще один простий приклад — лічильник:**

```
Змінна counter = 0
Потік A: counter = counter + 1
Потік B: counter = counter + 1

Очікуваний результат: counter = 2

Що може статися:
1. Потік A читає counter (0)
2. Потік B читає counter (0)     ← ще старе значення!
3. Потік A записує counter = 1
4. Потік B записує counter = 1   ← перезаписав результат A!

Фактичний результат: counter = 1 (замість 2)
```

**Чому гонки даних небезпечні:**
- Вони проявляються **непередбачувано** — програма може працювати правильно 99 разів з 100, а на 100-й дати помилку.
- Їх **дуже складно** виявити при тестуванні.
- Вони можуть призвести до катастрофічних наслідків: відомий випадок з апаратом променевої терапії Therac-25, де гонка даних призвела до загибелі пацієнтів.

### 7.2. Взаємне блокування (Deadlock)

**Взаємне блокування** (deadlock) — це ситуація, коли два або більше потоків заблоковані назавжди, бо кожен з них чекає ресурс, зайнятий іншим потоком.

**Аналогія з реального життя — два потяги на одній колії:**

```
Потяг A ──────►  ◄────── Потяг B
                 │
           Одна колія

Потяг A не може проїхати, бо на колії Потяг B.
Потяг B не може проїхати, бо на колії Потяг A.
Жоден не може від'їхати назад.
Результат: обидва стоять назавжди.
```

**Приклад з програмування:**

```
Потік A:                        Потік B:
1. Захопити ресурс X ✓          1. Захопити ресурс Y ✓
2. Захопити ресурс Y... чекаю   2. Захопити ресурс X... чекаю

Потік A тримає X і чекає Y.
Потік B тримає Y і чекає X.
Жоден не може продовжити = DEADLOCK!
```

**Ще одна аналогія:** Два водії зустрілися на вузькому мості, кожен з'їхав наполовину. Жоден не хоче здати назад — обидва стоять назавжди.

### 7.3. Чотири умови Коффмана

У 1971 році Едвард Коффман (Edward Coffman) з колегами сформулював **чотири умови**, які повинні виконуватися **одночасно** для того, щоб виникло взаємне блокування:

| № | Умова | Опис | Аналогія |
|---|---|---|---|
| 1 | **Взаємне виключення** (Mutual Exclusion) | Ресурс може використовувати лише один потік одночасно | Лише одна людина може сидіти на одному стільці |
| 2 | **Утримання та очікування** (Hold and Wait) | Потік утримує один ресурс і чекає на інший | Людина тримає виделку і чекає на ніж |
| 3 | **Неможливість примусового вилучення** (No Preemption) | Ресурс не можна відібрати у потоку силою — лише потік сам може його звільнити | Не можна відібрати виделку у людини, яка їсть |
| 4 | **Циклічне очікування** (Circular Wait) | Існує замкнутий ланцюжок потоків, де кожен чекає ресурс від наступного | A чекає B, B чекає C, C чекає A |

**Важливо:** Якщо порушити **хоча б одну** з цих чотирьох умов — deadlock стає неможливим. На цьому принципі базуються методи запобігання взаємним блокуванням.

**Приклад запобігання:** Вимагати, щоб усі потоки захоплювали ресурси **в одному й тому ж порядку**. Якщо і Потік A, і Потік B завжди захоплюють спочатку X, потім Y — циклічне очікування стає неможливим.

## 8. Синхронізація потоків

### 8.1. Навіщо потрібна синхронізація

**Синхронізація** — це механізми, які забезпечують коректну координацію роботи потоків при доступі до спільних ресурсів. Синхронізація потрібна, щоб:
- уникнути **гонок даних** (race conditions);
- запобігти **взаємним блокуванням** (deadlocks);
- гарантувати **правильність результатів** роботи програми.

Без синхронізації багатопотокова програма буде працювати неправильно та непередбачувано. Синхронізація — це "правила дорожнього руху" для потоків.

### 8.2. М'ютекс (Mutex)

**М'ютекс** (mutex, від англ. **mut**ual **ex**clusion — взаємне виключення) — це механізм синхронізації, який дозволяє **лише одному потоку** мати доступ до ресурсу в певний момент часу.

**Аналогія з ключем від туалетної кімнати:**
- В офісі є одна туалетна кімната і один ключ.
- Хто взяв ключ — той зайшов і зачинився (захопив м'ютекс).
- Інші чекають під дверима (потоки заблоковані).
- Коли людина вийшла — вона повертає ключ (звільняє м'ютекс).
- Наступний бере ключ і заходить.

```
Потік A:                           Потік B:
1. mutex.lock()     ← захопив      1. mutex.lock()   ← чекаю...
2. counter = counter + 1           (заблокований)
3. mutex.unlock()   ← звільнив     2. counter = counter + 1
                                   3. mutex.unlock()
```

**Ключова особливість м'ютекса:** Тільки потік, який **захопив** м'ютекс, може його **звільнити**. Це називається **суворе володіння** (strict ownership).

**Де використовується:** Захист будь-якого спільного ресурсу: змінної, файлу, бази даних, принтера.

### 8.3. Семафор (Semaphore)

**Семафор** — це механізм синхронізації, який підтримує **лічильник** і дозволяє **кільком потокам** (до певної кількості) одночасно мати доступ до ресурсу.

Семафор має дві атомарні операції:
- **wait()** (або acquire(), down(), P()) — зменшити лічильник на 1. Якщо лічильник стає менше 0 — потік блокується.
- **signal()** (або release(), up(), V()) — збільшити лічильник на 1. Якщо є заблоковані потоки — розблокувати один.

**Аналогія з парковкою:**
- Парковка на 5 місць (лічильник = 5).
- Автомобіль заїжджає — лічильник зменшується: 5 → 4 → 3 → 2 → 1 → 0.
- Коли лічильник = 0, нові автомобілі чекають на виїзді (потоки блокуються).
- Автомобіль виїхав — лічильник збільшується: 0 → 1, і наступний автомобіль може заїхати.

```
Семафор (ліміт = 3):

Потік 1: sem.wait() → лічильник = 2 ✓ (працює)
Потік 2: sem.wait() → лічильник = 1 ✓ (працює)
Потік 3: sem.wait() → лічильник = 0 ✓ (працює)
Потік 4: sem.wait() → лічильник < 0... ЧЕКАЮ!

Потік 1: sem.signal() → лічильник = 1
Потік 4: розблоковано! → лічильник = 0 ✓ (працює)
```

**Типи семафорів:**
- **Двійковий семафор** (binary semaphore) — лічильник може бути лише 0 або 1. Схожий на м'ютекс, але **без суворого володіння** (будь-який потік може зробити signal()).
- **Лічильний семафор** (counting semaphore) — лічильник може бути будь-яким невід'ємним числом.

### 8.4. Критична секція (Critical Section)

**Критична секція** — це ділянка коду, яка звертається до **спільного ресурсу** і тому повинна виконуватися лише **одним потоком** у кожен момент часу.

Критична секція — це не окремий механізм синхронізації, а **концепція**. Для захисту критичної секції використовуються м'ютекси, семафори або інші механізми.

```
// НЕ критична секція — кожен потік працює з власними даними
int local_variable = 10;
local_variable = local_variable + 5;

// КРИТИЧНА СЕКЦІЯ — потоки працюють зі спільною змінною
mutex.lock();           ← вхід у критичну секцію
shared_counter = shared_counter + 1;    // спільний ресурс
mutex.unlock();         ← вихід з критичної секції
```

**Правила для критичної секції:**
1. **Взаємне виключення:** Лише один потік може бути в критичній секції.
2. **Прогрес:** Якщо жоден потік не в критичній секції, потік, що хоче увійти, повинен мати таку можливість.
3. **Обмежене очікування:** Потік не повинен чекати нескінченно довго.

**У Windows API** існує окремий об'єкт `CRITICAL_SECTION`, який є спрощеним і швидшим аналогом м'ютекса, але працює лише в межах одного процесу (на відміну від м'ютекса, який може використовуватися між процесами).

### 8.5. Монітор (Monitor)

**Монітор** — це високорівневий механізм синхронізації, який об'єднує:
- **дані** (спільний ресурс);
- **процедури** (функції для роботи з цими даними);
- **механізм блокування** (м'ютекс);
- **умовні змінні** (condition variables) для координації потоків.

**Аналогія:** Монітор — це як банківська каса з одним вікном:
- Лише одна людина (потік) може стояти біля вікна (виконувати код монітора).
- Інші стоять у черзі (заблоковані).
- Якщо касир каже "зачекайте, поки прийдуть гроші" — людина відходить і чекає (condition variable: wait).
- Коли гроші приходять — людину викликають назад (condition variable: signal).

**Як працює монітор:**
1. Потік входить у монітор — автоматично захоплює блокування.
2. Виконує операції з даними.
3. Якщо потрібно чекати на умову — викликає `wait()` (блокується і звільняє монітор).
4. Інший потік входить у монітор, змінює дані, викликає `signal()` — будить заблокований потік.
5. Потік, що виходить з монітора, — автоматично звільняє блокування.

**Переваги монітора:**
- Безпечніший за м'ютекс — блокування захоплюється та звільняється **автоматично**.
- Менша ймовірність помилки програміста (забув звільнити м'ютекс).

**Приклади в мовах програмування:**
- **Java:** ключове слово `synchronized` — кожен об'єкт фактично є монітором.
- **C#:** ключове слово `lock` та клас `Monitor`.
- **Python:** `with threading.Lock()` — подібна ідея.

### 8.6. Порівняння механізмів синхронізації

| Механізм | Кількість потоків | Володіння | Рівень | Аналогія |
|---|---|---|---|---|
| М'ютекс | 1 | Суворе (хто захопив — той звільняє) | Низький | Ключ від кімнати |
| Семафор | N (визначається лічильником) | Немає суворого володіння | Низький | Парковка на N місць |
| Критична секція | 1 | Так | Низький | Ділянка коду, що потребує захисту |
| Монітор | 1 (з можливістю очікування) | Автоматичне | Високий | Банківська каса з одним вікном |

## 9. Як переглянути потоки в Windows

### 9.1. Диспетчер завдань

**Диспетчер завдань** (Task Manager) дозволяє переглянути загальну кількість потоків:

**Загальна статистика:**
1. Натиснути `Ctrl + Shift + Esc` (або `Ctrl + Alt + Del` → Диспетчер завдань).
2. Перейти на вкладку **"Продуктивність"** (Performance).
3. У розділі **"ЦП"** (CPU) знизу видно загальну кількість потоків у системі (наприклад, "Потоки: 2847").

**Потоки конкретного процесу:**
1. Перейти на вкладку **"Деталі"** (Details).
2. Натиснути правою кнопкою миші на заголовку стовпців.
3. Вибрати **"Вибрати стовпці"** (Select columns).
4. Поставити позначку біля **"Потоки"** (Threads).
5. Тепер поруч з кожним процесом відображатиметься кількість його потоків.

**Зверніть увагу:** Стандартний Диспетчер завдань показує лише **кількість** потоків для кожного процесу, але не дозволяє переглянути окремі потоки. Для детальнішого аналізу потрібен Process Explorer.

### 9.2. Process Explorer

**Process Explorer** — це безкоштовна утиліта від Microsoft (раніше Sysinternals), яка надає значно більше інформації про процеси та потоки, ніж стандартний Диспетчер завдань.

**Як переглянути потоки в Process Explorer:**
1. Завантажити Process Explorer з сайту Microsoft (пошук: "Process Explorer Sysinternals").
2. Запустити програму (не потребує встановлення).
3. Знайти потрібний процес у списку.
4. Двічі клікнути на процесі (або правою кнопкою → Properties).
5. Перейти на вкладку **"Threads"**.

**Що показує вкладка Threads:**
- **TID** (Thread ID) — унікальний ідентифікатор кожного потоку.
- **CPU** — відсоток процесорного часу, який використовує потік.
- **Cycles Delta** — кількість циклів процесора, спожитих потоком з моменту останнього оновлення.
- **Start Address** — адреса в пам'яті, з якої потік почав виконання.

**Додаткова можливість:** Двічі клікнувши на конкретному потоці, можна побачити його **стек викликів** (call stack) — ланцюжок функцій, які виконуються в цей момент. Це дуже корисно для діагностики "зависань" програм.

## 10. Підсумок

У цьому уроці ми вивчили ключові концепції багатопотокового виконання:

1. **Потік** — це найменша одиниця виконання, яка існує всередині процесу. Потоки ділять спільні ресурси процесу, але мають власний стек та регістри.

2. **Процес vs Потік:** Процес має ізольований адресний простір, а потоки одного процесу ділять спільну пам'ять. Потоки дешевші у створенні та перемиканні.

3. **Типи потоків:** Потоки рівня користувача (ULT) керуються бібліотекою, потоки рівня ядра (KLT) — операційною системою. Моделі відображення: N:1, 1:1, M:N.

4. **Багатопотоковість:** На одному ядрі — конкурентність (швидке перемикання), на багатьох ядрах — справжній паралелізм.

5. **Hyper-Threading** дозволяє одному фізичному ядру обробляти два потоки, використовуючи простої функціональних блоків.

6. **Гонка даних** — коли результат залежить від порядку виконання потоків. **Deadlock** — коли потоки заблоковані назавжди, чекаючи один на одного.

7. **Синхронізація** (м'ютекс, семафор, критична секція, монітор) — механізми, що гарантують коректну роботу потоків зі спільними ресурсами.

## 12. Глосарій ключових термінів

| Термін | Визначення |
|---|---|
| **Потік (Thread)** | Найменша одиниця виконання всередині процесу, яка може бути самостійно запланована операційною системою |
| **Легкий процес (Lightweight Process)** | Альтернативна назва потоку, що підкреслює меншу "вартість" порівняно з повноцінним процесом |
| **Головний потік (Main Thread)** | Перший потік, що створюється при запуску процесу; з нього можна створити додаткові потоки |
| **Потоки рівня користувача (ULT)** | Потоки, що створюються та керуються бібліотекою в просторі користувача, без участі ядра ОС |
| **Потоки рівня ядра (KLT)** | Потоки, що створюються та керуються безпосередньо ядром операційної системи |
| **Модель N:1** | Модель, де багато потоків користувача відображаються на один потік ядра |
| **Модель 1:1** | Модель, де кожен потік користувача відображається на один потік ядра |
| **Модель M:N** | Модель, де M потоків користувача відображаються на N потоків ядра (N <= M) |
| **Багатопотоковість (Multithreading)** | Здатність програми або ОС підтримувати виконання кількох потоків всередині одного процесу |
| **Конкурентність (Concurrency)** | Виконання кількох завдань впродовж одного періоду часу (не обов'язково одночасно) |
| **Паралелізм (Parallelism)** | Одночасне виконання кількох завдань на різних ядрах процесора |
| **Гіперпотоковість (Hyper-Threading)** | Технологія Intel (SMT), що дозволяє одному фізичному ядру обробляти два потоки |
| **Логічне ядро** | Віртуальне ядро, створене технологією Hyper-Threading; два логічні ядра на одному фізичному |
| **Гонка даних (Race Condition)** | Помилка, при якій результат роботи програми залежить від порядку виконання потоків |
| **Взаємне блокування (Deadlock)** | Ситуація, коли потоки заблоковані назавжди, бо кожен чекає ресурс, утримуваний іншим |
| **Умови Коффмана** | Чотири необхідні умови для виникнення deadlock: взаємне виключення, утримання і очікування, неможливість примусового вилучення, циклічне очікування |
| **Синхронізація** | Механізми координації роботи потоків при доступі до спільних ресурсів |
| **М'ютекс (Mutex)** | Об'єкт синхронізації, що дозволяє лише одному потоку мати доступ до ресурсу; має суворе володіння |
| **Семафор (Semaphore)** | Об'єкт синхронізації з лічильником, що дозволяє обмеженій кількості потоків доступ до ресурсу |
| **Двійковий семафор** | Семафор з лічильником 0 або 1, подібний до м'ютекса, але без суворого володіння |
| **Лічильний семафор** | Семафор з довільним невід'ємним значенням лічильника |
| **Критична секція (Critical Section)** | Ділянка коду, що звертається до спільного ресурсу і потребує захисту від одночасного доступу |
| **Монітор (Monitor)** | Високорівневий механізм синхронізації, що поєднує дані, процедури та автоматичне блокування |
| **Умовна змінна (Condition Variable)** | Механізм, що дозволяє потоку заснути, поки не виконається певна умова |
| **Квант часу (Time Slice)** | Короткий проміжок процесорного часу, що виділяється потоку планувальником |
| **Перемикання контексту (Context Switch)** | Збереження стану поточного потоку/процесу та завантаження стану іншого |
| **Process Explorer** | Безкоштовна утиліта Microsoft Sysinternals для детального аналізу процесів та потоків |
